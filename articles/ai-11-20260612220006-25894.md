## はじめに

僕は一人会社で、AIエージェントを「部門」として運用しています。CEO（=僕）の下に、開発・マーケ・営業・経理・法務・CS・人事・出版・コンサル・グロース・事業開発の11部門を作り、Claude Codeのサブエージェントとして実装しました。

結論から言うと、**11部門のうち、本当に売上に貢献したのは4部門だけ**でした。残りは「あったほうが安心」だが、なくても会社は回ります。

本記事では、実際に1年運用して見えた「いらない部門」と「必須の部門」を、当社の数字とともに共有します。

## 当社のスペック

まず前提として、当社の状況を共有します。

- 一人会社（CEOは僕）
- 2026年3月の売上: 受託 ¥1,937,635 + 書籍 ¥21,403 = **¥1,959,038**
- AI運用コスト: 月約¥20,000（Claude Code Max）
- 自動化率: 98%
- launchdジョブ: 17個
- SNS自動配信: 27件/日（9プロダクト×3チャネル）

この規模で「部門」を11個作るのは明らかにオーバーエンジニアリングでした。でも、作ってみないと「いらない」とは気づけなかったのです。

## 必須の部門（売上に直結した4部門）

### 1. 出版部門（Publisher）

これが**一番のヒット**でした。Zenn書籍の企画→執筆→品質チェック→公開→EPUB変換→Kindle登録までを一貫して自律実行します。

CEOへの中間確認はゼロ。完成品のURLだけ報告する設計にしました。

> 「AIは実行に使う。判断は人間。」

この原則を守りつつ、出版という「形にすれば売れる可能性が生まれる」領域に全振りしたのが正解でした。書籍売上は月¥21,403と決して大きくないですが、**信頼資産**として効いています。コンサル受注の決め手になることが多い。

### 2. コンサルティング事業部（Consulting VP）

受託売上の主力です。2026年3月は¥1,937,635のうち、ほぼ全額がコンサル経由。診断レポート・提案書の自動生成があるからこそ、僕は「商談」だけに集中できる。

### 3. コンテンツエンジン（Content Engine）

Zenn・Qiita・noteの3チャネルに毎日記事を配信しています。Zennは1日1コンテンツのレートリミットがあるので、スケジューリングも込み。

> 過去に1日複数記事+書籍を同時公開してデプロイ失敗。ロックファイルで1日1コンテンツに制限して解決。

オーガニック流入の生命線です。

### 4. CTO/開発部門

自動化の仕組みそのものを作る部門。これがないと、他10部門が動きません。「部門を運用する部門」なので、必須中の必須。

## いらなかった部門（解体・統合候補）

### 1. CHRO/人事部門

**完全に無駄**でした。

一人会社に「人事」はいない。エージェントの育成計画やパフォーマンス評価という発想は面白いのですが、結局**やることがない**。月1回も起動しません。

教訓: **組織図をマネしてはいけない**。大企業の組織を一人会社にコピーすると、99%の部門は空回りします。

### 2. CSO/営業部長

これも実質ゼロ稼働。営業はAutonomos（Sales Cruise）に一本化したため、CSOエージェントの出番がなくなりました。

> Autonomos一本化を決めた理由: drafts/にドラフトを作る運用は非効率で、結局CEOが手作業で確認することになっていた。

### 3. CS部長

問い合わせ自体が月に数件しかないので、専任部門は過剰。法務と統合してもいい。

### 4. グロースハッカー

「CVR改善・A/Bテスト・有料転換促進」を担当する設計でしたが、**そもそもトラフィックがA/Bテストに耐える量に達していない**。先にコンテンツとSEOで母数を作る方が先でした。

## 失敗の歴史（部門設計のアンチパターン）

部門を11個も作ってしまった背景には、いくつかの失敗があります。

### 失敗1: 10事業同時運用 → 9事業が売上ゼロ

SaaSを4つ作りました（Focalize, ShareToku, MochiQ, 元気ボタン）。**全部ユーザーゼロ**。

「AIなら何でも作れる」ことに気づいて、勢いで作りすぎた。各事業に1部門ずつアサインした結果、部門だけが増えていったのです。

2026年2月に3事業集中戦略に転換。**「AIは何でも作れるが、何でも売れるわけじゃない。」**

### 失敗2: AIメール自動送信で敬語事故

営業メールを確認なしで自動送信したら、取引先に敬語が不適切なメールが届きました。

ここから**承認パイプライン（draft→承認→実行）**を導入。「営業部門の自動化」はやりすぎ厳禁、と学びました。

### 失敗3: CLAUDE.mdが1,000行超え

ルールを追加し続けたら矛盾が発生し、AIが指示と逆の動作を始めた。

教訓: **200行以内のルール、詳細は.claude/agents/に分離**。部門を増やすほど、ルール整理コストが指数関数的に増えます。

### 失敗4: git push --force事故

AIが`git push --force`を自動実行し、ブランチが消えました。

> 自動化は「AIの能力を制限する」ためではなく「安心して委任するための仕組み」。

部門を増やす前に、**禁止リスト**を整備するほうが100倍重要です。

### 失敗5: HPリニューアル営業 → 成果ゼロで廃止

事業開発部門に「HPリニューアル営業」を担当させ、21社分析・提案書完成までやらせましたが、**1件もバウンスせず廃止**。

AIの進化でHP制作のハードルが下がり、事業価値そのものが消失していた。**部門に仕事を与える前に、その仕事に市場があるかを確認すべきでした**。

## 結論: 一人会社に必要なのは「4部門 + Orchestrator」

1年運用してたどり着いた最小構成はこうです。

| 部門 | 役割 |
|------|------|
| Orchestrator（CEO窓口） | 意図解釈・承認管理・部門間調整 |
| 開発 | 自動化の基盤を作る |
| 出版 | 信頼資産を生成（書籍） |
| コンテンツ | オーガニック流入を作る |
| コンサル | 売上を立てる |

人事・グロース・CS・営業・法務・経理・税務は、**必要なときだけ呼び出すユーティリティ**として残す。常駐部門にしてはいけない。

## CEOの仕事は「ボタンを押すこと」

最終的に、僕の仕事は朝の5分で全部門を把握し、承認ボタンを押すだけになりました。

> 「コストは10分の1になるが、売上が10倍になるわけではない。」

月¥20,000のAI投資で会社は回ります。ただし回すのはCEOの判断です。**部門を増やすことは、回す対象を増やすこと**。判断コストが増える方向の自動化は、本末転倒です。

これから一人会社でAIエージェントを組もうとしている方は、**まず4部門から始めて、足りなくなったら追加する**ことを強くおすすめします。
