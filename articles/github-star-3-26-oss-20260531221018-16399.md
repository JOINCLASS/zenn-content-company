## 結論から

僕が運用しているOSSのGitHub Starが、ある3日間でいきなり3→26に増えた。広告も打っていない、SNSでバズったわけでもない。けれど確かに増えた。

本記事は「何が効いたのか」を当社の計測データに基づいて分解し、**OSSマーケティングで本当に効くチャネルは何か**を再現可能な形でまとめたものです。結論を先に書くと、**一番効いたのは第三者が書いてくれたZenn記事1本**でした。X広告でも、Redditでもなく。

## 当社の状況（前提）

僕はAI-CEO Frameworkというリポジトリを公開しつつ、AIで会社運営を全自動化する仕組みを作っています。直近1ヶ月の運用実績は以下の通り。

- 受託売上: ¥1,937,635（2026年3月）
- 書籍売上: ¥21,403（Zenn + Amazon Kindle）
- AI運用コスト: 月¥20,000（Claude Code Max）
- 自動化率: 98%（launchdジョブ17本）

このうち**OSS自体は1円も売上を生んでいません**。OSSは「信頼の証拠」として置いてあるだけで、コンサル受注や書籍購入への動線として機能させる位置づけです。

そんなOSSのStarが、なぜか3日で26に跳ねた。これは検証する価値があります。

## 同時期に走らせていた施策

3日間でStarが3→26になる前後、当社で並行して走らせていたマーケ施策は以下です。

| チャネル | 投下コスト | 内容 |
|---------|-----------|------|
| X広告 | ¥6,000 | 書籍販売目的のキャンペーン |
| Reddit投稿 | ¥0 | r/ClaudeAI 等に英語で複数投稿 |
| Zenn記事（自社） | ¥0 | AI自動化系の記事を毎日1本 |
| 第三者Zenn記事 | ¥0（自然発生） | 他の方が当社OSSを紹介してくれた |

順番に「Starへの寄与」を見ていきます。

## 計測結果: 何がStarを生んだのか

### X広告（¥6,000）→ Star寄与ほぼゼロ

これは正直、痛い学びでした。X広告は¥6,000を投下しましたが、**Kindle販売の計測ができない**ことに途中で気づき、Starへの導線もほぼ作れずに終了。インプレッションは伸びましたが、リンククリック→GitHubアクセスはGA4上で数件レベル。Starに繋がった証跡は確認できませんでした。

教訓は明確で、**書籍やOSSのような「Amazon/GitHubに着地する商材」をX広告で売るのはやめたほうがいい**。計測可能性を事前にチェックしないと、¥6,000は一瞬で溶けます。

### Reddit投稿 → 反応はあるがStarには直結せず

Redditは英語圏のClaude Code関連サブレディットに投稿。コメントとUpvoteはついたものの、Starの増加タイミングと完全には一致しませんでした。Redditは「議論を呼ぶ」のは得意ですが、「ワンクリックでStarを押させる」までの動線が弱い印象です。

ただし、Redditで反応した人の一部が**後日Zenn記事や英語ブログ経由で再到達してStarを押している**痕跡はあり、複合効果としては効いていそう。単体評価が難しいチャネルです。

### 第三者Zenn記事 → これが本命だった

Star急増の3日間、GA4のリファラを追うと、明らかに**他の方が書いたZenn記事1本からの流入**が突出していました。当社が書いたものではなく、第三者の開発者が「AI-CEO Frameworkを試してみた」という体裁で紹介してくれた記事です。

この記事1本から、

- GitHubリポジトリへの直接遷移が増加
- そのうち体感で2〜3割がStarを押す
- ピーク時は1日10 Star以上

という形でStarが積み上がりました。**自分で書いた紹介記事より、第三者が書いた検証記事のほうが10倍効く**、というのが計測から導かれた結論です。

## なぜ第三者Zenn記事が強いのか

ここは僕の解釈ですが、3点あります。

### 1. 「使ってみた」の信頼性

自社が書く紹介記事は、どれだけ客観的に書いても「営業色」が滲みます。一方で第三者が「動かしてみたら良かった/悪かった」を書くと、読者は**判断の手間を省ける**。Starを押すかどうかの心理的ハードルが大きく下がります。

これは僕自身が普段OSSを評価するときの動線と完全に一致していて、「公式README→使ってみた記事→Star」という順で読むのが習慣化しています。

### 2. Zennは技術者の購買行動に近い

Zennの読者は、そもそも**コードを書く人**です。記事を読んでGitHubに飛ぶ動線が日常化していて、Starボタンの位置も知っている。RedditやXのように「いいねで終わる」プラットフォームと比べて、**1クリック分の摩擦が少ない**。

### 3. SEOで長期的に効く

Zenn記事はGoogle検索でも上位に出やすく、書かれた瞬間だけでなく**数ヶ月単位でジワジワ流入を生む**。3日間の急増の後も、当該記事経由のStarは細く長く続いています。X広告の¥6,000は1週間で消えますが、Zenn記事1本は半年効く。コスパが違いすぎる。

## では「第三者にZenn記事を書いてもらう」にはどうするか

ここが本記事の本題です。当社で機能した3つの仕掛けを共有します。

### 仕掛け1: READMEに「試しやすさ」を全振り

第三者が記事を書く最大の障壁は「動かすのが面倒」です。当社のリポジトリでは、

- 5分で動かせるクイックスタート
- コピペで動くサンプルコマンド
- 想定する失敗パターンとその対処

をREADMEの冒頭に置きました。書く側からすると**スクリーンショット1枚で記事になる**状態にしておくのが重要。

### 仕掛け2: 自社記事で「使い方」をひたすら公開

僕は自社Zenn記事で、AI-CEO Frameworkの内部実装・運用ノウハウ・失敗談を週次で出し続けています。これは直接Starを生まないのですが、**第三者が記事を書くときの「ネタの素材庫」**として機能します。

「あの会社の人がこう書いてた」を引用してもらえると、第三者の記事の説得力が上がり、結果としてStarに返ってきます。

### 仕掛け3: 失敗談を隠さない

これは僕の信念に近いのですが、**失敗体験を出すリポジトリほどStarが伸びる**。当社の場合、

- 10事業同時運用 → 9事業が売上ゼロ
- AIが`git push --force`を自動実行してブランチ消失
- AIメール自動送信で敬語が崩壊
- CLAUDE.mdが1,000行超えてAIが矛盾動作

といった失敗を全部公開しています。これらは記事化しやすい題材で、第三者にとっても「自分の経験と重ねやすい」コンテンツになる。

ちなみに上記の事故を踏まえて、当社は**承認パイプライン（draft→承認→実行）**と**絶対禁止リスト**を整備しました。この設計思想自体もOSSに含めてあり、これも記事化されやすいポイントです。

## 数字で振り返る「投資対効果」

3日間のStar増加（+23）を、チャネル別に概算で寄与配分するとこうなります。

| チャネル | 投下コスト | 推定Star寄与 | 1Starあたりコスト |
|---------|-----------|--------------|------------------|
| X広告 | ¥6,000 | 0〜1 | ¥6,000+ |
| Reddit投稿 | ¥0（時間のみ） | 3〜5 | ¥0 |
| 自社Zenn記事 | ¥0 | 3〜5 | ¥0 |
| 第三者Zenn記事 | ¥0 | 13〜17 | ¥0 |

**¥6,000のX広告より、¥0の第三者記事のほうが15倍以上効いている**。OSSマーケティングの常識として、これは覚えておく価値があります。

## 再現するための行動指針

僕がもう一度ゼロからOSSを伸ばすなら、こう動きます。

1. **READMEに5分クイックスタートを最優先で実装**（記事化のハードルを下げる）
2. **自社Zennで失敗談と内部実装を週1で公開**（第三者記事の素材を提供する）
3. **第三者が書いてくれた記事にきちんとお礼コメントを書く**（次の記事を呼ぶ）
4. **X広告にお金を入れるのは最後**（計測できる商材か事前確認する）
5. **RedditはSEO/権威付けの補助線として使う**（直接Star狙いにしない）

OSSは「作ってStarが付く」ものではなく、**第三者に書いてもらうための環境を整備する**ことが本質だと改めて感じました。
