## はじめに

ひとり社長で会社を回していると、問い合わせフォームは「数少ない売上の入り口」になります。だからこそ、そこに偽の投稿が混ざると地味に痛い。

僕の会社では、ある時期から問い合わせフォームに**月20件以上の偽投稿**が届くようになりました。存在しないメールアドレス、桁数の合わない電話番号、本文が「test」だけ、海外からの無差別投稿。最初は「ちょっと邪魔だな」程度の認識でしたが、放置していると本物のリードが偽投稿に埋もれ、対応の遅れに直結します。

ひとり社長にとって、問い合わせの一次対応は他の誰にも振れません。だからこそ、**入り口でゴミを止める**仕組みが売上機会を守ります。本記事では、僕が実際にやっている問い合わせフォームのスパム対策を、実装レベルで共有します。

## 実際に起きていたこと

最初に気づいたのは、月次の問い合わせ集計を見たときでした。

- 月の問い合わせ件数: 25件
- うち本物のリード: 3-5件
- 偽投稿: 20件前後

つまり、本物のリードの比率は2割を切っていました。残り8割は以下のような投稿です。

1. **存在しないメールアドレス** — 返信してもバウンスする
2. **ダミー電話番号** — `000-0000-0000` や `123-456-7890`
3. **本文が「test」「テスト」のみ** — ボットの動作確認
4. **海外SEO業者からの自動投稿** — 英語で「あなたのサイトを上位表示します」
5. **意味不明な文字列** — ランダムな英数字

問題は、これらが**本物の問い合わせと同じSlackチャネル・同じメールに通知される**ことです。通知が来るたびに確認し、偽物だと判断するコストが地味にかさみます。1件あたり1分でも、20件なら20分。ひとり社長の20分は、提案書1枚分の時間です。

## なぜスパムが急に増えたのか

これは僕の推測ですが、要因は3つありました。

1. **会社サイトのSEO流入が増えた** — Zenn記事経由で被リンクが付き、フォームURLがクローラーに発見された
2. **フォーム項目が単純すぎた** — name / email / phone / message の4項目だけ。スパムボットの「型」と完全一致
3. **バリデーションが緩かった** — 電話番号は文字列、メールは正規表現すら通っていなかった

つまり、**「見つけやすく、投稿しやすく、検出されにくい」フォーム**だったわけです。スパムから見れば理想的なターゲットです。

## やったこと(全部AIに実装させた)

僕は対策をすべて自分で書いていません。Claude Codeに「現状のフォーム実装を読み込んで、コスパの良いスパム対策を多層で入れて」と依頼しました。最終的に入った対策は次の通りです。

### 1. 入力バリデーションの厳格化

最初にやったのは、当たり前の入力チェックです。これだけで明らかに桁数のおかしい電話番号や、`@` のないメールアドレスは弾けます。

```typescript
// 電話番号: 日本の番号形式のみ許可
const phoneRegex = /^0\d{1,4}-?\d{1,4}-?\d{4}$/;

// メールアドレス: RFC準拠の簡易チェック + MXレコード確認は後段で
const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;

if (!phoneRegex.test(phone)) {
  return { ok: false, reason: "phone_invalid" };
}
if (!emailRegex.test(email)) {
  return { ok: false, reason: "email_invalid" };
}
```

ここで重要なのは、**バリデーションエラーを画面に詳細表示しない**ことです。「電話番号の形式が違います」と親切に教えると、スパムボットが学習します。「送信できませんでした」とだけ返して、サーバ側にだけ理由を記録しています。

### 2. Honeypot(蜜壺)フィールド

CAPTCHAは本物のユーザーに摩擦を生むので、まずは無料で効くHoneypotから入れました。人間には見えない隠しフィールドを置き、そこに入力があったらボット判定する手法です。

```tsx
<input
  type="text"
  name="website"
  tabIndex={-1}
  autoComplete="off"
  style={{ position: "absolute", left: "-9999px" }}
  aria-hidden="true"
/>
```

サーバ側で `website` フィールドに何か入っていたら、200を返したフリをして**何もしません**。ボットには「成功した」と見せかけ、再投稿を促さないのがコツです。

これだけで偽投稿のうち半分以上は消えました。

### 3. レート制限(IP + メールアドレス単位)

同じIPから短時間で複数投稿が来るパターンも多かったので、レート制限を入れました。Supabaseに `form_submissions` テーブルを切り、IP・メール・作成時刻を記録します。

```typescript
const since = new Date(Date.now() - 60 * 60 * 1000).toISOString();
const { count } = await supabase
  .from("form_submissions")
  .select("*", { count: "exact", head: true })
  .eq("ip", ip)
  .gte("created_at", since);

if ((count ?? 0) >= 3) {
  return { ok: false, reason: "rate_limited" };
}
```

「1時間に同じIPから3件以上」を閾値にしました。本物のユーザーが1時間に3回も問い合わせを送ってくることは、まずありません。

### 4. Cloudflare Turnstileによるボット検出

ここまでで偽投稿は8割減っていましたが、**残りの2割は明らかに高度なボット**でした。Honeypotを認識し、レート制限を回避するために複数IPから投稿してくる。

最後の砦としてCloudflare Turnstileを入れました。reCAPTCHAと違って**ユーザー側のクリック不要**、無料、プライバシー配慮もあるという三拍子。

```typescript
const verify = await fetch(
  "https://challenges.cloudflare.com/turnstile/v0/siteverify",
  {
    method: "POST",
    body: new URLSearchParams({
      secret: process.env.TURNSTILE_SECRET!,
      response: token,
      remoteip: ip,
    }),
  }
);
const result = await verify.json();
if (!result.success) {
  return { ok: false, reason: "turnstile_failed" };
}
```

これで本格的なボット投稿はほぼ止まりました。

### 5. NGワード・NGドメインのブラックリスト

最後に、**過去のスパム投稿から抽出したNGワードとNGドメイン**をブロックリスト化しました。`@mail.ru`、`SEO services`、`backlink` といったキーワードが含まれていたら自動でreject。

これは「スパムが届いた都度、リストに追加する」運用にしています。AIに「過去のSlack通知を読み込んでNGワードを抽出して」と頼めば、リストアップは数分で終わります。

## 結果

対策前後の月次比較がこれです。

| 指標 | 対策前 | 対策後 |
|---|---|---|
| 月次投稿数 | 25件 | 6件 |
| うち偽投稿 | 20件 | 1件以下 |
| 本物のリード | 3-5件 | 3-5件 |
| 一次対応にかかる時間 | 約30分/月 | 約5分/月 |

本物のリード数は変わっていません。当然です。スパムを止めても本物のユーザーは止めていないからです。**変わったのは「ノイズを掻き分ける時間」**でした。月25分の節約は小さく見えますが、ひとり社長にとっては提案書1本分のリソースです。

## ひとり社長が学んだこと

この一連の対策を通じて、3つのことを実感しました。

### 1. スパム対策は「売上機会を守る」投資

スパムは直接の被害がないように見えて、**本物のリードへの応答速度を下げる**形でビジネスに効きます。問い合わせから24時間以内に返信できるかどうかは、コンサル案件では特に成約率に直結します。偽投稿を掻き分けて本物を見つける時間は、見えない機会損失です。

### 2. 完璧を目指さず多層で防ぐ

スパム対策に銀の弾丸はありません。Honeypot、レート制限、Turnstile、NGワード — どれも単体では破られます。でも**4つ重ねれば、突破コストが上がってボットが諦めます**。これはbot検知側の経験(僕は自社の自動化で18層のbot検知対策を組んでいます)から逆算した発想です。攻撃側がどこまでコストをかけるかを見極めて、防御コストをそれ以上にすればいい。

### 3. 実装は全部AIに任せていい

今回の対策コードは、僕はほぼ書いていません。Claude Codeに「現状のフォームを読み込んで、低コストで効くスパム対策を多層で実装して」と頼んだだけです。実装、テスト、デプロイまで含めて2時間。コストはAI利用料の数十円。

ひとり社長にとって「AIは実行に使う、判断は人間がする」が基本方針です。何を守るか、どこまでコストをかけるかは人間が決める。実装はAIに任せる。この役割分担ができれば、対策のハードルは劇的に下がります。

## チェックリスト

最後に、ひとり社長が今日から始められるスパム対策チェックリストを置いておきます。

- [ ] 電話番号・メールアドレスのフォーマットバリデーション
- [ ] Honeypotフィールドの追加
- [ ] IP + メール単位のレート制限
- [ ] Cloudflare Turnstile(無料)の導入
- [ ] NGワード・NGドメインリストの整備
- [ ] バリデーションエラー詳細を画面に出さない
- [ ] サーバ側のログに偽投稿の理由コードを記録
- [ ] 月次でログを見て新しいスパムパターンをリスト追加

全部入れても半日で終わります。ひとり社長の半日は安くないですが、毎月20分を取り戻せると考えれば3ヶ月で元が取れる計算です。
