## 数字を正直に晒すところから始める

僕の会社の2026年3月の売上はこうだった。

- 受託売上: ¥1,937,635（X社¥880K、C社¥933K、A社¥110K、H社¥15K）
- 書籍売上: ¥21,403（Zenn ¥14,756 + Amazon ¥6,647）
- 合計: ¥1,959,038

受託が約99%、自社プロダクト（書籍）が約1%。よく「自社プロダクトで食えるようになりたい」と言う一人会社の経営者は多いけれど、当社の実態もまさにそれだ。180万円 vs 2万円。比率にして約90倍の差がある。

この記事では、この比率を変えていくために僕が今考えていること、やめたこと、これから賭けるところを正直に書く。同じく「受託依存をなんとかしたい」と思っている人の参考になれば嬉しい。

## なぜ受託が「強すぎる」のか

受託案件は強い。理由はシンプルで、**売上のリードタイムが短い**からだ。
契約してから2〜4週間で初回入金が走り、月次で安定する。営業コストも、既存クライアントのリピートが中心ならほぼゼロ。当社の場合、X社とC社の2社で月商の93%を占めている。

一方、自社プロダクト（当社の場合はZenn書籍とAmazon Kindle）は、執筆に数日〜数週間かけて公開しても、1冊あたりの月次売上は数百円〜数千円のオーダー。¥14,756という3月のZenn売上は、複数冊の合算だ。1冊あたりの貢献度は非常に小さい。

受託のROIは即時的で、自社プロダクトのROIは長期的かつ確率的。経営者として、目の前のキャッシュフローを優先する判断は合理的でもある。だからこそ、ほとんどの一人会社は受託に傾く。傾いたまま、5年経つ。

## 僕がやらかした「分散」の話

正直に書くと、僕は1年前に逆の方向に走って失敗している。

当時、「SaaSを作れば自社プロダクト売上が立つはず」と考えて、SaaSを4つ同時に作った。Focalize、ShareToku、MochiQ、元気ボタン。Claude Code Maxを契約して、AIに大量にコードを書かせて、4プロダクトを並行で開発した。

結果はどうだったか。**4つともユーザーゼロ**。1円も売れなかった。

僕はそこから「10事業同時運用 → 9事業が売上ゼロ」という現実を学んだ。AIで作る速度は10倍になっても、売る難しさは10倍にはならない。むしろ集中していない分、マーケティングが薄まって、認知すら取れなかった。

2026年2月、僕は3事業（受託、書籍出版、コンサル）に集中する戦略に切り替えた。SaaSは塩漬けにした。痛みはあったが、結果として受託180万円と書籍2万円という「健全な低空飛行」に戻ってこられた。

ここで言いたいのは、**自社プロダクト比率を上げたいからといって、闇雲に新プロダクトを増やすのは最悪手**ということだ。これは僕が実弾を撃って学んだことだ。

> 「AIは何でも作れるが、何でも売れるわけじゃない。」

## 受託180万円 vs 自社2万円という構造の正体

この比率を冷静に分析すると、3つの構造的な要因が見える。

### 1. 認知の蓄積差
受託は2社の既存クライアントとの信頼関係の上に立っている。これは過去2〜3年の積み上げ。書籍は公開してまだ数ヶ月〜半年程度。複利の時間軸が違う。

### 2. 単価差
受託は月¥88万〜¥93万の固定報酬。書籍は1冊¥500〜¥1,500、月数冊〜数十冊。単価が3桁違う。

### 3. 営業チャネルの太さ
受託は直接コミュニケーション、相手の予算は法人。書籍は不特定多数のZennユーザー・Amazon閲覧者で、購買単価は個人の小遣い帯。

つまり「自社プロダクト売上を増やす」とは、(1)認知の時間を待つ、(2)単価を上げる、(3)チャネルを太くする、のどれか・全部を同時に動かす必要がある。

## 僕が今賭けている3つの打ち手

### 打ち手①: コンサルメニューを書籍の延長線に置く

僕は現在、書籍を「コンサル受注のリード獲得装置」として再定義している。1冊¥1,000の本で利益は薄い。だが、本を読んだ人の中から月¥30万〜¥50万のコンサル契約が1件取れれば、それは書籍100〜500冊分のインパクトになる。

書籍は売上ではなく**信頼の証拠**だ。だからAmazon Kindle価格を記事に書かないし（価格差異クレームを避ける）、CTAは「もっと深く学びたい人向けの体系書」へ誘導する。書籍そのものをマネタイズの主役にしない。

### 打ち手②: 自動化原価を限界まで下げる

当社のランニングコストはAI約¥20,000、サーバー¥2,000、Google Workspace ¥680。合計でも¥25,000以下。自動化率98%、launchdジョブ17個、SNS自動配信27件/日。

「コストは10分の1になるが、売上が10倍になるわけではない。」これは僕が学んだ教訓だ。だが裏を返せば、**自社プロダクトの売上が月¥5万を超えれば、それは純粋な利益として残る**。固定費が薄いから、小さな売上でもプロダクト事業として黒字化する。受託売上に対する相対比率を上げるより、絶対額を底上げするほうが現実的だ。

### 打ち手③: 受託の収益を「自社の時間」に変換する

これが一番大事。受託180万円を全部CEOの労働時間で消費したら、自社プロダクトに使う時間は永遠にゼロだ。

僕は受託作業の中身も98%自動化した。AI（Claude Code Max）に実装させ、Autonomos（Sales Cruise）でリード対応をさせ、launchdで定型作業を回す。「AIは実行に使う。判断は人間。」朝の5分で全部門を把握し、承認ボタンを押すだけ。

これで受託180万円を維持しながら、自社プロダクト（書籍・コンサル）に投資する時間が生まれる。受託を縮小して自社にシフトするのではなく、**受託を自動化して、生まれた時間を自社プロダクトに投下する**。

## やめたこと

打ち手と同じくらい、「やめたこと」が比率を変える。

- **HPリニューアル営業**: 21社分析・提案書完成。だが、AIの進化でHP制作のハードルが急速に下がり、事業価値が消失と判断して廃止
- **複数SaaS並行開発**: 4つ作って全部ゼロ。集中なき分散の典型
- **新規広告チャネルの闇雲な投下**: X広告で書籍販売を試して¥6,000損失。Kindle売上計測が広告側でできず、ROIが検証不能だった
- **「自分で使っているから売れるはず」という前提の施策**: 自社体験の商品化は、市場検証なしには成立しない

これらは全部、**仮説検証なしに動いた施策**だ。今は新しい広告チャネル・新サービス・月¥5,000以上の継続投資を伴うものは、必ず仮説検証フェーズを挟んでから動かす運用に変えた。

## 比率を変える時間軸

正直、月商比率を「受託50% : 自社50%」にするまでには年単位の時間がかかる。短期で達成しようとすると、また「10事業同時運用」の再演になる。

僕の現実的な目標はこうだ。

- 6ヶ月以内: 書籍経由のコンサル受注で月¥30万
- 12ヶ月以内: 自社プロダクト系（書籍+コンサル）合計月¥50万
- 18ヶ月以内: 受託2社のうち1社を、自社プロダクト売上で代替できる規模に

受託180万円を「無理に減らさない」のがポイントだ。受託は土台。土台の上に、ゆっくり別の柱を建てる。180万円と2万円の比率を、180万円と50万円に変えるまでは、絶対額の積み上げに集中する。

## まとめ

「自社プロダクトで食いたい」というのは、多くの一人会社のロマンだ。だが受託で月180万円を回している現実を捨てて自社にオールインするのは、リスクが高すぎる。

僕の戦略は3つ。
1. 自社プロダクト（書籍）をリード獲得装置として再定義し、コンサルで単価を上げる
2. 固定費を月¥25,000以下に抑え、小さな売上でも黒字化する構造を維持する
3. 受託を自動化して、生まれた時間を自社プロダクトに投資する

受託と自社を二項対立で捉えない。**受託で稼いだ時間とキャッシュを、自社プロダクトの複利に変換する**。これが現時点での僕の答えだ。
