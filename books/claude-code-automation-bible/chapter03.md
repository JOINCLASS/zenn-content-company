# CLAUDE.mdで自動化の基盤を作る

:::message
**この章で学ぶこと**
- CLAUDE.mdが自動化において果たす役割
- Thin Orchestratorパターン（200行以内に収める設計思想）
- 権限レベル制御（read-only / draft / execute）の実装
- エージェント定義の分離（.claude/agents/）
- 実例: 9部門のエージェント定義
:::

## CLAUDE.mdは「AIへの永続的な指示書」

CLAUDE.mdは、Claude Codeがセッション開始時に自動で読み込むMarkdownファイルだ。プロジェクトのルートに置いておけば、毎回同じプロンプトを手打ちしなくても、AIが「自分は何者で、何をすべきか」を理解した状態で始まる。

手動時代、筆者は毎朝こんなプロンプトを打っていた。

```
あなたはAI-CEOフレームワークのオーケストレーターです。
9つの部門を管理しています。
部門は開発、マーケ、営業、経理、CS、法務、税務、コンサル、HRです。
各部門の状態は .company/departments/ にあります。
承認待ちは .company/approval-queue.md にあります。
（...以下50行くらい続く）
```

これを毎日打つのは無駄だ。CLAUDE.mdに書いておけば、一度書くだけで永続的に有効になる。

```
プロジェクトルート/
├── CLAUDE.md              ← これ。セッション開始時に自動読み込み
├── .claude/
│   ├── agents/            ← エージェント定義（後述）
│   ├── skills/            ← スキル定義（第5章）
│   └── settings.local.json ← Hooks・MCP設定（第4章・第6章）
└── .company/              ← 会社のデータ
    ├── STATE.md
    ├── departments/
    └── ...
```

CLAUDE.mdは3つの場所に配置できる。読み込み順と用途を理解しておこう。

| 配置場所 | パス | 用途 | 共有範囲 |
|---------|------|------|---------|
| グローバル | `~/.claude/CLAUDE.md` | 全プロジェクト共通のルール | 個人 |
| プロジェクト | `プロジェクトルート/CLAUDE.md` | プロジェクト固有のルール | チーム（git管理） |
| ディレクトリ | `サブディレクトリ/CLAUDE.md` | ディレクトリ固有のルール | チーム（git管理） |

自動化の文脈では、**プロジェクトルートのCLAUDE.md**が最も重要だ。ここにオーケストレーターの定義を書く。

## Thin Orchestratorパターン

### 問題: CLAUDE.mdが肥大化する

CLAUDE.mdに全ての指示を書こうとすると、すぐに数百行、数千行に膨れ上がる。9部門分のエージェント定義、全コマンドの実行フロー、権限設定、ブランドガイドライン...全部入れたら、Claude Codeのコンテキストウィンドウを圧迫する。

筆者も最初は全てをCLAUDE.mdに詰め込んでいた。結果、ファイルが800行を超え、Claude Codeの応答速度が落ち、指示の一部が無視されるようになった。

### 解決: 「薄い司令塔」パターン

Thin Orchestratorパターンは、CLAUDE.mdを**200行以内**に抑え、詳細を外部ファイルに分離する設計だ。

```
CLAUDE.md（200行以内）
├── 自分は誰か（5行）
├── 参照先ファイルの一覧（20行）
├── コマンド一覧（30行）
├── コマンド実行ルール（50行）
├── 権限制御ルール（20行）
├── エラー時の振る舞い（10行）
└── サブエージェントへの委任方法（15行）

詳細は全て外部ファイルに
├── .claude/agents/*.md     ← 各部門のエージェント定義
├── .company/steering/*.md  ← ブランド・技術・権限の詳細
└── .company/STATE.md       ← 現在の経営状態
```

核心は、**CLAUDE.mdにはファイルの中身を書かず、ファイルパスだけを書く**ことだ。

### 実装例: CLAUDE.mdの全体構造

以下は、筆者が実際に使っているCLAUDE.mdの構造だ（実コードから抜粋・簡略化）。

```markdown
# AI-CEO Framework — C-Suite Orchestrator

> あなたはAI-CEO Frameworkの「C-Suite Orchestrator」です。
> CEOの経営判断を支援し、9部門のAIエージェントを統括します。

## あなたの役割

あなたはCEOと直接対話する唯一のインターフェースです。

1. **CEOコマンドの受付と実行** — コマンドを解釈し、適切な部門に委任
2. **部門間調整** — 依存関係の解決
3. **承認管理** — draft成果物のCEOへの報告

## Thin Orchestratorの原則

- コンテキスト使用率を10-15%に維持する
- ファイルの中身を自分のコンテキストに読み込まず、ファイルパスだけを渡す
- 複雑なタスクは必ず .claude/agents/ のサブエージェントに委任する
- 自分で実作業（コーディング、文書作成等）を行わない

## 会社情報の参照先

- ビジョン・ミッション: `.company/VISION.md`
- 現在の経営状態: `.company/STATE.md`
- 権限・閾値設定: `.company/steering/permissions.md`
- 承認待ちキュー: `.company/approval-queue.md`
- 部門別状態: `.company/departments/`
```

ここで重要なのは「**自分で実作業を行わない**」という原則だ。オーケストレーターは指示を出すだけ。実際の作業はサブエージェントに委任する。

これにより、CLAUDE.mdのコンテキスト消費を最小限に抑えつつ、必要なときに必要なファイルだけを読み込む効率的な運用が可能になる。

## 権限レベル制御の実装

### なぜ権限制御が必要か

AIに全てを任せると、取り返しのつかないことが起きる。本番データベースの削除、顧客への誤ったメール送信、誤った請求書の発行...。

筆者の環境では、3段階の権限レベルを定義している。

```yaml
# .company/steering/permissions.md から抜粋

# read-only: 分析・レポート系。承認不要で即実行
# draft:     対外アクション。必ず承認を経由する
# execute:   閾値内の内部アクション。自動実行OK

auto_approve_limit: 5000    # 5,000円以下は自動承認
ceo_approval_above: 5000    # 5,000円超はCEO承認必要
monthly_ai_budget: 200      # 月間AI運用予算 $200上限
```

### 3段階の権限レベル

| レベル | 意味 | 例 | 承認 |
|--------|------|-----|------|
| **read-only** | 読み取り・分析のみ | 状態確認、レポート生成、KPI集計 | 不要 |
| **draft** | 対外影響のあるアクション | メール送信、SNS投稿、請求書、デプロイ | CEO承認必要 |
| **execute** | 内部アクション | バグ修正、テスト実行、リファクタリング | 不要（閾値内） |

### CLAUDE.mdへの権限ルール記述

```markdown
## 権限制御ルール

全てのアクションは `.company/steering/permissions.md` の閾値に従う。

- **read-only:** 分析・レポート系は承認不要で自動実行
- **draft:** 対外アクション（メール送信、請求書発行、SNS投稿、
  デプロイ等）は必ずdraftモードで生成し、approval-queue.md に追加
- **execute:** 閾値内の内部アクション（バグ修正、テスト実行等）
  は自動実行

**重要:** 対外に影響のあるアクションを直接実行してはならない。
必ずdraft → 承認 → 実行のパイプラインを通す。
```

この権限定義は単なる「お願い」ではない。CLAUDE.mdに書いておけば、Claude Codeは**毎回のセッションで**このルールに従う。手動でプロンプトに書き忘れる心配がない。これが自動化の基盤としてのCLAUDE.mdの真価だ。

### 承認フローの実装

権限制御を機能させるには、承認待ちキューの仕組みが必要だ。

```markdown
<!-- .company/approval-queue.md -->
# 承認待ちキュー

| ID | 部門 | 種類 | 内容 | ファイル | 登録日 |
|----|------|------|------|---------|--------|
| AQ-042 | マーケ | SNS投稿 | LinkedIn: AI活用事例の投稿 | drafts/linkedin-042.md | 2026-03-28 |
| AQ-043 | 営業 | メール | 新規顧客への提案メール | drafts/proposal-043.md | 2026-03-28 |
```

draftモードで生成されたアクションは、このキューに追加される。CEOが `/ai-ceo:approve AQ-042` と打てば実行される。却下なら `/ai-ceo:reject AQ-042 "トーンを修正して"` で差し戻す。

**重要なのは、このフロー全体がCLAUDE.mdの指示だけで回るということだ。** 特別なプラグインやサーバーは不要。Markdownファイルの読み書きだけで承認フローが実現できる。

## エージェント定義の分離（.claude/agents/）

### なぜ分離するのか

CLAUDE.mdに9部門分のエージェント定義を書くと、それだけで500行を超える。Thin Orchestratorの原則に反する。

解決策は、各部門のエージェント定義を `.claude/agents/` ディレクトリに分離することだ。

```
.claude/agents/
├── ai-ceo-morning.md         # 朝ダイジェストエージェント
├── ai-ceo-cto.md             # CTO/開発部長
├── ai-ceo-cmo.md             # CMO/マーケ部長
├── ai-ceo-cso.md             # CSO/営業部長
├── ai-ceo-cfo.md             # CFO/経理部長
├── ai-ceo-cs-lead.md         # CS部門リーダー
├── ai-ceo-legal.md           # 法務エージェント
├── ai-ceo-tax-accountant.md  # 税理士エージェント
├── ai-ceo-consulting-vp.md   # コンサルティング部門VP
├── ai-ceo-hr.md              # 人事エージェント
├── ai-ceo-content-engine.md  # コンテンツエンジン
├── ai-ceo-publisher.md       # 出版エージェント
├── ai-ceo-growth.md          # グロースエージェント
├── ai-ceo-biz-dev.md         # 事業開発エージェント
├── ai-ceo-openclaw-ops.md    # OpenClaw運用エージェント
└── ai-ceo-init.md            # 初期セットアップエージェント
```

### エージェント定義のテンプレート

各エージェントは、以下のフォーマットで定義する。

```markdown
---
name: ai-ceo-cto
description: CTO/開発部長エージェント。プロダクト開発全般を統括。
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---

# CTO / 開発部長エージェント

あなたは AI-CEO Framework の CTO（最高技術責任者）です。

## ペルソナ

経験豊富なテックリード。実用性を重視し、
オーバーエンジニアリングを避ける。

## 担当領域

- プロダクト開発全般（設計、実装、テスト、デプロイ）
- 技術的意思決定
- スプリント管理

## 権限レベル

- **execute:** コーディング、テスト実行、内部ドキュメント
- **draft:** 本番デプロイ、アーキテクチャの大幅変更

## 参照ファイル

- 技術スタック: `.company/steering/tech-stack.md`
- 開発部門状態: `.company/departments/dev/STATE.md`
- 権限設定: `.company/steering/permissions.md`
```

フロントマターの `tools` で使用可能なツールを制限しているのがポイントだ。CTOにはBash実行権限があるが、朝ダイジェストエージェントにはReadとGrepだけ。**最小権限の原則**をエージェントレベルで適用している。

### オーケストレーターからの委任方法

CLAUDE.mdには、サブエージェントへの委任方法を以下のように定義している。

```markdown
## サブエージェントへの委任方法

タスクをサブエージェントに委任する際は、以下の情報を渡す:
1. **タスクの目的** — 何を達成するか（1文で）
2. **参照ファイルパス** — 必要な入力ファイルのパス一覧
3. **成果物の出力先** — 出力ファイルのパスとフォーマット
4. **権限レベル** — read-only / draft / execute
5. **品質基準** — 完了条件と検証方法
```

この5項目を渡すだけで、エージェントは自律的にタスクを完了する。オーケストレーターはファイルの中身を読む必要がない。パスだけ渡せばいい。

## 実例: 朝ダイジェストエージェントの全体像

理論だけでは分かりにくいので、実際に稼働している朝ダイジェストエージェントを例に、CLAUDE.md + エージェント定義 + 外部スクリプトがどう連携するかを見てみよう。

### ステップ1: CLAUDE.mdのコマンド定義

```markdown
## CEOコマンド一覧

### 日次運用
- `/ai-ceo:morning` — 朝ダイジェスト生成。
  全部門のSTATE収集 + 承認待ち一覧 + KPIサマリ
```

CEOが `/ai-ceo:morning` と打つと、オーケストレーターが朝ダイジェストエージェントを起動する。

### ステップ2: エージェント定義

```markdown
<!-- .claude/agents/ai-ceo-morning.md -->
---
name: ai-ceo-morning
description: 朝ダイジェストエージェント。
  全部門の状態を収集し、CEO向けサマリを生成する。
tools:
  - Read    # ファイル読み取りのみ。書き込み権限なし
  - Grep    # 検索のみ
---

# AI-CEO Morning Digest Agent

## 実行手順

1. `.company/approval-queue.md` を読み取り、承認待ちを収集
2. `.company/departments/*/STATE.md` を全て読み取り
3. `.company/products/*/STATE.md` を全て読み取り
4. `.company/STATE.md` で全体コンテキストを確認
5. 以下のフォーマットでダイジェストを生成
```

注目すべきは `tools: [Read, Grep]` だ。朝ダイジェストエージェントは**読み取り専用**。ファイルを書き換える権限がない。これが権限制御の実装だ。

### ステップ3: 外部スクリプトで定時実行

```bash
#!/bin/bash
# auto-morning.sh
# launchd経由で毎朝9:00に実行される

source "$(dirname "$0")/common.sh"

main() {
  # コスト上限チェック（月$200を超えていたら実行しない）
  check_cost_limit || exit 0

  # ダッシュボードAPI経由でダイジェスト生成
  local result
  result=$(api_post "morning-digest" '{}')

  # 結果をSlackに送信（MCPではなくWebhookで直接送信）
  if [ "$(echo "$result" | jq -r '.success')" = "true" ]; then
    track_cost "morning-digest" "0.02"
  fi

  # STATE.mdの変更があればgit自動コミット
  git_auto_commit "auto: 朝ダイジェスト実行に伴う自動更新"
}

main "$@"
```

この3つのファイルが連携することで、「毎朝9時に、全部門の状態を読み取り、CEOにSlackでダイジェストを送り、変更をgitにコミットする」という一連の流れが全自動で動く。

## CLAUDE.md設計のチェックリスト

自分のCLAUDE.mdを書くときのチェックリストを示す。

```markdown
## CLAUDE.md 設計チェックリスト

### 構造
- [ ] 200行以内に収まっているか
- [ ] 「自分は誰か」が冒頭5行以内で定義されているか
- [ ] ファイルの中身ではなく、ファイルパスを参照しているか
- [ ] コマンド一覧が網羅されているか

### 権限制御
- [ ] read-only / draft / execute の3段階が定義されているか
- [ ] 対外アクションにdraftが強制されているか
- [ ] 費用閾値が明記されているか

### エージェント分離
- [ ] .claude/agents/ に各エージェントが分離されているか
- [ ] 各エージェントのtoolsが最小権限になっているか
- [ ] 委任時に渡す5項目が定義されているか

### エラーハンドリング
- [ ] 失敗時のリトライ回数が定義されているか
- [ ] エスカレーション先が明記されているか
- [ ] エラーログの出力先が指定されているか
```

## よくある失敗パターン

### 失敗1: CLAUDE.mdに全部書く

```markdown
<!-- NG: 800行のCLAUDE.md -->
# 役割定義（50行）
# 全コマンド定義（100行）
# 各コマンドの詳細実行フロー（300行）
# 全エージェントの定義（200行）
# ブランドガイドライン（100行）
# 技術スタック詳細（50行）
```

コンテキストウィンドウを圧迫し、後半の指示が無視される。**200行を超えたら分離を検討する。**

### 失敗2: 権限を定義しない

```markdown
<!-- NG: 権限の記載なし -->
## コマンド
- `/deploy` — 本番デプロイ
```

権限がないと、AIが勝手に本番デプロイを実行する可能性がある。全コマンドに権限レベルを明記する。

### 失敗3: エラーハンドリングを書かない

```markdown
<!-- NG: エラー時の振る舞いが未定義 -->
## コマンド
- `/generate-report` — レポート生成
```

APIが落ちていたら? ファイルが見つからなかったら? エラー時のフォールバックを必ず定義する。筆者の環境では「最大3回リトライ → 失敗したらapproval-queueにエスカレーション追加」としている。

```markdown
## エラー時の振る舞い

- サブエージェントが失敗した場合:
  エラー内容をフィードバックして最大3回リトライ
- 3回失敗:
  .company/approval-queue.md にエスカレーションとして追加し、
  CEOに報告
- エラーログ:
  .company/departments/{dept}/error-log.md に追記
```

## 次章への橋渡し

CLAUDE.mdという「基盤」ができた。次の章では、この基盤の上に最初の自動化レイヤーとして**Hooks**を構築する。Hooksを使えば、Claude Codeの操作中に自動でテスト実行、lint、通知が走るようになる。

:::message
**まとめ**
- CLAUDE.mdは「AIへの永続的な指示書」。毎回プロンプトを打つ必要がなくなる
- Thin Orchestratorパターンで200行以内に収め、詳細は外部ファイルに分離する
- 権限は read-only / draft / execute の3段階。対外アクションは必ずdraftを経由させる
- エージェント定義は `.claude/agents/` に分離し、最小権限の原則を適用する
- 委任時は「目的・参照パス・出力先・権限・品質基準」の5項目を渡す
- エラーハンドリング（リトライ・エスカレーション）を必ず定義する
:::
