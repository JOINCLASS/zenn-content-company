## Claude Codeは便利だ。でも、手動では限界がある

Claude Codeを使い始めたとき、衝撃を受けた読者は多いだろう。「このバグを直して」と言えばコードを修正してくれる。「テストを書いて」と言えばテストファイルが生成される。従来の開発体験とは次元が違う。

だが、しばらく使い続けると、ある種の「天井」にぶつかる。

筆者は合同会社ジョインクラスのCEOとして、1人で9つの部門（開発、マーケティング、営業、経理、CS、法務、税務、コンサル、HR）を運営している。Claude Codeを導入した当初、毎朝こんなことをやっていた。

```
# 毎朝やっていたこと（手動時代）
1. Claude Codeを起動する
2. 「各部門の状態を確認して」とプロンプトを打つ
3. 結果を読んで、次にやるべきことを考える
4. 「SNSの投稿を3件作って」とプロンプトを打つ
5. 出力をコピーして、LinkedInに貼り付ける
6. 「今月の経費を集計して」とプロンプトを打つ
7. 結果をスプレッドシートに転記する
8. ...以下、部門の数だけ繰り返す
```

これを9部門分やると、**毎朝2時間**がClaude Codeへの指示出しだけで消える。

Claude Codeの能力は十分に高い。問題は、その能力を引き出すために**人間がボトルネックになっている**ことだ。

## 手動運用の3つの壁

### 壁1: 繰り返しの壁 — 毎日同じプロンプトを打つ苦痛

「毎朝、全部門のSTATE.mdを読んで、承認待ちを一覧にして、今日のおすすめアクションを提案して」

このプロンプトを、筆者は3ヶ月間、毎朝手動で打っていた。内容はほぼ同じだ。にもかかわらず、毎回プロンプトを組み立て、出力を確認し、次のアクションに移る。

SNS投稿も同じだ。「LinkedInに投稿するAI活用のTipsを書いて。ハッシュタグは5つ以内で」と毎日打つ。出力をコピーしてブラウザに貼る。これを1日3件、7プラットフォーム分やっていた。

**毎日同じ指示を手動で打つのは、シェルスクリプトを毎回手で書き直すようなものだ。** エンジニアなら、それがいかに非効率かはわかるだろう。

### 壁2: スケールの壁 — 部門が増えると人間の処理能力が追いつかない

1部門なら手動でも回る。3部門でもなんとかなる。だが9部門になると、物理的に手が回らない。

筆者の場合、手動時代の1日のClaude Code操作は以下のようになっていた。

| 作業 | 所要時間 | 頻度 |
|------|---------|------|
| 朝のダイジェスト確認 | 15分 | 毎日 |
| SNS投稿作成・投稿 | 30分 | 毎日 |
| 技術記事の執筆依頼 | 20分 | 毎日 |
| 経理レポート確認 | 10分 | 毎日 |
| 営業メール作成 | 15分 | 随時 |
| コードレビュー依頼 | 15分 | 随時 |
| その他の部門タスク | 15分 | 随時 |
| **合計** | **約2時間** | |

2時間あればプロダクト開発が1つ進む。その時間を「AIへの指示出し」に費やすのは、本末転倒だ。

### 壁3: 一貫性の壁 — 人間の気分でプロンプトが変わる

手動運用の最も深刻な問題は、**プロンプトの品質が日によってブレる**ことだ。

月曜の朝、気合が入っているときは丁寧にプロンプトを書く。金曜の夕方、疲れているときは「適当にやっておいて」になる。結果として、AIの出力品質もブレる。

ブランドガイドラインに沿ったSNS投稿を求めたいのに、疲れている日はガイドラインの指定を忘れる。セキュリティレビューを通すべきコミットが、忙しい日にはスルーされる。

**自動化すれば、プロンプトの品質は常に一定だ。** 疲れていても、忙しくても、定義されたワークフローが同じ品質で実行される。

## Before/After — 著者のリアルな数字

ここからは、筆者が実際に自動化を構築した前後の数字を公開する。

### Before（手動運用期: 2025年7月〜12月）

```
# 手動運用時代の実績
- 1日のClaude Code操作時間: 約2時間
- SNS投稿数: 1日3-5件（手動コピペ）
- 技術記事公開頻度: 週1-2本
- 朝ダイジェスト: 気が向いたときだけ
- 月間コスト: Claude Pro $20 + 人件費（自分の2時間/日）
```

### After（自動化後: 2026年1月〜現在）

```
# 自動化後の実績
- 1日のCEO操作時間: 約30分（承認と意思決定のみ）
- SNS投稿数: 1日27件（7プラットフォーム、全自動スケジュール配信）
- 技術記事公開頻度: 毎日1本（Zenn 8:00 + Qiita 8:30 自動公開）
- 朝ダイジェスト: 毎朝9:00にSlack自動配信
- 月間コスト: Claude Code Max $200（約¥30,000）
- 運用部門数: 9部門
- 公開書籍数: 7冊（自動品質チェック付き）
```

**操作時間は2時間から30分に、SNS投稿数は5件から27件に。** しかも30分のうち大半は「承認」ボタンを押すだけだ。

### コスト構造の内訳

「月18,000円で9部門」の内訳はこうなっている。

| 項目 | 月額 | 備考 |
|------|------|------|
| Claude Code Max | ¥30,000 | 全AIエージェントの基盤 |
| Gemini API | ¥0 | 無料枠内で運用（ダッシュボード用） |
| Firebase | ¥0 | 無料枠内 |
| サーバー | ¥1,100 | レンタルサーバー |
| ドメイン | ¥500 | 年額按分 |
| **合計** | **約¥31,600** | |

厳密には約31,600円だが、Claude Code Maxの$200が中心で、それ以外はほぼ無料枠で収まっている。重要なのは、**追加の従量課金APIを使わずに、Claude Code Max単体で9部門分の自動化が回っている**ということだ。

人件費換算で考えると、1日2時間 x 20営業日 = 40時間/月。エンジニアの時給を3,000円とすれば月12万円。それが月3万円に置き換わっている。

### 本書のゴール

本書を読み終えたとき、あなたは以下を手に入れている。

1. **CLAUDE.mdベースのマルチエージェント設計** — 1つのリポジトリで複数のAIエージェントが協調動作する
2. **イベント駆動の自動化** — コミット、ファイル変更、エラーなどのイベントに応じて自動実行される仕組み
3. **定時実行の自動化** — launchdやcronで毎朝・毎時間・毎日の定型業務が自動で回る
4. **コスト管理の仕組み** — 月額上限を超えないよう自動監視し、超過前にアラートを出す

手動でプロンプトを打つ時代は終わりだ。次の章から、具体的な構築に入ろう。
