## この章で学ぶこと

- 全自動化に「承認」を挟むべき理由と、挟まなかった場合の実害
- 3段階の権限レベル（read-only / draft / execute）の設計
- `approval-queue.md` による承認キューの実装
- 朝ダイジェストとの連携で承認を日常業務に組み込む方法
- 同じタイプを3回承認したら自動化する「学習型承認」ルール

---

## 9.1 なぜ全自動ではダメなのか

自動化を進めていくと、ある段階で「全部自動にすればいいのでは」という誘惑に駆られる。筆者も最初はそう考えていた。しかし、実際に運用してみると**全自動は事故を起こす**ということを身をもって学んだ。

### git push --force 事故

開発エージェントにリファクタリングを委任したとき、エージェントがコンフリクト解消のために `git push --force` を実行した。結果、他のブランチで進行中だった3日分のコミットが消失した。復旧に半日かかった。

根本原因は明確で、「エージェントに破壊的操作の権限を与えていた」ことだ。コードの読み取りや分析は何度やっても被害はないが、外部に影響するアクションは取り消せない。

### 敬語メール事故

営業エージェントがクライアント候補に自動送信したメールで、「貴社のホームページ、正直ダメです」という直球表現が含まれていた。AIが「率直なフィードバックが効果的」と判断した結果だ。もちろん、このメールを送った相手からの返信はなかった。

これらの事故から導き出した原則がある。

> **対外に影響するアクションは、必ず人間の承認を通す。**

分析やレポート生成は自動でいい。しかし、メール送信、SNS投稿、デプロイ、請求書発行など「外部に影響が出る」アクションは、必ずドラフトを生成して人間が確認するパイプラインを通す。

## 9.2 3段階の権限レベル

筆者のAI-CEO Frameworkでは、すべてのアクションを3つの権限レベルに分類している。

### 権限レベルの定義

```yaml
# .company/steering/permissions.md から抜粋
# 3段階の権限レベル：何を自動実行し、何を人間が確認するかを明確にする

# Level 1: read-only — 分析・レポート系。何度実行しても被害なし
auto_execute_readonly:
  - analytics_report    # KPIレポート生成
  - kpi_update          # ダッシュボード更新
  - competitor_analysis # 競合調査
  - internal_docs       # 社内ドキュメント更新

# Level 2: draft — 対外アクション。必ず人間が確認してから実行
always_draft:
  - press_release       # プレスリリース
  - pricing_change      # 価格変更の告知
  - claim_response      # クレーム対応
  - proposal            # クライアント提案書
  - contract            # 契約書・NDA
  - invoice             # 請求書

# Level 3: execute — 閾値内の内部アクション。自動実行可
auto_execute:
  - bugfix              # バグ修正
  - minor_feature       # マイナー機能追加
  - test                # テスト追加・修正
  - refactor            # リファクタリング
  - docs                # ドキュメント更新
  - dependency_patch    # パッチ/マイナーの依存更新
```

### 判断基準はシンプル

「このアクションが間違っていた場合、取り消せるか？」を考える。

| 取り消し可能性 | 権限レベル | 例 |
|---|---|---|
| いつでも取り消せる | read-only | レポート生成、分析 |
| 社内で取り消せる | execute | バグ修正、テスト |
| 社外に影響が出る | draft | メール送信、デプロイ |

この基準を `permissions.md` に明文化しておくことで、新しいタスクが追加されたときも迷わず分類できる。

### 費用閾値との組み合わせ

```yaml
# 金額による自動判定も併用する
auto_approve_limit: 5000   # 5,000円以下は自動承認
ceo_approval_above: 5000   # 5,000円超はCEO承認必須
monthly_ai_budget: 200     # 月額$200のAI予算上限
```

たとえば、3,000円のドメイン更新は自動承認できるが、50,000円のサーバー契約はCEO承認が必要になる。権限レベルと金額閾値の二軸で安全性を担保する。

## 9.3 approval-queue.md の実装

承認キューは、あえてMarkdownファイルとして実装している。理由は3つある。

1. **gitで履歴管理できる** — 誰が・いつ・何を承認したかが全てコミットログに残る
2. **人間が直接読める** — データベースと違い、テキストエディタで即座に確認できる
3. **AIエージェントが操作しやすい** — 構造化されたMarkdownはLLMとの相性が良い

### ファイル構造

```markdown
# 承認待ちキュー

## 承認待ち (2件)

### [AQ-021] LinkedIn投稿：Claude Code全自動化バイブル告知
- **部門:** マーケティング
- **種別:** sns_post
- **ファイル:** .company/departments/marketing/drafts/linkedin-automation-book.md
- **概要:** 新刊の告知投稿ドラフト。ターゲットはエンジニア層。
- **登録日:** 2026-03-28
- **優先度:** medium

### [AQ-022] クライアント提案書：AI業務自動化コンサル
- **部門:** 営業
- **種別:** proposal
- **ファイル:** .company/departments/sales/proposals/ai-consulting-proposal.md
- **概要:** 製造業向けAI業務自動化の提案書ドラフト。
- **登録日:** 2026-03-28
- **優先度:** high

## 最近の承認/却下

### 2026-03-27: 承認
- [AQ-020] 既存取引先向けリリース告知メール送信 → **承認済み**
```

### エージェントからの登録フロー

サブエージェントがドラフトを生成すると、自動的に承認キューに追加される。この処理はダッシュボードのAPIを経由する。

```bash
#!/bin/bash
# 承認キューへの追加例（ダッシュボードAPI経由）
# エージェントが直接approval-queue.mdを編集するのではなく、
# APIを通すことで整合性を保つ

curl -sf -X POST \
  -H "Content-Type: application/json" \
  -d '{
    "title": "LinkedIn投稿：Claude Code全自動化バイブル告知",
    "department": "marketing",
    "type": "sns_post",
    "file": ".company/departments/marketing/drafts/linkedin-automation-book.md",
    "summary": "新刊の告知投稿ドラフト",
    "priority": "medium"
  }' \
  "http://localhost:4000/api/approvals/add"
```

### 承認・却下の操作

CEOが承認する場合は、CLI（`/ai-ceo:approve AQ-021`）またはダッシュボードのUIから操作する。

```bash
# 承認：キューから削除 + decisions/{month}.md に記録
/ai-ceo:approve AQ-021

# 却下：キューから削除 + 理由と共に記録 + 担当部門に差し戻し
/ai-ceo:reject AQ-022 "単価が低すぎる。月額30万円以上で再作成"
```

承認されたアイテムは `decisions/` ディレクトリに月ごとのMarkdownファイルとして記録される。これが経営判断のログになる。

## 9.4 朝ダイジェストとの連携

承認キューが溜まっていても、毎日能動的に確認するのは面倒だ。そこで、第7章で構築した朝ダイジェストに承認待ち一覧を組み込む。

### 朝ダイジェストのフォーマット

```
AI-CEO Morning Digest -- 2026-03-28

## 承認待ち (2件)
- [AQ-021] マーケ: LinkedIn投稿ドラフト | medium | 登録: 3/28
- [AQ-022] 営業: AI自動化コンサル提案書 | high | 登録: 3/28

## 部門状態サマリ
| 部門 | ステータス | 進行中タスク |
|------|-----------|------------|
| 開発 | 正常      | Kindle EPUB修正 |
| マーケ | 正常    | Autonomos 461件スケジュール済み |
| 営業 | 再編中    | AIコンサル営業集中 |

## 今日のおすすめアクション
1. [AQ-022] 高優先度の提案書を確認・承認してください
2. Zenn書籍の販売部数が11部に到達。次のプロモーション施策を検討
```

### Slack通知との連携

朝ダイジェストは毎朝9:00にSlackに自動送信される。承認待ちが3件以上溜まっている場合は、通知のアイコンを変えて注意を引く。

```bash
#!/bin/bash
# auto-morning.sh のSlack通知部分
# 承認待ち件数に応じてアイコンを変える

local approval_count=$(jq -r '.approvalCount' <<< "$result")

if [ "$approval_count" -ge 3 ]; then
  # 3件以上：警告アイコン
  icon=":warning:"
else
  # 通常：チャートアイコン
  icon=":bar_chart:"
fi

notify_slack "${icon} [AI-CEO] 朝ダイジェスト\n承認待ち: ${approval_count}件\n..."
```

実際の運用では、朝のSlack通知を見て、コーヒーを飲みながら承認操作をする。1件あたり30秒程度で判断できるので、5件溜まっていても3分で完了する。

## 9.5 自動承認ルール：学習型承認

同じタイプのアイテムを何度も手動承認するのは非効率だ。そこで「同じタイプを3回連続で承認したら、以降は自動承認する」というルールを導入している。

### 自動承認の仕組み

```bash
#!/bin/bash
# auto-approve.sh — 自動承認スクリプト
# permissions.md の閾値と承認履歴に基づき、承認待ちアイテムを自動承認する
# Cron: 毎日 09:30 JST

source "$(dirname "$0")/common.sh"

main() {
  log_info "=== 自動承認開始 ==="

  # コスト上限チェック（月$200を超えていたら実行しない）
  check_cost_limit || exit 0

  # ダッシュボードAPIで自動承認を実行
  # API内部で以下のロジックが走る:
  # 1. 承認待ちアイテムを取得
  # 2. 各アイテムの種別を確認
  # 3. 同じ種別で過去3回連続承認の履歴があれば自動承認
  # 4. 金額が auto_approve_limit 以下なら自動承認
  local result
  result=$(api_post "approvals/auto-approve" '{}')

  local approved_count
  approved_count=$(echo "$result" | jq -r '.approved | length')

  if [ "$approved_count" -gt 0 ]; then
    local approved_ids
    approved_ids=$(echo "$result" | jq -r '.approved | join(", ")')
    notify_slack ":white_check_mark: [AI-CEO] 自動承認完了: ${approved_ids} (${approved_count}件)"
  fi

  # 承認結果をgitにコミット（変更がある場合のみ）
  git_auto_commit "auto: 自動承認処理 (${approved_count}件)"
}

main "$@"
```

### 自動承認の条件

```yaml
# 自動承認が発動する条件（すべてAND）
auto_approve_conditions:
  # 条件1: 同じ種別で3回連続承認済み
  consecutive_approvals: 3

  # 条件2: 金額が閾値以下
  max_amount: 5000  # 円

  # 条件3: 以下の種別は自動承認の対象外（常にCEO確認）
  excluded_types:
    - contract        # 契約書は常に人間が確認
    - pricing_change  # 価格変更は常に人間が確認
    - claim_response  # クレーム対応は常に人間が確認
    - deploy_production  # 本番デプロイは常に人間が確認
```

### 実際の運用例

筆者の環境では、以下のアイテムが自動承認に移行した。

| 種別 | 3回承認した期間 | 自動化後の効果 |
|---|---|---|
| `scheduled_sns` | 2週間 | SNS投稿の確認が不要に |
| `tech_article` | 10日間 | Zenn/Qiita記事の公開が完全自動化 |
| `kpi_update` | 3日間 | ダッシュボード更新が完全自動化 |

一方で、`proposal`（提案書）は内容が毎回異なるため、20回以上手動承認しているが自動承認には移行していない。これは正しい判断だ。提案書は相手企業ごとにカスタマイズが必要で、AIの出力をそのまま送ると的外れになるリスクがある。

## 9.6 実際の運用フロー

ここまでの仕組みを統合した、1日の承認フローを示す。

### タイムライン

```
08:00  Zenn記事自動公開（自動承認済み → executeモード）
08:30  Qiita記事自動公開（自動承認済み → executeモード）
08:43  note記事自動投稿（自動承認済み → executeモード）
09:00  朝ダイジェストがSlackに届く
       → 承認待ち一覧を確認（通常1-3件）
09:05  CEOがスマホでSlackを確認
       → /ai-ceo:approve AQ-021
       → /ai-ceo:approve AQ-022
09:10  承認完了。以降はAIが自動実行
09:30  自動承認スクリプトが実行
       → 条件を満たすアイテムを自動処理
12:00  SNS投稿が自動配信（Autonomos経由）
19:00  夕方サマリがSlackに届く
       → 今日の実行結果を確認
```

### 承認にかかる時間

実測値として、承認操作に費やす時間は**1日平均5分以下**だ。朝のコーヒータイムで完了する。

全自動化と承認パイプラインの違いを数字で比較する。

| 指標 | 全自動（承認なし） | 承認パイプライン |
|---|---|---|
| 事故発生率 | 月2-3件 | 月0件（導入後3ヶ月） |
| 1日の承認時間 | 0分 | 5分 |
| 事故のリカバリ時間 | 平均2時間/件 | 0時間 |
| **月間のトータル工数** | **4-6時間** | **2.5時間** |

承認を挟む方が、トータルの工数は少ない。事故のリカバリは承認の何倍もコストがかかるからだ。

## まとめ

- 全自動化は事故を起こす。対外に影響するアクションには必ず承認を挟む
- 権限は3段階（read-only / draft / execute）に分け、`permissions.md` に明文化する
- 承認キューは `approval-queue.md` としてgit管理。AIエージェントが登録し、人間が承認する
- 朝ダイジェストに承認待ち一覧を組み込み、毎朝のルーティンとして承認を処理する
- 同じタイプを3回承認したら自動承認に移行することで、承認の手間を段階的に減らす
- 承認に1日5分を投資することで、月4-6時間の事故リカバリ工数をゼロにできる
