# 「AIは実行に、判断は人間に」— 承認パイプラインの設計

## AIに「全部任せる」は最悪の選択

前半6章で、日次レポート、問い合わせ対応、経理処理の自動化を解説しました。ここからは「自動化したあと、どう安全に運用するか」という話に入ります。

自動化で最も怖いのは、**AIが勝手に対外的なアクションを取ること**です。

たとえば、こんな事故が起きたらどうでしょうか。

- AIが取引先に誤った見積もりメールを自動送信した
- AIが顧客データを含むレポートを社外に共有した
- AIが間違った金額で請求書を発行した

「便利だから全部AIに任せよう」と考えた瞬間、こうしたリスクが現実になります。筆者自身、自社の自動化率は98%ですが、**対外的なアクションだけは必ず人間が確認するルール**を徹底しています。結果として、自動化を始めてからセキュリティ事故はゼロです。

この章では、「AIには実行させるが、判断は人間がする」という承認パイプラインの設計方法を解説します。

## 3段階の権限モデル: read-only / draft / execute

筆者が実際に使っている権限モデルは、たった3つの段階に分かれています。

### 第1段階: read-only（読み取り専用）

AIが情報を集めて分析するだけのモード。何も変更しない、何も送信しない。

**具体例:**
- 売上データの集計と可視化
- 競合他社の価格調査
- 社内文書の検索と要約
- 在庫数の確認
- ログファイルの分析

read-onlyのアクションは、**人間の確認なしで自動実行**します。データを見るだけで、外部に影響を与えないからです。

### 第2段階: draft（下書き）

AIが成果物を「下書き」として作成するモード。完成品を作るが、送信や公開はしない。

**具体例:**
- 取引先へのメール → 下書きとして保存 → 人間が確認して送信ボタンを押す
- 請求書 → PDFを生成 → 人間が金額を確認して送付
- SNS投稿文 → テキストを作成 → 人間が内容を確認して投稿
- 契約書のドラフト → 文面を作成 → 人間がレビューして署名

draftモードのポイントは、**AIが100%完成度の成果物を作る**ことです。人間は「確認して承認ボタンを押す」だけ。ゼロから書く必要はありません。

### 第3段階: execute（自動実行）

AIが判断から実行まで完全に自動で行うモード。人間は結果の報告を受けるだけ。

**具体例:**
- テストの実行 → 結果のみ報告
- バックアップの取得 → 完了通知のみ
- 日次レポートの生成と社内共有 → 自動配信
- データの定期集計 → 結果をダッシュボードに反映

executeにしてよいのは、**失敗しても被害が社内に限定される**アクションだけです。

### 判断の基準: どのモードにすべきか

迷ったら、次の質問に答えてください。

**「このアクションが間違っていた場合、社外の人に迷惑がかかるか？」**

- はい → **draft**（人間が確認してから実行）
- いいえ → **execute**（自動実行でOK）

もう1つの判断基準もあります。

**「このアクションは取り消し可能か？」**

- 取り消し不可（メール送信、請求書発行） → **draft**
- 取り消し可能（社内レポート、テスト実行） → **execute**

## 承認パイプラインの実装方法

### ステップ1: 業務をリストアップして分類する

まず、自動化する業務をすべてリストアップし、3段階に分類します。

| 業務 | 権限 | 理由 |
|------|------|------|
| 売上レポート生成 | execute | 社内向け、取り消し可能 |
| 取引先メール送信 | draft | 社外向け、取り消し不可 |
| 在庫データ確認 | read-only | 情報参照のみ |
| 請求書発行 | draft | 金銭に関わる、取り消し不可 |
| 社内チャット通知 | execute | 社内向け、取り消し可能 |
| SNS投稿 | draft | 社外向け、ブランドに影響 |
| データバックアップ | execute | 社内処理、取り消し可能 |
| 顧客への見積もり送付 | draft | 社外向け、金額に関わる |

この分類表を作るだけで、「何を自動にして、何を人間が確認するか」が明確になります。

### ステップ2: 承認キューを作る

draftモードの成果物を一覧管理する「承認キュー」を用意します。難しいツールは不要です。スプレッドシートやNotionのデータベースで十分です。

承認キューに必要な項目は以下の5つです。

- **ID**: 管理番号（AQ-001、AQ-002...）
- **種類**: メール / 請求書 / SNS投稿 / その他
- **内容の概要**: 「○○社への見積もり送付」など1行の説明
- **ファイルの場所**: 下書きファイルのパスやリンク
- **ステータス**: 待ち / 承認済み / 却下

筆者の会社では、この承認キューをMarkdownファイルで管理しています。毎朝のダイジェスト（第5章で解説）に「承認待ち○件」と表示されるので、見落としがありません。

### ステップ3: 承認のルーティンを組み込む

承認作業は、**毎日決まった時間に行う**のがコツです。

筆者のルーティンはこうです。

- **朝9時**: AIが生成した朝のダイジェストを確認（1-2分）
- **承認待ちがあれば**: 内容を確認して承認 or 却下（1件あたり30秒-1分）
- **午後は放置**: 承認キューが空なら、何もしない

重要なのは、**承認にかかる時間が極めて短い**ということです。AIが100%完成度の下書きを作ってくれているので、人間は「これで良いか悪いか」を判断するだけ。1日の承認作業は合計5分以内に収まります。

## CLAUDE.mdファイルによる権限管理

Claude Codeを使っている場合、`CLAUDE.md`というファイルにAIへの指示を書くことで、権限管理を自動化できます。

以下は筆者が実際に使っている設定の要約です。

```
# 権限ルール

## 自動実行OK（execute）
- データの読み取りと分析
- テストの実行
- 社内向けレポートの生成
- バックアップの取得

## 下書きモード（draft）— 必ず人間の承認後に実行
- メールの送信
- 請求書の発行
- SNSへの投稿
- 契約書の作成
- 対外的なファイルの共有

## 禁止事項
- パスワードやAPIキーの外部送信
- 顧客の個人情報を含むデータの社外共有
- 承認なしの金銭的コミットメント
```

この設定ファイルを用意するだけで、AIは自動的に権限レベルを判断します。メール送信の指示を受けた場合、AIは本文を作成して「下書きを作成しました。確認してください」と報告します。勝手に送信することはありません。

Claude Code以外のツールでも、同じ考え方は適用できます。Zapierなら「承認ステップ」を挟む。Power Automateなら「承認フロー」を追加する。ツールは違っても、原則は同じです。

## 品質ゲートの仕組み

承認パイプラインに加えて、**品質ゲート**を設けるとさらに安全性が高まります。

品質ゲートとは、「AIが成果物を作成した時点で、自動的にチェックをかける仕組み」です。

### 品質ゲートの例

**請求書の場合:**
- 金額が前月比150%を超えていないか → 超えていたら警告
- 宛先が既存の取引先リストに含まれているか → 含まれていなければ警告
- 消費税の計算が正しいか → 自動検算

**メールの場合:**
- 添付ファイルに個人情報が含まれていないか → スキャンして検出
- 宛先が社外か社内か → 社外なら自動的にdraftモードに変更
- 敬語の誤りがないか → 自動校正

**SNS投稿の場合:**
- 機密情報が含まれていないか → キーワード検出
- 投稿時間が適切か → 深夜投稿の防止
- 画像に著作権の問題がないか → 出典の確認

Claude Codeでは、「Hooks」という機能を使って品質ゲートを実装できます。Hooksとは、AIがアクションを実行する前後に自動で走るスクリプトです。

たとえば、「ファイルを保存する前に、個人情報が含まれていないかチェックする」というHookを設定できます。個人情報が検出されたら保存を中止し、警告を出します。

品質ゲートの設計で重要なのは、**「厳しすぎない」こと**です。すべてのアクションにゲートをかけると、承認が面倒になってAIを使わなくなります。「社外に影響があるもの」「金銭に関わるもの」に絞るのがコツです。

## 事故が起きた場合のリカバリー手順

どれだけ慎重に設計しても、事故が起きる可能性はゼロにはなりません。重要なのは、**事故が起きたときに素早くリカバリーできる体制**を整えておくことです。

### リカバリー手順の5ステップ

**ステップ1: 即時停止**
問題が発覚したら、まずAIの自動実行をすべて停止します。原因を特定する前に、これ以上の被害拡大を防ぎます。

**ステップ2: 影響範囲の特定**
「誰に」「何が」「いつ」影響したかを特定します。AIのログを確認すれば、実行したアクションの履歴がわかります。

**ステップ3: 関係者への連絡**
影響を受けた社外の関係者に、速やかに連絡します。隠そうとするのが最悪の対応です。「システムの不具合で誤った内容をお送りしました」と正直に伝えましょう。

**ステップ4: 原因の特定と修正**
権限設定に漏れがなかったか、品質ゲートが機能していたかを確認します。多くの場合、「draftにすべきだったものがexecuteになっていた」というシンプルな設定ミスが原因です。

**ステップ5: 再発防止策の実施**
権限の再設定、品質ゲートの追加、承認フローの見直しを行います。

### 筆者の経験: 事故ゼロを実現している理由

筆者がセキュリティ事故ゼロを維持できている理由は、シンプルです。

**「迷ったらdraft」のルールを徹底しているから**です。

新しい自動化を追加するとき、「これはexecuteでいいかな？」と迷ったら、必ずdraftにします。しばらく運用して問題がないことを確認してから、executeに昇格させます。

最初は面倒に感じるかもしれませんが、「事故が1回起きたときの対応コスト」と比較すれば、draftの手間は微々たるものです。

## 「承認が面倒」にならないための3つの設計原則

承認パイプラインの最大の敵は、**「面倒くさい」という感情**です。承認が面倒で放置される → 業務が止まる → 「やっぱりAIは使えない」となるパターンを何度も見てきました。

これを防ぐための設計原則を3つ紹介します。

### 原則1: 承認は1クリック（1アクション）で完了させる

承認のために別のツールを開く、ログインする、ファイルを探す...こうした手順が増えるほど、承認は放置されます。

理想は、**通知が来たら1クリックで承認できる**仕組みです。

筆者の場合、朝のダイジェストに承認待ち一覧が表示されるので、そこから直接承認できます。別のツールを開く必要はありません。

### 原則2: 承認が必要な件数を最小限に保つ

1日に20件の承認が溜まったら、誰でも嫌になります。

承認件数を減らすコツは、**「本当にdraftが必要なもの」を厳選する**ことです。

たとえば、「社内のSlack通知」にdraftは必要でしょうか？　社内向けで取り消し可能なら、executeで問題ありません。draftにするのは「社外向け」「金銭関連」「取り消し不可」のものだけに絞りましょう。

筆者の場合、1日の承認待ちは平均2-3件です。これなら5分以内に終わります。

### 原則3: 承認しなくても業務が止まらない設計にする

「承認待ち」のために他の業務が全停止する設計は危険です。

承認が遅れても、他の自動化は独立して動き続ける設計にしましょう。承認待ちのメールがあっても、日次レポートの生成や在庫データの集計は影響を受けません。

## 段階的に権限を緩和する方法

最初は厳しめの権限設定から始めて、実績に応じて徐々に緩和するのが安全なアプローチです。

**導入1ヶ月目:**
ほぼすべてのアクションをdraftモードに設定。人間が毎回確認。

**導入3ヶ月目:**
社内向けアクション（レポート配信、社内通知）をexecuteに昇格。確認作業が半減。

**導入6ヶ月目:**
定型的な社外アクション（定期レポートの顧客送付など）を「条件付きexecute」に昇格。内容が一定のテンプレートに従っている場合のみ自動実行。

**導入12ヶ月目:**
draftに残るのは「非定型の社外アクション」「金額が大きい取引」のみ。日常業務の95%が自動で回る。

この段階的なアプローチなら、リスクを最小限に抑えながら、自動化の範囲を広げていけます。

## この章のまとめ

AIに仕事を任せるときの原則は、**「実行はAIに、判断は人間に」**です。

3段階の権限モデル（read-only / draft / execute）を使えば、安全にAI自動化を運用できます。重要なのは「迷ったらdraft」。最初は厳しく、実績に応じて緩和していくのが失敗しないコツです。

## この章のアクションリスト

1. 自動化する業務をすべてリストアップし、read-only / draft / execute に分類する
2. draftモードの成果物を管理する「承認キュー」（スプレッドシートやNotionで可）を作成する
3. 毎朝5分の「承認タイム」をスケジュールに組み込む
4. 「社外向け」「金銭関連」「取り消し不可」のアクションは必ずdraftに設定する
5. 1ヶ月運用したら、問題がなかったものをexecuteに昇格できないか見直す
