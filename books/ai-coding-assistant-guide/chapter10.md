# アンチパターン集 -- AIコーディングでやってはいけないこと

## この章で学ぶこと

AIコーディングツールの活用法を学ぶだけでは、生産性は最大化できない。「やるべきこと」と同じくらい重要なのが「やってはいけないこと」を知ることだ。

筆者は6つのプロダクトの開発・運用を通じて、自身でも数多くの失敗を経験してきた。また、AIコーディングツールの導入支援を行う中で、多くのエンジニアが同じパターンの失敗を繰り返しているのを目の当たりにしてきた。

本章では、AIコーディングで特に多い7つのアンチパターンを、「問題」「なぜ起きるか」「対策」の構造で整理する。1つでも心当たりがあれば、今日から改善できるはずだ。


## アンチパターン1: AI生成コードを検証せずにマージする

### 問題

AIが生成したコードを「AIが書いたから正しいだろう」と思い込み、十分な検証をせずにマージしてしまう。結果として、微妙なバグ、パフォーマンスの問題、セキュリティホールが本番に混入する。

### 実例

以下は、AIが生成した一見正しそうなコードの例だ。

```typescript
// AIが生成したユーザー検索関数
async function searchUsers(query: string): Promise<User[]> {
  const users = await db.query(
    `SELECT * FROM users WHERE name LIKE '%${query}%'`
  );
  return users;
}
```

一見動くコードだが、**SQLインジェクションの脆弱性**がある。ユーザー入力を直接SQLに埋め込んでいるため、悪意のある入力でデータベースを操作される可能性がある。

AIはこのような脆弱性を含むコードを生成することがある。特に「動くコード」を素早く生成することに最適化されている場合、セキュリティの考慮が抜け落ちることがある。

### なぜ起きるか

- **AIへの過信**: 「最新のAIモデルだから間違えないだろう」という思い込み
- **スピード重視の文化**: 「速くリリースすること」が優先され、レビューが形骸化する
- **理解不足**: AI生成コードの内容を十分に理解していないため、問題に気づけない
- **レビューの負荷**: AI生成コードは量が多く、全てを丁寧にレビューする時間が取れない

### 対策

**1. 「AIが書いたコード」も人間が書いたコードと同じレビュー基準を適用する**

AIだからといってレビューを省略しない。むしろ、AIは「動くが最適ではないコード」を生成する傾向があるため、以下の観点で重点的にレビューする。

- セキュリティ: SQLインジェクション、XSS、認証・認可の抜け漏れ
- パフォーマンス: N+1クエリ、不要なループ、メモリリーク
- エッジケース: null/undefined処理、空配列、境界値

**2. テストを必ずセットで生成させる**

コードの生成と同時にテストも生成させる。テストが通ることはコードの正しさの最低限の保証になる。

```
悪い指示: 「ユーザー検索機能を実装して」
良い指示: 「ユーザー検索機能を実装して。ユニットテストも含めること。SQLインジェクション対策のテストケースも入れて」
```

**3. 段階的に信頼度を上げる**

最初は全てのAI生成コードを慎重にレビューし、徐々に「このパターンのコードはAIが正確に書ける」という信頼度を積み上げていく。信頼度が高い領域（ボイラープレート、定型的なCRUD等）はレビューを軽くし、信頼度が低い領域（セキュリティ関連、パフォーマンスクリティカルな箇所）は引き続き慎重にレビューする。


## アンチパターン2: プロンプトが曖昧すぎる

### 問題

「いい感じにして」「きれいにして」「バグを直して」のような曖昧な指示をAIに出し、期待と異なる結果が返ってきて時間を浪費する。

### 実例

```
悪い指示:
「このコンポーネントをリファクタリングして」

AIの反応:
- 何をどう改善すべきか不明
- 独自の判断で命名を変更、構造を変更、型定義を変更
- 結果、既存のテストが壊れる
- プロジェクトの規約に合わない書き方になる
```

### なぜ起きるか

- **AIの能力への過度な期待**: 「意図を汲み取ってくれるだろう」という楽観
- **指示の言語化に慣れていない**: 自分がやりたいことを具体的な言葉にする訓練が不足
- **CLAUDE.md/.cursorrules の未整備**: プロジェクトの前提情報がAIに伝わっていない

### 対策

**1. 5W1Hを意識した指示を書く**

| 要素 | 質問 | 例 |
|------|------|-----|
| What | 何をするのか | `SearchForm` コンポーネントのバリデーションロジックを分離する |
| Why | なぜそうするのか | テストしやすくするため |
| Where | どこを変更するのか | `src/components/SearchForm.tsx` |
| How | どういう方法で | カスタムフック `useSearchValidation` に抽出する |
| Constraints | 制約は何か | 既存のテストを壊さない。外部APIの呼び出しは変更しない |

**2. 具体的な指示の例**

```
良い指示の例:

src/components/SearchForm.tsx の中にあるバリデーションロジックを、
カスタムフック useSearchValidation に抽出してください。

条件:
- フックは src/hooks/useSearchValidation.ts に配置
- 既存のテスト（SearchForm.test.tsx）が引き続きパスすること
- フックのユニットテストも新たに作成すること
- エラーメッセージの文言は変更しないこと
```

**3. CLAUDE.md/.cursorrules を整備する**

第3章で解説した通り、プロジェクトの前提情報（技術スタック、コーディング規約、ディレクトリ構成）をCLAUDE.mdに記述しておけば、毎回の指示で前提を説明する手間が省ける。CLAUDE.mdは「AIへの指示のベースライン」だ。


## アンチパターン3: AIに全てを任せようとする

### 問題

設計判断、ビジネスロジックの決定、アーキテクチャの選定まで全てAIに委任しようとする。結果として、一貫性のないコードベースや、プロジェクトの方向性と合わない実装が生まれる。

### 実例

```
危険な委任の例:
「新しいユーザー管理機能を全部設計から実装まで任せます。
データベース設計、API設計、フロントエンド実装、テスト、全てお願いします」
```

この指示で生成されるコードは、技術的には動くかもしれない。しかし、以下の問題が起きやすい。

- 既存のデータベーススキーマとの整合性が取れない
- チームの認証・認可パターンと異なるアプローチが採用される
- ビジネス要件の優先順位が反映されない（全機能を等しく実装してしまう）
- 将来の拡張性を考慮しない設計になる

### なぜ起きるか

- **AIの能力の過大評価**: AIは「既存のパターンの延長」は得意だが、「ビジネスコンテキストを踏まえた判断」は苦手
- **効率化への焦り**: 「全部任せれば最速だ」という短絡的思考
- **設計スキルの未熟さ**: 自分で設計する力が不足しているため、AIに丸投げする

### 対策

**1. 人間が設計し、AIが実装する分業を徹底する**

| フェーズ | 担当 | 理由 |
|---------|------|------|
| 要件定義 | 人間 | ビジネス目標、ユーザーニーズの理解は人間にしかできない |
| アーキテクチャ設計 | 人間（AIの壁打ち活用） | トレードオフの判断は人間。AIには選択肢の列挙を依頼 |
| API設計 | 人間（AIのドラフト活用） | 一貫性の維持は人間。AIに叩き台を作らせるのは有効 |
| 実装 | AI | 設計が固まっていれば、AIは高速かつ正確に実装できる |
| テスト | AI + 人間 | AIにテストを生成させ、人間がエッジケースを追加 |
| コードレビュー | 人間 | 最終的な品質責任は人間 |

**2. 「AIに壁打ちを依頼する」パターンを活用する**

設計判断を全てAIに委ねるのではなく、「選択肢を提示してもらい、人間が判断する」パターンが効果的だ。

```
良い使い方:
「ユーザー管理のデータベース設計で、以下の2つのアプローチを比較してください。
A: 単一テーブルにroleカラムを持たせる
B: usersテーブルとrolesテーブルを分離する

比較ポイント: クエリのパフォーマンス、将来の拡張性、実装の複雑さ」
```

AIに「選択肢の比較」を依頼し、「どちらを選ぶか」は人間が決める。これが、AIの能力を最大限に活かしつつ、設計の一貫性を保つ方法だ。


## アンチパターン4: コンテキストを無視した指示

### 問題

CLAUDE.mdや.cursorrulesを設定せず、プロジェクトの前提情報なしにAIに指示を出す。結果として、プロジェクトの技術スタックや規約に合わないコードが生成される。

### 実例

Next.js + Tailwind CSS + shadcn/ui のプロジェクトでCLAUDE.mdを未設定のまま使った場合に起きる典型的な問題。

```typescript
// AIが生成したコード（CLAUDE.md未設定）
import React from 'react';
import styled from 'styled-components';  // プロジェクトではTailwindを使用
import { Button } from '@mui/material';   // プロジェクトではshadcn/uiを使用

const StyledContainer = styled.div`       // styled-componentsは使っていない
  padding: 16px;
  margin: 8px;
`;

export default function UserList() {      // プロジェクトはnamed exportが規約
  // ...
}
```

1つのコンポーネントに4つも規約違反が含まれている。手動で修正する手間を考えると、AIに書いてもらった意味がほとんどない。

### なぜ起きるか

- **初期設定の手間を惜しむ**: 「CLAUDE.mdを書く時間が惜しい」と考え、設定を後回しにする
- **効果を過小評価**: CLAUDE.mdの有無でAIの出力品質がどれほど変わるかを実感していない
- **既存プロジェクトへの途中導入**: 途中からAIツールを導入した場合、CLAUDE.mdの作成を忘れがち

### 対策

**1. CLAUDE.md/.cursorrules をプロジェクト初日に作る**

AIツールの導入と同時にCLAUDE.mdを作成する。最低限、以下の情報を含める。

```markdown
## 技術スタック
- Next.js 15 (App Router) + TypeScript
- Tailwind CSS + shadcn/ui
- Supabase (PostgreSQL)

## コーディング規約
- named export を使用（default export は page.tsx のみ）
- styled-components, CSS Modules は使わない
- UIコンポーネントは shadcn/ui を使用
```

この10行を書くだけで、AIの出力品質は劇的に変わる。投資対効果は絶大だ。

**2. AI自身にCLAUDE.mdを生成させる**

皮肉なことに、CLAUDE.mdの作成自体をAIに依頼するのが最も効率的だ。

```
指示例:
「このプロジェクトのコードベースを分析して、
CLAUDE.md（プロジェクトのコーディング規約、技術スタック、
ディレクトリ構成をまとめたファイル）を生成してください」
```

Claude Codeはプロジェクトのファイルをスキャンし、使用技術、コーディングパターン、ディレクトリ構成を自動で検出してCLAUDE.mdのドラフトを生成してくれる。人間はそのドラフトを確認・修正するだけだ。

**3. チームでCLAUDE.mdをバージョン管理する**

CLAUDE.mdはリポジトリにコミットし、チーム全員が同じ前提でAIを使えるようにする。新しい技術の追加や規約の変更があったら、CLAUDE.mdも合わせて更新する。コードの変更とCLAUDE.mdの変更を同じPRに含めるのがベストプラクティスだ。


## アンチパターン5: セキュリティを考慮しない

### 問題

AIとのやり取りの中で、APIキー、パスワード、個人情報などの機密情報を不用意に渡してしまう。また、AIが生成したコードにセキュリティ上の問題がないかを確認しない。

### 実例: よくある3つの事故パターン

**パターンA: プロンプトにAPIキーを含める**

```
悪い指示:
「このStripeのAPIキー sk_live_xxxxxxxxxx を使って、
決済処理のコードを書いてください」
```

APIキーがAIモデルのログに残る可能性がある。また、チャット履歴に残り、後からキーが漏洩するリスクがある。

**パターンB: .envファイルをコンテキストに含める**

Cursorで `.env` ファイルを開いた状態でAIチャットを使うと、環境変数の値がAIに送信される場合がある。

**パターンC: AI生成コードの脆弱性を見逃す**

アンチパターン1で示したSQLインジェクションの例に加え、以下のような脆弱性が含まれるケースがある。

```typescript
// AIが生成した認証チェック（不十分な例）
function isAdmin(user: User): boolean {
  return user.role === "admin";  // クライアントサイドの値を信頼している
}

// 正しいアプローチ: サーバーサイドでDBを参照して検証
async function isAdmin(userId: string): Promise<boolean> {
  const user = await db.query("SELECT role FROM users WHERE id = $1", [userId]);
  return user?.role === "admin";
}
```

### なぜ起きるか

- **便利さとセキュリティのトレードオフ**: 「環境変数をそのまま渡した方が早い」という判断
- **AIの安全性への過信**: 「AIに渡したデータは安全に処理されるだろう」という思い込み
- **セキュリティ知識の不足**: 何が機密情報に該当するか、どんな脆弱性パターンがあるかの知識が不十分

### 対策

**1. 機密情報はプロンプトに絶対に含めない**

環境変数、APIキー、パスワード、個人情報は、AIへの指示に含めない。代わりに、プレースホルダーを使う。

```
良い指示:
「Stripe決済処理のコードを書いてください。
APIキーは環境変数 STRIPE_SECRET_KEY から取得すること」
```

**2. .gitignore と .cursorignore を設定する**

AIツールがアクセスすべきでないファイルを明示的に除外する。

```
# .cursorignore
.env
.env.*
*.pem
credentials/
secrets/
```

**3. セキュリティレビューのチェックリスト**

AI生成コードに対して、以下のチェックリストでレビューする。

- [ ] SQLクエリはパラメータ化されているか（直接文字列結合していないか）
- [ ] ユーザー入力はサニタイズされているか
- [ ] 認証・認可のチェックはサーバーサイドで行われているか
- [ ] 機密情報がログに出力されていないか
- [ ] CORS設定は適切か
- [ ] エラーメッセージに内部情報が漏洩していないか


## アンチパターン6: AIの出力を理解しないまま使う

### 問題

AIが生成したコードの内容を理解しないまま、「動いたからOK」としてプロジェクトに取り込む。後でバグが発生した際に原因を追えない。また、スキルが向上しない。

### 実例

```typescript
// AIが生成した非同期処理のコード
const results = await Promise.allSettled(
  items.map(item =>
    retryWithBackoff(() => processItem(item), {
      maxRetries: 3,
      baseDelay: 1000,
      maxDelay: 10000,
    })
  )
);

const succeeded = results
  .filter((r): r is PromiseFulfilledResult<ProcessResult> => r.status === "fulfilled")
  .map(r => r.value);

const failed = results
  .filter((r): r is PromiseRejectedResult => r.status === "rejected")
  .map(r => r.reason);
```

このコードが何をしているか説明できるだろうか。`Promise.allSettled` と `Promise.all` の違いは何か。`retryWithBackoff` の指数バックオフはどう動作するか。型ガード `r is PromiseFulfilledResult<ProcessResult>` はなぜ必要か。

これらを理解せずに使うと、エラーハンドリングの挙動が期待と異なった時に対処できない。

### なぜ起きるか

- **時間的プレッシャー**: 「理解する時間より、次のタスクに進む方が重要」という判断
- **難易度のギャップ**: AIが自分のスキルレベルより高度なコードを生成し、理解が追いつかない
- **動作確認の誤解**: 「テストが通った = 正しく理解した」という勘違い

### 対策

**1. AIに説明させる**

生成されたコードの意味がわからない場合は、AIに説明を求める。これはスキルアップの絶好の機会でもある。

```
指示例:
「このコードを1行ずつ解説してください。
特に Promise.allSettled を使っている理由と、
Promise.all を使った場合との動作の違いを説明してください」
```

**2. 「理解してから使う」ルールを設ける**

チームルールとして「AI生成コードは、レビュアーに説明できるレベルまで理解してからマージする」というポリシーを設ける。説明できないコードは、理解するまでマージしない。

**3. 段階的に複雑さを上げる**

最初は自分が理解できるレベルの指示から始め、徐々に複雑な指示に進む。AIが高度すぎるコードを生成した場合は、「もっとシンプルなアプローチで書き直して」と指示する。

```
指示例:
「このリトライロジックを、async/awaitの基本的な構文だけで
書き直してください。ライブラリは使わずに、forループとtry-catchで実装してください」
```


## アンチパターン7: ツールの特性を無視した使い方

### 問題

各AIコーディングツールの得意・不得意を理解せず、全てのタスクに同じツールを使う。結果として、効率が上がらないか、かえって時間がかかる。

### 実例

**Claude CodeでCSS微調整を行う**

```
指示: 「このボタンのhover時の色を #3b82f6 に変更して、
角丸を 8px にして、影を追加して。
あ、やっぱり影は薄くして。もう少し右にずらして...」
```

Claude Codeはターミナルベースのツールだ。CSSの微調整のような「視覚的なフィードバックを見ながら少しずつ調整する」タスクには向いていない。Cursorでプレビューを確認しながら調整する方が圧倒的に早い。

**GitHub CopilotでプロジェクトD全体のアーキテクチャ変更を行う**

```
指示: 「認証システムを全面的に書き換えて。
Supabase AuthからClerk認証に移行して、
関連する全ファイルを変更して」
```

Copilotはインライン補完とチャットが主な機能だ。プロジェクト全体にまたがる大規模な変更は、Claude Codeのようにプロジェクト全体を把握してサブエージェントに分割実行するツールが適している。

### なぜ起きるか

- **ツール学習のコスト**: 複数ツールを学ぶ手間を惜しみ、1つのツールで全てを済ませようとする
- **使い慣れへの偏り**: 最初に習得したツールに依存する
- **ツール特性の未理解**: 第2章の比較表を読んでいない（または忘れている）

### 対策

**1. タスクとツールのマッピングを作る**

チーム内で、以下のようなマッピングを共有する。

| タスクの特性 | 最適ツール | 理由 |
|------------|-----------|------|
| プロジェクト横断の大規模変更 | Claude Code | プロジェクト全体の理解 + サブエージェント |
| 日常的なコーディング | Cursor | インライン補完 + Composer |
| CSSやUIの微調整 | Cursor | 視覚的フィードバック |
| PRレビューと要約 | GitHub Copilot | GitHubとのネイティブ統合 |
| CI/CDの構築 | Claude Code | ターミナル操作との親和性 |
| チーム標準ツール | GitHub Copilot | 低い学習コスト + チーム管理 |

**2. 「このタスクにはどのツールが最適か」を指示の前に5秒考える**

習慣にするだけで、ツール選択のミスマッチによる時間の浪費を大幅に削減できる。

**3. 週に1回、別のツールを試す**

普段Claude Codeだけ使っている人は、週に1回はCursorで作業してみる。普段Cursorだけの人は、Claude Codeで大きなタスクを試してみる。体験することで、各ツールの特性が体感的にわかるようになる。

## この章のまとめ

7つのアンチパターンに共通するのは、「AIツールの力を借りすぎて、人間としての判断力と責任感を手放してしまう」ことだ。

AIコーディングツールは、あくまで**人間のエンジニアの能力を増幅する道具**だ。道具の使い方を誤れば、生産性は上がるどころか下がる。

重要なのは、AIに任せることと人間が判断することの線引きを明確にすることだ。

- **AIに任せること**: 実装、テスト生成、ボイラープレート、ドキュメント生成
- **人間が判断すること**: 設計、セキュリティ、品質の最終確認、ビジネス要件の解釈

この線引きを守りつつ、AIの出力を常に検証し、理解した上で使う。この姿勢こそが、AIコーディング時代に長期的に生産性を高め続けるための基盤だ。
