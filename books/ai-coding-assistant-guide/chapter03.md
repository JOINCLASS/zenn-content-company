# Claude Code 実践ガイド -- ターミナルからプロジェクト全体を操る

## この章で学ぶこと

前章では、3つのAIコーディングツールの比較と使い分けの指針を示した。本章からは各ツールの実践ガイドに入る。

まずはClaude Codeだ。筆者が6つのプロダクトを1人で運営できている最大の理由が、このツールにある。

本章では以下を扱う。

1. Claude Code のインストールと初期設定
2. CLAUDE.md の設計 -- プロジェクト固有の指示を最適化する
3. サブエージェント（Task tool）の活用 -- 探索・計画・実装の委任
4. Permission設定 -- 安全に使うための権限管理

## インストールと初期設定

### 前提条件

Claude Codeを使うには、以下が必要だ。

- Node.js 18以上
- ターミナル（macOS Terminal、iTerm2、Windows Terminal、WSL等）
- Anthropicのアカウント（Claude Maxプラン推奨）

### インストール

npmでグローバルインストールする。

```bash
npm install -g @anthropic-ai/claude-code
```

インストール後、ターミナルで `claude` コマンドが使えるようになる。

```bash
claude
```

初回起動時にAnthropicアカウントでの認証が求められる。ブラウザが開くので、ログインして認可すればセットアップは完了だ。

### 初回起動時の動作確認

プロジェクトのディレクトリに移動して `claude` を起動する。

```bash
cd /path/to/your/project
claude
```

起動すると対話型のインターフェースが表示される。まずは簡単な指示で動作を確認しよう。

```
> このプロジェクトの構造を教えて
```

Claude Codeがディレクトリをスキャンし、プロジェクトの概要を返してくれる。ここで正確にプロジェクト構造を把握できていれば、セットアップは成功だ。

### プラン選択のガイド

Claude Codeの利用プランにはいくつかの選択肢がある。

| プラン | 月額 | 特徴 | 推奨用途 |
|--------|------|------|---------|
| API直接利用 | 従量課金 | 使った分だけ | 軽い使用、試用 |
| Claude Pro | $20/月 | Claude.ai経由、使用量制限あり | ライトユーザー |
| Claude Max 5x | $100/月 | 高い使用量上限 | 日常的な開発 |
| Claude Max 20x | $200/月 | 非常に高い使用量上限 | ヘビーユーザー、プロジェクト全体の自動化 |

筆者はClaude Max（$100-200/月）を利用している。6つのプロダクトの開発・保守を行う上で、1日に数十回の大きなタスクを実行するため、この投資は十分にペイしている。

始めたばかりの段階では、API直接利用で感触を掴み、使用頻度が上がってきたらMaxプランへの移行を検討するのが合理的だ。


## CLAUDE.md の設計 -- AIに「プロジェクトの常識」を教える

### CLAUDE.md とは何か

CLAUDE.mdは、Claude Codeに対するプロジェクト固有の指示書だ。リポジトリのルートディレクトリに配置する。

人間のエンジニアがチームに参加した時のことを思い出してほしい。最初にオンボーディングドキュメントを読み、コーディング規約を学び、アーキテクチャの概要を把握するだろう。CLAUDE.mdは、それと同じ情報をClaude Codeに伝えるためのファイルだ。

CLAUDE.mdがないままClaude Codeを使うのは、ドキュメントなしで新人エンジニアにタスクを振るようなものだ。動くコードは書けるかもしれないが、プロジェクトの規約やパターンに沿わない可能性が高い。

### CLAUDE.md の配置場所

CLAUDE.mdは複数の場所に配置でき、それぞれスコープが異なる。

| 配置場所 | スコープ | 用途 |
|---------|--------|------|
| `~/.claude/CLAUDE.md` | グローバル（全プロジェクト共通） | 個人の作業スタイル、共通の品質基準 |
| `プロジェクトルート/CLAUDE.md` | プロジェクト全体 | コーディング規約、アーキテクチャ、技術スタック |
| `サブディレクトリ/CLAUDE.md` | 特定のディレクトリ以下 | モジュール固有のルール |

複数の場所にCLAUDE.mdがある場合、全てがマージされてClaude Codeに渡される。グローバルで共通ルールを定義し、プロジェクトルートで固有ルールを追加し、必要に応じてサブディレクトリで詳細ルールを上書きするという階層構造が使える。

### CLAUDE.md に書くべき内容

効果的なCLAUDE.mdは、以下の要素を含んでいる。

**1. プロジェクトの概要**

```markdown
## プロジェクト概要
Focalize は、営業チーム向けの Instagram 投稿管理 SaaS。
Next.js 15 (App Router) + Supabase + Stripe で構築。
多言語対応（日本語・英語）、Freeプラン/Proプランの2層構成。
```

この情報があるだけで、Claude Codeは「営業向けSaaS」「Next.js + Supabase」というコンテキストを踏まえた提案を行えるようになる。

**2. 技術スタック**

```markdown
## 技術スタック
- フレームワーク: Next.js 15 (App Router)
- 言語: TypeScript (strict mode)
- DB: Supabase (PostgreSQL + RLS)
- 認証: Supabase Auth
- 決済: Stripe
- UI: Tailwind CSS + shadcn/ui
- テスト: Jest + Testing Library
- CI/CD: GitHub Actions
```

使用している技術を明示することで、「Prismaではなく Supabase Client を使う」「MUIではなく shadcn/ui を使う」といった技術選定の一貫性が保たれる。

**3. コーディング規約**

```markdown
## コーディング規約
- 関数コンポーネントのみ使用（クラスコンポーネント禁止）
- named export を使用（default export は page.tsx のみ）
- 型定義は interface を優先（type alias は union/intersection のみ）
- 不要な複雑さを排除: 早期return、ガード節を活用してネストを減らす
- コメントは「なぜ」を書く。「何を」しているかはコード自体で表現する
```

**4. ディレクトリ構成**

```markdown
## ディレクトリ構成
src/
  app/           -- Next.js App Router のルート定義
  components/    -- 共通UIコンポーネント
    ui/          -- shadcn/ui コンポーネント
    landing/     -- LP固有のコンポーネント
  features/      -- 機能モジュール（ドメインごと）
  lib/           -- ユーティリティ、Supabase Client等
  hooks/         -- カスタムフック
  types/         -- 型定義
```

ディレクトリ構成を伝えることで、「新しいコンポーネントはどこに作るべきか」「ユーティリティはどこに置くか」をClaude Codeが正しく判断できる。

**5. 開発ワークフローのルール**

```markdown
## 開発ルール
- コードを書いた後、レビュー前に自動でシンプル化できる箇所がないか確認する
- 未使用の import、変数、コメントアウトされたコードは削除する
- 新機能はテストと一緒に実装する
- 破壊的変更は段階的に行い、一度に大量のファイルを変更しない
```

### CLAUDE.md のアンチパターン

効果的なCLAUDE.mdを書くために、避けるべきパターンも知っておこう。

**長すぎるCLAUDE.md**

CLAUDE.mdの内容は毎回Claude Codeのコンテキストに読み込まれる。10,000字を超えるような長大なCLAUDE.mdは、コンテキストウィンドウを圧迫し、本来のタスクに使えるトークン数を減らしてしまう。核心的な情報に絞り、詳細は別ファイルに分離して必要時に参照させる方が効率的だ。

**曖昧な指示**

「きれいなコードを書いて」のような曖昧な指示は効果がない。「早期returnを使ってネストを最大3段に抑える」のように、具体的で検証可能なルールにする。

**矛盾する指示**

「シンプルにコードを書くこと」と「全てのエッジケースを網羅的にハンドリングすること」のように、トレードオフが存在するルールを両方書くと、Claude Codeは一貫性のない出力をする。優先順位を明示するか、どちらか一方に絞る。


## サブエージェント（Task tool）の活用

### サブエージェントとは

Claude Codeの最も強力な機能の1つが、サブエージェント（Task tool）だ。これは、Claude Codeが複雑なタスクを実行する際に、自律的に「子タスク」を起動する仕組みだ。

イメージとしては、プロジェクトマネージャーが複数の担当者にタスクを振るのに近い。メインのClaude Codeセッションが「PM」として全体を管理し、個別の調査や実装をサブエージェントに委任する。

### 実際の動作フロー

たとえば「この機能のテストカバレッジを改善して」と指示した場合、以下のような流れで実行される。

```
メインエージェント（PM）
  ├── サブエージェント1: 現在のテストカバレッジを調査
  ├── サブエージェント2: テストが不足しているモジュールを特定
  ├── サブエージェント3: ユニットテストを生成
  └── サブエージェント4: E2Eテストを生成
```

各サブエージェントは独立したコンテキストを持つため、メインエージェントのコンテキストウィンドウが圧迫されない。これが「Thin Orchestrator」（薄い統括者）パターンと呼ばれる設計思想で、Claude Codeのスケーラビリティを支えている。

### Thin Orchestrator パターン

サブエージェントを効果的に活用するための設計パターンがThin Orchestratorだ。

**原則**: メインのClaude Codeセッションは、自分ではファイルの詳細を読み込まず、サブエージェントに「何を調べるか」「何を実装するか」の指示だけを出す。

これにより、メインセッションのコンテキスト使用率を低く保ちつつ、プロジェクト全体にわたる複雑なタスクを実行できる。

CLAUDE.mdにこの考え方を明示しておくと、Claude Codeは自律的にサブエージェントを活用するようになる。

```markdown
## Thin Orchestratorの原則
- コンテキスト使用率を10-15%に維持する
- ファイルの中身を自分のコンテキストに読み込まず、ファイルパスだけを渡す
- 複雑なタスクは必ずサブエージェントに委任する
```

### サブエージェントが特に有効なシーン

| シーン | 使い方 |
|--------|-------|
| 大規模リファクタリング | サブエージェントにモジュール単位でリファクタリングを委任 |
| コードベース調査 | 「このパターンが使われている箇所を全て探して」とサブエージェントに依頼 |
| テスト一括生成 | 複数モジュールのテストを並行してサブエージェントに生成させる |
| ドキュメント更新 | API仕様の変更に合わせて、関連するドキュメントの更新をサブエージェントに委任 |


## Permission設定 -- 安全に使うための権限管理

### なぜ権限管理が必要なのか

Claude Codeは非常に強力なツールだ。ファイルの作成・編集・削除、シェルコマンドの実行、外部APIの呼び出しまで実行できる。

この強力さは諸刃の剣でもある。「データベースのマイグレーションを実行して」と不用意に指示すれば、本番環境のデータを破壊する可能性もある。Permission設定は、そうした事故を防ぐための安全装置だ。

### Permission の基本概念

Claude Codeの権限は大きく以下の3段階で制御できる。

| 権限レベル | 動作 | 適用例 |
|-----------|------|--------|
| **Allow（許可）** | 確認なしで自動実行 | ファイルの読み取り、lsコマンド |
| **Ask（確認）** | 実行前にユーザーに確認 | ファイルの書き込み、テストの実行 |
| **Deny（拒否）** | 実行を禁止 | 本番環境へのデプロイ、破壊的コマンド |

### 推奨設定

筆者が実際に使用しているPermission設定の方針を紹介する。

**自動許可（Allow）にしてよい操作**:
- ファイルの読み取り（Read）
- ディレクトリ内容の一覧（ls）
- gitの参照系コマンド（git status, git log, git diff）
- テストの実行（npm test, pytest等）
- 型チェック（tsc --noEmit）
- リンター（eslint, prettier --check）

**確認を求める（Ask）べき操作**:
- ファイルの書き込み・作成
- gitの変更系コマンド（git add, git commit）
- パッケージのインストール（npm install）
- ビルド（npm run build）

**拒否（Deny）すべき操作**:
- 本番環境への直接アクセス
- `rm -rf` などの破壊的コマンド
- 環境変数やシークレットの出力
- 外部APIへのリクエスト（明示的に許可した場合を除く）

### 設定ファイルの配置

Permission設定は `.claude/settings.json` に記述する。

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Bash(git status)",
      "Bash(git log*)",
      "Bash(git diff*)",
      "Bash(npm test*)",
      "Bash(npx tsc --noEmit)"
    ],
    "deny": [
      "Bash(rm -rf*)",
      "Bash(*--force*)",
      "Bash(*prod*)",
      "Bash(curl*)",
      "Bash(wget*)"
    ]
  }
}
```

`allow` に含まれない操作は、デフォルトでユーザーに確認が求められる（Ask動作）。まずは保守的な設定から始めて、信頼度が高まったら徐々にallowの範囲を広げるのが安全なアプローチだ。


## 実践例 -- AI-CEO Frameworkで使用しているCLAUDE.md

ここからは、筆者が実際に運用しているCLAUDE.mdの構造を紹介する。合同会社ジョインクラスでは「AI-CEO Framework」と呼ぶ仕組みで、Claude Codeを経営管理にまで活用している。

### 全体構造

```
プロジェクトルート/
  CLAUDE.md              -- メインの指示書（Orchestrator定義）
  .claude/
    agents/              -- サブエージェント定義
      content-engine.md  -- コンテンツ作成エージェント
      growth-hacker.md   -- グロースハックエージェント
      dev-lead.md        -- 開発リードエージェント
    settings.json        -- Permission設定
  .company/
    STATE.md             -- 現在の経営状態
    ROADMAP.md           -- 四半期ロードマップ
    steering/            -- 経営方針・ブランドガイド
    departments/         -- 部門別の状態と成果物
    products/            -- プロダクト別の状態
```

### CLAUDE.md の中身（抜粋・解説付き）

メインのCLAUDE.mdは、Claude Codeを「C-Suite Orchestrator（経営統括者）」として定義している。

```markdown
# AI-CEO Framework -- C-Suite Orchestrator

> あなたはAI-CEO Frameworkの「C-Suite Orchestrator」です。
> CEOの経営判断を支援し、6部門のAIエージェントを統括します。

## あなたの役割
1. CEOコマンドの受付と実行
2. 部門間調整
3. 承認管理
4. プロダクト横断管理

## Thin Orchestratorの原則
- コンテキスト使用率を10-15%に維持する
- ファイルの中身を自分のコンテキストに読み込まず、ファイルパスだけを渡す
- 複雑なタスクは必ずサブエージェントに委任する
- 自分で実作業（コーディング、文書作成等）を行わない
```

ポイントは以下の3つだ。

**1. 役割の明確な定義**

「C-Suite Orchestrator」という名前と役割を最初に宣言することで、Claude Codeは自分の振る舞いの範囲を理解する。これがないと、「何でも自分でやろうとする」動作になりがちだ。

**2. Thin Orchestratorの原則の明示**

「自分で実作業をしない」「サブエージェントに委任する」というルールを明文化している。これにより、コンテキストウィンドウの効率的な利用が実現する。

**3. 参照先の一覧**

CLAUDE.mdの中でファイルの内容を書くのではなく、「どこを見ればよいか」のパスだけを示す。必要な時にClaude Codeが自分でファイルを読みに行く設計だ。

### サブエージェント定義の例

`.claude/agents/` ディレクトリに、各部門のエージェント定義を配置している。

```markdown
# コンテンツエンジンエージェント（content-engine.md 抜粋）

あなたは AI-CEO Framework のコンテンツ量産エンジンです。
あなたの目的は、コンテンツを通じてトラフィックと売上を生み出すことです。

## 専門知識
- SEOライティング: 検索意図分析、タイトル最適化、E-E-A-T対策
- コンバージョンコピーライティング: AIDA/PASフレームワーク
- プラットフォーム別最適化: Zenn、X、Instagram

## 品質基準
1. すべてのコンテンツにプロダクトへの導線があるか
2. タイトル、見出しがSEO最適化されているか
3. 数字と実例が含まれているか
```

各エージェントが「専門家」として定義されているため、Claude Codeがタスクを委任する際に、適切なコンテキストとルールが引き継がれる。

### この仕組みで実現していること

この構造により、以下のような運用が1人で可能になっている。

- 朝: `claude` を起動して「今日の状態を教えて」と聞くだけで、全プロダクト・全部門の状態が一覧化される
- 記事執筆: 「SEO記事を書いて」と指示するだけで、コンテンツエンジンエージェントがキーワード分析から執筆まで実行する
- コードレビュー: PRのURLを渡すだけで、開発リードエージェントがレビューを実行する
- 経営判断: 「来月の優先事項を提案して」と聞くと、各部門の状態とKPIを踏まえた提案が返ってくる


## この章のまとめ

Claude Codeを最大限に活用するための4つの柱を解説した。

1. **初期設定**: インストールは簡単だが、プランの選択は利用頻度に合わせて慎重に
2. **CLAUDE.md**: プロジェクトの「常識」をAIに伝える最重要ファイル。具体的で検証可能なルールを書く
3. **サブエージェント**: Thin Orchestratorパターンで複雑なタスクを効率的に分解・実行する
4. **Permission設定**: 安全に使うための権限管理。保守的な設定から始めて徐々に広げる

Claude Codeは、単なるコード補完ツールではない。プロジェクト全体を理解し、複数のファイルにまたがる複雑なタスクを自律的に実行できるAIエージェントだ。CLAUDE.mdとPermission設定を適切に行えば、信頼できる開発パートナーとして日々の開発を大幅に効率化してくれる。
