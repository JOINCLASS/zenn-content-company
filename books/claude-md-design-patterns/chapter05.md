# ワークフロー自動化

CLAUDE.mdにビルドやテストのコマンドを書くだけでなく、**ワークフロー全体**を定義することで、Claude Codeに一連の作業を任せることができます。

この章では、開発ワークフローの自動化パターンを紹介します。

## ワークフローとは

ワークフローは「こういう作業をするときは、この手順で進めてほしい」というClaude Codeへの指示です。

例えば、「新しいAPIエンドポイントを追加するとき」のワークフローは以下のようになります：

```markdown
## 新しいAPIエンドポイント追加の手順

1. `app/api/`にルートファイルを作成
2. リクエスト/レスポンスのZodスキーマを`schemas/`に作成
3. ビジネスロジックを`services/`に作成
4. ルートハンドラでスキーマ検証 → サービス呼び出し → レスポンス返却
5. テストを`__tests__/api/`に作成
6. `pnpm test`で全テストがパスすることを確認
7. `pnpm typecheck`で型エラーがないことを確認
```

この手順をCLAUDE.mdに書いておくと、「ユーザー一覧APIを作って」と依頼するだけで、Claude Codeがこの手順に従って実装してくれます。

## ビルド確認パターン

コードを書いた後にビルドが通ることを確認するワークフローです。

```markdown
## コード変更後のチェック手順

コードを変更したら、以下を順番に実行する:

1. `pnpm typecheck` — 型エラーがないこと
2. `pnpm lint` — リントエラーがないこと
3. `pnpm test` — 全テストがパスすること
4. `pnpm build` — ビルドが成功すること

エラーが出た場合は、修正してから再度チェックする。
全てパスするまでコミットしない。
```

## テスト駆動パターン

テストを先に書いてから実装するパターンです。

```markdown
## 新機能実装の手順（TDD）

1. まずテストファイルを作成し、期待する動作をテストケースとして書く
2. `pnpm test`でテストが失敗することを確認（Red）
3. テストをパスする最小限の実装を書く（Green）
4. `pnpm test`でテストがパスすることを確認
5. コードをリファクタリングする（Refactor）
6. `pnpm test`で再度パスすることを確認
```

## データベース変更パターン

データベーススキーマの変更は影響範囲が大きいため、慎重な手順が必要です。

```markdown
## データベーススキーマ変更の手順

1. `prisma/schema.prisma`を編集
2. `pnpm prisma migrate dev --name <変更内容>`でマイグレーション生成
3. 生成されたSQLファイルを確認（意図しない変更がないか）
4. 影響を受けるクエリ関数を`lib/db/`で更新
5. 影響を受けるAPIルートを更新
6. テストを更新・追加
7. `pnpm test`で全テストがパスすることを確認
```

## コンポーネント作成パターン

UIコンポーネントを追加する際のワークフローです。

```markdown
## 新しいUIコンポーネントの作成手順

1. `components/ui/`に新しいファイルを作成（PascalCase.tsx）
2. 同ファイル内でProps型を定義
3. コンポーネントを実装（Tailwind CSSでスタイリング）
4. `components/ui/index.ts`にexportを追加
5. Storybookストーリーを`stories/`に作成（使っている場合）
6. テストを`__tests__/components/`に作成
```

## 環境別デプロイパターン

ステージングと本番で異なるデプロイ手順がある場合のワークフローです。

```markdown
## デプロイ手順

### ステージング
1. `develop`ブランチにマージ
2. GitHub Actionsが自動でステージング環境にデプロイ
3. ステージングURLで動作確認

### 本番
IMPORTANT: 本番デプロイはCLAUDE.mdの指示では実行しない。
手動でGitHub上からリリースを作成する。
```

本番デプロイのような重大な操作は、CLAUDE.mdで「実行しない」と明示しておくことが重要です。

## 条件付きワークフロー

特定の条件で手順が分岐するパターンです。

```markdown
## バグ修正の手順

1. バグを再現するテストを書く
2. テストが失敗することを確認
3. 修正を実装する
4. テストがパスすることを確認
5. 既存テストが全てパスすることを確認

### セキュリティバグの場合
- 通常のPRフローに加え、修正内容にセキュリティに関する説明を含める
- 依存ライブラリの脆弱性の場合は`pnpm audit`の結果も記載する
```

## パフォーマンスチェックパターン

パフォーマンスに影響する変更をする際のワークフローです。

```markdown
## パフォーマンス影響のある変更

データベースクエリ、API呼び出し、レンダリングロジックを変更する場合:

1. 変更前の状態でベンチマークを取る（該当するテストの実行時間を記録）
2. 変更を実装する
3. 変更後のベンチマークを取る
4. 大幅な劣化（20%以上）がある場合は代替手段を検討する
```

## ワークフロー設計のコツ

### 番号付きリストを使う

手順の順序が重要な場合は、番号付きリストを使います。Claude Codeは番号順に実行してくれます。

### 検証ステップを入れる

各ワークフローに「確認」のステップを入れると、Claude Codeが途中で問題を検知して報告してくれます。

### 粒度を適切に

1ステップが大きすぎると、Claude Codeが途中で迷います。逆に細かすぎると冗長です。「1つのコマンドまたは1つの判断」を1ステップの目安にしてください。

## まとめ

- ワークフローは手順を番号付きリストで書く
- ビルド確認、TDD、DB変更、デプロイなどパターン別に整理
- 検証ステップを必ず含める
- 本番影響のある操作は「実行しない」と明示する
