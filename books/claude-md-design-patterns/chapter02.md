# 基本構文とセクション設計

CLAUDE.mdの書き方に厳密なフォーマットはありません。Markdownで書けば、Claude Codeは内容を理解してくれます。しかし、構造を整理しておくと、AIの理解精度が上がり、メンテナンスも楽になります。

この章では、実践的なセクション構成のパターンを紹介します。

## セクションの基本構造

効果的なCLAUDE.mdは、以下の5つのセクションで構成します。

```markdown
# プロジェクト概要
（何をするプロジェクトか、1-2文で）

## 技術スタック
（使用するフレームワーク・ライブラリ）

## コーディング規約
（書き方のルール）

## コマンド
（ビルド・テスト・デプロイの実行方法）

## 禁止事項
（やってはいけないこと）
```

順番に見ていきましょう。

## プロジェクト概要

最初のセクションは、プロジェクトが何をするものかをAIに伝えます。

```markdown
# ECサイト管理ダッシュボード

Next.js 15 + Supabaseで構築した、中小企業向けECサイトの管理画面。
商品管理、注文管理、顧客管理、売上分析の4機能を提供する。
```

ここで重要なのは、**ビジネスコンテキスト**を含めることです。「ECサイトの管理画面」と書くだけで、Claude Codeは「商品にはSKUがある」「注文にはステータスがある」といった業務知識を前提に動いてくれます。

## 技術スタック

使用するフレームワークとバージョンを明示します。

```markdown
## 技術スタック

- フレームワーク: Next.js 15 (App Router)
- 言語: TypeScript 5.x (strict mode)
- DB: Supabase (PostgreSQL)
- 認証: Supabase Auth
- スタイリング: Tailwind CSS v4
- テスト: Vitest + Testing Library
- パッケージマネージャ: pnpm
```

バージョン番号を書くことで、Claude Codeが古いAPIを使うミスを防げます。例えば「Next.js 15」と書けば、Pages Routerではなく、App Routerのコードを生成してくれます。

## コーディング規約

プロジェクト固有のコーディングルールを箇条書きで列挙します。

```markdown
## コーディング規約

- コンポーネントは関数コンポーネントで書く。classコンポーネントは使わない
- 状態管理はReact Server Componentsを基本とし、クライアント状態が必要な場合のみ`use`を使う
- APIルートは`app/api/`以下にRESTfulに配置する
- データベースアクセスは`lib/db/`にクエリ関数をまとめる。コンポーネントから直接SQLを書かない
- エラーハンドリングは`lib/errors.ts`のカスタムエラークラスを使う
- 日付はDay.jsではなく、標準のIntl APIとDate objectを使う
```

ポイントは **「なぜそうするか」よりも「何をするか」を明確に書く** ことです。AIは理由を知る必要はなく、ルールに従えればよいのです。

## コマンド

ビルド、テスト、デプロイのコマンドを記述します。

```markdown
## コマンド

- 開発サーバー: `pnpm dev`
- ビルド: `pnpm build`
- テスト全体: `pnpm test`
- テスト（単一ファイル）: `pnpm test -- path/to/file.test.ts`
- リント: `pnpm lint`
- 型チェック: `pnpm typecheck`
- DBマイグレーション: `pnpm supabase migration new <name>`
```

Claude Codeはこのセクションを参照して、テスト実行やビルド確認を自動で行います。パッケージマネージャがpnpmなのにnpmを使う、という間違いも防げます。

## 禁止事項

AIにやってほしくないことを明示します。このセクションは、予期しないトラブルを防ぐために重要です。

```markdown
## 禁止事項

- `main`ブランチに直接コミットしない。必ずfeatureブランチを切る
- `.env`ファイルをコミットに含めない
- `node_modules/`や`dist/`内のファイルを変更しない
- 既存のDBマイグレーションファイルを変更しない（新しいマイグレーションを作る）
- `package.json`の依存関係を勝手にアップグレードしない
```

## 書き方のコツ

### 箇条書きを使う

長い文章よりも、箇条書きの方がAIの理解精度が高くなります。

```markdown
# よくない例
このプロジェクトではTypeScriptを使っており、strictモードを有効にしています。
テストはVitestで書いていて、カバレッジは80%以上を目標にしています。
コミットメッセージは日本語で、Conventional Commitsの形式に従ってください。

# よい例
- TypeScript strict mode
- テスト: Vitest、カバレッジ80%以上
- コミットメッセージ: 日本語、Conventional Commits形式
```

### 具体例を添える

ルールだけでなく、具体例を示すとAIの出力品質が安定します。

```markdown
## ファイル命名規則

- コンポーネント: PascalCase（例: `UserProfile.tsx`）
- ユーティリティ: camelCase（例: `formatDate.ts`）
- 定数ファイル: SCREAMING_SNAKE_CASE（例: `API_ENDPOINTS.ts`）
- テスト: テスト対象と同名 + `.test`（例: `formatDate.test.ts`）
```

### IMPORTANTキーワード

特に重要なルールには`IMPORTANT`をつけると、Claude Codeがより確実に従います。

```markdown
IMPORTANT: 本番データベースに対してDELETE文を直接実行しない。必ず論理削除（deleted_atカラム更新）を使う。
```

## セクション設計のアンチパターン

### 情報が多すぎる

CLAUDE.mdに何百行も書くと、優先度が曖昧になります。本当に重要なルールに絞りましょう。目安は50-100行です。

### 矛盾するルール

「簡潔に書く」と「詳細なコメントを書く」のように矛盾する指示があると、AIの出力が不安定になります。ルールを追加するたびに、既存のルールと矛盾しないか確認しましょう。

### 古い情報

CLAUDE.mdが古くなると、AIが間違ったルールに従います。ライブラリのアップグレード、ディレクトリ構成の変更があったら、CLAUDE.mdも更新してください。

## まとめ

- 5つのセクション（概要・技術スタック・規約・コマンド・禁止事項）で構成する
- 箇条書きで簡潔に書く
- 具体例を添える
- 50-100行を目安にする
- 矛盾するルールを書かない
