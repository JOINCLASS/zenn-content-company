# マルチエージェント構成

Claude Codeには、メインのAIに加えて「サブエージェント」を起動する機能があります。複数のエージェントを連携させることで、大規模なタスクを並列に処理したり、専門分野ごとに役割を分けたりできます。

## サブエージェントとは

Claude Codeのサブエージェントは、メインのエージェントから起動される独立したAIです。メインエージェントのコンテキスト（会話履歴）を消費せずに、特定のタスクを実行して結果を返します。

```
メインエージェント
├── サブエージェント A（コード検索）
├── サブエージェント B（テスト実行）
└── サブエージェント C（ドキュメント生成）
```

サブエージェントが持つ情報は、起動時に渡されたプロンプトのみです。メインの会話履歴は自動的には共有されません。

## CLAUDE.mdでのサブエージェント設計

### Orchestratorパターン

1つのメインエージェントが、複数のサブエージェントに指示を出す構成です。

```markdown
## エージェント構成

### メインエージェント（Orchestrator）
- タスクの分解と委任を担当
- サブエージェントの結果を統合して最終回答を生成
- 自分ではコーディングしない

### サブエージェント: コードリサーチ
- コードベースの調査・検索を担当
- Grep, Glob, Readツールを使用
- 結果をサマリとして返す

### サブエージェント: 実装
- 実際のコーディングを担当
- Edit, Write, Bashツールを使用
- テスト実行まで含む

### サブエージェント: レビュー
- 実装結果のレビューを担当
- コーディング規約の遵守チェック
- セキュリティチェック
```

### 役割分担の指示

CLAUDE.mdに各エージェントの役割を明記しておくと、Claude Codeが適切にサブエージェントを使い分けてくれます。

```markdown
## タスク委任のルール

以下の場合にサブエージェントを起動する:

1. **コードベース調査**: 3つ以上のファイルを横断して調査が必要な場合
2. **並列実行**: 独立した複数のタスク（テスト実行 + リント + 型チェック）
3. **大量のコード生成**: 5ファイル以上の変更が必要な場合

以下の場合はサブエージェントを使わない:

1. **単一ファイルの修正**: メインで直接対応
2. **簡単な質問**: メインで直接回答
3. **依存関係のあるタスク**: 順番に処理が必要な場合
```

## 実践: プロジェクト横断タスク

大きな機能を実装する際に、サブエージェントで並列に作業するパターンです。

### 例: 認証機能の実装

```markdown
## 大規模機能の実装手順

1. まずサブエージェントで現在のコードベースを調査:
   - 既存の認証関連コードがあるか
   - 使用しているフレームワークの認証パターン
   - データベーススキーマの確認

2. 調査結果を元に実装計画を立てる

3. 独立した部分をサブエージェントで並列実装:
   - エージェントA: データベーススキーマとマイグレーション
   - エージェントB: 認証ミドルウェア
   - エージェントC: ログイン/ログアウトUI

4. 結果を統合し、結合テストを実行
```

## Thin Orchestratorの原則

マルチエージェント構成で重要なのは、メインエージェント（Orchestrator）のコンテキストを軽く保つことです。

```markdown
## Orchestratorのルール

- ファイルの中身を自分のコンテキストに読み込まない
- ファイルパスだけを把握し、内容の調査はサブエージェントに委任する
- コンテキスト使用率を低く維持する
- 自分で実作業（コーディング、文書作成等）を行わない
```

メインエージェントが全てのファイルを読み込むと、コンテキストウィンドウが溢れて、長い会話が続けられなくなります。Thin Orchestratorは「指揮者」に徹し、実作業は「演奏者」であるサブエージェントに任せます。

## 情報の受け渡し

サブエージェントにタスクを渡す際に、必要な情報を明確に指定します。

```markdown
## サブエージェントへの委任フォーマット

タスクをサブエージェントに委任する際は、以下の情報を渡す:

1. **目的** — 何を達成するか（1文で）
2. **参照ファイル** — 必要な入力ファイルのパス一覧
3. **出力先** — 結果をどこに出力するか
4. **制約** — 禁止事項、品質基準
```

## チーム構成パターン

プロジェクトの性質に応じて、エージェントの構成を変えます。

### Webアプリ開発チーム

```markdown
## エージェント構成

- Orchestrator: タスク管理、PR作成
- Frontend Agent: Reactコンポーネント、スタイリング
- Backend Agent: API設計、DB操作
- Test Agent: テスト作成、カバレッジ確認
```

### データ処理パイプライン

```markdown
## エージェント構成

- Orchestrator: パイプライン全体の管理
- ETL Agent: データ抽出・変換・ロード
- Validation Agent: データ品質チェック
- Report Agent: 結果レポート生成
```

### ドキュメント生成

```markdown
## エージェント構成

- Orchestrator: ドキュメント構成の管理
- Research Agent: コードベース調査、API仕様の抽出
- Writer Agent: 文書作成
- Review Agent: 校正、一貫性チェック
```

## 注意点

### コンテキストの独立性

サブエージェントはメインの会話履歴を知りません。必要な情報はプロンプトで明示的に渡す必要があります。

### コスト

サブエージェントを起動するたびにAPI呼び出しが発生します。小さなタスクにサブエージェントを使うと、かえって非効率です。

### 結果の統合

複数のサブエージェントが同じファイルを変更すると競合が起きます。ファイルの担当を明確に分けるか、順次実行にします。

## まとめ

- サブエージェントはメインのコンテキストを消費しない独立したAI
- Orchestratorパターンでメインは指揮に徹する
- 並列実行できるタスクを見極めてサブエージェントに委任
- 委任時は目的・参照ファイル・出力先・制約を明示する
- ファイル変更の競合に注意し、担当を分ける
