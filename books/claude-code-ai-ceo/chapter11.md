# 失敗と学び -- やってはいけなかった3つのこと

## 失敗を語る理由

成功事例は派手だ。「AIで経営を自動化した」「月5分で全社管理」。こうした話は読者の興味を引くが、実践の参考にはなりにくい。なぜなら、成功に至るまでの道筋は状況ごとに異なるが、失敗のパターンは驚くほど共通しているからだ。

この章では、AI-CEO Frameworkの構築過程で経験した3つの重大な失敗を、具体的な状況、原因、そして改善策とともに記録する。これからAIエージェント経営に取り組む読者が、同じ轍を踏まないために。

## 失敗1: 過度な自動化 -- 承認なしでメール送信してしまった事件

### 何が起きたか

AI-CEO Frameworkの構築初期、営業部門のエージェントにメール送信機能を持たせていた時期がある。リード獲得のために、潜在顧客リストに対して自動でアプローチメールを送る仕組みだ。

テスト段階では問題なかった。テスト用のメールアドレスに送信し、内容を確認し、「これなら大丈夫だ」と判断した。問題は、テストから本番に切り替えたときに起きた。

営業エージェントが、CEOの承認なしに5通のメールを実際のクライアント候補に送信してしまったのだ。

内容自体は問題なかった。丁寧な挨拶メールで、失礼な文面ではなかった。だが、問題は2つあった。

1. **タイミングが不適切だった**: 深夜2時にメールが送信されていた。AIはバッチ処理として深夜に実行されたが、受信者にとっては不審なタイミングである
2. **CEOが把握していなかった**: 翌朝のダッシュボードで初めて気づいた。送信済みのメールは取り消せない

幸い、受信者からのクレームはなく、実害はゼロだった。だが「AIが勝手にクライアントにメールを送った」という事実は、背筋が凍る経験だった。

### 根本原因

原因は明確だ。権限設定の不備である。

当時のpermissions.mdには、メール送信に関する明確なルールがなかった。「対外コミュニケーションはdraft」という原則は頭にあったが、それがファイルに明文化されておらず、営業エージェントの定義にも反映されていなかった。

AIは指示されたことを忠実に実行する。「リード獲得のためにメールを送れ」と指示して、「ただし送信前にCEOの承認を得ろ」と明記していなければ、AIは指示通りにメールを送る。AIに「常識で判断してくれ」と期待するのは、設計の怠慢だ。

### 改善策

この事件の翌日、以下の改善を行った。

**1. permissions.mdにメール送信権限を明記**

```yaml
email:
  internal_notification: execute    # 社内通知は自動送信可
  client_communication: draft       # クライアント連絡はCEO確認後
  marketing_email: draft            # マーケティングメールはCEO確認後
  system_notification: execute      # システム通知は自動（テンプレート限定）
```

**2. 営業エージェントからメール送信ツールを除外**

エージェント定義の `tools` セクションから、メール送信に関するツールを削除した。エージェントはメールの「下書き」を生成してファイルに保存するが、実際の送信はできない。送信はCEOが承認した後、別のフローで行う。

**3. 「常にdraft」カテゴリの新設**

対外的な影響のあるアクションを網羅的にリストアップし、「always_draft」カテゴリに分類した。

```yaml
always_draft:
  - press_release
  - pricing_change
  - claim_response
  - proposal
  - contract
  - invoice
  - client_email        # 追加
  - marketing_email     # 追加
  - sns_post            # 追加
```

この「always_draft」リストは、AI-CEO Frameworkの安全装置だ。ここに列挙されたアクションは、どんな状況でもCEOの承認なしには実行されない。

### 教訓

**「AIに何ができるか」ではなく「AIに何をさせないか」を先に設計せよ。**

自動化の興奮に駆られると、つい「あれもこれも自動化しよう」と考える。だが、自動化すべきでない領域を先に明確にすることのほうが重要だ。対外コミュニケーション、金銭処理、法的文書——これらは人間の判断を必ず介在させるべき領域であり、それを明文化することが最優先だ。

## 失敗2: コンテキスト爆発 -- CLAUDE.mdに全情報を詰め込んで機能不全

### 何が起きたか

AI-CEO Frameworkの初期バージョンでは、CLAUDE.md（メインのOrchestrator定義）に可能な限りの情報を詰め込んでいた。

- 6部門のエージェント定義
- 全コマンドの詳細仕様
- 各プロダクトの概要と現在の状態
- 権限設定の全文
- 過去の意思決定ログ
- ブランドガイドライン
- 技術スタックの詳細

ファイルサイズは約15,000トークンに膨張していた。

何が起きたか。Claude Codeの応答品質が著しく低下した。

具体的には以下の症状が出た。

1. **コマンドの誤解釈**: `/ai-ceo:morning` を実行したのに、マーケティング部門のタスクが実行された
2. **情報の混同**: ShareTokuのKPIを聞いたのに、FocalizeのKPIが返ってきた
3. **指示の無視**: 「サブエージェントに委任して」と明記しているのに、自分で作業を始めた
4. **応答の遅延**: 考える時間が長くなり、レスポンスに30秒以上かかるようになった

特に深刻だったのは3番目だ。Orchestratorが自分で作業を始めると、コンテキストがさらに膨張し、品質がさらに低下するという悪循環に陥った。

### 根本原因

LLM（大規模言語モデル）には「迷子になる」現象がある。コンテキストウィンドウに大量の情報が詰まると、AIは「どの情報が今のタスクに関連するか」の判断を誤り始める。人間でも、100ページの資料を渡されて「この中から必要な情報を探して」と言われたら、効率は落ちる。AIも同じだ。

さらに、CLAUDE.mdの情報が増えると、1回の対話あたりのトークン消費が増える。CLAUDE.mdはすべての対話で読み込まれるからだ。1回あたり15,000トークンの「固定コスト」が発生し、月間のトークン使用量が爆発した。

### 改善策

「Thin Orchestrator」パターンへの移行だ。3日かけて以下のリファクタリングを行った。

**1. CLAUDE.mdのスリム化**

CLAUDE.mdに残す情報を以下に絞った。

- コマンド一覧と実行ルール
- サブエージェントへの委任方法
- ファイルパスの一覧（中身は含めない）
- 権限制御の概要（詳細はpermissions.mdを参照）
- エラー時の振る舞い

結果、CLAUDE.mdのサイズは約4,000トークンまで縮小した。

**2. 情報の外部化**

CLAUDE.mdに含まれていた情報を、それぞれ適切なファイルに分離した。

```
CLAUDE.mdから除外 → 移動先
───────────────────────────────────
部門エージェント定義 → .claude/agents/ai-ceo-{dept}.md
権限設定の詳細 → .company/steering/permissions.md
ブランドガイドライン → .company/steering/brand.md
技術スタック → .company/steering/tech-stack.md
プロダクト状態 → .company/products/{name}/STATE.md
部門状態 → .company/departments/{dept}/STATE.md
意思決定ログ → .company/decisions/{month}.md
```

**3. 参照ルールの明文化**

CLAUDE.mdに「何を直接保持し、何をファイル参照にするか」のルールを明記した。

```markdown
## Thin Orchestratorの原則
- コンテキスト使用率を10-15%に維持する
- ファイルの中身を自分のコンテキストに読み込まず、ファイルパスだけを渡す
- 複雑なタスクは必ず .claude/agents/ のサブエージェントに委任する
- 自分で実作業（コーディング、文書作成等）を行わない
```

この改善後、応答品質は劇的に向上した。コマンドの誤解釈はなくなり、レスポンス速度も改善し、月間のトークン消費も40%以上削減できた。

### 教訓

**「AIに多く伝える」のではなく「AIが必要なときに必要な情報にアクセスできる構造」を設計せよ。**

これはソフトウェアアーキテクチャの原則と同じだ。モノリシックな巨大ファイルは保守性が低い。マイクロサービスのように、情報を適切な粒度で分割し、必要なときにだけ参照する設計が正解だ。

CLAUDE.mdは「インデックス」であるべきだ。百科事典そのものではなく、百科事典の目次であるべきだ。

## 失敗3: エージェント定義の曖昧さ -- 部門間の責任範囲が重複して矛盾した出力

### 何が起きたか

初期のエージェント定義では、各部門の責任範囲が曖昧だった。特にマーケティング部門（CMO）と営業部門（CSO）の境界が不明確だった。

マーケティング部門のエージェント定義には「リード獲得の施策立案と実行」と書いていた。営業部門のエージェント定義にも「リード獲得の戦略策定」と書いていた。

ある日、`/ai-ceo:mkt:campaign "Focalize リード獲得"` と `/ai-ceo:sales:proposal "Focalize"` を同日に実行したところ、以下の事態が発生した。

1. マーケ部門が「Instagram広告でリードを獲得し、LPに誘導する」戦略を提案
2. 営業部門が「LinkedInでDMアプローチし、無料トライアルを提案する」戦略を提案
3. 両者のリード定義が異なっていた（マーケ: LP訪問者、営業: 商談アポ獲得者）
4. 予算配分も重複していた（両方がInstagram広告予算を含めていた）

CEOとしては「全体最適なリード獲得戦略」がほしかったのに、部門ごとに矛盾する2つの戦略が出てきてしまった。

さらに困ったのは、この矛盾を解消するためにOrchestratorに調整を依頼したところ、Orchestratorも混乱して「両方の戦略を統合した折衷案」を出してきたことだ。折衷案は、どちらの戦略の強みも薄まった、中途半端なものだった。

### 根本原因

エージェント定義における「責任範囲」と「成果物」が曖昧だったことが原因だ。

組織設計において、部門間の責任分界点（Boundary）は明確でなければならない。人間の組織でも部門間の責任が曖昧だと「うちの仕事じゃない」「そっちがやると思っていた」という問題が起きる。AIエージェントの場合、さらに厄介だ。AIは遠慮しないので、両方の部門が同じ領域のタスクを堂々と実行してしまう。

### 改善策

**1. RACIマトリクスの導入**

各タスクについて、Responsible（実行）、Accountable（責任）、Consulted（相談）、Informed（通知）を明確にした。

| タスク | マーケ | 営業 | 開発 | 経理 |
|--------|--------|------|------|------|
| リード獲得戦略 | R/A | C | I | I |
| 広告運用 | R/A | I | I | C |
| アウトリーチ | C | R/A | I | I |
| LP改善 | C | I | R/A | I |
| 提案書作成 | I | R/A | C | I |
| 価格設定 | C | C | I | R/A |

これにより、「リード獲得」はマーケ部門がResponsible、営業部門はConsulted（意見を聞かれるが実行はしない）と明確になった。

**2. エージェント定義にRACIを明記**

各エージェントの定義ファイルに、以下のセクションを追加した。

```markdown
## 責任範囲（RACI）

### Responsible（あなたが実行する）
- コンテンツマーケティング戦略の立案と実行
- SEO記事の企画・執筆・公開
- 広告運用と最適化
- リード獲得施策の設計

### Consulted（あなたに相談が来る）
- 営業資料のブランディング確認
- プロダクトのメッセージング

### NOT your responsibility（あなたの範囲外）
- 個別クライアントへのアプローチ（→営業部門）
- 提案書の作成（→営業部門）
- 価格設定の最終決定（→経理部門）
```

特に重要なのは「NOT your responsibility」セクションだ。何をやるかだけでなく、何をやらないかを明示することで、部門間の越境を防ぐ。

**3. 部門間調整のプロトコル**

部門横断タスク（例: 新プロダクトのリリース）では、Orchestratorが「Wave実行計画」を生成し、各部門の担当範囲を事前に定義するルールを設けた。

```markdown
## Wave実行計画: Focalize リード獲得キャンペーン

### Wave 1: マーケ部門
- LP改善のコピー作成
- Instagram広告のクリエイティブ作成
- 成果物: 広告素材、LPコピー（draft）

### Wave 2: 開発部門
- LPコピーの実装・デプロイ
- 成果物: デプロイ済みLP

### Wave 3: 営業部門
- LP経由のリードに対するフォローアップメール作成
- 成果物: フォローアップテンプレート（draft）
```

このWave計画により、各部門が何をいつやるかが明確になり、重複や矛盾が発生しなくなった。

### 教訓

**AIエージェントの部門設計は、人間の組織設計と同じ厳密さが必要だ。**

「AIだから曖昧でも大丈夫」ということはない。むしろ、AIは曖昧な指示を「自分なりに解釈して」実行するため、人間以上に明確な責任分界点が必要だ。

RACIマトリクスを作成し、各エージェントの「NOT your responsibility」を明示し、部門横断タスクにはWave計画を用いる。この3つのルールで、部門間の矛盾はほぼ解消された。

## 失敗を防ぐためのチェックリスト

上記3つの失敗から得た教訓を、チェックリストとしてまとめる。新しいエージェントを定義するとき、新しい自動化フローを追加するとき、このリストを確認してほしい。

### 権限チェック

- [ ] このエージェントが実行できる「最も危険なアクション」は何か
- [ ] そのアクションにdraft/承認フローが設定されているか
- [ ] permissions.mdに該当する権限ルールが明記されているか
- [ ] エージェントのtoolsに不要な権限が含まれていないか（最小権限の原則）
- [ ] 対外コミュニケーション（メール、SNS、請求書等）はすべてalways_draftになっているか

### コンテキストチェック

- [ ] CLAUDE.mdのサイズが肥大化していないか（目安: 5,000トークン以下）
- [ ] ファイルの中身ではなくパスを参照しているか
- [ ] サブエージェントに委任すべきタスクを自分でやっていないか
- [ ] STATE.mdの完了タスクをアーカイブしているか（月次）

### 責任範囲チェック

- [ ] 新しいエージェントの責任範囲が既存エージェントと重複していないか
- [ ] RACIマトリクスが更新されているか
- [ ] 「NOT your responsibility」が明記されているか
- [ ] 部門横断タスクにWave計画が用意されているか

## 「完璧を目指さない」というマインドセット

この章で紹介した3つの失敗はいずれも、AI-CEO Frameworkを「完璧な自動化システム」にしようとする欲求から生じたものだ。

- 完璧な自動化を目指して → メール送信まで自動化してしまった
- 完璧な情報共有を目指して → CLAUDE.mdに全情報を詰め込んだ
- 完璧な組織設計を目指して → 部門の守備範囲を広くしすぎた

ここで重要なのは、「完璧を目指さない」ことだ。

AI-CEO Frameworkは「完璧な自動化システム」ではない。「CEOの意思決定を支援するシステム」だ。最終的な判断は常に人間（CEO）が行う。AIは情報収集、整理、提案を行うが、実行の最終決定は人間が持つ。

この原則を守る限り、失敗のダメージは限定的だ。メール事件も、CEOが送信前に確認していれば防げた。CLAUDE.mdの肥大化も、定期的にレビューしていれば早期に気づけた。

### 段階的に自動化する

自動化は段階的に進めるべきだ。以下の3ステップを推奨する。

**ステップ1: 観察モード（1-2週間）**

新しいエージェントは、最初はread-only権限のみで稼働させる。レポート生成、状態確認、分析だけを行い、実行はすべて人間が行う。この期間で、エージェントの出力品質と安定性を確認する。

**ステップ2: ドラフトモード（2-4週間）**

品質に問題がなければ、draft権限を追加する。エージェントが成果物を生成し、承認キューに登録する。CEOが確認してから実行する。この期間で、承認フローが適切に機能しているか確認する。

**ステップ3: 自動実行モード（条件付き）**

承認していたタスクの中で、「毎回承認している」「判断に迷うことがない」ものは、auto_executeに昇格させる。ただし、対外コミュニケーションと金銭処理は永久にdraftのままにする。

### 失敗は「設計のフィードバック」だ

失敗したとき、最初の反応は「AIが間違えた」ではなく「設計に不備があった」であるべきだ。AIは指示通りに動く。予期しない動作をしたなら、それは指示（エージェント定義、権限設定、CLAUDE.md）に問題がある。

失敗のたびにpermissions.mdを改善し、エージェント定義を精緻化し、CLAUDE.mdをスリム化する。失敗は、システムをより堅牢にするための情報源だ。

この姿勢を持ち続ける限り、AI-CEO Frameworkは失敗のたびに進化する。完璧なシステムは最初から存在しない。運用しながら育てるシステムだけが、実際に機能するシステムになる。

## まとめ

3つの失敗と、そこから得た教訓を整理する。

| 失敗 | 根本原因 | 教訓 |
|------|---------|------|
| メール自動送信 | 権限設定の不備 | 「何をさせないか」を先に設計する |
| コンテキスト爆発 | CLAUDE.mdの肥大化 | Thin Orchestrator、情報の外部化 |
| 部門間の矛盾 | 責任範囲の曖昧さ | RACI、NOT your responsibility の明記 |

失敗は避けられない。だが、失敗のダメージを最小限に抑え、学びを最大化する仕組みは作れる。承認フロー、権限設定、チェックリスト——これらの安全装置が、AIエージェント経営を持続可能なものにする。
