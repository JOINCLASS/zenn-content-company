# コスト管理 -- 月額$100-200でどこまでできるか

## AIコストは「人件費」と比較すべきだ

AI-CEO Frameworkの運用で最も聞かれる質問が「月いくらかかるのか」だ。

答えは明快だ。月額$100から$200（約15,000円から30,000円）。これで6部門のAIエージェントが稼働し、6つのプロダクトの並行管理が可能になる。

この金額が高いか安いかは、何と比較するかで変わる。ChatGPT Plusの$20と比較すれば高い。だが、正社員一人の人件費（月額40万円以上）と比較すれば破格だ。パートタイムのアシスタントを雇うにしても月額10万円は必要だろう。

しかし単純な金額比較だけでは不十分だ。この章では、Claude Code Maxプランの費用構造、トークン使用量の実態、そして最も重要なROI（投資対効果）を、実データを元に解説する。

## Claude Code Maxプランの費用構造

### プラン概要

2026年2月時点のClaude Code Maxプランは以下の構成だ。

| プラン | 月額 | 特徴 |
|--------|------|------|
| Claude Max 5x | $100/月 | 通常の5倍の使用量 |
| Claude Max 20x | $200/月 | 通常の20倍の使用量 |

AI-CEO Frameworkの運用には、最低でもMax 5x（$100）が必要だ。通常のProプラン（$20）では、1日の使用量上限にすぐ到達してしまい、朝のダッシュボード生成すらままならない。

### $100プランと$200プランの使い分け

結論から言うと、通常運用は$100プランで十分だ。$200プランが必要になるのは以下のケースである。

1. **書籍の集中執筆期間**: 1日で3-4チャプターを書く場合
2. **新プロダクトの立ち上げ**: 初期設計からプロトタイプまで一気に進める場合
3. **大規模リファクタリング**: 既存コードベース全体を対象にした改善

筆者の運用では、月の3/4は$100プラン、集中開発期間だけ$200プランに切り替えている。ただし、プランの切り替えには注意が必要だ。月の途中でアップグレードすると差額のみ請求されるが、ダウングレードは翌月の更新タイミングまで反映されない場合がある。プランの変更タイミングは事前に確認しておくべきだ。

## トークン使用量の分析

### 1ヶ月の実際のトークン使用量

ここで、2026年2月の実際のトークン使用量を共有する。

| 用途 | 頻度 | 1回あたりの推定トークン数 | 月間合計 |
|------|------|--------------------------|---------|
| morning digest | 毎日 | 15,000-25,000 | 450,000-750,000 |
| コードレビュー・修正 | 週3-5回 | 30,000-50,000 | 360,000-600,000 |
| 記事執筆 | 週2回 | 40,000-60,000 | 320,000-480,000 |
| 承認操作 | 週5-10回 | 3,000-5,000 | 60,000-200,000 |
| 戦略検討（ask） | 週1-2回 | 20,000-40,000 | 80,000-320,000 |
| サブエージェント起動 | 随時 | 10,000-30,000 | 200,000-600,000 |
| **合計** | | | **約1,500,000-2,950,000** |

Max 5xプラン（$100）の実質的な月間上限を超えそうな月は、以下の戦略で対処する。

### トークン最適化の7つの手法

**手法1: Thin Orchestratorパターン**

これはAI-CEO Frameworkの設計原則であり、最大のコスト削減手法でもある。

メインのCLAUDE.md（Orchestrator）は、ファイルの中身を読み込まない。ファイルパスだけを管理し、実際のファイル読み取りはサブエージェントに委任する。

なぜこれがコスト削減になるのか。Claude Codeのトークン消費で最も高コストなのは「コンテキストウィンドウの膨張」だ。CLAUDE.mdに全ファイルの内容を含めると、1回の対話あたりのトークン数が倍増する。それが毎回の対話で課金される。

```
# 悪い例（Fat Orchestrator）
CLAUDE.mdに全部門のSTATE.md内容を直接記述
→ 1回の対話: 50,000-80,000トークン

# 良い例（Thin Orchestrator）
CLAUDE.mdにはファイルパスのみ記述
→ 1回の対話: 8,000-15,000トークン
```

この差が毎日20-30回の対話に掛け算されると、月間で数百万トークンの差になる。

**手法2: サブエージェントの活用**

サブエージェントは独自のコンテキストウィンドウを持つ。タスクが完了するとコンテキストが破棄される。つまり、長時間の対話で蓄積されるコンテキストのコストがリセットされる。

メインセッションで10ファイルを読み込んで作業するより、サブエージェントを起動して10ファイルの作業を委任し、結果だけを受け取るほうが、トータルのトークン消費は少なくなる。

```
# メインセッションで直接作業
対話1: 10,000トークン
対話2: 15,000トークン（前回の会話が含まれる）
対話3: 20,000トークン（さらに蓄積）
対話4: 25,000トークン
合計: 70,000トークン

# サブエージェントに委任
メインセッション: 5,000トークン（指示のみ）
サブエージェント: 30,000トークン（独立したコンテキスト）
合計: 35,000トークン
```

**手法3: STATE.mdのサイズ管理**

各部門のSTATE.mdは定期的に「アーカイブ」する。完了したタスクは月次で別ファイルに移動し、STATE.mdには進行中のタスクだけを残す。

これにより、morningエージェントが読み取るファイルのサイズが一定に保たれる。放置しておくとSTATE.mdが肥大化し、ダッシュボード生成のトークン消費が徐々に増えていく。

**手法4: プロンプトの簡潔化**

AIへの指示は、長ければ良いというものではない。以下は同じ結果を得る2つの指示の比較だ。

```
# 冗長な指示（200トークン）
「現在のFocalizeプロダクトの状態を確認して、特に登録ユーザー数と有料プラン
契約数のKPIを見て、目標に対してどの程度達成しているかを分析してほしい。
もし目標未達であれば、改善策を3つ提案してほしい。」

# 簡潔な指示（50トークン）
「Focalize KPIレビューと改善策3つ。STATE.md参照。」
```

1回の差は小さいが、毎日数十回の対話を重ねると無視できない差になる。

**手法5: 不要なファイル読み込みの回避**

サブエージェントに渡すファイルパスは、本当に必要なものだけに絞る。「念のため」で全ファイルを渡すと、トークンの無駄遣いになる。morningエージェントにはReadとGrepだけを与え、Writeを与えていないのも、不要な操作（=トークン消費）を防ぐ設計だ。

**手法6: 定型タスクのテンプレート化**

繰り返し行うタスクは、プロンプトをテンプレート化してエージェント定義に含める。毎回ゼロから指示を書くより、テンプレートを呼び出すほうが指示部分のトークンが少なくなる。

**手法7: 対話の分割**

1つの長い対話セッションより、短い対話を複数回行うほうが効率的な場合がある。Claude Codeのコンテキストウィンドウには上限があり、上限に近づくとパフォーマンスが低下する。早めにセッションを切り替えることで、品質を維持しながらトークン効率も上げられる。

## Thin Orchestratorパターンの詳細

この書籍で何度も登場する「Thin Orchestrator」パターンについて、コスト管理の観点から詳しく解説する。

### コンテキスト使用率の目標

CLAUDE.md（メインのOrchestrator）のコンテキスト使用率を10-15%に維持する。これは設計上の目標として明記している。

```
## Thin Orchestratorの原則
- コンテキスト使用率を10-15%に維持する
- ファイルの中身を自分のコンテキストに読み込まず、ファイルパスだけを渡す
- 複雑なタスクは必ず .claude/agents/ のサブエージェントに委任する
- 自分で実作業（コーディング、文書作成等）を行わない
```

「10-15%」という数字は経験則から来ている。これより多いとコンテキストウィンドウの余裕がなくなり、AIの応答品質が下がる。これより少ないと、Orchestratorが十分な判断材料を持てず、サブエージェントへの指示が曖昧になる。

### 情報の3層構造

Thin Orchestratorを機能させるために、情報を3層に分けている。

```
第1層: Orchestratorが保持する情報（10-15%）
  - コマンド一覧と実行ルール
  - 部門構成とファイルパス
  - 権限制御ルール
  - エラー時の振る舞い

第2層: サブエージェントが参照する情報（必要に応じて読み込み）
  - 各部門のSTATE.md
  - 各プロダクトのSTATE.md
  - 承認キュー
  - KPIデータ

第3層: アーカイブ情報（通常は参照しない）
  - 過去のdecisionsログ
  - 完了済みタスクの履歴
  - 旧バージョンの設計文書
```

この3層構造により、日常的な操作ではトークン消費を最小限に抑えつつ、必要なときには深い情報にもアクセスできる。

## 他のAIツールとの併用コスト

Claude Code Maxだけで完結するわけではない。実際の開発・経営では複数のAIツールを使い分けている。

### 現在の月間AIコスト内訳

| ツール | 月額 | 用途 |
|--------|------|------|
| Claude Code Max 5x | $100 | AI-CEO Framework、コード生成、記事執筆 |
| GitHub Copilot | $10 | エディタ内のコード補完 |
| **合計** | **$110** | |

Claude Code Maxがメインのツールであり、コストの大部分を占める。GitHub Copilotは細かなコード補完に使う。Claude Codeでファイル全体の設計や生成を行い、Copilotでエディタ内の細かい修正を行う、という使い分けだ。

### ツール選定の基準

AI関連のツールは次々と登場するが、導入の基準は明確にしている。

1. **既存ワークフローに統合できるか**: 新しいUIを覚える学習コストがかかるツールは避ける
2. **Claude Code Maxで代替できないか**: 専用ツールの機能が、Claude Codeのプロンプトで再現できるなら導入しない
3. **月額コストの総額が$200を超えないか**: AI運用予算の上限は$200/月と決めている

この基準により、「便利そうだから入れる」という衝動を抑え、本当に必要なツールだけを使っている。

## ROI計算: AI投資$200 vs 人件費換算

ここが最も重要な部分だ。月額$200（約30,000円）のAI投資が、どれだけの人的作業を代替しているか。

### AI-CEO Frameworkが代替している作業の時間換算

| 作業 | 人間がやった場合 | AIでの所要時間 | 削減時間/月 |
|------|----------------|---------------|------------|
| 朝のダッシュボード生成 | 30分/日 | 5分/日 | 約12.5時間 |
| 記事執筆（月4本） | 4時間/本 | 1時間/本 | 約12時間 |
| コードレビュー・修正 | 2時間/回 x 週4 | 30分/回 x 週4 | 約24時間 |
| LP・広告コピーの作成 | 3時間/回 | 45分/回 | 約2.25時間 |
| 経理帳簿の整理 | 4時間/月 | 1時間/月 | 約3時間 |
| 提案書・営業資料作成 | 3時間/件 | 45分/件 | 約2.25時間 |
| **合計削減時間** | | | **約56時間/月** |

月56時間。これを時給に換算する。

フリーランスエンジニアの一般的な時給を5,000円とすると、56時間 x 5,000円 = 280,000円相当の作業が、月額30,000円のAI投資で代替されている。

**ROI = (280,000 - 30,000) / 30,000 = 833%**

もちろんこの計算は概算であり、AIの出力品質と人間の作業品質は完全に同一ではない。AIの出力をチェック・修正する時間も含まれていない。それでも、投資対効果が極めて高いことは明らかだ。

### 「見えないROI」も忘れてはならない

上記の計算に含まれていない効果がある。

**1. 意思決定の速度向上**

ダッシュボードにより、問題の発見から対処までのリードタイムが短縮された。以前は週次のレビューで初めて気づいていた問題に、毎朝気づけるようになった。この「早期発見」の価値は、金額換算しにくいが極めて大きい。

**2. 並行タスクの管理能力向上**

人間1人では3-4個のプロジェクトを並行管理するのが限界だ。AI-CEO Frameworkにより、6つのプロダクト + 受託開発を同時に管理できている。これは「やらなかった場合に失われていた売上」として計算すべきだが、正確な算出は難しい。

**3. 精神的負荷の軽減**

「あれもやらなきゃ、これも忘れてるかも」という焦燥感が大幅に減った。AIが全部門を網羅的にチェックしてくれるという安心感は、経営者のメンタルヘルスに直結する。これもROI計算には含まれないが、持続的な経営に不可欠な要素だ。

## コスト管理のベストプラクティス

最後に、AI運用コストを管理するためのベストプラクティスをまとめる。

### 月次コストレビューの実施

毎月末に以下を確認する。

1. 今月のAIツール合計コスト
2. 主要な使用パターン（何にトークンを多く使ったか）
3. 来月の予定（集中開発期間があるか）
4. プランのアップグレード/ダウングレードが必要か

これを `.company/departments/finance/STATE.md` に記録し、morningダッシュボードから参照できるようにしている。

### コスト警告の仕組み

permissions.mdに月間AI運用予算の上限を設定している。

```yaml
monthly_ai_budget: 200  # USD
```

予算の80%に到達した場合、CEOに通知するルールが設定されている。これにより、月末に「想定以上にコストがかかっていた」という事態を防ぐ。

### スケーリングの判断基準

事業が成長し、AI運用の需要が増えたとき、どのタイミングでプランをアップグレードするか。判断基準は以下の通りだ。

1. **$100プランで月3回以上の使用量制限に到達** → $200プランを検討
2. **$200プランでも制限に到達** → 作業の効率化（プロンプト最適化、不要タスクの削減）を先に行う
3. **効率化しても足りない** → AI投資によるROIが維持できるか再計算

重要なのは、「コストが増えたから減らす」のではなく、「ROIが維持できるか」で判断することだ。月額$500かかっても、それで月100時間の作業が代替され、売上に貢献しているなら安い投資だ。

## まとめ

AI-CEO Frameworkの運用コストは月額$100-200だ。この投資で月56時間以上の作業を代替し、ROIは800%を超える。

コスト管理の要点は3つある。

1. **Thin Orchestratorパターンでトークン消費を最小化する** — コンテキスト使用率10-15%を維持する
2. **サブエージェントを活用してコンテキストの蓄積を防ぐ** — タスクごとにコンテキストをリセットする
3. **ROIで判断する** — 絶対額ではなく、投資対効果でプランを選択する

ひとり社長が月3万円で「6部門のAIチーム」を持てる。この事実の意味は大きい。従来なら数百万円の人件費が必要だった経営機能が、個人でも手の届くコストで実現できる時代が来ている。
