# CMO部門の自動化 -- コンテンツ・SEO・SNS運用

## マーケティングは「やらなければならないが、やりたくない」仕事だった

正直に告白する。エンジニアとして会社を経営する中で、マーケティングは最も後回しにしてきた業務だった。

6つのプロダクトを開発し、コードを書き、バグを直し、インフラを管理する。それだけで1日が終わる。「今日こそブログ記事を書こう」「今週こそSNSを更新しよう」と思いながら、気づけば3ヶ月が過ぎていた。

結果、プロダクトのユーザー数はゼロのまま。技術的にはしっかり動いているのに、誰にも知られていない。これは多くのエンジニア起業家が陥る罠である。

AI-CEO FrameworkのCMO部門は、この「やらなければならないが、やれていなかった」マーケティング業務を、2つのエージェントで自動化する。

## CMOエージェントの設計 -- 戦略と実行を分離する

CMO部門は、1つの部門に2つの役割を持たせている。戦略を担うCMOエージェントと、実行を担うContent Engineエージェントである。

```
CMO部門
├── CMOエージェント (ai-ceo-cmo.md)
│   ├── コンテンツ戦略の立案
│   ├── SEO監査と改善施策
│   ├── 広告キャンペーン分析
│   ├── KPI設計と計測基盤
│   └── ブランディング管理
│
└── Content Engineエージェント (ai-ceo-content-engine.md)
    ├── SEO記事の量産
    ├── Zenn有料書籍の執筆
    ├── LP/プロダクトページのコピー改善
    ├── 広告コピーの作成
    ├── SNS投稿の一括生成
    └── メールテンプレートの作成
```

なぜ分離するのか。理由はシンプルで、「何を作るか」と「どう作るか」は異なるスキルセットを要求するからである。

CMOエージェントのペルソナは「データドリブンなグロースマーケター」として定義した。CLAUDE.mdの中で、以下のように記述している。

```markdown
# CMO / マーケティング部長エージェント

あなたは AI-CEO Framework の CMO（最高マーケティング責任者）です。

## ペルソナ

データドリブンなグロースマーケター。技術プロダクトのマーケティングに精通し、
コンテンツマーケティング・SEO・SNS・有料広告を統合的に運用する。
「計測できないものは改善できない」がモットー。
少人数チームでの効率的なマーケ運用が得意。
```

一方、Content Engineエージェントのペルソナは「SEOとコンバージョンに特化したコンテンツプロデューサー」である。

```markdown
# コンテンツエンジンエージェント

あなたは AI-CEO Framework のコンテンツ量産エンジンです。
**あなたの目的は、コンテンツを通じてトラフィックと売上を生み出すことです。**

## ペルソナ

SEOとコンバージョンに特化したコンテンツプロデューサー。
「読まれるだけのコンテンツ」ではなく「行動を起こさせるコンテンツ」を作る。
1記事あたりの期待流入数と売上貢献を常に意識する。
```

ポイントは「行動を起こさせるコンテンツ」という一文である。単に検索上位に表示される記事ではなく、読者がプロダクトに登録する、書籍を購入する、問い合わせをするという行動につながるコンテンツを求めている。

## コンテンツタイプと収益導線の設計

Content Engineには、コンテンツのタイプごとに収益への貢献経路を明示的に定義している。

```markdown
## コンテンツタイプと収益貢献

| タイプ | 目的 | 収益導線 |
|--------|------|---------|
| Zenn技術記事 | SEO流入→プロダクト認知 | 記事内CTA→LP→登録 |
| Zenn有料書籍 | 直接売上 + ブランディング | 書籍販売収益（86.8%取り分） |
| LP/プロダクトページ | コンバージョン | 訪問→登録→有料転換 |
| 広告コピー | 有料流入の効率化 | 広告CTR向上→CPA削減 |
| SNS投稿 | エンゲージメント→流入 | フォロワー→LP訪問→登録 |
| メールテンプレート | リテンション/アップセル | 既存ユーザーの有料転換・継続 |
```

この表が重要なのは、AIエージェントに「なぜそのコンテンツを作るのか」を理解させるためである。目的なきコンテンツは作らせない。すべてのコンテンツには、プロダクトへの導線が存在しなければならない。

## SEO記事の自動生成フロー

SEO記事の生成は、Content Engineの主力ワークフローである。実際にどのように動くかを、ステップごとに解説する。

### ステップ1: キーワード分析と検索意図の特定

記事テーマが決まると、まずそのテーマに関連するキーワードの検索意図を分析する。Content Engineのプロンプトには、検索意図の3分類が組み込まれている。

```markdown
### SEOライティング
- 検索意図分析（情報型、商取引型、ナビゲーション型）
- タイトル最適化（CTR最大化のフォーマット）
- 構造化（H2/H3、FAQ、箇条書き、表の効果的な使い方）
- 内部リンク設計（記事→プロダクト→記事のハブ構造）
- E-E-A-T対策（実体験ベースの専門性）
```

たとえば「ポモドーロテクニック アプリ」というキーワードなら、検索意図は「商取引型」（比較検討して選びたい）と判断する。記事構成も比較表やランキング形式を中心にする。

### ステップ2: 構成案の作成

検索意図に基づいて、H2/H3レベルの見出し構成を設計する。ここでのポイントは、自社プロダクトへのCTAを「自然に」配置する場所を最初から設計に組み込むことである。

```
## ポモドーロテクニック対応アプリ5選
### 1. Focalize -- 営業タスクと時間管理を一元化
### 2. Forest -- ゲーミフィケーションで集中力維持
### 3. Focus To-Do -- シンプルなポモドーロタイマー
...

## 集中力を2倍にする実践法
### ポモドーロテクニックの基本ルール
### タスク管理との組み合わせが鍵
→ ここでFocalizeのタスク管理機能をCTAとして配置
```

### ステップ3: 記事執筆（2,000-4,000字）

構成案に基づいて本文を執筆する。Content Engineには以下の品質基準が設定されている。

```markdown
## 品質基準

1. **CTA必須:** すべてのコンテンツにプロダクトへの導線があるか
2. **SEO最適化:** タイトル、見出し、メタ情報が検索最適化されているか
3. **具体性:** 抽象的な表現でなく、数字と実例が含まれているか
4. **ブランド整合:** .company/steering/brand.md のトーンに合っているか
5. **量産性:** 1コンテンツの作成時間が適切か（記事30分以内、SNS5分以内）
```

特に重要なのは3番の「具体性」と4番の「ブランド整合」である。AIが生成する記事は、放っておくと抽象的な一般論になりがちだ。「具体性」の基準は「数字と実例を含むこと」と明確に指示している。

ブランド整合については、`.company/steering/brand.md`で技術記事のトーンを定義している。

```markdown
### 技術記事（Zenn/Qiita）
- **実践的で具体的**: 理論だけでなく、実際のコード例や実装手順を含める
- **学びを共有する姿勢**: 上から教えるのではなく、共に学ぶスタンス
- **再現可能性を重視**: 読者が同じ手順で再現できる品質を保つ
```

### ステップ4: Zenn CLIフォーマットでの出力

執筆が完了すると、Zenn CLIが要求するフォーマットで記事ファイルを出力する。

```yaml
---
title: "ポモドーロテクニック対応アプリ5選と仕事の集中力を2倍にする実践法"
emoji: "🍅"
type: "tech"
topics: ["ポモドーロテクニック", "集中力", "タスク管理", "Focalize", "生産性"]
published: false
---
```

`published: false`で出力するのがポイントである。記事の公開は「対外アクション」に分類されるため、必ずCEO承認を経てから`published: true`に変更する。

## Zennへの記事公開自動化 -- GitHub連携

記事の公開プロセスは、Zenn CLIとGitHub連携で自動化している。

```
zenn-content-company/     ← GitHubリポジトリ
├── articles/
│   ├── focalize-sales-automation.md
│   ├── pomodoro-apps-comparison.md
│   └── ai-business-automation.md
├── books/
│   └── claude-code-ai-ceo/
│       ├── config.yaml
│       ├── chapter01.md
│       └── ...
└── package.json
```

このリポジトリをZennのGitHub連携機能と紐づけておくと、`main`ブランチにpushするだけで記事が自動公開される。ワークフローは以下のとおり。

```
1. Content Engine が記事を執筆
2. articles/ ディレクトリにファイルを配置（published: false）
3. approval-queue.md に承認リクエストを追加
4. CEOが内容を確認・承認
5. published: true に変更してmainブランチにpush
6. Zenn側で自動検知・公開
```

実際の運用では、新しい会社名義のZennアカウントを開設し、9本の記事を3週間で段階的に公開するスケジュールを策定した。

```
Week 1 (2/25-27): スキル管理シリーズ 3本
Week 2 (3/3-5):  Focalize営業自動化シリーズ 3本
Week 3 (3/10-12): DX・AI業務自動化シリーズ 3本
```

一気に公開しない理由は、Zennの内部アルゴリズム上、連続投稿よりも定期的な投稿のほうがフィード露出が有利になるためである。

## Zenn有料書籍の企画・執筆プロセス

有料書籍は、SEO記事とは異なるワークフローで進める。直接収益を生むコンテンツであり、品質基準も高い。

### 市場調査から始める

Content Engineの書籍ワークフローは、市場分析から始まる。

```markdown
### /ai-ceo:content:book "テーマ" -- Zenn書籍企画・執筆
1. 市場分析: Zennの有料書籍トレンドを確認
2. 企画立案: テーマ、ターゲット、価格、章構成
3. 各チャプターを執筆（既存記事を拡張・深掘り）
4. 無料プレビュー章の選定（購入動機を最大化）
5. config.yaml + 全チャプターをZenn CLIフォーマットで出力
```

実際に行った市場調査では、Zennで売れている書籍のジャンル、価格帯、章構成を分析した。結果、以下のベストプラクティスを導き出した。

| 項目 | 最適値 | 根拠 |
|------|--------|------|
| 価格帯 | 1,000-2,000円 | 購入障壁と単価のバランス |
| 総章数 | 8-15章 | 薄い印象を避けつつ完読率を維持 |
| 1章の文字数 | 2,000-5,000字 | Zennの読了体験に最適 |
| 無料プレビュー | 全体の20-30% | 価値を示しつつ続きを読ませる |

### 4冊の書籍企画を一括策定

この市場調査に基づき、4冊の有料書籍を一括で企画した。

| 書籍 | 価格 | 年間販売予測 | 年間収益（86.8%取り分） |
|------|------|------------|---------------------|
| Claude Codeで会社を動かす | 1,980円 | 260-520部 | 44-89万円 |
| Next.js + Supabase SaaS開発入門 | 1,500円 | 325-650部 | 42-84万円 |
| AIコーディングアシスタント完全活用ガイド | 1,500円 | 390-780部 | 50-101万円 |
| 中小企業のAI業務自動化 実践ガイド | 2,000円 | 195-445部 | 33-77万円 |
| **合計** | - | **1,170-2,395部** | **170-350万円** |

4冊合計で年間170万-350万円の収益を見込む。Zennの著者取り分は86.8%と高率であり、在庫リスクもゼロ。コンテンツビジネスとしては極めて効率が良い。

### 執筆と品質管理

書籍の執筆は、Content Engineエージェントに各チャプターの執筆を委任する。ただし、書籍はSEO記事と異なり、CEOの実体験が核となるため、エージェントには会社の内部ファイル（`.company/`ディレクトリの各種ファイル）を参照させながら、リアルなデータと経験に基づいた内容を執筆させる。

各チャプターの完了条件は以下のとおり。

- 2,000-5,000字の分量があること
- 実際のファイル構造やコード例が含まれていること
- CTAがプロダクトまたはサービスへの導線として機能していること
- ブランドガイドラインのトーンに合っていること

## SNSカレンダーの自動生成と予約投稿

SNS運用もCMO部門の重要な業務である。Content Engineには、SNS投稿の一括生成ワークフローが定義されている。

```markdown
### /ai-ceo:content:sns-batch -- SNS投稿一括生成
1. 今週公開する記事・書籍・プロダクト更新を確認
2. 各コンテンツに対して:
   - X投稿（3パターン: 告知、学び共有、質問形式）
   - ハッシュタグ選定
   - 投稿タイミングの推奨
3. 1週間分をまとめて出力
```

3パターンの投稿を生成するのが特徴である。同じコンテンツでも、告知型、学び共有型、質問形式型で異なる訴求ができる。

```
【告知型】
Focalizeのメール転送商談管理、Zennで詳しく解説しました。
メールを転送するだけで商談が自動整理される仕組みとは？
https://zenn.dev/joinclass/articles/...
#営業自動化 #SaaS

【学び共有型】
営業事務に月20時間かけていた時の話。
メールの転送先を変えるだけで、商談情報の手入力がゼロになった。
詳しい仕組みはこちら：
https://zenn.dev/joinclass/articles/...

【質問形式型】
営業メールの手動管理、まだやってますか？
メール転送だけで商談管理が完了する方法をまとめました。
https://zenn.dev/joinclass/articles/...
```

実際に7週間分のSNSコンテンツカレンダーを生成し、`.company/departments/marketing/sns-content-calendar.md`に保存した。投稿タイミングの推奨も含まれている。

```markdown
### 投稿タイミング
| メールタイプ | 推奨曜日 | 推奨時間帯 |
|------------|---------|-----------|
| 告知型 | 火・水 | 8:00-9:00 |
| 学び共有型 | 月・木 | 12:00-13:00 |
| 質問形式型 | 金 | 18:00-19:00 |
```

## CMO部門の権限設計

CMO部門のアクションは、以下の権限レベルで制御される。

**自動実行可（execute）:**
- 分析レポート、コンテンツカレンダー、SEO監査、A/Bテスト設計
- 記事執筆、書籍チャプター執筆、コピー作成、SNS下書き

**CEO承認必要（draft）:**
- 記事公開、SNS投稿、広告キャンペーン変更、LP変更のデプロイ
- 書籍公開

すべての対外的なコンテンツ公開は、必ずCEO承認を経る。これは第8章で詳しく述べる承認パイプラインの仕組みの一部である。

## まとめ -- CMO部門自動化で変わったこと

CMO部門の自動化により、マーケティングは「やりたくない仕事」から「AIに任せて成果を確認する仕事」に変わった。

**Before（自動化前）:**
- 3ヶ月間ブログ記事ゼロ
- SNS更新は思い出した時だけ
- SEOは聞いたことはあるが実施したことがない
- プロダクトのユーザー数はゼロ

**After（自動化後）:**
- 1日で9記事+4冊の書籍企画を生成
- 7週間分のSNSカレンダーを事前策定
- 全プロダクトのSEO監査を完了
- 3週間で記事を段階的に公開するスケジュールを確立

次章では、もう1つの「やりたくない仕事」であるCFO部門（経理・財務）の自動化について解説する。freeeとの連携、月次レポートの自動生成、そしてAI運用コスト月額$100-200の内訳管理を実現した方法を紹介する。
