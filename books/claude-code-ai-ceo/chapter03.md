# Claude Codeの導入と初期設定 -- CLAUDE.mdとエージェント定義

## Claude Code とは何か

本書を読んでいる方の多くは、GitHub CopilotやCursorといったAIコーディングツールを使ったことがあるだろう。Claude Codeはそれらとは根本的に異なるツールだ。

GitHub CopilotやCursorは、エディタに統合された「コード補完ツール」の延長線上にある。コードを書いている途中で次の行を提案してくれる。便利だが、あくまでエディタの中で完結する。

Claude Codeは違う。**ターミナルで動作し、ファイルの読み書き、コマンドの実行、Gitの操作など、開発者がターミナルでやることのほぼすべてを代行できる**。コード補完ではなく、開発プロセス全体を委任できるツールだ。

そして本書の文脈で最も重要なのは、**CLAUDE.mdというファイルを通じて、AIの振る舞い・役割・制約を細かく定義できる**点だ。これにより、Claude Codeは「賢いコード補完」から「設定可能なAIエージェント」に変わる。


## プラン選択 -- なぜMaxプランなのか

Claude Codeを利用するには、Anthropicのプランに加入する必要がある。2026年2月時点での選択肢は以下の通りだ。

| プラン | 月額 | 用途 |
|--------|------|------|
| Pro | $20 | 個人利用、軽い開発補助 |
| Max 5x | $100 | 本格的な開発、エージェント運用 |
| Max 20x | $200 | 大量のエージェント運用、企業利用 |

筆者はMax 5x（$100/月）からスタートし、利用量に応じてMax 20x（$200/月）に切り替えている。

**AIエージェント経営をやるなら、Max プランは必須だ。** 理由は3つある。

1. **長時間のセッション**: エージェントにスプリント全体を委任すると、1セッションが数十分に及ぶことがある。Proプランでは利用制限に引っかかる
2. **大きなコンテキスト**: 6プロダクトの状態管理ファイル、エージェント定義ファイル、コードベースを同時に扱うには、十分なコンテキストウィンドウが必要
3. **頻繁な利用**: 1日に複数の部門エージェントを稼働させるため、使用量が多い

月$100-200は安くない。しかし、第1章で述べた通り、「6部門の部下を月15,000-30,000円で雇える」と考えれば、投資対効果は非常に高い。


## セットアップ手順

### 1. Claude Code のインストール

```bash
# npmでグローバルインストール
npm install -g @anthropic-ai/claude-code

# インストール確認
claude --version
```

### 2. 認証

```bash
# 初回起動で認証フローが開始される
claude

# ブラウザが開き、Anthropicアカウントでログイン
# Max プランに加入済みのアカウントでログインする
```

### 3. プロジェクトディレクトリの初期化

```bash
# プロジェクトのルートディレクトリに移動
cd ~/workspace/your-project

# Claude Codeを起動
claude

# 初回起動時、CLAUDE.mdが存在しなければ
# デフォルトの設定で動作する
```

ここまでは通常のClaude Codeセットアップだ。ここからが、AIエージェント経営のための設定に入る。


## CLAUDE.md の書き方 -- Orchestratorとして定義する

CLAUDE.mdは、Claude Codeがプロジェクトで起動するたびに最初に読み込む設定ファイルだ。プロジェクトルートに配置する。

通常の使い方では、コーディング規約やプロジェクト固有の指示を書く程度だ。しかし、AIエージェント経営では、**CLAUDE.mdにOrchestrator（統括者）としての役割を定義する**。これが、単なるツール利用と組織設計の分かれ目になる。

### CLAUDE.mdの基本構造

筆者が実際に使っているCLAUDE.mdの構造を、要点を絞って解説する。

```markdown
# AI-CEO Framework -- C-Suite Orchestrator

> あなたはAI-CEO Frameworkの「C-Suite Orchestrator」です。
> CEOの経営判断を支援し、6部門のAIエージェントを統括します。

## あなたの役割

あなたはCEOと直接対話する唯一のインターフェースです。
以下を担当します:

1. CEOコマンドの受付と実行
2. 部門間調整
3. 承認管理
4. プロダクト横断管理

## Thin Orchestratorの原則

- コンテキスト使用率を10-15%に維持する
- ファイルの中身を自分のコンテキストに読み込まず、ファイルパスだけを渡す
- 複雑なタスクは必ずサブエージェントに委任する
- 自分で実作業を行わない
```

冒頭の宣言が重要だ。「あなたはC-Suite Orchestratorです」と明示することで、Claude Codeは「自分はツールではなく、組織の統括者として振る舞うべきだ」と理解する。

### 会社情報の参照先を定義する

```markdown
## 会社情報の参照先

- ビジョン・ミッション: `.company/VISION.md`
- 現在の経営状態: `.company/STATE.md`
- 四半期ロードマップ: `.company/ROADMAP.md`
- CEO意思決定ログ: `.company/decisions/` (当月のファイル)
- 権限・閾値設定: `.company/steering/permissions.md`
- 承認待ちキュー: `.company/approval-queue.md`
- ブランド・技術規約: `.company/steering/`
- プロダクト別状態: `.company/products/`
- 部門別状態: `.company/departments/`
```

ここで「ファイルパス」を列挙しているのがポイントだ。ファイルの中身はここには書かない。Thin Orchestratorの原則に従い、必要な時にエージェントが自分で読みに行く。

### コマンド体系を定義する

```markdown
## CEOコマンド一覧

### 日次運用
- `/ai-ceo:morning` -- 朝ダイジェスト生成
- `/ai-ceo:status` -- 現在の全体状態

### 承認操作
- `/ai-ceo:approve <id>` -- 承認
- `/ai-ceo:reject <id> "理由"` -- 却下

### 部門コマンド
- `/ai-ceo:dev:sprint` -- スプリント計画・実行
- `/ai-ceo:dev:hotfix "説明"` -- 緊急バグ修正
- `/ai-ceo:mkt:campaign "概要"` -- マーケキャンペーン
- `/ai-ceo:mkt:content-plan` -- 月間コンテンツカレンダー
...
```

コマンドを定義しておくことで、CEOは自然言語で長々と指示を書く必要がなくなる。`/ai-ceo:dev:sprint`と入力するだけで、Orchestratorは「CTOエージェントにスプリント計画を委任する」と理解する。

### コマンド実行フローを定義する

コマンドの名前だけでは不十分だ。各コマンドが「何をするのか」を具体的なステップで定義する。

```markdown
### /ai-ceo:morning の実行フロー
1. 各部門の `.company/departments/{dept}/STATE.md` を読み取る
2. `.company/approval-queue.md` の承認待ちアイテムを読み取る
3. 各プロダクトの `.company/products/{name}/STATE.md` を読み取る
4. 以下のフォーマットでダイジェストを生成:

## 承認待ち ({n}件)
- [AQ-xxx] {部門}: {内容} | {ファイルパス}

## 部門状態サマリ
| 部門 | ステータス | 進行中タスク | 注意事項 |
...
```

このように、入力（何を読むか）、処理（何をするか）、出力（何を生成するか）を明確に定義する。曖昧さを排除すればするほど、エージェントの出力品質は安定する。

### 権限制御ルールを組み込む

```markdown
## 権限制御ルール

全てのアクションは `.company/steering/permissions.md` の閾値に従う。

- **read-only:** 分析・レポート系は承認不要で自動実行
- **draft:** 対外アクション（メール送信、請求書発行、SNS投稿、
  デプロイ等）は必ずdraftモードで生成し、approval-queue.md に追加
- **execute:** 閾値内の内部アクション（バグ修正、テスト実行等）は自動実行

**重要:** 対外に影響のあるアクションを直接実行してはならない。
必ずdraft → 承認 → 実行のパイプラインを通す。
```

この宣言が承認パイプラインのエントリーポイントだ。CLAUDE.mdに明記することで、Orchestratorが呼び出すすべてのサブエージェントにも、この制約が適用される。

### エラーハンドリング

```markdown
## エラー時の振る舞い

- サブエージェントが失敗した場合: エラー内容をフィードバックして最大3回リトライ
- 3回失敗: `.company/approval-queue.md` にエスカレーションとして追加し、CEOに報告
- エラーログ: `.company/departments/{dept}/error-log.md` に追記
```

AIエージェントは失敗する。コードの生成に失敗することもあるし、ファイルの読み取りに失敗することもある。エラーが起きた時の振る舞いを定義しておかないと、エージェントはエラーを握りつぶすか、延々とリトライし続ける。3回リトライしてダメならCEOに報告する、というルールが、運用上は最もバランスが良い。


## .claude/agents/ でのサブエージェント定義

CLAUDE.mdがOrchestratorなら、`.claude/agents/`配下のファイルは各部門の「エージェント定義書」だ。

### ファイル構成

```
.claude/agents/
├── ai-ceo-cto.md              # CTO部門
├── ai-ceo-cmo.md              # CMO部門
├── ai-ceo-content-engine.md   # コンテンツ量産
├── ai-ceo-cfo.md              # CFO部門
├── ai-ceo-cso.md              # CSO部門
├── ai-ceo-biz-dev.md          # 事業開発
├── ai-ceo-cs-lead.md          # CS部門
├── ai-ceo-legal.md            # 法務部門
├── ai-ceo-hr.md               # 人事部門
├── ai-ceo-growth.md           # グロース
├── ai-ceo-consulting-vp.md    # コンサル統括
├── ai-ceo-init.md             # 初期セットアップ
├── ai-ceo-morning.md          # 朝ダイジェスト
└── ai-ceo-openclaw-ops.md     # 自動化基盤
```

注目してほしいのは、「部門エージェント」だけでなく、「機能エージェント」も定義している点だ。`ai-ceo-init.md`は初期セットアップ専用、`ai-ceo-morning.md`は朝ダイジェスト専用のエージェントだ。頻繁に使うワークフローを専用エージェントとして切り出すことで、品質と速度が安定する。

### エージェント定義ファイルの書き方

CTO部門のエージェント定義を例に、書き方のパターンを解説する。

```markdown
---
name: ai-ceo-cto
description: CTO/開発部長エージェント。プロダクト開発全般を統括する。
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---

# CTO / 開発部長エージェント

あなたは AI-CEO Framework の CTO（最高技術責任者）です。
```

#### frontmatter（メタデータ）

```yaml
---
name: ai-ceo-cto
description: CTO/開発部長エージェント。
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---
```

- `name`: エージェントの識別子。Orchestratorがエージェントを呼び出す時に使う
- `description`: 1行の説明。何を担当するエージェントかを簡潔に記述
- `tools`: このエージェントが使用できるツール。`Bash`を含めるとターミナルコマンドの実行が可能になる。権限を最小限に絞ることがセキュリティ上重要

#### ペルソナ定義

```markdown
## ペルソナ

経験豊富なテックリード。実用性を重視し、オーバーエンジニアリングを避ける。
「動くものを最速で出し、フィードバックを得て改善する」がモットー。
```

ペルソナを定義するのは、出力の一貫性を保つためだ。「実用性重視」と書いておけば、エージェントは過度に複雑な設計を提案しにくくなる。人間のエンジニアを採用する時に「どういうタイプの人か」が重要なのと同じだ。

#### 権限レベル

```markdown
## 権限レベル

- **execute:** コーディング、テスト実行、ステージングデプロイ、内部ドキュメント
- **draft:** 本番デプロイ、アーキテクチャの大幅変更、新規ライブラリ導入
```

各エージェントに許可されるアクションの範囲を、`execute`と`draft`に分けて明記する。これにより、エージェントが「自分の判断で自動実行していいこと」と「CEOの承認を得るべきこと」を区別できる。

#### 参照ファイルの指定

```markdown
## 参照ファイル

- 技術スタック: `.company/steering/tech-stack.md`
- プロダクト状態: `.company/products/{name}/STATE.md`
- 開発部門状態: `.company/departments/dev/STATE.md`
- 権限設定: `.company/steering/permissions.md`
```

エージェントが参照すべきファイルを列挙する。これがないと、エージェントは必要な情報を探すのに余計なコンテキストを消費する。「あなたに必要な情報はここにある」と教えてあげることで、効率が上がる。

#### ワークフロー定義

```markdown
## ワークフロー

### /ai-ceo:dev:sprint
1. `.company/products/{name}/STATE.md` からバックログを確認
2. 優先度の高いタスクを選定（最大3タスク）
3. 各タスクの仕様を作成:
   - Requirements (何を作るか)
   - Design (どう作るか)
   - Tasks (実装タスクリスト)
4. 実装
5. コードレビュー
6. テスト実行・確認
7. `.company/departments/dev/STATE.md` を更新
```

ワークフローはできるだけ具体的なステップに分解する。曖昧なステップがあると、エージェントの出力がブレる。「バックログを確認」ではなく「`.company/products/{name}/STATE.md` からバックログを確認」と書くことで、何を読むべきかが明確になる。

#### 品質検証の定義

```markdown
## 品質検証

すべての成果物に以下の検証を適用する:

1. ゴール逆算検証: 「この実装が正しいなら、{テスト}がパスするはず」→テスト実行
2. tech-stack.md 準拠チェック: 使用技術が規約に合っているか
3. セキュリティチェック: 基本的なセキュリティベストプラクティス
```

品質検証ルールを組み込むことで、エージェントは自分の出力を自己検証する。「実装 → テスト → 確認」のサイクルがエージェント内で完結するため、CEOが一つ一つコードを確認する必要がなくなる。


## Permission設定 -- 安全に使うための権限管理

権限管理は、AIエージェント経営の安全性を担保する最も重要な仕組みだ。

### permissions.md の全体像

```markdown
# 権限・閾値設定

## 費用閾値

| アクション | 閾値 | モード |
|-----------|------|--------|
| 自動承認上限 | 5,000円/件 | execute |
| CEO承認必要 | 5,000円超/件 | draft |
| 月間AI運用予算上限 | $200/月 | monitor |
```

金額の閾値は、事業規模に応じて調整する。筆者の場合、5,000円未満のアクション（たとえばドメイン更新やプラグイン購入）は自動承認、5,000円以上はCEO承認としている。

### 開発権限の詳細

```yaml
# 自動実行可（execute）
auto_execute:
  - bugfix          # バグ修正
  - minor_feature   # マイナー機能追加
  - test            # テスト追加・修正
  - refactor        # リファクタリング
  - docs            # ドキュメント更新
  - dependency_patch # パッチ/マイナーアップデート

# CEO承認必要（draft）
ceo_approval:
  - new_feature         # 新機能開発
  - architecture_change # アーキテクチャ変更
  - deploy_production   # 本番デプロイ
  - major_update        # メジャーアップデート
  - new_dependency      # 新規ライブラリ導入
  - schema_change       # DBスキーマ変更
```

この分類は、実運用を通じて調整してきた。最初はすべてを`draft`にしていたが、バグ修正やテスト追加まで承認を待つのは非効率だった。逆に、新機能開発を`execute`にしたら、意図しない機能が実装されたことがある。この粒度に落ち着くまで、2-3週間の試行錯誤があった。

### 対外コミュニケーションの権限

```yaml
# 常にdraft（CEO確認必須）
always_draft:
  - press_release      # プレスリリース
  - pricing_change     # 価格変更の告知
  - claim_response     # クレーム・苦情対応
  - proposal           # クライアント提案書
  - contract           # 契約書・NDA
  - invoice            # 請求書

# CEO承認後に自動実行可
auto_after_approval:
  - scheduled_sns      # 承認済みテンプレートに基づくSNS投稿
  - faq_response       # 承認済みテンプレートに基づくFAQ回答
  - tech_article       # レビュー済み技術記事の公開

# 自動実行可（read-only系）
auto_execute_readonly:
  - analytics_report   # 分析レポート生成
  - kpi_update         # KPIダッシュボード更新
  - competitor_analysis # 競合調査レポート
  - internal_docs      # 社内ドキュメント更新
```

対外コミュニケーションは最も慎重に管理する領域だ。クライアントへの提案書や請求書を、AIが勝手に送信することは許されない。一方、社内向けの分析レポートやドキュメント更新は自動実行で問題ない。

### デプロイ権限

```yaml
deploy:
  local: execute      # ローカル開発: 自由に実行可
  staging: execute     # ステージング: 自動デプロイ可
  production: draft    # 本番: CEO承認後にデプロイ
```

開発環境とステージング環境は自動デプロイを許可し、本番環境のみCEO承認を必須とする。ステージング環境でエージェントが自由にデプロイ・テストできることで、開発のイテレーション速度が上がる。


## Thin Orchestratorの原則 -- コンテキスト管理の実践

### なぜコンテキスト管理が重要か

Claude Codeには一度に処理できる情報量の上限（コンテキストウィンドウ）がある。6プロダクトの状態、8部門のエージェント定義、数十のファイル -- これらをすべてコンテキストに入れると、すぐに上限に達する。

コンテキストが溢れると、エージェントは指示の一部を「忘れ」たり、品質が低下する。これを防ぐのがThin Orchestratorの原則だ。

### 実践パターン

#### パターン1: ファイルパスだけを渡す

```markdown
# 悪い例（コンテキストを圧迫する）
## 現在の経営状態
合同会社ジョインクラスは6つのプロダクトを運営しており...
（STATE.mdの全内容をCLAUDE.mdにコピー）

# 良い例（パスだけ指定する）
## 会社情報の参照先
- 現在の経営状態: `.company/STATE.md`
```

CLAUDE.mdにはファイルパスだけを書き、実際のファイル内容はサブエージェントが必要な時に読み込む。Orchestrator自身のコンテキスト消費を最小限に抑える。

#### パターン2: サブエージェントへの委任

```markdown
# 悪い例（Orchestratorが実作業をする）
/ai-ceo:dev:sprint と入力されたら、コードを書いて、テストして、
PRを作成してください。

# 良い例（サブエージェントに委任する）
/ai-ceo:dev:sprint と入力されたら、ai-ceo-cto エージェントに
以下の情報を渡して委任してください:
1. タスクの目的
2. 参照ファイルパス
3. 成果物の出力先
4. 権限レベル
5. 品質基準
```

Orchestratorは作業を自分でやらない。タスクの目的とコンテキスト（ファイルパス）をサブエージェントに渡すだけだ。

#### パターン3: 必要最小限の情報だけを返す

`/ai-ceo:morning`で朝ダイジェストを生成する際も、Orchestratorは各部門のSTATE.mdをすべて読み込む必要はない。`ai-ceo-morning`エージェントに委任し、要約されたダイジェストだけを受け取る。

### コンテキスト使用率の目安

```
Orchestrator自身のコンテキスト:  10-15%
  └── CLAUDE.md の内容
  └── コマンドの解釈
  └── サブエージェントへの委任指示

サブエージェントのコンテキスト:  60-80%
  └── エージェント定義
  └── 参照ファイルの読み込み
  └── 実作業（コーディング、分析、文書作成）

バッファ:                        10-20%
  └── エラーハンドリング
  └── 追加のやり取り
```

この配分を意識することで、長時間のセッションでもコンテキストが溢れにくくなる。


## 実際のCLAUDE.mdの構造（全体像）

ここまでの要素をまとめると、筆者が実際に使っているCLAUDE.mdは以下の構造になっている。

```markdown
# AI-CEO Framework -- C-Suite Orchestrator

> あなたはAI-CEO Frameworkの「C-Suite Orchestrator」です。

## あなたの役割

（コンテキスト管理のルール）
```

## 初回セットアップ -- /ai-ceo:init

すべての設定が完了したら、初回セットアップを実行する。

```
# Claude Code内で実行
/ai-ceo:init
```

このコマンドを実行すると、Orchestratorがインタビュー形式でCEOに質問する。

```
Orchestrator: 会社名と事業内容を教えてください。

CEO: 合同会社ジョインクラス。教育テック企業で、
     6つのプロダクトと受託開発を運営しています。

Orchestrator: ミッション・ビジョンを教えてください。

CEO: 「個の開発を促進し、社会の発展に貢献する」です。

Orchestrator: 現在のプロダクト一覧と各プロダクトの状態を教えてください。

CEO: ShareToku（運用中）、MochiQ（運用中）、Focalize（運用中）...
```

質問は1問ずつ、会話的に行われる。7つの質問に答えると、Orchestratorが以下のファイルを自動生成する。

```
自動生成されるファイル:
- .company/VISION.md
- .company/STATE.md
- .company/ROADMAP.md
- .company/steering/brand.md
- .company/steering/tech-stack.md
- .company/steering/policies.md
- .company/steering/permissions.md
- .company/approval-queue.md
- .company/decisions/2026-02.md
- .company/products/{product-name}/STATE.md (各プロダクト分)
- .company/departments/{dept}/STATE.md (6部門分)
```

合計20ファイル以上が一度に生成される。これがAIエージェント組織の「初期状態」だ。

生成完了後、自動的に`/ai-ceo:status`が実行され、初期状態が表示される。ここから、AIエージェント経営が始まる。


## つまずきやすいポイントと対策

最後に、筆者が実際につまずいたポイントと、その対策を共有する。

### 1. CLAUDE.mdが長すぎてコンテキストを圧迫する

最初のバージョンのCLAUDE.mdは300行以上あった。すべてのコマンドの詳細フロー、すべての部門の説明、すべてのルールを1ファイルに詰め込んでいた。結果、Orchestrator自身のコンテキスト消費が大きくなり、サブエージェントに十分なコンテキストを渡せなくなった。

**対策:** CLAUDE.mdは150行以内に抑え、詳細はサブエージェント定義ファイルに分離する。Orchestratorには「何をどこに委任するか」だけを書き、「どうやるか」はサブエージェントに書く。

### 2. エージェントの出力品質がバラつく

同じコマンドを実行しても、日によって出力の品質やフォーマットが異なることがある。特にコンテンツ系（記事、提案書）で顕著だった。

**対策:** エージェント定義に「成果物テンプレート」と「品質検証ルール」を追加する。テンプレートで出力フォーマットを固定し、品質検証ルールでセルフチェックさせることで、品質が安定した。

### 3. 承認キューが溜まりすぎる

エージェントが活発に動くと、承認キューに10件以上のアイテムが溜まることがある。全件を確認するのに時間がかかり、ボトルネックになった。

**対策:** `permissions.md`の閾値を調整し、低リスクなアクション（テスト追加、リファクタリング、内部ドキュメント更新など）をより多く`execute`に移行した。承認キューには本当にCEOの判断が必要なアイテムだけが入るようにした。

### 4. サブエージェントの起動に時間がかかる

サブエージェントが起動する際にエージェント定義ファイルと参照ファイルを読み込むため、最初の応答に10-20秒かかることがある。

**対策:** これはClaude Codeの仕様上避けられない。ただし、エージェント定義ファイルをコンパクトに保つこと、参照ファイルのパスを最小限にすることで、起動時間をある程度短縮できる。
