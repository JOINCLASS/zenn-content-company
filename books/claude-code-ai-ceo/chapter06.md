# CFO部門の自動化 -- freee連携と月次レポート

## エンジニアが最も苦手とする業務

コードを書くのは得意だが、経理は苦手。これはエンジニア出身の経営者に共通する悩みだろう。

毎月の請求書処理、経費精算、月次決算、確定申告の準備。いずれも「やらないと法的に問題になる」にもかかわらず、開発に比べるとモチベーションが湧かない。freeeにログインして仕訳を入力する作業は、1つひとつは5分で終わるのに、なぜか後回しにしてしまう。

AI-CEO FrameworkのCFO部門は、この経理業務の「面倒くささ」を解消するために設計した。完全自動化は規制や会計基準の制約上、現時点では難しい。しかし「データ収集→レポート生成→CEOが確認→freeeに記帳」というフローの中で、最も時間のかかる「データ収集」と「レポート生成」をAIに任せることで、CEOの負担を大幅に軽減できる。

## CFOエージェントの役割定義

CFOエージェントは、以下のペルソナで定義している。

```markdown
# CFO / 経理部長エージェント

あなたは AI-CEO Framework の CFO（最高財務責任者）です。

## ペルソナ

スタートアップの財務管理に精通した実務型CFO。少人数法人の税務・会計を効率化し、
キャッシュフローの可視化と将来予測でCEOの意思決定を支援する。
「1円の無駄も見逃さない」と「投資すべきところには投資する」のバランスを取る。
```

ポイントは「実務型」という表現である。大企業向けの高度な財務戦略は求めていない。合同会社という小規模法人の日々の経理業務を効率的に回すことが目的である。

CFOエージェントの専門知識は3つの領域に分けている。

```markdown
## 専門知識

### 会計・税務
- 合同会社の法人税務（法人税、消費税、源泉徴収）
- freee会計を活用した仕訳・月次処理の自動化
- 確定申告・決算準備

### コスト管理
- SaaSプロダクトのユニットエコノミクス（LTV、CAC、MRR、チャーン）
- クラウドインフラコストの最適化（GCP、Firebase、Vercel、お名前.com）
- AI運用コストの追跡と最適化（Claude API、OpenAI API）

### 収支予測
- ランウェイ計算と資金繰り表
- プロダクト別P/L
- シナリオ分析（ベース/楽観/悲観）
```

## freeeのデータを活用した月次決算の効率化

### 月次レポートのワークフロー

CFOエージェントの最も重要なワークフローは月次決算レポートの生成である。

```markdown
### /ai-ceo:fin:monthly-report -- 月次決算レポート
1. freee のデータを参照（CEO入力ベース）
2. プロダクト別の売上・コストを集計
3. 以下のレポートを生成:
   - 月間P/L サマリ
   - プロダクト別収支
   - インフラコスト内訳
   - AI運用コスト内訳
   - 前月比較と異常値フラグ
4. .company/departments/finance/monthly-{YYYY-MM}.md に出力
```

現状、freee APIとの直接連携は行っていない。APIを使えばデータ取得を完全自動化できるが、合同会社の規模ではAPI連携のセットアップコストに見合わない。代わりに、freeeの画面から取得したデータをCEOが月1回入力し、CFOエージェントがレポートを自動生成する方式を取っている。

## KPIダッシュボードの自動生成 -- STATE.mdの設計

経理部門の自動化で最も価値が高いのは、KPIダッシュボードの自動生成である。STATE.mdのKPIセクションがそれに該当する。

```markdown
## KPIダッシュボード

### Focalize
| KPI | 現在値 | Q1末目標 | 状態 |
|-----|--------|----------|------|
| 登録ユーザー数 | 1 | 30 | Instagram広告開始 |
| 有料プラン契約数 | 0 | 5 | Free→Pro転換施策が次 |
| MRR | ¥0 | ¥15,000 | 計測開始 |

### 全社収支（Q1末目標）
| 項目 | 現在値 | Q1末目標 |
|------|--------|----------|
| 全社売上合計 | ¥0 | ¥345,000/月 |
| 全社コスト合計 | ¥0 | 把握完了 |
| 営業利益 | ¥0 | ¥200,000/月 |
```

このKPIダッシュボードは、`/ai-ceo:morning`コマンドで毎朝表示される。CEOは毎朝5分で全プロダクトの収益状況を把握できる。

重要なのは、KPIの「状態」欄にアクションの示唆が含まれている点である。単に「0」と表示するのではなく、「Instagram広告開始」「Free→Pro転換施策が次ステップ」のように、現状への対処方針を明記する。これにより、数値を見ただけで「次に何をすべきか」が即座に判断できる。

## コスト分析レポートの自動生成

### コストトラッキング体制の設計

CFO部門の最初の仕事は、コストトラッキング体制の構築だった。`.company/departments/finance/cost-tracking.md`に、以下の構造を定義した。

```markdown
## 1. 月次コスト管理テンプレート

### 1-1. AI関連コスト
| 項目 | サービス | プラン/単位 | 月額見込み(USD) | 実績(USD) | 備考 |
|------|---------|------------|----------------|----------|------|
| LLM基盤 | Claude Code Max | 月額固定 | $100-200 | - | メイン |
| LLM API | Claude API | 従量課金 | $0 | - | 必要時のみ |
| LLM API | OpenAI API | 従量課金 | $0 | - | 必要時のみ |

### 1-2. インフラコスト
| 項目 | サービス | プラン/単位 | 月額見込み(JPY) | 実績(JPY) | 備考 |
|------|---------|------------|----------------|----------|------|
| ホスティング | お名前.com | レンタルサーバー | ¥X,XXX | - | 共有 |
| BaaS | Firebase | Spark/Blaze | ¥0 | - | 無料枠内 |
| ドメイン | お名前.com | 年額按分 | ¥XXX | - | 按分 |

### 1-3. SaaS/ツールコスト
| 項目 | サービス | プラン | 月額(JPY) | 実績(JPY) | 備考 |
|------|---------|------|----------|----------|------|
| 会計 | freee | 法人プラン | ¥X,XXX | - | 経理基盤 |
| コラボ | Google Workspace | Business | ¥X,XXX | - | メール等 |
| ソース管理 | GitHub | Free | ¥0 | - | 無料プラン |
```

### プロダクト別コスト配分

6つのプロダクトを運営しているため、共有リソースのコスト配分が問題になる。大企業なら精緻なABC（Activity Based Costing）を行うが、1人法人ではそこまでの精度は不要である。

```markdown
### 共有リソースの按分方法

CEO 1名 + AI運営のため、複雑な配賦計算は避け、シンプルなルールで按分する。

#### AI利用料（Claude Code Max）の按分
| 按分方法 | 算出基準 | 適用条件 |
|---------|---------|---------|
| 均等割り | 総額 / アクティブプロダクト数 | デフォルト |
| 工数ベース | 月間の作業時間比率で按分 | 工数記録がある場合 |
| 売上ベース | 各プロダクトの売上比率で按分 | 収益が安定している場合 |

**当面は「均等割り」を採用。**
```

この判断自体も、CFOエージェントの提案に基づいている。最初から完璧な配賦を目指すのではなく、まず「把握できる状態」を作り、データが蓄積されたら精度を上げる。この段階的アプローチは、リソースの限られたスタートアップにおいて合理的である。

### コスト異常検知ルール

CFOエージェントには、コストの異常を自動検知するルールも組み込んでいる。

```markdown
### コスト異常検知ルール

以下の条件に該当した場合、承認キューにエスカレーションとして追加する:

| ルール | 条件 | 重要度 |
|--------|------|--------|
| 前月比急増 | いずれかのカテゴリで前月比 +20% 以上 | 高 |
| 予算超過 | AI関連コストが月額$200を超過 | 緊急 |
| 新規課金発生 | 過去に$0だったサービスで課金が発生 | 中 |
| Firebase従量超過 | Firebase Blaze プランの無料枠超過 | 高 |
| サーバー容量超過 | レンタルサーバーのディスク/転送量の80%超過 | 中 |
```

特にFirebaseの従量課金は要注意である。Sparkプラン（無料）からBlazeプラン（従量課金）に移行した場合、セキュリティルールの設定ミスによる不正リクエストで予想外の課金が発生するリスクがある。CFOエージェントはこのリスクを監視し、閾値を超えた場合に即座にCEOに通知する仕組みである。

## 請求書ドラフトの自動作成

DX支援や受託開発の案件では、請求書の発行が必要になる。CFOエージェントは、請求書のドラフトを自動生成する。

```markdown
### /ai-ceo:fin:invoice "対象" -- 請求書ドラフト
1. 対象クライアント・案件情報を確認
2. 請求書ドラフトを作成（品目、金額、支払条件）
3. approval-queue に追加
```

ここで重要なのは、請求書が**必ず**approval-queueに追加される点である。permissions.mdでは、請求書発行は「常にdraft」に分類されている。

```yaml
always_draft:
  - press_release
  - pricing_change
  - claim_response
  - proposal
  - contract
  - invoice  # 請求書は常にCEO承認が必要
```

金額の間違いや宛先の間違いは、対外的な信用に直結する。AIが生成したドラフトをそのまま送付するのは危険すぎる。

## freee連携の実際 -- 勘定科目の設計

freeeとの連携で最も実務的に役立つのが、勘定科目のマッピング設計である。CFOエージェントは、AI運用という新しいコストカテゴリに対して、適切な勘定科目を提案した。

```markdown
### freee 勘定科目の設定例

| 支出項目 | freee勘定科目 | 補助科目 | 税区分 | 備考 |
|---------|-------------|--------|--------|------|
| Claude Code Max | 通信費 | AI利用料 | 課税仕入（国外取引） | USD建て |
| Claude API | 通信費 | AI利用料 | 課税仕入（国外取引） | USD建て |
| お名前.com | 通信費 | サーバー利用料 | 課税仕入 | JPY建て |
| Firebase | 通信費 | サーバー利用料 | 課税仕入（国外取引） | USD建て |
| freee | 支払手数料 | 会計ソフト | 課税仕入 | JPY建て |
```

AI利用料を「通信費」と「外注費」のどちらに分類するかは、実は判断が分かれるところである。CFOエージェントは以下のように整理した。

> AI利用料は「通信費」に統一する（クラウドサービス利用の位置づけ）。
> 「外注費」とする判断もあるが、AIツールはあくまでソフトウェア利用のため「通信費」が妥当。
> 税理士確認時に変更の可能性あり。

この判断は合理的である。Claude Code Maxはソフトウェアのサブスクリプションであり、人間に業務を委託する「外注」とは性質が異なる。ただし、AIエージェントが業務を「遂行」しているという観点では「外注費」も成り立つ。この曖昧さを認めつつ、暫定方針を示した上で税理士確認を推奨している点が、CFOエージェントの実用性を示している。

### 国外取引の注意点

Claude、Firebase、Google Workspaceなど、海外サービスの利用は「国外取引」として消費税の処理が必要になる。

```markdown
**国外取引の注意点:**
- USD建てのサービスは消費税の「課税仕入（国外取引）」として処理
- リバースチャージ方式が適用される場合がある（売上規模により判断）
- 為替差損益が発生する場合はfreeeの自動処理に任せるか、
  クレカ明細の円額で計上
```

これはAIエージェントに教えてもらうまで正直意識していなかった部分である。USD建てのサービスを複数利用している場合、為替レートの違いによる差損益が発生する。freeeのクレジットカード自動連携を使えば、カード会社の為替レートで自動処理されるため、手動で為替計算する必要はない。

## AI運用コストの管理 -- 月額$100-200の内訳と最適化

### Claude Code Maxプランの費用構造

AI-CEO Frameworkの運用コストの大部分は、Claude Code Maxプランの月額料金である。

```markdown
## リソース状況

- **AI運用予算**: 月額 $100-200（Claude Code Maxプラン）
```

permissions.mdでは、AI運用コストの月額上限を$200に設定している。

```yaml
monthly_ai_budget: 200  # USD
```

この$200という数字は、1人法人の経営に対するAI投資として妥当な水準だと判断した。日本円で約3万円。フルタイムの従業員を1人雇用すれば月額30万円以上かかることを考えると、10分の1のコストで6部門のAIエージェントを運用できることになる。

### $200上限の監視方法

CFOエージェントは、AI運用コストの閾値監視を行う。

```markdown
### $200上限の監視方法

| 監視ポイント | 閾値 | アクション |
|-------------|------|-----------|
| AI月額コスト | $160（80%） | CEOに予算警告を通知 |
| AI月額コスト | $180（90%） | CEOに緊急警告。追加API利用の一時停止を提案 |
| AI月額コスト | $200（100%） | CEOに即時報告。追加利用は承認なしでは不可 |
```

Claude Code Maxプランは固定月額であるため、通常は$200以内に収まる。追加でClaude APIを直接呼び出す場合（たとえばOpenClawエージェントからのAPI利用）のみ超過リスクがある。

### コスト最適化のための「Thin Orchestrator」戦略

AI運用コストを抑えるための最も重要な設計判断は、第2章で紹介した「Thin Orchestrator」原則である。

```markdown
## Thin Orchestratorの原則

- **コンテキスト使用率を10-15%に維持する**
- ファイルの中身を自分のコンテキストに読み込まず、**ファイルパスだけを渡す**
- 複雑なタスクは必ずサブエージェントに委任する
- 自分で実作業（コーディング、文書作成等）を行わない
```

中央のOrchestrator（C-Suite Orchestrator）がすべてのファイルを読み込むと、コンテキストウィンドウが爆発し、トークン消費量が急増する。代わりに、Orchestratorはファイルパスだけを管理し、実際の作業はサブエージェントに委任する。

この設計により、1回のOrchestrator呼び出しあたりのトークン消費を最小化し、月額予算内で多くのタスクを処理できる。

### 月次コスト確認フロー

実際の運用では、月次のコスト確認を以下のフローで行う。

```
1. [自動] fin-lead が各サービスの請求情報を収集
   - お名前.com: コントロールパネル → 請求額確認
   - Firebase: Firebase Console → Usage and billing
   - Claude: Anthropic Console → Billing
   - freee, Google Workspace: 請求メール or 管理画面
   - GitHub: Settings → Billing

2. [自動] コストトラッキングテンプレートを当月データで更新

3. [自動] 異常検知チェック（ルール適用）

4. [自動] コストレポートを approval-queue.md に追加（確認依頼）

5. [CEO] レポート確認 → 承認 or 質問・指示

6. [自動/CEO] freeeへの仕訳入力
```

ステップ1-4はCFOエージェントが自動実行する。CEOが行うのはステップ5のレビューとステップ6のfreee記帳のみである。記帳自体も、クレジットカードの自動連携を設定していれば、freee上での勘定科目の確認・修正だけで完了する。

## コスト最適化チェックリスト

CFOエージェントは、月次のコスト確認時に以下のチェックリストを自動実行する。

```markdown
### 不要リソースの棚卸し
- [ ] お名前.com: 使用していないサーバー/サブドメインがないか
- [ ] Firebase: 不要なFirebaseプロジェクトが残っていないか
- [ ] Firebase: Cloud Functionsで不要なファンクションが稼働していないか
- [ ] GitHub: プライベートリポジトリで有料プランが必要なものがないか
- [ ] ドメイン: 更新不要なドメインの自動更新がONになっていないか
- [ ] API: 利用していないAPIキーが有効なままになっていないか

### プラン最適化
- [ ] Firebase: Sparkプラン（無料）で十分なプロダクトがBlazeになっていないか
- [ ] Google Workspace: 利用していない追加ライセンスがないか
- [ ] freee: プランの見直し（上位プランの機能を使っていない場合）

### 従量課金の最適化
- [ ] Firebase: Firestoreの読み取り/書き込み回数が想定範囲内か
- [ ] GitHub Actions: ビルド時間が増加傾向にないか
- [ ] API利用: Claude API等の従量課金が想定を超えていないか
```

このチェックリストは「当たり前のこと」に見えるかもしれない。しかし、6つのプロダクトを並行運用していると、使っていないFirebaseプロジェクトや、自動更新がONのまま放置されたドメインが意外と多い。CFOエージェントが毎月このチェックを行うことで、月に数千円の無駄を削減できている。

## CFO部門のディレクトリ構造

```
.company/departments/finance/
├── STATE.md                    ← 部門の現在状態
├── cost-tracking.md            ← コストトラッキング体制の定義
├── monthly-2026-02.md          ← 月次レポート（2月）
├── monthly-2026-03.md          ← 月次レポート（3月）
└── ...
```

シンプルな構造である。月次レポートが月ごとに蓄積されていくため、過去のデータとの比較も容易に行える。

## CFO部門の権限設計

```markdown
## 権限レベル

- **execute:** コスト分析、収支レポート、予算追跡
- **draft:** 請求書発行、支払い承認、予算変更
```

分析・レポート系は自動実行可、金銭に関わるアクション（請求書発行、支払い承認）はCEO承認必須。この設計は、経理業務の特性に合致している。

## CFO部門の品質検証

CFOエージェントが生成するすべての成果物は、以下の4つの基準で品質検証される。

```markdown
## 品質検証

1. **正確性:** 金額に誤りがないか（桁数、税込/税抜の統一）
2. **完全性:** すべてのコスト項目が漏れなく計上されているか
3. **適時性:** 月次処理が翌月5日以内に完了しているか
4. **閾値チェック:** AI運用コストが月$200以内に収まっているか
```

特に1番の「正確性」は、経理業務において最も重要である。AIが生成するレポートの金額に1桁の間違いがあれば、経営判断を誤る可能性がある。そのため、CFOエージェントの出力は必ずCEOが目視で確認する運用としている。

## まとめ -- CFO部門は「自動化」より「半自動化」

CFO部門のアプローチは、他の部門と比べて慎重である。完全自動化ではなく「半自動化」を選んだ。

**自動化した部分:**
- コストデータの収集と集計
- 月次レポートのフォーマット生成
- 異常値の検知と通知
- コスト最適化チェックリストの実行
- 勘定科目のマッピング提案

**手動のままにした部分:**
- freeeへの仕訳入力（最終確認を含む）
- 請求書の最終確認と送付
- 税理士とのやり取り
- 銀行振込の実行

この分担は、「間違いが許されない業務はCEOが最終確認する」という原則に基づいている。経理は、他の部門と異なり、ミスが法的リスクに直結する。AIの出力を100%信頼するのは、現時点では早すぎる。

次章では、CSO部門（営業部長）の自動化について解説する。提案書の自動生成、リード獲得戦略の自動策定、そしてパイプライン管理の仕組みを紹介する。営業はエンジニアにとってもう1つの「やりたくない仕事」だが、AIエージェントがその壁を大幅に下げてくれる。
