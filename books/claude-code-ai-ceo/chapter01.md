# CEO1名で6プロダクトを回す -- なぜAIエージェント経営なのか

## はじめに -- この本は「やってみた記事」ではない

本書を手に取っていただいた方に、最初にはっきり伝えておきたいことがある。

これは「Claude CodeでAIエージェントを作ってみました」という技術検証の記録ではない。**実際に会社を経営するために、AIエージェント群を組織として設計し、日々の業務で運用し続けている人間が書いた、経営実践の記録**である。

筆者は合同会社ジョインクラスのCEOだ。従業員は自分1人。だが、社内には6つの部門が存在し、CTO、CMO、CFO、CSO、カスタマーサポート、法務のそれぞれにAIエージェントが配置されている。彼らは毎日、コードレビューをし、SEO記事を書き、経理処理を補助し、営業提案書を下書きする。

「AIで仕事を自動化する」という話はもう珍しくない。しかし、**AIエージェントを組織構造として設計し、承認パイプラインを通じて経営判断に組み込み、複数プロダクトを横断管理する** -- ここまで踏み込んだ事例は、少なくとも筆者が知る限り、まだほとんど存在しない。

本書では、その全貌を包み隠さず公開する。成功も失敗も、設計思想も泥臭い運用の現実も、すべてだ。


## 著者と会社の紹介

合同会社ジョインクラスは、「個の開発を促進し、社会の発展に貢献する」をミッションに掲げる教育テック企業だ。

もともとはフリーランスエンジニアとして受託開発をしていた筆者が法人化し、自社プロダクトの開発と受託事業を並行して進める形で成長してきた。フルスタックエンジニアとして、フロントエンド（Next.js/React）、バックエンド（Firebase/Node.js）、モバイル（Flutter）、インフラ（GCP）まで一通り対応できることが、1人経営の前提条件になっている。

現在運営しているプロダクトとサービスは以下の通りだ。

6つのプロダクトに加えて受託開発、コンサルティング、書籍執筆。冷静に数えると、9つの事業ラインを1人で運営していることになる。


## 1人で9事業を回す1日 -- AIエージェント導入前

AIエージェント経営を導入する前、筆者の1日がどうだったかを振り返ってみる。

### 典型的な1日（導入前）

```
06:00  起床、メール・Slack確認
06:30  受託案件の開発作業
09:00  クライアントとのミーティング
10:00  受託案件の開発作業（続き）
12:00  昼食（食べながらプロダクトのバグ報告確認）
13:00  ShareTokuの機能改善
14:30  MochiQのバグ修正
15:30  Focalizeの新機能実装
17:00  元気ボタンアプリのApp Store審査対応
17:30  freeeで経理処理
18:00  スキル管理SaaSの問い合わせ対応
18:30  SNS投稿（X）を考えて投稿
19:00  技術記事のネタ出しと下書き
20:00  翌日のタスク整理
20:30  受託案件の見積もり作成
21:00  終了（のはずが、緊急バグ対応で23時になることも）
```

この生活を続けていると、確実にあることが起きる。**すべてが中途半端になる**。

6つのプロダクトがあっても、1日に触れるのは2-3個が限界だ。残りのプロダクトは放置される。ShareTokuのSEO改善を進めたい週に、MochiQで緊急バグが出れば、ShareTokuは後回しになる。後回しになったSEO改善は、翌週も翌月も着手されないまま、バックログの奥に沈んでいく。

受託案件は期限があるから最優先で対応するが、そのぶん自社プロダクトの開発時間が削られる。自社プロダクトが成長しないから収益が伸びず、受託への依存度が下がらない。典型的な悪循環だ。

経理処理は月末にまとめてやるが、レシートの仕分けと勘定科目の確認に丸1日かかる。技術記事は「今月こそ書く」と毎月思うが、月2本のペースすら維持できない。SNSは思い出した時に投稿するだけで、戦略的な発信にはほど遠い。

**物理的に、1人の人間が1日24時間で処理できるタスク量を、完全に超えていた。**


## 転機 -- Claude Code との出会い

2025年後半、Claude Code Maxプラン（$100-200/月）を使い始めたことが転機になった。

最初は普通の開発補助として使っていた。コードの雛形生成、バグの原因調査、テストコードの作成。それだけでも開発効率は大幅に上がった。

しかし、使い込むうちに気づいたことがある。Claude Codeは単なるコード生成ツールではない。**CLAUDE.mdという設定ファイルを通じて、AIの振る舞いそのものを定義できる**。つまり、「こういう役割で、こういうルールに従って、こういう品質基準で仕事をしろ」と指示できるのだ。

この瞬間、頭の中でパズルのピースがはまった。

「これ、組織を作れるのではないか。」

CTO役のエージェントにコードレビューを任せ、CMO役のエージェントにコンテンツ戦略を任せ、CFO役のエージェントに経理処理を任せる。各エージェントには専門知識と権限を定義し、成果物は承認パイプラインを通じてCEO（自分）が最終確認する。

1人の人間がすべてを処理するのではなく、**1人のCEOが複数のAIエージェントを統括する**。これなら、6プロダクトの同時並行管理が物理的に可能になる。

こうして、AI-CEO Frameworkの構想が始まった。


## AIエージェント経営の全体像

構想を具体化するのに、それほど時間はかからなかった。組織設計の知識と、Claude Codeの仕組みを組み合わせれば、必要な設計は自ずと見えてくる。

最終的に構築したシステムの全体像は以下の通りだ。

```
CEO（人間）
  │
  ├── CLAUDE.md（C-Suite Orchestrator）
  │     └── AIとの唯一のインターフェース
  │
  ├── .claude/agents/（サブエージェント群）
  │     ├── ai-ceo-cto.md     ← 開発部門
  │     ├── ai-ceo-cmo.md     ← マーケ部門
  │     ├── ai-ceo-cfo.md     ← 経理部門
  │     ├── ai-ceo-cso.md     ← 営業部門
  │     ├── ai-ceo-cs-lead.md ← CS部門
  │     └── ai-ceo-legal.md   ← 法務部門
  │
  └── .company/（会社の状態管理）
        ├── VISION.md          ← ミッション・ビジョン
        ├── STATE.md           ← 現在の経営状態
        ├── ROADMAP.md         ← 四半期ロードマップ
        ├── approval-queue.md  ← 承認待ちキュー
        ├── steering/          ← 経営方針・規約
        ├── products/          ← プロダクト別状態
        ├── departments/       ← 部門別状態
        └── decisions/         ← 意思決定ログ
```

ポイントは3つある。

**1. Orchestratorパターン**

CLAUDE.mdに定義した「C-Suite Orchestrator」が、CEOとAIエージェント群の間に立つ。CEOはOrchestratorに指示を出し、Orchestratorが適切な部門エージェントにタスクを委任する。CEOが個々のエージェントと直接やり取りする必要はない。

**2. ファイルベースの状態管理**

会社の状態はすべて`.company/`ディレクトリ内のMarkdownファイルで管理される。データベースは不要。Gitでバージョン管理でき、変更履歴も追える。エージェントはファイルを読み書きすることで「仕事」をする。

**3. 承認パイプライン**

対外的な影響があるアクション（デプロイ、メール送信、SNS投稿など）は、必ず`draft`モードで生成され、`approval-queue.md`に追加される。CEOが承認するまで実行されない。これにより、AIの暴走を防ぎつつ、CEOは「判断」に集中できる。


## Before/After -- 具体的に何が変わったか

### 開発効率

| 指標 | Before | After |
|------|--------|-------|
| 1日に触れるプロダクト数 | 2-3個 | 全6個を同時管理 |
| コードレビュー | 自分でやる（見落とし多数） | CTO エージェントが一次レビュー |
| テスト作成 | 後回しにしがち | エージェントが自動生成 |
| CI/CD設定 | プロダクトごとにバラバラ | 全プロダクト統一フロー |
| バグ修正の初動 | 自分が気づくまで放置 | エージェントが検出・修正案を提示 |

実例を一つ挙げよう。元気ボタンアプリにCI/CDを追加する作業があった。導入前なら、GitHub Actionsのワークフロー作成、テスト環境の設定、デプロイスクリプトの作成、動作確認まで、少なくとも半日はかかっていた。

AI-CEO Framework導入後は、`/ai-ceo:dev:sprint`コマンド一つでCTOエージェントがタスクを受け取り、ワークフローファイルの作成からPR作成まで自動で実行した。筆者がやったのは、PRの内容を確認して承認ボタンを押すことだけだ。所要時間は15分。

### マーケティング

| 指標 | Before | After |
|------|--------|-------|
| 技術記事の公開ペース | 月0-1本（書く時間がない） | 月4本以上（エージェント下書き→CEO編集） |
| SEO監査 | やったことがない | 全プロダクトLPを監査済み |
| SNS投稿 | 思い出した時だけ | 週間カレンダーに基づく計画的投稿 |
| LP改善 | 年1回程度 | 月次でCVR分析→改善サイクル |
| Zenn有料書籍 | 0冊 | 4冊企画・執筆進行中 |

特に大きいのが技術記事だ。「書かなきゃ」と思いながら書けなかった記事が、コンテンツエンジンエージェントとの協業で量産できるようになった。エージェントが構成案と初稿を作成し、筆者が自分の実体験を加筆して仕上げる。このフローなら、1記事あたり30分-1時間で公開できる。

### 経営管理

| 指標 | Before | After |
|------|--------|-------|
| 全プロダクトの状態把握 | 頭の中（よく忘れる） | STATE.mdで一元管理 |
| 四半期計画 | Notionに書いて忘れる | ROADMAP.mdで進捗追跡 |
| 意思決定の記録 | なし | decisions/月別ログに自動記録 |
| 朝の状態確認 | 各ツールを巡回（30分） | `/ai-ceo:morning`で2分 |


## AIエージェント経営のリアルな制約

ここまで書くと、夢のような話に聞こえるかもしれない。しかし、正直に制約も書いておく。

### コスト

Claude Code Maxプランは月額$100-200だ。日本円で約15,000-30,000円。フリーランスや1人社長にとっては安くない金額である。ただし、「月15,000円で6部門の部下を雇える」と考えれば、破格の投資効率だと筆者は判断している。人間のアルバイトを1人雇うだけでも月10万円はかかる。

### 判断力の限界

AIエージェントは指示された業務を高品質にこなすが、「何をすべきか」の判断は苦手だ。どのプロダクトに注力すべきか、どの機能を優先すべきか、いつピボットすべきか。こうした経営判断は、依然として人間のCEOがやる必要がある。

AI-CEO Frameworkでは、この制約を前提に設計している。エージェントは「実行」を担当し、CEOは「判断」を担当する。承認パイプラインは、この役割分担を仕組みとして担保するものだ。

### コンテキストの制約

Claude Codeには一度に処理できる情報量（コンテキストウィンドウ）の制約がある。6プロダクトの全情報をエージェントに渡すことはできない。そこで「Thin Orchestratorの原則」を採用した。Orchestratorはファイルの中身を読み込まず、ファイルパスだけを管理する。必要な情報は、タスクを実行するサブエージェントが、その都度必要なファイルだけを読み込む。

### セキュリティ

AIエージェントに会社の情報を扱わせる以上、セキュリティは無視できない。本フレームワークでは、権限レベル（read-only / draft / execute）を厳密に定義し、対外アクション（メール送信、デプロイ、SNS投稿等）は必ずCEOの承認を経るようにしている。APIキーや認証情報へのアクセスも制限している。


## この本で伝えたいこと

本書は、以下の読者を想定している。

- **技術力のあるひとり社長**: 自分でコードが書けるが、経営のあらゆる側面を1人で回すことに限界を感じている人
- **フリーランスから法人化した人**: 組織を持たずに法人としての体裁と品質を維持したい人
- **スタートアップのCTO**: 少人数チームの生産性を劇的に上げたい人

伝えたいメッセージはシンプルだ。

**「AIエージェントは、ツールではなく組織として設計すると、その価値が桁違いに上がる。」**

1つのAIツールを1つのタスクに使うだけでは、作業効率が少し上がるだけだ。しかし、複数のAIエージェントを組織構造として設計し、状態管理と承認パイプラインで結合すると、**1人の人間が10人分の組織を運営できる**ようになる。

本書では、その設計思想、具体的な実装方法、そして日々の運用で得た知見を、すべて公開する。
