# AIエージェント経営の全体設計図 -- 6部門フレームワーク

## なぜ「組織構造」が必要なのか

AIエージェントを業務に導入する際、多くの人が最初にやるのは「タスク単位の自動化」だ。コードレビューを自動化する。議事録を要約させる。メールの下書きを作らせる。

これは間違いではないが、スケールしない。

タスクが10個、20個と増えていくと、プロンプトの管理が煩雑になり、エージェントへの指示が一貫性を失い、成果物の品質がバラつく。何より、「全体として何が進んでいて、何が滞っているのか」を把握できなくなる。

筆者がたどり着いた解は、**従来の企業組織構造をAIエージェントで再現する**というアプローチだった。

企業組織には長年の知恵が詰まっている。部門分け、権限委譲、報告ライン、承認フロー。これらは「複数の人間が協調して仕事をする」ために最適化されてきた仕組みだ。AIエージェントの協調にも、同じ仕組みが有効だと考えた。

結果的に、この判断は正しかった。「組織」という構造があるからこそ、各エージェントの責任範囲が明確になり、成果物の品質が安定し、全体の進捗を一元管理できるようになった。


## 6部門の設計

### 部門構成の全体像

```
CEO（人間）
  │
  ├── C-Suite Orchestrator（CLAUDE.md）
  │
  ├── CTO部門（開発）
  │     └── ai-ceo-cto.md
  │
  ├── CMO部門（マーケティング）
  │     ├── ai-ceo-cmo.md
  │     └── ai-ceo-content-engine.md
  │
  ├── CFO部門（経理・財務）
  │     └── ai-ceo-cfo.md
  │
  ├── CSO部門（営業）
  │     ├── ai-ceo-cso.md
  │     └── ai-ceo-biz-dev.md
  │
  ├── CS部門（カスタマーサポート）
  │     └── ai-ceo-cs-lead.md
  │
  └── Legal部門（法務）
        └── ai-ceo-legal.md
```

加えて、部門横断のサポートエージェントも配置している。

```
├── ai-ceo-hr.md          ← 人事（エージェント管理・育成）
├── ai-ceo-growth.md      ← グロース（部門横断の成長施策）
├── ai-ceo-consulting-vp.md ← コンサルティング事業統括
└── ai-ceo-openclaw-ops.md ← 自動化基盤運用
```

### なぜこの6部門なのか

部門構成は「1人社長が実際に兼務している役割」から逆算した。

筆者が日々やっている仕事を分類すると、以下の6つの領域に整理できた。

1. **開発（CTO）**: コードを書く、レビューする、テストする、デプロイする
2. **マーケティング（CMO）**: 記事を書く、SEOを改善する、SNSで発信する、広告を運用する
3. **経理（CFO）**: 請求書を処理する、経費を仕分ける、月次決算する
4. **営業（CSO）**: リードを獲得する、提案書を作る、見積もりを出す
5. **CS**: 問い合わせに対応する、バグ報告を受け付ける、FAQ を整備する
6. **法務**: 契約書をレビューする、利用規約を更新する、NDAを確認する

1人で全部やっていたからこそ、「ここはAIに任せられる」「ここは人間の判断が必要」という境界が明確に見えた。この境界に基づいて、各部門エージェントの責任範囲と権限レベルを設計した。


### 各部門の役割定義

#### CTO部門 -- 開発の要

```yaml
担当: プロダクト開発全般（設計、実装、テスト、デプロイ）
権限:
  execute: コーディング、テスト実行、ステージングデプロイ、内部ドキュメント
  draft: 本番デプロイ、アーキテクチャの大幅変更、新規ライブラリ導入
ペルソナ: 実用性重視のテックリード。オーバーエンジニアリングを避ける。
```

CTO部門は最も稼働頻度が高い部門だ。6つのプロダクトすべての保守・改善を担当する。日常的にはバグ修正、テスト追加、リファクタリングを自動実行し、新機能開発や本番デプロイは`draft`モードでCEOの承認を待つ。

#### CMO部門 -- 認知と集客

```yaml
担当: コンテンツ戦略、SEO、SNS運用、広告分析、Zenn書籍管理
権限:
  execute: 分析レポート、コンテンツカレンダー、SEO監査
  draft: 記事公開、SNS投稿、広告キャンペーン変更、LP変更のデプロイ
ペルソナ: データドリブンなグロースマーケター。計測できないものは改善できない。
```

CMO部門は2つのエージェントで構成している。`ai-ceo-cmo.md`が戦略立案と分析を担当し、`ai-ceo-content-engine.md`が記事・書籍・SNS投稿の実制作を担当する。戦略と実行を分けることで、量産しつつも品質を担保する設計だ。

#### CFO部門 -- 数字の管理

```yaml
担当: 経理処理、月次決算、予算管理、KPIダッシュボード
権限:
  execute: 分析レポート、KPI更新
  draft: 請求書発行、経費精算、予算変更
ペルソナ: 堅実な管理会計のプロ。数字に基づく意思決定を支援する。
```

CFO部門は、freee（クラウド会計ソフト）との連携を前提に設計している。現時点ではfreee APIとの直接連携は実装していないが、月次の経理処理フローの型化と、KPIの集計・可視化を担当している。

#### CSO部門 -- 売上を作る

```yaml
担当: リード獲得、提案書作成、見積書作成、クライアント対応
権限:
  execute: 市場調査、競合分析、リード候補リストの作成
  draft: 提案書、見積書、クライアントへのメール
ペルソナ: ソリューション営業のプロ。技術とビジネスの橋渡し役。
```

CSO部門にも`ai-ceo-biz-dev.md`というサブエージェントを配置している。BizDevエージェントは新規事業開発、パートナーシップ構築、市場調査を専門に担当する。

#### CS部門 -- 顧客の声を聞く

```yaml
担当: 問い合わせ対応、FAQ整備、フィードバック収集、エスカレーション管理
権限:
  execute: FAQ更新、フィードバック分析
  draft: ユーザーへの返信、サービス変更の告知
ペルソナ: 共感力の高いサポート担当。技術的な問い合わせにも対応できる。
```

正直に言うと、CS部門は現時点ではまだ本格稼働していない。プロダクトのユーザー数が増えてから本格化する予定だ。しかし、組織構造としては最初から定義しておくことが重要だ。ユーザーが増えてから急いで設計するのでは遅い。

#### Legal部門 -- リスクを守る

```yaml
担当: 契約書レビュー、利用規約更新、NDA確認、コンプライアンス
権限:
  execute: 契約書の読み取りと分析
  draft: 契約書の修正案、利用規約の改定
ペルソナ: リスク感度の高い法務担当。ビジネスの推進とリスク回避のバランスを取る。
```

Legal部門も、必要に応じて外部の専門家と連携する前提だ。AIエージェントが行うのはあくまで一次的なレビューと論点整理であり、最終的な法的判断は人間（場合によっては弁護士）が行う。


## Orchestrator（統括者）パターンの設計

### なぜOrchestratorが必要か

6つの部門エージェントがバラバラに動いては、組織として機能しない。部門間の依存関係を解決し、優先順位を調整し、CEOへの報告を統合する役割が必要だ。

これを担うのが「C-Suite Orchestrator」であり、`CLAUDE.md`ファイルに定義される。

### Orchestratorの責務

```
1. CEOコマンドの受付と実行
   └── /ai-ceo:* コマンドを解釈し、適切な部門エージェントに委任

2. 部門間調整
   └── 依存関係の解決、Wave実行計画の生成

3. 承認管理
   └── draft成果物の承認キュー管理、CEOへの報告

4. プロダクト横断管理
   └── 複数プロダクト間のリソース配分・優先度判断
```

### Thin Orchestratorの原則

Orchestratorの設計で最も重要な原則がある。**Orchestratorはできるだけ「薄く」保つ**ということだ。

Claude Codeにはコンテキストウィンドウの制約がある。Orchestratorが全部門の情報を読み込んでしまうと、すぐにコンテキストを使い切ってしまう。

そこで採用したのが「Thin Orchestrator」パターンだ。

```
原則:
- コンテキスト使用率を10-15%に維持する
- ファイルの中身を自分のコンテキストに読み込まず、ファイルパスだけを渡す
- 複雑なタスクは必ずサブエージェントに委任する
- Orchestrator自身が実作業（コーディング、文書作成等）を行わない
```

Orchestratorは「交通整理役」に徹する。実際の作業は各部門エージェントが行い、Orchestratorはどのエージェントにどのタスクを割り振るかだけを判断する。

### コマンド体系

CEOがOrchestratorに指示を出すためのコマンド体系も、設計の重要な要素だ。

```
# 日次運用
/ai-ceo:morning          ← 朝ダイジェスト（全部門STATE + 承認待ち + KPI）
/ai-ceo:status           ← 現在の全体状態

# 承認操作
/ai-ceo:approve <id>     ← 承認待ちアイテムを承認
/ai-ceo:reject <id> "理由" ← 却下

# 部門コマンド
/ai-ceo:dev:sprint       ← スプリント計画・実行
/ai-ceo:dev:hotfix "説明" ← 緊急バグ修正
/ai-ceo:mkt:campaign "概要" ← マーケキャンペーン
/ai-ceo:mkt:content-plan ← 月間コンテンツカレンダー
/ai-ceo:sales:proposal "対象" ← 提案書自動生成
/ai-ceo:fin:monthly-report ← 月次決算レポート

# 戦略指示
/ai-ceo:new-product "概要" ← 新プロダクト開発開始
/ai-ceo:pivot "方針"      ← 戦略変更
```

コマンドは名前空間で分類されている。`dev:`は開発部門、`mkt:`はマーケ部門、`sales:`は営業部門、`fin:`は経理部門。CEOはコマンドを入力するだけで、適切な部門エージェントにタスクが委任される。


## .companyディレクトリの構造

AIエージェント組織が機能するためには、「会社の状態」を一元管理する仕組みが必要だ。人間の組織であればSlack、Notion、Jira、Google Driveなど複数のツールに情報が分散するが、AIエージェント組織ではすべてをファイルシステム上のMarkdownファイルで管理する。

### ディレクトリ構造の全体像

```
.company/
├── VISION.md                    # ミッション・ビジョン・コアバリュー
├── STATE.md                     # 現在の経営状態（全社サマリ）
├── ROADMAP.md                   # 四半期ロードマップ
├── approval-queue.md            # 承認待ちキュー
│
├── steering/                    # 経営方針・規約
│   ├── brand.md                 # ブランドガイドライン
│   ├── tech-stack.md            # 技術スタック規約
│   ├── permissions.md           # 権限・閾値設定
│   └── policies.md              # 各種ポリシー
│
├── products/                    # プロダクト別状態
│   ├── sharetoku/STATE.md
│   ├── mochiq/STATE.md
│   ├── focalize/STATE.md
│   ├── genki-button/STATE.md
│   ├── skill-management-saas/STATE.md
│   └── dx-consulting/STATE.md
│
├── departments/                 # 部門別状態
│   ├── dev/STATE.md
│   ├── marketing/STATE.md
│   ├── finance/STATE.md
│   ├── sales/STATE.md
│   ├── cs/STATE.md
│   └── legal/STATE.md
│
└── decisions/                   # 意思決定ログ
    └── 2026-02.md
```

### 各ファイルの役割

#### VISION.md -- 組織の北極星

```markdown
# 合同会社ジョインクラス -- ビジョン & ミッション

## ミッション
「個の開発を促進し、社会の発展に貢献する」

## ビジョン
テクノロジーと教育の力で、企業と個人が持つ可能性を最大限に引き出し、
持続的な成長を実現する社会をつくる。

## コアバリュー
1. 技術で課題を解決する実行力
2. 一人ひとりの成長が社会を変えるという信念
3. 小さくても高品質なプロダクトを届ける
4. 継続的な学習と進化
5. テクノロジーの民主化
```

VISIONは滅多に変わらない。しかし、エージェントが「判断に迷った場合の指針」として参照できるよう、明文化しておくことが重要だ。たとえばCMOエージェントがコンテンツの方向性に迷った時、「個の開発を促進する」というミッションに立ち返ることで、一貫したブランドメッセージを維持できる。

#### STATE.md -- 経営の現在地

STATE.mdは最も頻繁に更新されるファイルだ。全プロダクトの状態、各部門の稼働状況、KPIの現在値がすべて記録されている。

このファイルがあることで、`/ai-ceo:morning`コマンドを実行するだけで、CEOは朝一番に全社の状態を把握できる。

```markdown
## プロダクト状態

| プロダクト | フェーズ | ステータス | 次のマイルストーン |
|-----------|---------|-----------|------------------|
| ShareToku | 運用・成長 | 運用中 | SEO最適化 |
| MochiQ | 運用・改善 | 運用中 | AI学習機能の改善 |
| Focalize | 運用・成長 | LP CVR改善中 | CVR 3-7%達成 |
| 元気ボタン | 運用 | Google Play公開済み | App Store承認待ち |
| ...

## 部門状態

| 部門 | ステータス | 主要タスク |
|------|-----------|-----------|
| 開発 | 稼働 | CI/CD完了、保守・改善 |
| マーケ | 稼働 | 記事6本承認済み、書籍4冊企画承認 |
| 営業 | 準備中 | リード獲得戦略策定済み |
| ...
```

#### ROADMAP.md -- 四半期の航路図

ROADMAPは「今この四半期で何を達成するか」を定義する。マイルストーンごとに担当部門と状態が記録されている。

エージェントはROADMAPを参照することで、「今やるべきタスク」と「後回しにしていいタスク」を判断できる。これがないと、エージェントが優先度を見誤って、重要でないタスクに時間を使ってしまうことがある。

#### permissions.md -- 安全装置

```markdown
## 費用閾値
| アクション | 閾値 | モード |
|-----------|------|--------|
| 自動承認上限 | 5,000円/件 | execute |
| CEO承認必要 | 5,000円超/件 | draft |
| 月間AI運用予算上限 | $200/月 | monitor |

## 開発閾値
### 自動実行可（execute）
- バグ修正、マイナー機能追加、テスト追加、リファクタリング

### CEO承認必要（draft）
- 新機能開発、アーキテクチャ変更、本番デプロイ
```

permissions.mdは組織全体の安全装置だ。金額の閾値、開発の権限レベル、対外コミュニケーションの承認フロー。すべてがここに定義されている。エージェントはアクションを実行する前にこのファイルを参照し、権限レベルに応じて自動実行するか、承認キューに追加するかを判断する。

#### approval-queue.md -- CEOの判断を待つ列

```markdown
# 承認待ちキュー

## 待機中

- [AQ-001] 開発: Focalize本番デプロイ | .company/departments/dev/deploy-focalize.md
- [AQ-002] マーケ: Zenn記事3本公開 | .company/departments/marketing/articles/
- [AQ-003] 営業: DX支援提案書 | .company/departments/sales/proposals/dx-proposal-001.md
```

承認キューは、CEOが朝一番に確認するファイルだ。ここにリストされたアイテムを1件ずつ確認し、`/ai-ceo:approve AQ-001`で承認するか、`/ai-ceo:reject AQ-001 "理由"`で差し戻す。

このフローにより、CEOは「何を判断すべきか」を一箇所で把握でき、判断に集中できる。


## 承認パイプライン -- draft → 承認 → 実行

### パイプラインの全体フロー

```
エージェントがタスクを受け取る
  ↓
permissions.md を参照
  ↓
┌─────────────────────┐
│ 権限チェック         │
│                     │
│ execute? ──→ 自動実行│
│ draft?   ──→ ↓      │
│ read-only? → 自動実行│
└─────────────────────┘
  ↓ (draftの場合)
成果物をdraftファイルとして生成
  ↓
approval-queue.md にアイテム追加
  ↓
CEOに通知（/ai-ceo:morning で確認）
  ↓
CEO が approve / reject
  ↓
┌────────────────────────────┐
│ approve → 実行（デプロイ等）│
│ reject  → 差し戻し・修正   │
└────────────────────────────┘
  ↓
decisions/{month}.md に記録
```

### なぜこの仕組みが必要か

AIエージェントに仕事を任せる上で、最も怖いのは「意図しないアクションが対外的に実行される」ことだ。

- バグのあるコードが本番にデプロイされる
- 不適切な内容のSNS投稿が公開される
- 間違った金額の請求書がクライアントに送られる

こうしたリスクを防ぐのが承認パイプラインだ。対外的な影響があるアクションは、必ずCEOの目を通す。一方で、内部的なアクション（テスト実行、分析レポート生成、ドキュメント更新など）は自動実行を許可する。

この「判断の粒度」が重要だ。すべてを承認必須にすると、CEOがボトルネックになって自動化の意味がなくなる。すべてを自動実行にすると、リスクが管理できない。permissions.mdで閾値を明文化することで、このバランスを最適に保っている。


## 1人社長がCEOとして意思決定に集中できる仕組み

AI-CEO Frameworkが目指すゴールは、**1人社長を「全タスクの実行者」から「意思決定者」に変える**ことだ。

### フレームワーク導入前のCEOの1日

```
実行 : ████████████████████████░░ 92%
判断 : ██░░░░░░░░░░░░░░░░░░░░░░░░ 8%
```

### フレームワーク導入後のCEOの1日

```
実行 : ████░░░░░░░░░░░░░░░░░░░░░░ 15%
判断 : ██████████████░░░░░░░░░░░░ 55%
戦略 : ████████░░░░░░░░░░░░░░░░░░ 30%
```

「実行」が15%に減ったのは、大半のタスクがエージェントに委任されたからだ。CEOが自分で実行するのは、最終的なコンテンツの編集、クライアントとの直接ミーティング、エージェントでは対応できない創造的な作業に限られる。

「判断」が55%に増えたのは、承認パイプラインを通じて意思決定が構造化されたからだ。毎朝の`/ai-ceo:morning`で全社状態を把握し、承認キューの各アイテムに対してapprove/rejectの判断を下す。

「戦略」が30%を占めるようになったのが、最も大きな変化だ。以前はタスクの消化に追われて「そもそも何をすべきか」を考える時間がなかった。今は、プロダクト間の優先順位、新規事業の可能性、中長期の技術選定について、腰を据えて考える時間がある。

### 実際のモーニングルーティン

筆者の現在の朝のルーティンを紹介する。

```
06:00  起床
06:15  /ai-ceo:morning を実行（2分で全社状態を把握）
06:20  承認キューを確認（3-5件、各1-3分で判断）
06:40  今日の戦略的な意思決定（何に注力するか）
07:00  エージェントへの指示出し
       例: /ai-ceo:dev:sprint
       例: /ai-ceo:content:article "Claude Code CLAUDE.md 設計パターン"
07:30  受託案件の開発作業（自分の手を動かす時間）
```

朝の30分で「CEOの仕事」が終わる。その後は、受託案件の開発や、自社プロダクトの創造的な設計作業など、人間にしかできない仕事に集中できる。

エージェントたちは並行して仕事を進め、成果物が出来上がったらapproval-queue.mdに追加される。昼休みや仕事の合間に承認キューを確認して、approve/rejectするだけだ。


## 設計上の重要な判断

最後に、このフレームワークを設計する上で行った重要な判断をいくつか共有する。

### 1. Markdownファイルを選んだ理由

会社の状態管理にデータベースやSaaSではなくMarkdownファイルを使っている理由は3つある。

- **Gitでバージョン管理できる**: いつ、誰が（どのエージェントが）、何を変更したかが追跡できる
- **AIエージェントが直接読み書きできる**: API連携やデータベースクエリが不要
- **人間も読みやすい**: 特別なツールなしで、テキストエディタで確認・編集できる

### 2. エージェント定義をファイルに分離した理由

すべてのエージェント定義をCLAUDE.mdに書くこともできるが、ファイルを分けた。理由は、Thin Orchestratorの原則に基づく。CLAUDE.mdにすべてを書くと、コンテキストウィンドウを圧迫する。`.claude/agents/`にファイルを分離し、Orchestratorは必要な時に必要なエージェントだけを呼び出す設計にした。

### 3. 部門の粒度

6部門は「多すぎないか」という疑問があるかもしれない。実際、CS部門とLegal部門は現時点ではほとんど稼働していない。しかし、最初から組織構造を完成形で定義しておくことで、事業が成長した時にスムーズにスケールできる。逆に、後から部門を追加すると、既存の権限設定やコマンド体系との整合性を取るのが面倒になる。
