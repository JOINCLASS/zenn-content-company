# 朝のダッシュボードと意思決定フロー

## CEO の朝は「5分の儀式」から始まる

経営者の朝は忙しい。メールを確認し、Slackの未読を消化し、各プロジェクトの進捗を確認し、財務状況を把握する。一般的な経営者なら、この「状況把握」だけで1時間は消える。6つのプロダクトを同時運用しているなら、なおさらだ。

AI-CEO Frameworkを導入してから、この朝のルーティンが劇的に変わった。ターミナルを開き、1つのコマンドを打つだけで、全社の状態が1画面に集約される。

```
/ai-ceo:morning
```

所要時間は5分。内訳はこうだ。

- ダッシュボード生成: 30秒（AIの処理時間）
- 内容確認: 2分
- 承認操作: 1-2分
- 指示出し: 1分

この章では、`/ai-ceo:morning` コマンドの設計思想から実装の詳細、そして実際の出力例まで、CEOの朝を変えるダッシュボードの全貌を解説する。

## `/ai-ceo:morning` の設計思想

朝ダッシュボードを設計するにあたって、3つの原則を定めた。

**原則1: 1分で全体像を把握できること**

6部門、6プロダクト、数十のタスクがある。それらすべての詳細を見ていたら朝が終わる。必要なのは「異常検知」だ。正常に動いている部分は「正常です」の一言でいい。問題がある部分だけ目立たせる。信号機の色分け（緑・黄・赤）を採用したのはこの理由からである。

**原則2: アクション可能な情報だけを出すこと**

「登録ユーザー数が増えています」という報告は不要だ。「承認待ちが3件あります。今日中に承認しないと公開スケジュールに影響します」。これが意思決定に繋がる情報である。ダッシュボードに表示されるすべての項目は、CEOの具体的なアクションに紐づいている。

**原則3: 承認操作がその場でできること**

ダッシュボードを見て、別のツールを開いて、該当ファイルを探して、承認する——このフローでは意味がない。ダッシュボード上で `/ai-ceo:approve AQ-003` と打てば承認が完了する。情報確認と意思決定が同一のインターフェースで完結する設計にした。

## 全部門STATE.mdの自動収集

`/ai-ceo:morning` コマンドが最初に行うのは、全部門のSTATE.mdファイルを自動収集することだ。

### 収集対象のファイル構造

```
.company/
├── STATE.md                          # 全社状態
├── approval-queue.md                 # 承認待ちキュー
├── departments/
│   ├── engineering/STATE.md          # 開発部門
│   ├── marketing/STATE.md            # マーケティング部門
│   ├── sales/STATE.md                # 営業部門
│   ├── finance/STATE.md              # 経理部門
│   ├── cs/STATE.md                   # カスタマーサポート
│   ├── legal/STATE.md                # 法務
│   ├── consulting/STATE.md           # コンサルティング
│   └── hr/STATE.md                   # 人事
└── products/
    ├── sharetoku/STATE.md
    ├── mochiq/STATE.md
    ├── focalize/STATE.md
    ├── genkibutton/STATE.md
    ├── skill-tracker/STATE.md
    └── dx-consulting/STATE.md
```

このファイル構造がAI-CEO Frameworkの「状態管理層」を形成している。各サブエージェントがタスクを完了するたびに、自部門のSTATE.mdを更新する。`/ai-ceo:morning` エージェントは、これらのファイルをすべて読み取り、横断的なサマリを生成する。

### STATE.mdの標準フォーマット

各部門のSTATE.mdは以下のフォーマットに統一している。

```markdown
# {部門名} -- 部門状態

## ステータス: {🟢 正常 / 🟡 注意 / 🔴 問題あり / ⚪ 未稼働}

## 進行中タスク
- [ ] {タスク名} — {状態} — 期限: {YYYY-MM-DD}
- [x] {完了タスク} — 完了: {YYYY-MM-DD}

## 注意事項
- {問題や懸念があれば記載}

## KPI（該当部門のみ）
| 指標 | 現在値 | 目標値 | 状態 |
|------|--------|--------|------|

## 最終更新: {YYYY-MM-DD}
```

このフォーマットが統一されていることが重要だ。morningエージェントはマークダウンの構造を頼りに情報を抽出する。フォーマットがバラバラだと、抽出精度が下がり、ダッシュボードの品質が低下する。第11章で詳しく述べるが、このフォーマット統一に至るまでにはいくつかの失敗があった。

### 収集の仕組み

morningエージェントの定義ファイルは `.claude/agents/ai-ceo-morning.md` に格納されている。

```yaml
---
name: ai-ceo-morning
description: 朝ダイジェストエージェント。全部門の状態を収集し、CEO向けサマリを生成する。
tools:
  - Read
  - Grep
---
```

ポイントは、このエージェントがRead（ファイル読み取り）とGrep（検索）しか持たないことだ。Write権限がない。つまり、このエージェントは「見るだけ」のエージェントであり、状態を変更することは決してできない。

これは意図的な設計である。朝のダッシュボード生成時に、何かのファイルが書き換えられてしまうリスクをゼロにしたかった。状態の変更は、CEOの承認操作を通じてのみ行われるべきだ。

## 承認待ちキューの一覧表示

ダッシュボードの最も重要なセクションが「承認待ちキュー」だ。

### 承認キューの仕組み

各部門のサブエージェントが対外的なアクション（記事公開、メール送信、デプロイなど）を準備すると、成果物を生成して `.company/approval-queue.md` に登録する。

```markdown
### [AQ-007] Zenn技術記事「Claude Codeの始め方」公開
- **部門:** マーケ
- **種別:** tech_article
- **ファイル:** /workspace/zenn-content-company/articles/claude-code-getting-started.md
- **概要:** Claude Codeの導入手順を解説する技術記事。Focalizeの事例をCTAとして含む。
- **登録日:** 2026-02-22
- **優先度:** medium
```

morningダッシュボードでは、これがこう表示される。

```
🔔 承認待ち (1件)
  [AQ-007] マーケ: Zenn技術記事「Claude Codeの始め方」公開 | 期限: なし
  → ファイル: /workspace/zenn-content-company/articles/claude-code-getting-started.md
  → 承認: /ai-ceo:approve AQ-007  却下: /ai-ceo:reject AQ-007 "理由"
```

承認操作のコマンドが直接表示されているのがミソだ。CEOはこの行をコピーしてペーストするだけで承認できる。

### 権限レベルと承認フロー

どのアクションが承認を必要とするかは、`permissions.md` で一元管理している。

```yaml
always_draft:           # 常にCEO承認必須
  - press_release
  - pricing_change
  - claim_response
  - proposal
  - contract
  - invoice

auto_after_approval:    # 一度承認されたテンプレートに基づくもの
  - scheduled_sns
  - faq_response
  - tech_article

auto_execute_readonly:  # 自動実行可（分析・レポート系）
  - analytics_report
  - kpi_update
  - competitor_analysis
  - internal_docs
```

金額にも閾値がある。5,000円以下の支出は自動承認、超過するものはCEO承認が必要だ。この閾値は事業規模に応じて調整すればよい。月商が100万円を超えたら、自動承認の上限を10,000円に引き上げるかもしれない。

## KPIサマリの自動生成

ダッシュボードの「プロダクト状態」セクションは、全プロダクトのKPIを一覧表示する。

### KPI収集ロジック

各プロダクトの `STATE.md` にはKPIセクションがある。morningエージェントは、これらのKPIから以下の情報を抽出する。

1. **現在値と目標値の乖離** — 目標に対して何%達成しているか
2. **前回からの変化** — 前日（または前週）と比較して上昇・下降・横ばいか
3. **異常値の検出** — 急激な変動がないか

これらを統合して、以下のようなテーブルを生成する。

```
📦 プロダクト状態
| プロダクト | フェーズ | 進捗 | 次のマイルストーン |
|-----------|---------|------|------------------|
| ShareToku | 運用・成長 | 🟡 ユーザー獲得が目標未達 | SEO最適化の効果測定 |
| MochiQ | 運用・改善 | 🟢 安定稼働中 | AI学習機能の改善リリース |
| Focalize | 運用・成長 | 🟡 登録1名/目標30名 | CVR 3-7%達成 |
| 元気ボタン | 運用 | 🟢 Google Play公開済み | App Store審査完了待ち |
| スキル管理 | 保守モード | ⚪ 最低限保守 | 保守モード維持 |
| DX支援 | 運用 | 🟢 稼働中 | 新規案件獲得 |
```

ここで重要なのは、「保守モード」のプロダクトが目立たないようになっている点だ。積極的に成長を目指しているプロダクトと、維持だけでよいプロダクトでは、見せ方を変えるべきだ。情報の優先度設計がダッシュボードの使い勝手を大きく左右する。

### KPIの異常検知

morningエージェントは単に数字を並べるだけではない。以下のルールで異常を検知する。

- **目標達成率が50%未満**: 黄色信号をつける
- **前週比で20%以上の下落**: 赤信号をつけ、注意事項に記載
- **3週間以上データ更新がない**: 「計測が止まっている可能性」として警告

この異常検知が、CEOの「見落とし」を防ぐ。6つのプロダクトを同時に追っていると、どうしても注意が特定のプロダクトに偏る。AIが全プロダクトを均等にチェックし、人間が見落としやすい「静かに悪化しているプロダクト」を浮かび上がらせてくれる。

## 「今日のおすすめアクション」の提案ロジック

ダッシュボードの最後のセクション「今日のおすすめアクション」は、AIが経営判断の優先順位を提案する部分だ。

### 提案の優先順位ルール

1. **承認待ちがあれば最優先** — 承認の滞留はチーム全体のブロッカーになる
2. **期限付きタスクの期限が近い** — 2日以内に期限を迎えるタスク
3. **長期停滞タスク** — 1週間以上進捗がないタスク
4. **KPI異常** — 目標に対して大幅に未達のプロダクトへの対策
5. **何もなければ** — 「順調です。戦略的タスクに集中しましょう」

このルールは `.claude/agents/ai-ceo-morning.md` に明示的に記述している。AIに「自由に提案して」と言うと、毎回違う基準で提案してきて一貫性がなくなる。ルールを明示することで、毎朝同じ基準で優先順位が判定される。

### 実際の提案例

```
💡 今日のおすすめアクション
1. [承認] AQ-007: Zenn技術記事の公開を承認してください。
   公開予定日は明日です。
   → /ai-ceo:approve AQ-007

2. [対策] Focalizeの登録ユーザーが目標30名に対して1名。
   Instagram広告のCVR改善が急務です。
   → /ai-ceo:mkt:campaign "Focalize CVR改善"

3. [確認] DX支援の新規リード獲得施策が2週間停滞中。
   営業部門の状態を確認してください。
   → /ai-ceo:status
```

各アクションにはコマンドが添えられている。CEOは「なるほど、了解」とコマンドを打つだけでよい。判断→実行のサイクルが極限まで短縮されている。

## morningエージェントの実装詳細

morningエージェントの実装で工夫した点を、もう少し詳しく解説する。

### Thin Orchestratorとの連携

第10章で詳しく述べるが、AI-CEO Frameworkは「Thin Orchestrator」パターンを採用している。メインのOrchestrator（CLAUDE.md）は、ファイルの中身を自分のコンテキストに読み込まない。代わりに、ファイルパスをmorningエージェントに渡し、エージェントが実際のファイル読み取りと要約を行う。

```
[CEO] → /ai-ceo:morning
  ↓
[Orchestrator] → morningエージェントを起動
  ↓
[morningエージェント]
  → .company/approval-queue.md を読み取り
  → .company/departments/*/STATE.md を全て読み取り
  → .company/products/*/STATE.md を全て読み取り
  → .company/STATE.md を読み取り
  → ダッシュボードを生成して返却
  ↓
[Orchestrator] → CEOに表示
```

この分離により、Orchestratorのコンテキストが膨張しない。morningエージェントはタスク完了後にコンテキストが破棄されるため、毎朝クリーンな状態で実行される。

### 曜日による表示内容の変化

平日と休日でダッシュボードの内容を微調整している。

- **月曜日**: 先週の振り返りサマリを追加。週末に変化があったデータを強調
- **金曜日**: 来週の予定を追加。週末中に承認が必要なものがあれば警告
- **月末**: 経理処理のリマインドを追加。月次締めの作業一覧を表示

これらはmorningエージェントのプロンプト内に条件分岐として記述している。エージェント定義がシンプルなマークダウンファイルであるため、こうしたロジックの追加も容易だ。

### エラーハンドリング

ファイルが存在しない場合や、フォーマットが壊れている場合の処理も定義している。

- STATE.mdが存在しない部門: 「⚠ 状態ファイルなし」と表示
- フォーマットが不正: 「⚠ STATE.mdのフォーマットを確認してください」と表示
- 最終更新が7日以上前: 「⚠ 長期間未更新」と表示

これにより、ダッシュボードの生成自体がファイル群のヘルスチェック機能を兼ねている。更新を忘れているファイルや、壊れたフォーマットが自然と発見される。

## CEOの朝5分ルーティン

`/ai-ceo:morning` を中心にした朝のルーティンを、具体的なフローとして整理する。

### ステップ1: ダッシュボード確認（2分）

ターミナルを開いて `/ai-ceo:morning` を実行する。出力されたダッシュボードを上から順に見る。見るべきポイントは3つだけだ。

1. **承認待ちの件数**: ゼロなら次へ。1件以上あれば確認
2. **赤信号の有無**: 部門状態とプロダクト状態で赤がなければ次へ
3. **おすすめアクション**: 最初の1-2件だけ確認

### ステップ2: 承認操作（1-2分）

承認待ちがあれば、内容を確認して承認または却下する。

```
/ai-ceo:approve AQ-007
```

これだけだ。承認するとapproval-queue.mdからアイテムが削除され、decisions/{month}.mdに記録される。却下する場合は理由をつける。

```
/ai-ceo:reject AQ-007 "タイトルのキーワードを変更。'始め方'ではなく'入門'に"
```

却下された場合は、該当部門のエージェントに差し戻し指示が飛ぶ。

### ステップ3: 指示出し（1分）

おすすめアクションを見て、今日やるべきことを決める。緊急性が高いものがあればその場でコマンドを打つ。そうでなければ、頭の中で優先順位を決めて、通常の業務に移る。

### このルーティンの効果

導入前は、各プロダクトのGitHubリポジトリを開き、Slackを確認し、freeeを確認し、Google Analyticsを確認し......と、30分以上かかっていた。それが5分になった。

しかも、情報の「漏れ」がなくなった。以前は忙しいときに特定のプロダクトの確認を忘れることがあった。AIは毎朝すべてのプロダクトを均等にチェックする。人間が苦手な「網羅的で退屈な確認作業」を、AIが肩代わりしてくれている。

## ダッシュボードの進化

最初のダッシュボードは、今よりもずっとシンプルだった。承認待ちと部門状態だけの、テキスト羅列だ。運用を重ねるうちに、以下の改善を加えていった。

### v1: テキスト羅列（初期版）

```
承認待ち: 0件
開発部門: 正常
マーケ部門: 正常
...
```

情報が少なすぎて、結局個別のファイルを見に行く必要があった。

### v2: テーブル追加（1週間後）

テーブル形式にして、進行中タスクと注意事項を追加。情報量は増えたが、まだ「だから何をすべきか」が分からなかった。

### v3: おすすめアクション追加（2週間後）

「今日のおすすめアクション」を追加。これでダッシュボードを見るだけで次のアクションが分かるようになった。

### v4: 承認コマンド直接表示（現在）

承認操作のコマンドをダッシュボード内に直接表示。コピー&ペーストで承認できるようになり、操作がさらに高速になった。

この段階的な改善は、AI-CEO Framework全体に通じるアプローチだ。最初から完璧を目指さず、毎日使う中で足りない部分を足していく。AIシステムの設計は、一度作って終わりではなく、運用しながら育てていくものである。

## ダッシュボードを「経営判断の起点」にする

ダッシュボードは単なるレポートではない。経営判断の「トリガー」として機能させることが重要だ。

### パターン認識による判断支援

毎朝のダッシュボードを2-3週間蓄積すると、パターンが見えてくる。

- Focalizeの登録数が3週連続で未達 → マーケティング施策の根本的な見直しが必要
- 開発部門が2週間ずっと「正常」→ チャレンジングなタスクが振られていない可能性
- 承認待ちが毎朝3件以上 → 権限の閾値を下げて自動承認範囲を広げるべき

こうしたパターンは、将来的にはAI自身が検出して提案するようにしたい。現時点では人間がダッシュボードの推移を見て気づくレベルだが、decisions/{month}.mdにログが蓄積されていけば、AIが過去のパターンと照合して「前回と同じ傾向です。前回はこう対処しました」と提案できるようになるだろう。

### 判断のテンプレート化

ダッシュボードを見て繰り返し行う判断は、ルール化してpermissions.mdに追加していく。

例えば、「技術記事の公開は、レビュー済みならすべて承認」というパターンに気づいたら、permissions.mdの `auto_after_approval` に追加する。すると、次回からは承認キューに上がってこなくなり、自動的に公開される。

CEOの仕事は、毎朝の判断を行うことだけではない。判断のパターンを見抜き、ルール化し、徐々に自分の判断を不要にしていくことだ。これこそがAI-CEO Frameworkの本質的な価値である。

## まとめ

`/ai-ceo:morning` は、AI-CEO Frameworkの「最前線のインターフェース」だ。

- 全部門のSTATE.mdを自動収集し、1画面に集約する
- 承認待ちキューを表示し、その場で承認操作ができる
- KPIの異常を検知し、黄色・赤の信号で知らせる
- 「今日のおすすめアクション」でCEOの判断を支援する
- 所要時間は5分。情報確認→判断→実行のサイクルを極限まで短縮する

たかがダッシュボードと思うかもしれない。だが、経営者にとって「全体像の把握」は最も時間を食い、最も価値の高い活動だ。それをAIが毎朝30秒で生成してくれる。この差は、日々積み重なると途方もなく大きい。
