# 承認パイプラインで品質を担保する

## なぜAIに全権委任してはいけないのか

AI-CEO Frameworkの構築初期、1つの判断ミスを犯しかけた。

「AIエージェントが優秀なら、すべて自動でやらせればいいのではないか」

6部門のエージェントを定義し、各部門が自律的に動く仕組みを設計する中で、この誘惑は常にあった。記事を書いたら自動で公開する。提案書を作ったら自動で送信する。コードを書いたら自動でデプロイする。人間のレビューを挟まなければ、処理速度は飛躍的に上がる。

しかし、これは危険な考えである。以下の3つの失敗シナリオを考えてみてほしい。

**シナリオ1: 事実誤認の記事を公開する**

AIが生成したSEO記事に、競合他社の機能について誤った記述が含まれていたとする。自動公開していた場合、その誤った情報がZennに公開され、競合他社の関係者に見つかり、SNSで炎上する。修正しても、スクリーンショットは残る。

**シナリオ2: 不適切な価格で提案書を送信する**

CSOエージェントが生成した提案書で、本来¥500,000のプロジェクトを¥50,000と記載していた（桁を1つ間違えた）。自動送信していた場合、クライアントはその価格で合意したと主張する可能性がある。

**シナリオ3: 壊れたコードを本番デプロイする**

CTOエージェントが実装したコードにバグがあり、ユーザーの個人データが漏洩する脆弱性を含んでいた。自動デプロイしていた場合、気づくまでの間にデータ漏洩が発生する。

いずれも、人間が一度レビューしていれば防げた事故である。AIの出力は「99%正しいかもしれないが、1%の間違いが致命的になりうる」。この1%のリスクを管理するのが、承認パイプラインの役割である。

## draft/承認/実行の3段階パイプライン設計

AI-CEO Frameworkでは、すべてのアクションを3つの権限レベルで分類している。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  read-only  │────▶│    draft     │────▶│   execute   │
│  (自動実行)  │     │ (CEO承認待ち) │     │  (実行完了)  │
└─────────────┘     └─────────────┘     └─────────────┘
    分析・レポート         対外アクション          内部作業
    KPI集計              デプロイ              バグ修正
    競合調査             メール送信             テスト実行
    社内文書更新          SNS投稿              リファクタリング
                        請求書発行
                        記事公開
```

### read-only（自動実行）

分析レポートの生成やKPIの集計など、外部に影響を与えない読み取り系のアクション。これらは承認不要で自動実行される。

```yaml
auto_execute_readonly:
  - analytics_report    # 分析レポート生成
  - kpi_update          # KPIダッシュボード更新
  - competitor_analysis # 競合調査レポート
  - internal_docs       # 社内ドキュメント更新
```

### draft（CEO承認待ち）

対外的に影響のあるアクション。必ずドラフトとして生成され、CEOの承認を得てから実行される。

```yaml
always_draft:
  - press_release       # プレスリリース
  - pricing_change      # 価格変更の告知
  - claim_response      # クレーム・苦情への対応
  - proposal            # クライアントへの提案書
  - contract            # 契約書・NDA
  - invoice             # 請求書
```

### execute（自動実行可）

閾値内の内部アクション。開発部門のバグ修正やテスト追加など、外部に直接影響しない作業。

```yaml
auto_execute:
  - bugfix              # バグ修正
  - minor_feature       # マイナー機能追加
  - test                # テスト追加・修正
  - refactor            # リファクタリング
  - docs                # ドキュメント更新
  - dependency_patch    # 依存パッケージのパッチ更新
```

## approval-queue.md の構造と運用方法

承認パイプラインの中核となるのが、`.company/approval-queue.md`ファイルである。すべてのdraftアイテムは、このファイルに集約される。


### 承認アイテムのフォーマット

各アイテムは統一フォーマットで登録される。

```markdown
### [AQ-{連番}] {タイトル}
- **部門:** {担当部門}
- **種別:** {draft_type}
- **ファイル:** {成果物のファイルパス}
- **概要:** {1-2文の説明}
- **登録日:** {YYYY-MM-DD}
- **優先度:** {high / medium / low}
```

このフォーマットの設計で重要なのは、**ファイル**フィールドである。CEOは承認判断をする際に、実際の成果物を読んで内容を確認する必要がある。ファイルパスが明記されていることで、「この提案書を読んで問題なければ承認」という判断が迅速にできる。

### 承認・却下の操作

```markdown
### /ai-ceo:approve <id>
1. approval-queue.md から該当アイテムを削除
2. decisions/{month}.md に承認記録を追加
3. 該当部門に実行指示を送信

### /ai-ceo:reject <id> "理由"
1. approval-queue.md から該当アイテムを削除
2. decisions/{month}.md に却下記録と理由を追加
3. 該当部門に差し戻し指示を送信
```

却下の場合は「理由」を必ず付与する。理由なしの却下はしない。これは、AIエージェントが次回のドラフト生成時に改善できるよう、フィードバックを提供するためである。

## 権限レベルの設計 -- permissions.md

permissions.mdは、AI-CEO Framework全体の権限ルールを一元管理するファイルである。

### 費用閾値

```yaml
auto_approve_limit: 5000  # 円。この金額以下は自動承認
ceo_approval_above: 5000  # 円。この金額以上はCEO承認必要
monthly_ai_budget: 200    # USD。月間AI運用予算の上限
```

5,000円という閾値は、1人法人の実態に合わせた設定である。大企業なら数十万円の閾値になるだろうが、当社の規模では5,000円を超える支出はCEOが把握すべきと判断した。

### 開発閾値

```yaml
auto_execute:
  - bugfix              # バグ修正は即座に対応
  - minor_feature       # 小さな機能追加も自動
  - test                # テストの追加・修正
  - refactor            # リファクタリング
  - docs                # ドキュメント更新
  - dependency_patch    # パッチバージョンの依存更新

ceo_approval:
  - new_feature          # 新機能はCEOの方向性確認が必要
  - architecture_change  # アーキテクチャ変更は影響が大きい
  - deploy_production    # 本番デプロイは常にCEO確認
  - major_update         # メジャーバージョン更新はリスクが高い
  - new_dependency       # 新しいライブラリの導入は方針に関わる
  - schema_change        # DB変更は不可逆な場合がある
```

この分類のポイントは、**不可逆性**と**影響範囲**で判断していることである。

- バグ修正やリファクタリング → 問題があればすぐ戻せる → 自動実行可
- 本番デプロイやDB変更 → 戻すのが困難 or ユーザーに影響 → CEO承認必要

### デプロイ権限

```yaml
deploy:
  local: execute       # ローカル環境は自由
  staging: execute     # ステージングも自動デプロイ可
  production: draft    # 本番は常にCEO承認
```

ステージング環境への自動デプロイを許可しているのは、テスト目的であり、外部ユーザーに影響しないためである。本番環境は絶対にCEO承認を必須とする。

### メール送信権限

```yaml
email:
  internal_notification: execute    # 社内通知は自動
  client_communication: draft      # クライアント連絡はCEO確認
  marketing_email: draft           # マーケメールもCEO確認
  system_notification: execute     # システム通知は自動
```

メール送信は特に慎重に扱う。「社内通知」と「システム通知」は自動送信を許可しているが、「クライアント連絡」と「マーケメール」は必ずCEO確認を経る。

### 権限設計のマトリクス

各部門のアクションと権限レベルをまとめると、以下のマトリクスになる。

```
             | read-only | draft | execute |
─────────────┼───────────┼───────┼─────────┤
 開発部門    | コード分析  | 本番デプロイ | バグ修正 |
             | テスト結果  | 新機能     | テスト追加 |
             | 依存監査   | DB変更     | リファクタ |
─────────────┼───────────┼───────┼─────────┤
 マーケ部門   | SEO監査   | 記事公開   | 記事執筆  |
             | 競合分析   | SNS投稿   | SNS下書き |
             | KPI集計   | LP変更     | コピー作成 |
─────────────┼───────────┼───────┼─────────┤
 営業部門    | パイプライン| 提案送付   | 提案書作成 |
             | 分析      | 価格交渉   | 競合調査  |
─────────────┼───────────┼───────┼─────────┤
 経理部門    | コスト分析 | 請求書発行  | 収支レポート|
             | 予算追跡  | 支払い承認  |           |
```

## 対外アクションのガバナンス

### 「対外」の定義

permissions.mdでは、以下を「対外アクション」として定義している。

1. **公開コンテンツ:** Zenn記事の公開、SNS投稿、ブログ更新
2. **顧客コミュニケーション:** 提案書送付、メール送信、契約書
3. **プロダクトリリース:** 本番デプロイ、価格変更
4. **金銭的なアクション:** 請求書発行、支払い実行

これらはすべて、一度実行すると取り消しが困難か、または取り消し自体が信用毀損につながるものである。

### ガバナンスの3原則

**原則1: 対外アクションは必ずdraftモードで生成する**

エージェントの定義ファイルに、権限レベルが明記されている。例えばCMO部門の場合：

```markdown
## 権限レベル

- **execute:** 分析レポート、コンテンツカレンダー、SEO監査、A/Bテスト設計
- **draft:** 記事公開、SNS投稿、広告キャンペーン変更、LP変更のデプロイ
```

「記事公開」「SNS投稿」が明示的にdraftに分類されている。エージェントはこの指示に従い、記事を`published: false`で生成し、approval-queueに追加する。

**原則2: 承認前に実行指示を出してはならない**

CLAUDE.mdのルールとして、以下を明記している。

```markdown
**重要:** 対外に影響のあるアクションを直接実行してはならない。
必ずdraft → 承認 → 実行のパイプラインを通す。
```

この一文が、AI-CEO Framework全体の安全装置として機能している。

**原則3: 承認済みテンプレートに基づく場合のみ自動実行可**

唯一の例外は、CEOが事前に承認したテンプレートに完全一致するケースである。

```yaml
auto_after_approval:
  - scheduled_sns     # 承認済みテンプレートに基づくSNS投稿
  - faq_response      # 承認済みテンプレートに基づくFAQ回答
  - tech_article      # レビュー済みの技術記事
```

たとえば、7週間分のSNSコンテンツカレンダーを一括で承認した場合、個々の投稿はスケジュールどおりに自動投稿できる。ただし、テンプレートから逸脱する内容の場合は、再度draftとして承認を求める。

## decisions/{month}.md による意思決定ログの運用

### なぜ意思決定をログに残すのか

承認・却下の結果は、`.company/decisions/{月}.md`に記録される。これは単なるログではなく、経営判断の蓄積である。

```markdown
# CEO意思決定ログ -- 2026年02月

## 2026-02-17: 技術記事3本の公開を承認 (AQ-001, AQ-002, AQ-003)

- **決定:** コンテンツマーケティング用技術記事3本のZenn/Qiita公開を承認
- **理由:** スキル管理SaaSのユーザー獲得施策として、ターゲット層への
  リーチを開始する
- **影響範囲:** マーケティング部門
- **承認アイテム:**
  - [AQ-001] 「エクセルでのスキル管理がなぜ破綻するのか？」
  - [AQ-002] 「従業員スキルマップを可視化して、プロジェクトアサインを
    最適化する方法」
  - [AQ-003] 「SmartHR・カオナビ不要？小規模企業向けスキル管理ツールの
    選び方」
- **次のアクション:** CEOがZenn/Qiitaに手動で投稿
```

### 意思決定ログの構造

各決定は以下のフォーマットで記録される。

```markdown
## {日付}: {決定の要約}

- **決定:** {何を決定したか}
- **理由:** {なぜその判断をしたか}
- **影響範囲:** {どの部門・プロダクトに影響するか}
- **詳細/承認アイテム:** {具体的な内容}
- **次のアクション:** {この決定を受けて何をするか}
```

「理由」フィールドが最も重要である。半年後、1年後に振り返った時、「なぜあの時この判断をしたのか」を思い出せるようにするためだ。

### 意思決定ログの活用方法

このログは3つの用途で活用される。

**1. 経営判断の一貫性チェック**

過去の決定と矛盾する判断をしていないかを確認できる。たとえば、2月に「スキル管理SaaSの優先度を下げる」と決定した後、3月に同SaaSの大規模機能追加を承認しようとしたら、過去の決定と矛盾していることに気づける。

```markdown
## 2026-02-17: スキル管理SaaS 優先度引き下げ

- **決定:** 保守モードに移行
- **理由:** 動画配信機能の運用ハードルが高く、売上見込みが薄い
```

**2. AIエージェントのコンテキスト提供**

エージェントが過去の意思決定を参照することで、CEOの方針に沿った提案ができる。CFOエージェントが予算配分を提案する際に、「スキル管理SaaSは保守モード」という決定を踏まえて、同SaaSへの追加投資を提案しない、という判断ができる。

**3. 四半期レビューの材料**

Q1の終わりに、decisionsフォルダ内の全決定をレビューすることで、四半期のハイライトと反省点を抽出できる。これはロードマップの更新やQ2の計画策定に直結する。

## 実践例: 記事公開の承認フロー（AQ-001〜006の実例）

ここまでの概念を、実際の承認フローで具体化する。

### Phase 1: ドラフト生成（AQ-001〜003）

Content Engineエージェントが3本のSEO記事を執筆した。

```
Content Engine → 記事3本を執筆
  → articles/draft-excel-skill-management.md
  → articles/draft-skill-visualization.md
  → articles/draft-skill-tool-comparison.md
  → approval-queue.md に3件のアイテムを追加
```

approval-queue.mdに追加された内容：

```markdown
### [AQ-001] 技術記事「エクセルでのスキル管理がなぜ破綻するのか？」
- **部門:** マーケティング
- **種別:** tech_article
- **ファイル:** .company/departments/marketing/articles/draft-excel-skill-management.md
- **概要:** Excel管理の限界を解説し、skill-tracker.jpへのCTAを設置
- **登録日:** 2026-02-17
- **優先度:** high
```

### Phase 2: CEOレビュー

CEOは`/ai-ceo:morning`コマンドで承認待ちアイテムの一覧を確認する。

```
承認待ち (3件)
- [AQ-001] マーケ: 技術記事「エクセルでのスキル管理」
- [AQ-002] マーケ: 技術記事「従業員スキルマップの可視化」
- [AQ-003] マーケ: 技術記事「小規模企業向けスキル管理ツール」
```

各記事のファイルを開き、内容を確認する。確認ポイントは以下のとおり。

- 事実関係に誤りがないか
- 自社プロダクトへのCTAが適切に配置されているか
- 競合他社についての記述が公正か
- ブランドガイドラインのトーンに合っているか

### Phase 3: 承認実行

問題なければ、`/ai-ceo:approve AQ-001`を実行する。

```
/ai-ceo:approve AQ-001
/ai-ceo:approve AQ-002
/ai-ceo:approve AQ-003
```

これにより、以下が自動で行われる。

1. approval-queue.mdから3件のアイテムが削除される
2. decisions/2026-02.mdに承認記録が追加される
3. 記事が公開準備完了の状態になる

## エスカレーションルール

permissions.mdには、異常事態のエスカレーションルールも定義されている。

```markdown
## エスカレーションルール

1. 自動実行タスクが3回連続で失敗した場合 → CEO通知
2. 予算の80%に到達した場合 → CEO通知
3. セキュリティインシデントの疑いがある場合 → 即座にCEO通知
4. ユーザーからのクレーム → 即座にCEO通知
5. OpenClawが許可外リソースへのアクセスを試行した場合 → 即時停止 + CEO通知
```

特に5番目のOpenClaw関連のルールは重要である。OpenClawは自律型のエージェントであり、ファイルシステムやネットワークへのアクセスが許可されている。このエージェントが許可外のリソースにアクセスしようとした場合、即時停止してCEOに通知する。

CLAUDE.mdにも、エラー時の振る舞いを定義している。

```markdown
## エラー時の振る舞い

- サブエージェントが失敗した場合: エラー内容をフィードバックして最大3回リトライ
- 3回失敗: .company/approval-queue.md にエスカレーションとして追加し、CEOに報告
- エラーログ: .company/departments/{dept}/error-log.md に追記
```

3回リトライして失敗したら諦める。無限にリトライさせると、トークン消費が爆発し、月額予算を圧迫する。この3回という上限は、コスト管理の観点からも合理的である。

## 承認パイプラインの設計思想 -- 信頼は段階的に構築する

承認パイプラインの設計思想をまとめると、1つの原則に集約される。

**AIへの信頼は段階的に構築する。**

最初はすべてのアクションをCEO承認にする。運用を重ねてAIの出力品質に確信が持てたら、一部のアクションを自動実行に昇格させる。逆に、問題が発生したら権限を引き下げる。

具体的なステップとして、以下の段階を想定している。

**現在（Phase 1）: 保守的な権限設定**
- 対外アクションはすべてCEO承認
- 内部作業のうち低リスクなもののみ自動実行

**将来（Phase 2）: 実績に基づく権限拡大**
- 過去に問題なく承認された記事のパターンに合致するものは自動公開
- 定型的なコスト追跡レポートは自動実行に昇格
- 承認済みテンプレートに基づくSNS投稿は自動投稿

**さらに将来（Phase 3）: AIの自律性拡大**
- 閾値内の提案書は自動送信
- 閾値内の請求書は自動発行
- ステージング環境のテスト結果が良好な場合の自動本番デプロイ

現時点ではPhase 1にいる。Phase 2への移行は、少なくとも3ヶ月間の問題のない運用実績が前提条件である。

## まとめ -- 承認パイプラインは「ブレーキ」ではなく「品質保証」

承認パイプラインを「面倒な手続き」と捉えるべきではない。これは品質保証の仕組みである。

人間の開発チームにもコードレビューがある。PRをマージする前に、別のエンジニアがコードを読んでフィードバックする。これと同じことを、AIエージェントの出力に対してCEOが行っている。

違いは、レビュー対象がコードだけでなく、記事、提案書、請求書、SNS投稿、デプロイなど、経営活動のすべてに及ぶことである。

**承認パイプラインの3つの効果:**
1. **品質担保:** AIの出力ミスを本番環境に到達させない
2. **意思決定の記録:** すべての判断が理由付きで保存される
3. **段階的な信頼構築:** 実績に基づいてAIの自律性を徐々に拡大できる

次章では、この承認パイプラインの出力を活用する「朝のダッシュボード」について解説する。毎朝5分で全社状況を把握し、承認待ちアイテムを処理し、その日の優先事項を決定する。CEO1名で6部門を統括するための、最も重要な日次ルーティンである。
