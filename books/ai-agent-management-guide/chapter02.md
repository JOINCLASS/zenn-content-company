# 必要なツールと初期費用 -- 最小構成で始めるAIエージェント経営

前章では、AIエージェント経営の全体像と「なぜ1人で6プロダクトを回せるのか」を説明した。本章では、それを実現するための具体的なツール選定と費用を解説する。

「面白そうだけど、何を用意すればいいのかわからない」という方のために、最小構成から始められるガイドをまとめた。結論から言えば、月3万円あれば始められる。月5万円あれば十分に実用的な体制が整う。


## AIエージェント経営の技術スタック

### コアツール: Claude Code

AIエージェント経営の中心に位置するのは、Anthropicが提供する**Claude Code**だ。ターミナル上で動作するAIエージェントで、プロジェクト全体のファイル構造を理解し、コードの読み書きだけでなく、文書作成、分析、意思決定の補助まで行える。

なぜClaude Codeなのか。理由は3つある。

**1. プロジェクト全体を理解する**

Claude Codeはプロジェクトディレクトリ内のすべてのファイルにアクセスできる。経営文書、設定ファイル、コード、データ -- どれも同じコンテキストの中で扱える。ChatGPTのようにファイルを1つずつアップロードする必要はない。

```bash
# Claude Codeはプロジェクトルートで起動するだけ
cd ~/my-company
claude
```

起動した瞬間、Claude Codeはプロジェクト内の構造を把握する。「先月の売上データを分析して」と言えば、該当ファイルを自分で見つけて読み取り、分析結果を返してくれる。

**2. CLAUDE.mdによるカスタマイズ**

プロジェクトルートに`CLAUDE.md`というファイルを置くと、Claude Codeはそれを「指示書」として読み込む。会社のルール、部門構成、プロダクト情報、優先事項 -- すべてをこのファイルに定義しておけば、毎回説明する必要がない。

```markdown
# CLAUDE.md の例（最小版）

あなたは合同会社○○のAI経営アシスタントです。

## 会社の基本情報
- 事業内容: Webアプリケーション開発
- 従業員: 代表1名
- 主要プロダクト: SaaS「○○」

## 参照先
- 経営状態: .company/STATE.md
- 財務データ: .company/departments/finance/
```

このファイルがあるだけで、Claude Codeは「社内の事情を知っている社員」のように振る舞う。

**3. サブエージェントによる並列処理**

Claude Codeは、複数の作業を同時に走らせる「サブエージェント」機能を持つ。開発、マーケティング、経理など、異なる部門のタスクを並列で処理できる。1つのターミナルから、複数の部門に指示を出して結果を集約する -- これがAIエージェント経営の核心だ。

### 他のツールとの比較

AIコーディングツールは複数あるが、「経営タスク」に使う観点で比較するとClaude Codeの優位性が際立つ。

| 観点 | Claude Code | ChatGPT | Cursor | GitHub Copilot |
|------|-------------|---------|--------|----------------|
| ファイルシステムへのアクセス | プロジェクト全体 | 手動アップロード | エディタ内のみ | エディタ内のみ |
| 非コードタスク（文書作成・分析） | 得意 | 得意 | 不向き | 不向き |
| カスタム指示の永続化 | CLAUDE.md | Custom Instructions | .cursorrules | 限定的 |
| ターミナルコマンドの実行 | 直接実行 | 不可 | 限定的 | 不可 |
| バッチ処理・自動化 | 対応 | 不可 | 不可 | 不可 |
| サブエージェント | 対応 | 不可 | 不可 | 不可 |

ChatGPTは汎用的な対話には優れているが、ファイルシステムを直接操作できない。Cursorは開発作業に特化したエディタで、経営文書の管理には向いていない。GitHub Copilotはコード補完が主な用途で、経営タスクの範囲外だ。

Claude Codeが経営タスクに向いている理由をまとめると、「プロジェクト全体を見渡しながら、ファイルの読み書きとコマンド実行ができる唯一のツール」だからだ。経営とは、複数の領域を横断してファイルを管理し、意思決定を行うことに他ならない。その作業モデルに最もフィットするのがClaude Codeである。

1プロダクトから始めて、徐々に部門を増やしていく段階ではこのプランで十分だ。

**Claude Max 20x（$200/月）**

筆者のように6プロダクトを並行で運用し、毎日複数の部門エージェントを動かす場合はこのプランが必要になる。サブエージェントを多用する高度な使い方をする場合や、大量のコンテンツを一気に生成する場合に有効だ。

最初から20xを契約する必要はない。5xで始めて、利用量が増えてきたタイミングでアップグレードすればよい。

**API直接利用（従量課金）**

定期的に自動実行するスクリプト（例: 毎朝の経営ダイジェスト生成、定期的なデータ分析）を組む場合に使う。Claude Code自体はサブスクリプションで使いつつ、一部の自動化処理だけAPIで補完する形が現実的だ。

### 推奨: Max 5xから始める

迷ったら**Claude Max 5x（$100/月）**から始めることを推奨する。理由は以下の通りだ。

1. 日常的な経営タスクに十分な利用量がある
2. サブエージェント機能を制限なく使える
3. 不足を感じたら月単位で20xにアップグレードできる
4. Pro（$20）では制限に引っかかり、中途半端な体験になりやすい


## 補助ツールの選定

Claude Codeだけでもかなりのことができるが、いくつかの補助ツールを併用すると効率が上がる。

### 必須: Git / GitHub

AIエージェント経営では、経営文書もコードもすべてファイルで管理する。変更履歴の追跡にはGitが不可欠だ。

```bash
# Gitがインストールされているか確認
git --version

# インストールされていなければ
# macOS
xcode-select --install

# GitHubアカウントの設定
git config --global user.name "あなたの名前"
git config --global user.email "your@email.com"
```

GitHubを使う理由は3つある。

1. **変更履歴の完全な追跡**: 「いつ、誰が（AIが）、何を変更したか」が常に記録される
2. **ロールバック**: AIが誤った変更を加えた場合、即座に元に戻せる
3. **ブランチ戦略**: 試験的な変更を本番に影響させずに試せる

### 必須: ターミナル

Claude Codeはターミナル上で動作する。以下のいずれかを使う。

| OS | 推奨ターミナル | 備考 |
|----|--------------|------|
| macOS | iTerm2 または標準Terminal.app | iTerm2のほうが視認性と操作性に優れる |
| Windows | Windows Terminal | WSL2（Windows Subsystem for Linux）と併用推奨 |
| Linux | 標準ターミナル | そのまま使える |

ターミナルに慣れていない方も心配はいらない。Claude Code自体が対話的に操作できるため、複雑なコマンドを覚える必要はない。最低限、`cd`（ディレクトリ移動）と`ls`（ファイル一覧表示）がわかれば始められる。

### 推奨: テキストエディタ

Claude Codeが生成・編集したファイルを確認する際に、テキストエディタがあると便利だ。

**VS Code**が最も推奨だ。無料で、ファイルツリーの表示、Markdownプレビュー、Git差分の確認が1画面でできる。Claude Codeで生成したファイルをVS Codeで確認し、必要に応じて手動で微修正を加えるというワークフローが効率的だ。

Cursorを開発用に使っている場合は、そちらでも構わない。

### オプション: MCP連携ツール

MCP（Model Context Protocol）は、Claude Codeが外部サービスと連携するための仕組みだ。導入は任意だが、以下のような連携が可能になる。

| 連携先 | できること | 導入難易度 |
|--------|-----------|-----------|
| Slack | メッセージの送受信、通知 | 低 |
| Google Drive | ドキュメントの読み書き | 中 |
| データベース | データの直接クエリ | 中 |
| freee | 会計データの取得 | 高 |

最初から全部入れる必要はない。まずはClaude Code単体で運用し、「この作業を自動化したい」という具体的なニーズが出てきた段階でMCPを追加していくのが現実的だ。

## 環境構築の手順

では、実際に環境を構築していこう。以下の手順は、macOSを前提に書いているが、Linux/Windowsでも基本は同じだ。

### Step 1: Claude Codeのインストール

Node.js（v18以上）が必要だ。インストールされていない場合は先にインストールする。

```bash
# Node.jsのバージョン確認
node --version

# Node.jsがなければインストール（macOS）
brew install node

# Claude Codeのインストール
npm install -g @anthropic-ai/claude-code
```

### Step 2: 認証設定

Claude Codeを初めて起動すると、認証フローが始まる。

```bash
# 任意のディレクトリでClaude Codeを起動
claude

# ブラウザが開き、Anthropicアカウントでのログインを求められる
# ログイン後、ターミナルに戻れば認証完了
```

Claude MaxプランのサブスクリプションをAnthropic公式サイトで契約しておくこと。認証時に有効なサブスクリプションが紐づいていれば、そのまま利用できる。

### Step 3: プロジェクトディレクトリの作成

AIエージェント経営の「本社」となるディレクトリを作成する。

```bash
# プロジェクトディレクトリの作成
mkdir -p ~/my-company
cd ~/my-company

# Gitリポジトリとして初期化
git init

# 基本ディレクトリ構造の作成
mkdir -p .company/{departments,products,steering,decisions}
```

### Step 4: 最初のCLAUDE.mdの作成

プロジェクトルートに`CLAUDE.md`を作成する。最初は最小限の内容で構わない。

```markdown
# AI経営アシスタント

あなたは株式会社○○のAI経営アシスタントです。
CEOの経営判断を支援し、各部門の業務を代行します。

## 会社の基本情報
- 会社名: 株式会社○○
- 事業内容: （ここに記載）
- 代表者: （ここに記載）
- 設立: （ここに記載）

## 参照すべきファイル
- 経営状態: .company/STATE.md
- 各部門の状態: .company/departments/
- プロダクト情報: .company/products/

## 基本ルール
- 対外的なアクション（メール送信、SNS投稿等）は必ず下書きを作成し、承認を得てから実行する
- 重要な意思決定は .company/decisions/ に記録する
- すべての変更はGitでコミットする
```

ポイントは「基本ルール」のセクションだ。AIに自律的に動いてもらいつつも、対外的な行動は必ず承認を経るようにしておく。この「ガードレール」がないと、AIが勝手にメールを送ったり、SNSに投稿したりするリスクがある。

### Step 5: 動作確認

設定が完了したら、簡単な動作確認を行う。

```bash
# プロジェクトルートでClaude Codeを起動
cd ~/my-company
claude

# 以下のような指示を出してみる
> 会社の現在のディレクトリ構造を確認して、何が足りないか教えて

> .company/STATE.md を作成して。現在は創業直後で、プロダクトはまだない状態

> 今日の日付で decisions/ に意思決定ログファイルを作成して
```

Claude Codeが指示に従ってファイルを作成・編集してくれれば成功だ。ファイルの内容をVS Codeで確認し、問題なければGitでコミットしておく。

```bash
git add .
git commit -m "AIエージェント経営の初期構成を作成"
```


## ディレクトリ構成の設計

環境構築の最後に、`.company/`ディレクトリの設計思想を解説する。

### 基本構造

```
.company/
├── STATE.md              # 会社全体の現在の状態（KPI、フェーズ等）
├── VISION.md             # ミッション・ビジョン
├── ROADMAP.md            # 四半期ロードマップ
├── approval-queue.md     # 承認待ちアイテムの一覧
├── departments/          # 部門別の情報
│   ├── dev/              # 開発部門
│   │   └── STATE.md
│   ├── marketing/        # マーケティング部門
│   │   └── STATE.md
│   ├── finance/          # 経理・財務部門
│   │   └── STATE.md
│   ├── sales/            # 営業部門
│   │   └── STATE.md
│   ├── cs/               # カスタマーサポート部門
│   │   └── STATE.md
│   └── legal/            # 法務部門
│       └── STATE.md
├── products/             # プロダクト別の情報
│   └── my-product/
│       └── STATE.md
├── steering/             # 全社的なルール・方針
│   ├── brand.md          # ブランドガイドライン
│   ├── tech-stack.md     # 技術スタック定義
│   ├── permissions.md    # 権限・閾値設定
│   └── policies.md       # 社内ポリシー
└── decisions/            # 意思決定ログ（月次）
    └── 2026-03.md
```

### 各ディレクトリの役割

**departments/**

6つの部門それぞれが独自の`STATE.md`を持つ。各部門のAIエージェントは、自分の担当ディレクトリ内の情報を参照して業務を遂行する。部門間の情報はオーケストレーター（CLAUDE.md）が橋渡しする。

**products/**

プロダクトが増えるたびにサブディレクトリを追加する。各プロダクトの状態、設計書、バックログなどをここに格納する。

**steering/**

全社的なルールや方針を定義する。ブランドガイドライン、技術スタック、権限設定など、どの部門のエージェントも参照する共通設定だ。

**decisions/**

月ごとにファイルを作り、重要な意思決定とその理由を記録する。「なぜこの判断をしたのか」を後から振り返るためのログだ。

### なぜファイルベースで管理するのか

データベースやSaaSツールではなく、テキストファイル（Markdown）で経営情報を管理する理由は3つある。

**1. Git追跡が可能**

すべてのファイルがGitで管理されるため、いつ、何が、どう変わったかの完全な履歴が残る。「先月の経営状態と今月を比較したい」「あの判断をいつ行ったか確認したい」といったニーズに即座に応えられる。

```bash
# 経営状態の変更履歴を確認
git log --oneline .company/STATE.md

# 特定時点の状態を確認
git show HEAD~5:.company/STATE.md
```

**2. AIが読み書きしやすい**

テキストファイルはAIにとって最も扱いやすいフォーマットだ。JSONやYAMLのようにフォーマットが厳密すぎず、かといって非構造的でもない。Markdownは見出し、リスト、表によって十分な構造を持ちながら、自然言語として読み書きできる。

**3. 透明性と可搬性**

特定のSaaSに依存しないため、ベンダーロックインのリスクがない。ファイルをテキストエディタで開けば誰でも中身を確認できる。SaaSのダッシュボードを見なくても、`cat .company/STATE.md`で会社の状態がわかる。

この設計は、一見するとローテクに見えるかもしれない。しかし、Gitという数十年の実績を持つバージョン管理システムの上に経営情報を置くことは、実は極めて堅牢な選択だ。データベースのクラッシュやSaaSのサービス終了に怯える必要がない。
