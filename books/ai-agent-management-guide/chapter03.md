# 6部門エージェントの設計図 -- AIに「組織」を作る

前章では、CLAUDE.mdというたった1つのファイルが「AI経営チーム」の核になることを見てきた。この章では、そのチームに「部門」という構造を持たせる設計を解説する。

ここが本書の技術的な核心部分だ。なぜ1つのAIに全部任せるのではなく、6つの専門部門に分けるのか。そしてそれぞれの部門をどう定義し、どう連携させるのか。実際に稼働しているコードと設計図を見せながら説明する。

## なぜ「部門」で分けるのか

### 人間の組織と同じ原理

会社組織がなぜ「経理部」「営業部」「開発部」に分かれているのか。答えは単純で、1人の人間がすべての業務を高いレベルでこなすのは不可能だからである。経理のプロに営業をやらせても成果は出ないし、その逆も然りだ。

AIエージェントでもこの原理はまったく同じように働く。

### 1つのAIに全部やらせるとどうなるか

最初は筆者もそうしていた。CLAUDE.mdに「あなたは会社の全業務を担当するAIアシスタントです。開発、マーケティング、営業、経理、カスタマーサポート、法務のすべてを管理してください」と書いていた時期がある。

結果は散々だった。具体的には3つの問題が発生する。

**1. コンテキスト汚染**

Claude Codeのコンテキストウィンドウには上限がある。経理の仕訳データ、営業の顧客リスト、開発のコード、マーケのSNS投稿案を1つのセッションに詰め込むと、あっという間にコンテキストが埋まる。そうなると、AIの回答品質が目に見えて劣化する。「さっき話した経費の件、もう一度確認して」と言っても、既に開発コードの議論で上書きされていて、正確に思い出せない。

**2. ペルソナの混乱**

経理担当者に求められる慎重さと、マーケ担当者に求められる攻めの姿勢は正反対だ。1つのAIに両方の役割を与えると、どっちつかずの中途半端な回答が増える。SEO記事を書かせたいのに「リスク管理の観点から...」と経理モードの慎重な文章が出てきたりする。

**3. 指示の肥大化**

すべての業務をカバーしようとすると、CLAUDE.mdが膨大になる。筆者の場合、1つのファイルに500行以上の指示を書いていた時期があった。当然、AIがすべての指示を正確に守ることは不可能で、重要なルールが無視されることが頻発した。

### Thin Orchestratorパターン

この問題を解決するのが、「Thin Orchestrator（薄い統括者）」パターンである。

考え方はシンプルだ。ルートのCLAUDE.md（統括者）は、自分では何も作業しない。各部門エージェントに指示を出し、結果を受け取って判断するだけ。人間のCEOと同じ役割である。

```
CEO（あなた）
  ↓ 指示
C-Suite Orchestrator（ルートCLAUDE.md）
  ↓ 委任          ↓ 委任         ↓ 委任
開発部門        マーケ部門      営業部門 ...
```

統括者のコンテキスト使用率を10-15%に抑えることで、常に余裕を持って判断できる状態を維持する。実作業はすべて専門の部門エージェントに任せる。これが「Thin」の意味だ。

## 6部門の全体設計

筆者が実際に運用している6つの部門を紹介する。スモールビジネスにおいて、これが必要十分な構成だと考えている。

### 各部門の役割一覧

| 部門 | 略称 | 責任者ロール | 主要タスク |
|------|------|------------|-----------|
| 開発部門 | dev | CTO | プロダクト開発、バグ修正、コードレビュー、デプロイ管理 |
| マーケティング部門 | marketing | CMO | コンテンツ作成、SEO、SNS運用、広告分析、LP最適化 |
| 営業部門 | sales | CSO | 提案書作成、リード管理、アウトリーチメール、成約管理 |
| 経理・財務部門 | finance | CFO | 請求書管理、経費管理、収支レポート、税務準備 |
| カスタマーサポート部門 | cs | CS Lead | FAQ管理、問い合わせ対応、エスカレーション管理 |
| 法務部門 | legal | Legal Counsel | 契約書レビュー、利用規約管理、コンプライアンス確認 |

### 各部門の入出力マップ

どの部門が何を入力として受け取り、何を出力するかを明確にしておくと、部門間の連携がスムーズになる。

| 部門 | 主な入力 | 主な出力 |
|------|---------|---------|
| 開発 | 機能要件、バグ報告、技術スタック定義 | コード、デプロイ済み機能、テスト結果 |
| マーケ | プロダクト情報、アクセス解析データ | 記事、SNS投稿、広告レポート、LP改善案 |
| 営業 | リードリスト、プロダクト情報、価格表 | 提案書、アウトリーチメール、成約レポート |
| 経理 | 取引データ、請求情報、経費レシート | 仕訳データ、請求書、月次レポート、確定申告書類 |
| CS | 問い合わせ内容、FAQデータベース | 回答ドラフト、FAQ更新、エスカレーション通知 |
| 法務 | 契約書ドラフト、利用規約、OSSリスト | レビュー結果、リスク指摘、改善提案 |

この入出力を見ると、部門間でデータが流れていることがわかるだろう。例えば、開発部門が新機能をリリースすれば、その情報がマーケ部門の入力になり、告知記事やSNS投稿が出力される。営業部門が成約すれば、経理部門の入力になり、請求書が出力される。

### 各階層の役割

**ルートCLAUDE.md（統括者）** は、CEOコマンドの受付と部門への委任だけを行う。ここに書くのは「どのコマンドをどの部門に渡すか」のルーティングルールだ。統括者が自分でコードを書いたり、記事を書いたりすることは絶対にない。

**.claude/agents/（部門エージェント）** は、各部門の専門知識とワークフローを定義する。Claude Codeのサブエージェント機能を使って、統括者から呼び出される。それぞれが独立したコンテキストを持つため、互いのデータで汚染されることがない。

**.company/（共有データ領域）** は、全部門がアクセスする共有ファイルシステムだ。各部門は自分の `departments/{dept}/` 配下で作業し、成果物をここに出力する。他部門はそのファイルを読み取って自分の作業に使う。これがファイルベースの部門間連携の仕組みである。

### データフロー

```
CEO → ルートCLAUDE.md → .claude/agents/{dept}.md → 作業実行
                                                       ↓
                                              .company/departments/{dept}/
                                                       ↓
                                              他部門が参照して連携
```

重要なのは、部門エージェント同士が直接やり取りしない点だ。必ず共有ファイルシステム（`.company/`）を介して連携する。これにより、各部門の独立性が保たれ、デバッグや改善がしやすくなる。

## エージェント定義の書き方

ここが実務上もっとも重要なパートだ。部門エージェントの定義ファイルに何を書くかで、AIの仕事の質が決まる。

### 定義ファイルの構造

すべてのエージェント定義は、以下の5つの要素で構成される。

1. **役割の宣言** -- 「あなたは何者か」を明確に伝える
2. **専門知識の列挙** -- その部門に必要な知識領域を具体的に書く
3. **入力と出力の定義** -- 何を受け取り、何を生成するか
4. **権限レベル** -- 何を自動実行でき、何に承認が必要か
5. **品質基準** -- 成果物の検証方法

順番に解説する。

### 要素1: 役割の宣言

最初の一文が最も重要だ。AIに「あなたは誰で、何をする人か」を明確に伝える。

悪い例:
```
あなたはAIアシスタントです。開発関連のタスクを手伝ってください。
```

良い例:
```
あなたは AI-CEO Framework の CTO（最高技術責任者）です。
経験豊富なテックリード。実用性を重視し、オーバーエンジニアリングを避ける。
「動くものを最速で出し、フィードバックを得て改善する」がモットー。
```

違いは明確だ。良い例では、役職名、性格、判断基準まで定義している。これにより、AIの回答のトーンと意思決定の方向性が安定する。

### 要素2: 専門知識の列挙

その部門が持つべき知識を具体的にリストアップする。AIは「何を知っているか」を明示されると、その知識を積極的に活用するようになる。

### 要素3: ワークフローの定義

各コマンドに対して、ステップバイステップの手順を書く。これが曖昧だと、AIが毎回異なる手順で作業し、品質がばらつく。

### 具体例: 開発部門エージェント（CTO）

実際に稼働しているCTOエージェントの定義を見てみよう。

```markdown
---
name: ai-ceo-cto
description: CTO/開発部長エージェント。プロダクト開発全般を統括する。
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---

# CTO / 開発部長エージェント

あなたは AI-CEO Framework の CTO（最高技術責任者）です。

## ペルソナ

経験豊富なテックリード。実用性を重視し、オーバーエンジニアリングを避ける。
「動くものを最速で出し、フィードバックを得て改善する」がモットー。

## 担当領域

- プロダクト開発全般（設計、実装、テスト、デプロイ）
- 技術的意思決定
- スプリント管理
- コードレビュー・セキュリティ

## 権限レベル

- **execute:** コーディング、テスト実行、ステージングデプロイ、内部ドキュメント
- **draft:** 本番デプロイ、アーキテクチャの大幅変更、新規ライブラリ導入

## 参照ファイル

- 技術スタック: `.company/steering/tech-stack.md`
- プロダクト状態: `.company/products/{name}/STATE.md`
- 開発部門状態: `.company/departments/dev/STATE.md`

## ワークフロー

### /ai-ceo:dev:sprint
1. バックログからタスクを選定（最大3タスク）
2. 各タスクの仕様をCC-SDD式で作成
3. Wave パターンで実装（依立タスクは並列実行）
4. コードレビュー + テスト実行
5. 部門STATE.mdを更新

## 品質検証

1. **ゴール逆算検証:** テストがパスするか
2. **tech-stack.md 準拠チェック:** 使用技術が規約に合っているか
3. **セキュリティチェック:** 基本的なベストプラクティスの遵守
```

注目してほしいポイントがいくつかある。

**frontmatter（`---`で囲まれた部分）** で、エージェント名と使用するツールを宣言している。Claude Codeはこのメタデータを解釈してエージェントの能力を制御する。

**参照ファイル** で、このエージェントが読むべきファイルを明示している。これにより、不要なファイルを読み込んでコンテキストを浪費することを防ぐ。

**ワークフロー** は番号付きのステップで書かれている。AIはこの手順に沿って作業するため、毎回同じ品質の成果物が出てくる。

### 具体例: マーケティング部門エージェント（CMO）

```markdown
---
name: ai-ceo-cmo
description: CMO/マーケティング部長エージェント。コンテンツ戦略、SEO、SNS運用を統括する。
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---

# CMO / マーケティング部長エージェント

あなたは AI-CEO Framework の CMO（最高マーケティング責任者）です。

## ペルソナ

データドリブンなグロースマーケター。技術プロダクトのマーケティングに精通し、
コンテンツマーケティング・SEO・SNS・有料広告を統合的に運用する。
「計測できないものは改善できない」がモットー。

## 専門知識

### SEO
- テクニカルSEO（Core Web Vitals, 構造化データ, サイトマップ）
- コンテンツSEO（キーワード戦略, E-E-A-T, 検索意図分析）

### コンテンツマーケティング
- Zenn/Qiita/note の各プラットフォーム特性と投稿戦略
- 技術記事のSEOライティング（構成設計、CTA配置、内部リンク）
- 有料書籍（Zenn Books）の企画・価格戦略

### 広告運用
- Meta広告のターゲティング・クリエイティブ最適化
- Google広告のキーワード戦略
- 広告 → LP → CVのファネル分析と改善

## 権限レベル

- **execute:** 分析レポート、コンテンツカレンダー、SEO監査
- **draft:** 記事公開、SNS投稿、広告キャンペーン変更、LP変更

## ワークフロー

### /ai-ceo:mkt:content-plan
1. 各プロダクトのSTATE.mdから訴求ポイントを抽出
2. SEOキーワード分析に基づいて記事テーマを選定
3. 月間カレンダーを生成（Zenn記事 週1本 + X投稿 平日毎日）
4. 記事公開 → X告知 → 有料書籍誘導の導線を設計

## 品質検証

1. **データ根拠:** すべての施策提案はデータに基づいているか
2. **ROI意識:** 費用対効果が明確か
3. **ブランド整合:** brand.mdのトーンに合っているか
```

CTOエージェントと比較すると、ペルソナがまったく異なることがわかる。CTOは「動くものを最速で」という実装寄りの性格だが、CMOは「計測できないものは改善できない」というデータ寄りの性格だ。この違いが、同じAIモデルから異なる品質の成果物を引き出す鍵になる。

## 承認パイプラインの設計

6部門のエージェントが自律的に動くとして、最大の問題は「勝手にメールを送ったり、コードを本番にデプロイしたりしないか」である。

答えは、承認パイプラインで制御する。

### なぜ承認が必要か

AIエージェントに完全な自律性を与えるのは危険だ。特に以下のケースでは人間のチェックが不可欠である。

- **対外コミュニケーション:** 顧客へのメール、SNS投稿、プレスリリース
- **金銭に関わる操作:** 請求書発行、価格変更、広告予算の変更
- **不可逆な操作:** 本番デプロイ、データベーススキーマ変更、契約締結

逆に、以下のケースでは承認なしで自動実行させて問題ない。

- **分析・レポート生成:** アクセス解析、KPIダッシュボード更新
- **内部ドキュメント:** 社内メモ、議事録、タスク管理
- **軽微な修正:** バグ修正、テスト追加、リファクタリング

### 3段階の権限レベル

この考え方を整理すると、3つの権限レベルに分類できる。

```yaml
# .company/steering/permissions.md の核心部分

# 承認不要で自動実行
auto_execute:
  - bugfix              # バグ修正
  - test                # テスト追加
  - refactor            # リファクタリング
  - analytics_report    # 分析レポート
  - internal_docs       # 社内ドキュメント

# 必ずCEO承認が必要
always_draft:
  - proposal            # 提案書
  - contract            # 契約書
  - invoice             # 請求書
  - press_release       # プレスリリース
  - deploy_production   # 本番デプロイ

# 一度承認されたテンプレートに基づくものは自動実行
auto_after_approval:
  - scheduled_sns       # スケジュール済みSNS投稿
  - faq_response        # FAQ定型回答
  - tech_article        # レビュー済み技術記事
```

この3段階により、「安全なことは高速に、危険なことは慎重に」を実現する。

### approval-queue.md の仕組み

draftモードで生成された成果物は、すべて `approval-queue.md` に登録される。

```markdown
# 承認待ちキュー

## 待機中

- [AQ-042] 営業部門: キングループ様への提案書
  | ファイル: .company/departments/sales/proposals/king-group-proposal.md
  | 作成日: 2026-03-04
  | 優先度: 高

- [AQ-043] マーケ部門: X投稿（AI経営の記事告知）
  | ファイル: .company/departments/marketing/drafts/x-ai-ceo-post.md
  | 作成日: 2026-03-04
  | 優先度: 中
```

CEOは毎朝の `/ai-ceo:morning` コマンドでこのキューを確認し、以下のコマンドで承認・却下する。

```
/ai-ceo:approve AQ-042      # 承認 → 実行可能に
/ai-ceo:reject AQ-043 "トーンがカジュアルすぎる。もう少し専門的に"  # 却下 → 差し戻し
```

却下された場合、該当部門のエージェントがフィードバックを受けて修正版を作成し、再度キューに登録する。このサイクルが品質を担保する仕組みだ。

### 権限マトリクス

各部門の権限をまとめると以下のようになる。

| 部門 | execute（自動実行） | draft（承認必要） |
|------|-------------------|-----------------|
| 開発 | バグ修正、テスト、リファクタ、ステージングデプロイ | 本番デプロイ、新機能、アーキテクチャ変更 |
| マーケ | 分析レポート、SEO監査、カレンダー作成 | 記事公開、SNS投稿、広告変更、LP変更 |
| 営業 | リード分析、競合調査 | 提案書送付、契約書、アウトリーチメール |
| 経理 | 仕訳チェック、経費分析 | 請求書発行、振込実行、税務申告 |
| CS | FAQ検索、内部メモ | 顧客への回答、エスカレーション対応 |
| 法務 | 契約書分析、OSSライセンス監査 | 契約書送付、利用規約変更 |

法則は明快だ。**社外に出るものはすべてdraft、社内で完結するものはexecute。** これだけ覚えておけば、新しいタスクが出てきたときにもどちらの権限にすべきか迷わない。

## 部門間連携の設計

6つの部門がバラバラに動いていても意味がない。実際のビジネスでは、1つの案件に複数の部門が関わる。

### 実例: HP制作の営業フロー

筆者が実際に運用しているHP制作の営業フローを例に説明する。

```
1. 営業部門: リードを分析し、提案書を作成（draft）
2. CEO: 提案書を承認
3. 営業部門: 承認済み提案書をメールで送付（draft）
4. CEO: メール文面を承認
5. マーケ部門: 成約した場合、事例記事の下書きを作成
6. 経理部門: 請求書を発行（draft）
7. CEO: 請求書を承認
```

このフローでは、営業→マーケ→経理と3つの部門が順番に連携している。各ステップの成果物は `.company/` 配下のファイルとして保存されるため、次の部門はそのファイルを読み取るだけで作業を開始できる。

### Wave実行: 並列処理で高速化

すべてのタスクが順番に実行される必要はない。依存関係のないタスクは同時に実行できる。

```
Wave 1（並列実行）:
  - 開発部門: 新機能のステージングデプロイ
  - マーケ部門: 今月のコンテンツカレンダー作成
  - 経理部門: 先月の経費レポート生成

Wave 2（Wave 1の結果に依存）:
  - マーケ部門: 新機能のリリース記事を作成（← 開発部門の完了を待つ）
  - 営業部門: 新機能を盛り込んだ提案書を更新（← 開発部門の完了を待つ）

Wave 3（Wave 2の結果に依存）:
  - マーケ部門: X投稿でリリースを告知（← リリース記事の完了を待つ）
```

Wave 1の3タスクは互いに依存関係がないため、同時に実行される。Wave 2は開発部門の成果物が必要なので、Wave 1の完了後に実行される。このパターンにより、6部門すべてを直列に実行する場合と比べて、処理時間を大幅に短縮できる。

### ファイルベース連携の利点

部門間の連携をファイルベースにしている理由は3つある。

**1. 追跡可能性:** すべてのやり取りがファイルとして残るため、「誰が何をいつ作ったか」が完全に追跡できる。Gitで管理すれば変更履歴も残る。

**2. 非同期処理:** 部門エージェントは同時に動く必要がない。営業部門が提案書を `.company/departments/sales/proposals/` に保存しておけば、マーケ部門は好きなタイミングでそれを読み取れる。

**3. デバッグの容易さ:** 問題が起きたとき、中間ファイルを確認すれば、どの部門のどの段階でおかしくなったかがすぐにわかる。APIベースの連携と違って、データが見える形で残っている。

## この章のまとめ

- 1つのAIに全部やらせるとコンテキスト汚染、ペルソナ混乱、指示肥大化が起きる
- Thin Orchestratorパターンで「統括者は指示だけ、実作業は部門エージェント」に分離する
- 6部門（開発、マーケ、営業、経理、CS、法務）がスモールビジネスの必要十分な構成
- エージェント定義は「役割、専門知識、入出力、権限、品質基準」の5要素で書く
- 承認パイプラインで「社外に出るものはdraft、社内はexecute」を徹底する
- 部門間はファイルベースで連携し、Wave実行で並列処理する

次章では、この6部門構成を実際に動かすための「1日の運用フロー」を解説する。朝のダイジェスト確認から、承認処理、各部門への指示出しまで、筆者の実際の運用を時系列で追っていく。
