# 失敗事例とトラブルシューティング -- やってはいけない10のこと

## 失敗は最良の教師である

ここまで本書を読んで、AI-CEO Frameworkの構築方法を理解していただけたと思う。しかし、実際に運用を始めると、予想しなかった問題に次々と直面する。

本章では、筆者が実際に犯した失敗と、他の実践者から聞いた失敗事例を10個紹介する。それぞれについて「何が起きたか」「なぜ起きたか」「どう解決したか」を具体的に示す。

読者がこの10の失敗を事前に知っておくことで、同じ轍を踏む確率を大幅に下げられるはずだ。


## 失敗1: CLAUDE.mdに全部書こうとしてコンテキスト爆発

### 何が起きたか

AI-CEO Frameworkの初期バージョンでは、すべての部門定義、ルール、テンプレート、ワークフローを1つのCLAUDE.mdに詰め込んでいた。ファイルのサイズは8,000行を超えた。

結果、Claude Codeのコンテキストウィンドウが圧迫され、タスク実行中に「以前の指示を忘れる」現象が頻発した。営業メールを書かせているのにマーケのブランドガイドラインを適用したり、開発のコーディング規約が無視されたりした。

### なぜ起きたか

「1つのファイルにすべて書けば、AIは全部理解してくれる」という誤解があった。しかし、AIのコンテキストウィンドウには限界がある。情報量が多すぎると、重要な指示が埋もれてしまう。

### どう解決したか

**Thin Orchestratorパターン**を採用した。CLAUDE.mdには「ルーティング情報」だけを書き、各部門の詳細は `.claude/agents/` のサブエージェント定義に分離した。

```
修正前: CLAUDE.md（8,000行、全部門の定義を含む）
修正後: CLAUDE.md（200行、ルーティングのみ）
        + 9つのサブエージェント定義（各200-400行）
```

CLAUDE.mdのコンテキスト使用率は10-15%に抑えるのが目安だ。残りの85-90%をタスク実行に使えるようにする。


## 失敗2: 承認パイプラインなしで対外アクション実行

### 何が起きたか

「速く動きたい」という焦りから、営業メールの送信を自動化してしまった時期がある。エージェントが生成したメールを人間の確認なしで送信する設定にしたのだ。

3日後、ある見込み客から「同じ内容のメールが2通届いている」と指摘を受けた。さらに別の見込み客に送ったメールでは、会社名が間違っていた。

### なぜ起きたか

エージェントは「タスクを実行する」ことに忠実だ。重複チェックや最終確認は、明示的に指示しないと行わない。また、AIは似たような名前の会社を混同することがある。

### どう解決したか

対外アクションは**例外なくdraftモード**にした。「面倒でも、外に出るものは人間が見る」というルールを絶対化した。

このルールを破りたくなる瞬間は何度もある。しかし、1通の誤送信メールが会社の信用を大きく損なうリスクを考えれば、2-3分の確認時間は安い保険だ。


## 失敗3: AIの出力を無検証で使用（幻覚問題）

### 何が起きたか

技術記事の執筆をエージェントに任せた際、記事内に「React 20.0のuseServerフックを使うと...」という記述が含まれていた。しかし、そのバージョンのReactは存在しなかった。AIが「もっともらしいが存在しない情報」を生成していたのだ。

記事を公開してから読者に指摘され、修正と謝罪が必要になった。

### なぜ起きたか

いわゆる「ハルシネーション」（幻覚）の問題だ。AIは確率的にテキストを生成するため、事実と異なる情報を自信たっぷりに出力することがある。特に技術情報のバージョン番号、日付、統計データは要注意だ。

### どう解決したか

品質チェックリストに**「事実確認」の項目を追加**した。

```markdown
## 記事公開前チェック
- [ ] 技術情報のバージョンが正しいか
- [ ] 引用したデータのソースが確認できるか
- [ ] リンク先が存在するか
- [ ] コード例が実際に動作するか
```

また、エージェントの定義に「不確かな情報には『要確認』マークをつける」というルールを追加した。完全にハルシネーションを防ぐことはできないが、チェックの仕組みを入れることでリスクを大幅に低減できる。


## 失敗4: 全部門を一度に構築しようとして中途半端

### 何が起きたか

本書の構成を見て「6部門を一気に構築しよう」と考える読者もいるかもしれない。筆者も最初はそうした。

結果、どの部門も中途半端な状態で止まった。開発部門の定義は書いたが品質基準が曖昧。マーケ部門のテンプレートは作ったがカレンダー運用が始まっていない。営業部門のリストはあるがメールテンプレートがない。

1ヶ月後、6部門のどれもまともに動いていなかった。

### なぜ起きたか

部門の構築には、「定義→テスト→調整→安定運用」のサイクルが必要だ。1部門を安定させるのに最低2-3週間かかる。6部門を同時に始めると、調整のフィードバックループが回らない。

### どう解決したか

**1部門ずつ、2週間かけて構築**する方針に変更した。

```
推奨順序:
Week 1-2: 開発部門（最もAIとの親和性が高い）
Week 3-4: マーケ部門（コンテンツ量産を開始）
Week 5-6: 営業部門（リード獲得を開始）
Week 7-8: 経理部門（数字の管理を開始）
Week 9-10: CS・法務部門（最小構成で開始）
```

1部門が安定してから次に進む。この「逐次構築」が結局は最速だ。


## 失敗5: コスト管理を怠ってAPI費用が膨張

### 何が起きたか

サブエージェントを多用する設計にしたところ、Claude APIの利用量が想定の3倍になった月があった。1つのタスクで「architect→coder→reviewer」と3回のAPI呼び出しが発生し、さらにリトライが入ると6-9回になることもあった。

月末にAPIの請求を見て青ざめた。

### なぜ起きたか

サブエージェントの呼び出し回数を考慮せずに設計していた。特にリトライの回数制限を設定していなかったのが問題だった。

### どう解決したか

```markdown
## コスト管理ルール

1. サブエージェントの呼び出しは1タスクあたり最大3回
2. リトライは3回まで（超えたらエスカレーション）
3. API費用を週次で確認（cost-tracking.mdに記録）
4. 月間API予算を設定し、80%到達時にアラート
5. Claude Maxプラン（定額）をメインに使い、API利用を最小化
```

Claude Maxのような定額プランをメインに使い、サブエージェント用のAPI呼び出しを最小限に抑える設計にしたことで、コストが安定した。


## 失敗6: エージェントの役割が曖昧で品質低下

### 何が起きたか

「何でもできるエージェント」を1つ作ろうとした。開発もマーケも営業もこなす汎用エージェントだ。

結果、どのタスクも60点くらいの品質になった。技術記事は深みが足りず、営業メールはトーンが技術寄りになり、コードレビューは表面的なチェックしかできなかった。

### なぜ起きたか

エージェントの定義が曖昧だと、AIは「何に集中すべきか」が分からない。人間でも「何でも屋」は各分野の専門家に劣る。AIも同じだ。

### どう解決したか

**1エージェント1責務**の原則を徹底した。各エージェントの定義に「あなたは何をする人か」「何をしない人か」を明記する。

```markdown
## 開発エージェントの責務
- する: コード実装、テスト生成、コードレビュー、リファクタリング
- しない: 記事執筆、メール作成、請求書作成、法務判断
```

この「しないこと」の明記が重要だ。AIは親切なので、頼まれればなんでもやろうとする。明示的に境界を引くことで、品質が安定する。


## 失敗7: Git管理なしで変更が追跡不能

### 何が起きたか

初期の頃、`.company/` ディレクトリのファイルをGit管理していなかった。エージェントがSTATE.mdやcost-tracking.mdを更新するが、「いつ、何が変わったか」が分からない。

ある日、cost-tracking.mdの数字が明らかにおかしいことに気づいたが、いつ間違いが混入したかを特定できなかった。結局、1ヶ月分のデータを手動で再入力するはめになった。

### なぜ起きたか

「Gitはコード管理のツール」という固定観念があった。しかし、AI-CEO Frameworkではすべてのファイルがテキストベースであり、Git管理の対象になる。

### どう解決したか

`.company/` ディレクトリ全体をGit管理に含めた。エージェントがファイルを更新するたびにコミットが作成される。

```
git log --oneline .company/departments/finance/cost-tracking.md

a3b2c1d 2026-03-05 update: 3月変動費を追加
f4e5d6c 2026-03-03 update: 固定費を更新
1a2b3c4 2026-03-01 feat: 3月のcost-tracking開始
```

これにより、いつでも過去の状態に遡れるようになった。`git diff` で変更点を確認し、`git blame` で誰（どのエージェント）が変更したかを追跡できる。


## 失敗8: 人間のレビューなしで品質基準が形骸化

### 何が起きたか

品質チェックリストを設定し、エージェントに「このチェックリストをすべてクリアしてから提出して」と指示した。エージェントはすべての項目に「チェック済み」と報告してきた。

しかし、実際に成果物を見ると、明らかにチェックが通っていない項目があった。テストカバレッジが80%未達なのに「達成」と報告されていたり、CTAが配置されていないのに「配置済み」と報告されていたりした。

### なぜ起きたか

AIは「チェックリストを埋めること」と「実際に基準を満たすこと」を混同することがある。特に、チェックの手段が曖昧な場合に起きやすい。

### どう解決したか

品質チェックを**2段階**にした。

```
Stage 1: エージェントによるセルフチェック
  → チェック結果と根拠（数値、該当箇所の引用）を記載

Stage 2: CEOによるサンプルチェック
  → 週1回、ランダムに3件の成果物を抽出して検証
  → 不正確なセルフチェックが見つかったら、定義を修正
```

「信頼するが検証する（Trust but verify）」の精神だ。毎回すべてを検証する必要はないが、定期的にサンプルチェックすることで品質を維持できる。


## 失敗9: 10事業を同時に回そうとして全部売上ゼロ（実体験）

### 何が起きたか

AIエージェントの力を過信し、「10個のプロダクトを同時に立ち上げれば、どれか1つは当たるだろう」と考えた。SaaSプロダクト4つ、Zenn書籍3冊、受託事業2つ、コンサル1つを同時並行で進めた。

3ヶ月後、10事業すべての売上がほぼゼロだった。

### なぜ起きたか

AIエージェントは「作業の実行」を効率化してくれるが、「戦略の判断」は効率化してくれない。10事業を回すということは、10の市場分析、10のマーケ戦略、10の営業パイプラインが必要だということだ。CEOの判断力は1人分しかないのに、10倍の判断を求めていた。

さらに、各事業に投入できるマーケリソースが分散しすぎた。SEO記事を10テーマに分散して書いても、どのテーマでも検索上位は取れない。

### どう解決したか

事業を3つに絞った。「すでに収益がある事業」「最も伸びしろがある事業」「自分の強みが最も活きる事業」の3軸で評価し、残りの7つを休止した。

```
残した事業:
1. 受託開発（すでに収益あり）
2. HP制作（伸びしろ大、営業の仕組みが機能し始めた）
3. Zenn書籍（自分の強み、ブランディング効果大）

休止した事業:
4-10. いったん凍結。1-3が安定したら再開を検討
```

3つに集中したことで、各事業のKPIが明確になり、マーケリソースも集中できるようになった。結果、2ヶ月後に初めてまとまった売上が立った。

教訓は明確だ。**AIは実行力を10倍にしてくれるが、判断力は1倍のままだ。** 判断力に合わせて事業数を絞ることが、成功の前提条件だ。


## 失敗10: AIに完璧を求めて導入が進まない

### 何が起きたか

これは筆者ではなく、知人の事例だ。「AIエージェントの出力が完璧になるまで公開しない」という方針で、3ヶ月間ずっとCLAUDE.mdの調整を繰り返していた。1本の記事も公開されず、1通のメールも送信されなかった。

### なぜ起きたか

完璧主義だ。AIの出力には必ず改善の余地がある。しかし、90点を95点にする労力と、60点を90点にする労力はまったく違う。前者に時間をかけすぎると、何もアウトプットできない。

### どう解決したか

**「80点で出して、フィードバックで改善する」** というルールを設定した。

```
判断基準:
- 事実に間違いはないか？ → Yes
- 読者/顧客に失礼な内容はないか？ → No
- 最低限の品質基準を満たしているか？ → Yes
→ 80点以上。公開する。

残りの20点は:
- 公開後の反応を見て改善する
- 次回の記事/メールに活かす
- 週次レビューで振り返る
```

AIエージェント経営で最も重要なのは「動き続けること」だ。完璧な仕組みを作ってから動き出すのではなく、動きながら仕組みを改善する。


## 10の失敗から学ぶ3原則

これら10の失敗を振り返ると、3つの原則に集約できる。

### 原則1: 段階的に始める

全部門を一度に構築しない。10事業を同時に回さない。1つずつ、小さく始めて、安定したら次に進む。

### 原則2: 人間が最終判断する

対外アクションの承認、事実確認、品質チェック。これらはAIに丸投げしない。「AIが準備、人間が判断」の分業を徹底する。

### 原則3: 80点で動かす

完璧を求めない。80点で公開し、フィードバックで改善する。動き続けることが、停滞に勝る。
