# CS・法務部門の構築 -- サポートとリスク管理を自動化する

## なぜ2部門をまとめるのか

本書では、CS（カスタマーサポート）と法務を1つの章にまとめている。理由は明快だ。ひとり会社のフェーズでは、この2部門の負荷が相対的に小さいからだ。

開発部門は毎日コードを書く。マーケ部門は毎日コンテンツを出す。営業部門は毎日メールを送る。経理部門は毎日お金を記録する。しかし、CSの問い合わせは初期段階ではそれほど多くないし、法務の出番も月に数回程度だ。

だからといって「不要」ではない。1件のサポート対応の失敗がSNSで炎上することもあるし、利用規約の不備が法的トラブルに発展することもある。**頻度は低いが、インパクトが大きい**のがこの2部門の特徴だ。

エージェントで仕組みを作っておけば、いざという時に迅速に対応できる。そのための準備として、本章では両部門の基本設計を解説する。


## CS部門の設計

### CSエージェントの定義

```markdown
# CS部門エージェント

## 役割
ユーザーからの問い合わせ対応、FAQ管理、フィードバック分析を行う。

## タスク分類

| カテゴリ | 内容 | 権限レベル |
|---------|------|-----------|
| FAQ管理 | よくある質問と回答の整備 | execute |
| 問い合わせ対応 | ユーザーへの返信ドラフト作成 | draft |
| エスカレーション | 対応困難な案件をCEOに上げる | draft |
| フィードバック分析 | ユーザーの声を分析・集約 | read-only |
| ドキュメント更新 | ヘルプページの改善 | execute |

## 対応ポリシー
- 返信は24時間以内（ドラフト生成は即時）
- 技術的な問題は開発部門にエスカレーション
- 返金要求はCEO判断（金額に関わらず）
- ユーザーの感情に配慮した丁寧なトーン
```

### FAQ管理の仕組み

CSの負荷を下げる最も効果的な方法は、FAQの充実だ。同じ質問に何度も答える時間はひとり会社にはない。

```markdown
# FAQ管理

## 構造

faq/
  general.md          # 一般的な質問
  billing.md          # 料金・支払い関連
  technical.md        # 技術的な質問
  {product}/
    getting-started.md  # 始め方
    troubleshooting.md  # トラブルシューティング

## 更新ルール
1. 同じ質問が2回来たらFAQに追加する
2. プロダクトの機能変更時にFAQを見直す
3. 月1回、アクセス解析でよく見られるFAQを確認する
```

### エスカレーションフロー

CSエージェントが単独で解決できない問題は、段階的にエスカレーションする。

```
Level 1: CSエージェントが自動回答（FAQ該当の場合）
    ↓ 解決しない場合
Level 2: CSエージェントが回答ドラフトを作成 → CEO承認
    ↓ 技術的問題の場合
Level 3: 開発部門にエスカレーション
    ↓ ビジネス判断が必要な場合
Level 4: CEOが直接対応
```

Level 1で解決できる問い合わせの割合を上げることが、CS効率化の鍵だ。そのためにFAQの充実が重要になる。

### フィードバック分析

ユーザーの声はプロダクト改善の最大のヒントだ。CSエージェントが問い合わせ内容を分類・集約する。

```markdown
# ユーザーフィードバック月次レポート

## 問い合わせ分類

| カテゴリ | 件数 | 割合 | トレンド |
|---------|------|------|---------|
| 使い方の質問 | 8 | 40% | → |
| バグ報告 | 5 | 25% | ↑ |
| 機能リクエスト | 4 | 20% | → |
| 料金の質問 | 2 | 10% | → |
| その他 | 1 | 5% | → |

## 上位の機能リクエスト
1. ダッシュボードのカスタマイズ（3件）
2. CSV エクスポート機能（2件）
3. Slack連携（1件）

## 開発部門への推奨
- バグ報告が増加傾向。品質強化スプリントを推奨
- ダッシュボードカスタマイズの要望が最多。次期開発の候補に
```

このレポートを `/ai-ceo:morning` に統合することで、CEOはユーザーの声を日常的に把握できる。


## 法務部門の設計

### 法務エージェントの定義

```markdown
# 法務部門エージェント

## 役割
契約書レビュー、利用規約管理、コンプライアンスチェックを行う。
法的助言の最終判断は人間（CEO or 弁護士）が行う。

## タスク分類

| カテゴリ | 内容 | 権限レベル |
|---------|------|-----------|
| 契約書レビュー | 受託契約書のリスクポイント抽出 | read-only |
| 利用規約管理 | Terms/Privacyの更新管理 | draft |
| コンプライアンス | 法令準拠のチェック | read-only |
| OSSライセンス | 使用ライブラリのライセンス監査 | read-only |
| 契約書テンプレート | 定型契約書の雛形管理 | execute |

## 重要ルール
- 法的判断をAIだけで確定しない
- リスクが高い契約は弁護士に相談するよう推奨する
- 法改正情報は定期的に確認する
```

### 契約書レビューの自動化

受託開発やHP制作の案件では、契約書のやり取りが発生する。法務エージェントが契約書をレビューし、リスクポイントを抽出する。

```markdown
# 契約書レビュー結果

## 対象: HP制作業務委託契約書（A社）

## リスク評価: 中

## チェック結果

| 条項 | 内容 | リスク | コメント |
|------|------|--------|---------|
| 第3条 納期 | 「納品後7日以内に検収」 | 低 | 標準的 |
| 第5条 瑕疵担保 | 「納品後1年間」 | 中 | 6ヶ月が一般的。交渉推奨 |
| 第8条 損害賠償 | 「上限なし」 | 高 | 受託金額を上限に交渉すべき |
| 第10条 著作権 | 「成果物の著作権はA社に帰属」 | 中 | 汎用部分は留保すべき |
| 第12条 競業禁止 | 「契約終了後2年間」 | 高 | 期間が長すぎる。1年以内に |

## 推奨アクション
1. 第8条の損害賠償上限を設定するよう交渉
2. 第12条の競業禁止期間を1年以内に短縮
3. 第10条の著作権について汎用部分の留保を追加
4. リスクが高い項目が2件あるため、弁護士への相談を推奨
```

このような分析を人間がゼロから行うと、契約書を読み込むだけで30分以上かかる。エージェントが下準備をしてくれるだけで、CEOは「何を交渉するか」の判断に集中できる。

### 利用規約の管理

プロダクトを公開している場合、利用規約（Terms of Service）とプライバシーポリシーの管理が必要だ。

```markdown
# 利用規約管理

## 管理対象

| ドキュメント | 最終更新 | 次回レビュー |
|------------|---------|------------|
| 利用規約 | 2026/01/15 | 2026/04/15 |
| プライバシーポリシー | 2026/01/15 | 2026/04/15 |
| 特定商取引法に基づく表記 | 2026/02/01 | 必要時 |
| Cookie ポリシー | 2026/01/15 | 2026/04/15 |

## 更新トリガー
- 新機能の追加でデータ取り扱いが変わった場合
- 法改正があった場合
- 3ヶ月ごとの定期レビュー
```

法務エージェントが更新トリガーを監視し、レビューが必要なタイミングでCEOに通知する仕組みだ。

### OSSライセンス監査

開発で使用するオープンソースライブラリのライセンスは、見落としがちだがリスクがある領域だ。

```markdown
# OSSライセンス監査

## コマンド
/ai-ceo:legal:oss-audit

## チェック内容
1. package.json（npm）の全依存パッケージのライセンスを確認
2. 以下のリスクレベルで分類:
   - 低リスク: MIT, Apache 2.0, BSD
   - 中リスク: MPL, LGPL
   - 高リスク: GPL, AGPL（自社コードの公開義務が発生しうる）
   - 不明: ライセンス記載なし
3. 高リスク・不明のパッケージを報告

## 出力例
| パッケージ | バージョン | ライセンス | リスク |
|-----------|----------|-----------|-------|
| react | 19.0.0 | MIT | 低 |
| some-lib | 2.1.0 | GPL-3.0 | 高 |
| another-lib | 1.0.0 | 不明 | 高 |
```


## 実践例: 利用規約の変更をAIが検出・更新提案

実際に役立った事例を紹介する。

プロダクトに分析機能を追加した際、ユーザーの利用データを集計する処理が含まれていた。法務エージェントがこの変更を検出し、以下のアラートを出した。

```
[法務アラート] プライバシーポリシーの更新が必要です

理由: 新機能「利用分析ダッシュボード」がユーザーの利用データを
集計・表示する機能を含んでいます。現在のプライバシーポリシーには
「利用状況の統計的分析」に関する記載がありません。

推奨: プライバシーポリシーに以下を追加
- 収集するデータの種類
- データの利用目的
- データの保持期間

ドラフトを生成しますか？ → approval-queue に追加済み
```

このように、開発部門の変更が法務に影響する場合を自動検出できる。部門間の連携がエージェントによって実現されている例だ。


## 2部門の最小構成

ひとり会社が最初からフル装備で構築する必要はない。以下の最小構成から始めて、必要に応じて拡張すればよい。

### CS部門の最小構成

```
必須:
- FAQ文書（1ファイル）
- 問い合わせ対応テンプレート（1ファイル）
- エスカレーションルール（エージェント定義内に記載）

後で追加:
- フィードバック分析の自動化
- チャットボット連携
```

### 法務部門の最小構成

```
必須:
- 契約書レビューチェックリスト（エージェント定義内に記載）
- 利用規約・プライバシーポリシー管理台帳
- OSSライセンス監査コマンド

後で追加:
- 法改正の自動モニタリング
- 契約書テンプレートの充実
- コンプライアンスチェックの自動化
```


## CS・法務部門構築のチェックリスト

### CS部門
- [ ] CSエージェントの定義ファイルを作成した
- [ ] FAQドキュメントの初期版を作成した
- [ ] エスカレーションフローを定義した
- [ ] 問い合わせ対応テンプレートを用意した
- [ ] 対外返信はすべてdraftモードにした

### 法務部門
- [ ] 法務エージェントの定義ファイルを作成した
- [ ] 利用規約とプライバシーポリシーを整備した
- [ ] 契約書レビューのチェック項目を設定した
- [ ] OSSライセンス監査を1回実行した
- [ ] 法的判断はAIだけで確定しないルールを明記した

CS・法務は「保険」のようなものだ。日常的に大きな負荷はかからないが、いざという時に仕組みがあるかないかで、対応速度が大きく変わる。最小構成でよいので、早めに構築しておくことを勧める。
