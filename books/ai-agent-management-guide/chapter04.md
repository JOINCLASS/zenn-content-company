# 開発部門の構築 -- CTOエージェントでプロダクトを回す

## 開発部門はAIエージェント経営の心臓部

AI-CEO Frameworkにおいて、開発部門は最も投資対効果が高い部門である。

理由はシンプルだ。ソフトウェア開発のタスクは「テキスト入力→テキスト出力」の構造を持つものが大半であり、AIエージェントが最も得意とする領域だからだ。設計書からコードを生成する、コードからテストを生成する、バグ報告から修正パッチを生成する。いずれもAIの能力と親和性が高い。

筆者の会社では、開発部門のエージェント構築に最初に取り組んだ。結果として、1人で6つのプロダクトを同時にメンテナンスできる体制が実現している。もちろんすべてを完全自動化しているわけではないが、定型作業の80%以上をエージェントに委任できている実感がある。

本章では、開発部門エージェントの設計から実装まで、実際に動いている定義ファイルをベースに解説する。


## 開発部門エージェントの全体設計

### ディレクトリ構成

まず、開発部門に関連するファイルの配置を確認しよう。

```
.company/
  departments/
    dev/
      STATE.md              # 開発部門の現在状態
      error-log.md          # エラーログ
      sprint-backlog.md     # スプリントバックログ
.claude/
  agents/
    dev-agent.md            # CTO/開発エージェント定義
    architect-agent.md      # 設計サブエージェント
    coder-agent.md          # 実装サブエージェント
    reviewer-agent.md       # レビューサブエージェント
```

### CTO エージェントの定義

開発部門のトップレベルエージェントは、CTOの役割を担う。このエージェントが開発タスクを受け取り、適切なサブエージェントに振り分ける。

```markdown
# 開発部門エージェント（CTO）

## 役割
プロダクト開発の統括。タスクの分類・優先度判断・サブエージェントへの委任を行う。

## タスク分類

| カテゴリ | 例 | 委任先 | 権限レベル |
|---------|-----|-------|-----------|
| 新機能開発 | 機能設計→実装→テスト | architect → coder | draft |
| バグ修正 | 報告→原因特定→修正 | coder | execute |
| コードレビュー | PR分析→指摘→改善提案 | reviewer | read-only |
| テスト生成 | 既存コードのテスト追加 | coder | execute |
| リファクタリング | 技術的負債の解消 | architect → coder | execute |
| デプロイ | 本番環境への反映 | - | draft（CEO承認必須） |

## 品質基準
- テストカバレッジ: 新規コードは80%以上
- TypeScript strict mode: エラーゼロ
- コミットメッセージ: Conventional Commits準拠
```

ここで重要なのは「権限レベル」の列だ。バグ修正やテスト追加は`execute`（自動実行）だが、新機能開発は`draft`（レビュー待ち）、デプロイは必ずCEO承認が必要としている。これにより、日常的な開発作業は自動で回しつつ、重要な判断は人間が握る構造になる。


## サブエージェントの設計

CTOエージェント1つですべてをこなすのは非効率だ。Claude Codeのサブエージェント機能を活用して、役割を分割する。

### architect（設計エージェント）

```markdown
# 設計エージェント

## 役割
機能要件から技術設計書を生成する。

## 入力
- 機能の概要説明
- 既存のコードベース構造

## 出力
- 技術設計書（実装方針、ファイル構成、API設計）
- 影響範囲の分析

## ルール
- 既存のアーキテクチャパターンに従う
- 新しいライブラリの導入は理由を明記する
- 影響範囲が3ファイル以上の場合は必ず設計書を作成
```

### coder（実装エージェント）

```markdown
# 実装エージェント

## 役割
設計書またはタスク指示に基づいてコードを実装する。

## 入力
- 設計書またはバグ報告
- 関連する既存コード

## 出力
- 実装コード
- ユニットテスト
- 変更の説明

## ルール
- 1つのコミットは1つの論理的変更に対応させる
- テストを書いてから実装する（TDDの精神）
- 既存のコーディング規約に従う
- console.logを本番コードに残さない
```

### reviewer（レビューエージェント）

```markdown
# レビューエージェント

## 役割
コード変更のレビューを行い、品質を担保する。

## 入力
- git diff または PRの変更内容

## 出力
- レビューコメント（問題点、改善提案）
- 承認 or 差し戻し判定

## チェック項目
- [ ] セキュリティリスクはないか
- [ ] パフォーマンスへの影響はないか
- [ ] テストは十分か
- [ ] 命名は適切か
- [ ] エラーハンドリングは適切か
```

この3層構造にすることで、「設計→実装→レビュー」という開発サイクルをエージェント内で完結できる。人間のCEOがレビューするのは最終成果物だけでよい。


## 実践例: 新機能の設計から実装まで

実際にどう動くのかを、具体例で示す。ここでは「ダッシュボードにKPIウィジェットを追加する」というタスクを例にとる。

### Step 1: タスクの投入

CEOがコマンドを実行する。

```
/ai-ceo:dev:sprint
→ タスク: ダッシュボードにKPIウィジェットを追加
```

CTOエージェントがタスクを受け取り、タスク分類を行う。

```
分類結果:
- カテゴリ: 新機能開発
- 権限レベル: draft
- 委任先: architect → coder → reviewer
- 見積もり: 中規模（影響ファイル5-10）
```

### Step 2: 設計フェーズ

architectエージェントが起動し、既存コードを分析した上で設計書を生成する。

```markdown
## KPIウィジェット 技術設計書

### 概要
ダッシュボード画面にKPIサマリーウィジェットを追加する。

### 実装方針
- コンポーネント: `KpiWidget.tsx` を新規作成
- データ取得: 既存の `/api/kpi` エンドポイントを利用
- 状態管理: React Query でサーバー状態を管理

### ファイル構成
- src/components/dashboard/KpiWidget.tsx（新規）
- src/components/dashboard/KpiWidget.test.tsx（新規）
- src/pages/dashboard.tsx（既存・修正）

### 影響範囲
- dashboard.tsxのレイアウト変更のみ
- API変更なし
```

### Step 3: 実装フェーズ

coderエージェントが設計書に基づいてコードを実装する。テストも同時に生成される。

### Step 4: レビューフェーズ

reviewerエージェントが変更差分をチェックし、問題がなければ承認する。

### Step 5: 承認待ち

CTOエージェントが成果物をまとめ、`approval-queue.md` に追加する。

```markdown
- [AQ-015] 開発: KPIウィジェット追加 | PR #42 | draft
```

CEOが `/ai-ceo:approve AQ-015` を実行すれば、マージとデプロイが進む。

この一連の流れが、1つのコマンドから自動的に進行する。CEOが介入するのは最終承認の1回だけだ。


## Permission設定の詳細

開発部門のPermission設定は、速度と安全性のバランスが重要だ。

```markdown
# 開発部門 Permission設定

## 自動実行（execute）
- バグ修正（影響範囲3ファイル以下）
- テスト追加
- リファクタリング（ロジック変更なし）
- ドキュメント更新
- 依存パッケージのパッチバージョン更新

## 承認必要（draft）
- 新機能開発
- API仕様の変更
- データベーススキーマの変更
- 依存パッケージのメジャーバージョン更新
- デプロイ（staging / production）

## 禁止
- 本番データベースへの直接操作
- 環境変数・シークレットの変更
- git force push
```

筆者の経験では、この設定が一番バランスが良い。バグ修正を毎回承認していたらCEOの時間が足りないし、逆にデプロイを自動化すると事故が起きたときに取り返しがつかない。

### コツ: 段階的に権限を広げる

最初は「すべてdraft」から始めて、エージェントの出力品質を確認しながら段階的にexecuteに切り替えるのがおすすめだ。筆者も最初の2週間はすべてのコード変更をレビューしていたが、品質が安定してからバグ修正とテスト追加を自動実行に切り替えた。


## CLAUDE.mdの開発ルール設計

開発部門のエージェントが高品質なコードを出すために、CLAUDE.mdに開発ルールを明記しておくことが重要だ。

```markdown
# 開発ルール

## コーディング規約
- TypeScript strict mode を使用
- 変数名はキャメルケース、型名はパスカルケース
- 関数は50行以下に収める
- ネストは3段階まで（早期リターンを使う）

## Git規約
- コミットメッセージ: Conventional Commits
  - feat: 新機能
  - fix: バグ修正
  - refactor: リファクタリング
  - test: テスト追加
  - docs: ドキュメント

## テスト規約
- 新規関数にはユニットテスト必須
- テストファイルは `*.test.ts` で同一ディレクトリに配置
- テスト名は「何をしたら何が起きるか」の形式

## PR規約
- タイトル: Conventional Commits形式
- 本文: 変更の背景、実装方針、テスト方法を記載
- 1PRは1つの論理的変更に対応
```

これらのルールをCLAUDE.mdに書いておくだけで、エージェントの出力品質が劇的に安定する。ルールが曖昧だと、毎回異なるスタイルのコードが生成されてメンテナンスが困難になる。


## 開発部門構築のチェックリスト

本章の内容を実践する際のチェックリストをまとめる。

- [ ] CTOエージェントの定義ファイルを作成した
- [ ] サブエージェント（architect, coder, reviewer）を定義した
- [ ] タスク分類と権限レベルを設定した
- [ ] CLAUDE.mdに開発ルールを記載した
- [ ] 最初の1週間はすべてdraftモードで運用する
- [ ] エージェントの出力品質を確認し、段階的にexecuteに切り替える

開発部門が安定して回り始めると、CEOは「何を作るか」の判断に集中できるようになる。コードを書く時間が減る分、プロダクトの方向性を考える時間が増える。これがAIエージェント経営の最大のメリットだ。
