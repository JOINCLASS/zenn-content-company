# 承認パイプラインの運用 -- 品質と速度を両立する

## AIに「全自動」は危険すぎる

第4章から第8章まで、6つの部門のエージェントを構築してきた。各部門が独立して動き、日常のタスクを自動的に処理する。しかし、ここで1つの問いが生まれる。

**「AIの出力をどこまで信用するか？」**

答えは明確だ。**対外的なアクションは、必ず人間が確認する。** メールの送信、契約書の提出、コードのデプロイ、SNSの投稿。これらを無検証で自動実行するのは、会社の信用を賭けたギャンブルだ。

一方で、すべてのアクションにCEOの承認を求めたら、エージェントを導入した意味がなくなる。「AIが生成→人間が承認→実行」のサイクルに5分以上かかるなら、自分でやったほうが早い場面も出てくる。

この「品質」と「速度」のバランスを取るのが、承認パイプラインだ。AI-CEO Frameworkの中核をなす仕組みである。


## approval-queue.md の設計

### ファイル構造

```markdown
# 承認待ちキュー

## 承認待ちアイテム

| ID | 部門 | 内容 | ファイルパス | 優先度 | 追加日 |
|----|------|------|------------|--------|-------|
| AQ-031 | 営業 | A歯科 アウトリーチメール | .company/departments/sales/hp-renewal/outreach/01-a-dental-email.md | 高 | 2026-03-05 |
| AQ-032 | マーケ | Zenn記事「CLAUDE.md設計パターン」 | .company/departments/marketing/drafts/zenn-claude-md.md | 中 | 2026-03-05 |
| AQ-033 | 開発 | KPIウィジェット PR #42 | PR #42 | 中 | 2026-03-04 |
| AQ-034 | 経理 | B社 請求書 | .company/departments/finance/invoices/b-corp-202603.md | 高 | 2026-03-05 |

## 却下・差し戻し履歴

| ID | 部門 | 内容 | 理由 | 日付 |
|----|------|------|------|------|
| AQ-029 | 営業 | C整骨院メール | 金額の記載誤り。修正後に再提出 | 2026-03-04 |
```

### フィールドの設計思想

各フィールドには意図がある。

- **ID**: 一意の識別子。`AQ-{連番}` 形式。コマンドで指定する際に使用する
- **部門**: どの部門からの承認依頼か。部門ごとにまとめて処理するときに便利
- **内容**: 1行で何の承認か分かる要約
- **ファイルパス**: 実際の成果物の場所。CEOが中身を確認するために必須
- **優先度**: 高/中/低。期限が近い、金額が大きい、対外影響が大きいものは「高」
- **追加日**: いつ追加されたか。古いものから処理するための目安


## 承認/却下のワークフロー

### 承認フロー

```
1. CEOが /ai-ceo:morning を実行
   → 承認待ち一覧が表示される

2. CEOがファイルの中身を確認
   → ファイルパスを見て内容をレビュー

3. 問題なければ /ai-ceo:approve AQ-031
   → approval-queue.md から該当行を削除
   → decisions/{month}.md に承認記録を追加
   → 該当部門に「承認済み」を通知

4. 部門エージェントが実行
   → メール送信、デプロイ、公開などを実行
```

### 却下フロー

```
1. CEOが問題を発見
   → 金額の誤り、トーンの問題、事実誤認など

2. /ai-ceo:reject AQ-029 "金額が間違っている。15万円ではなく10万円"
   → approval-queue.md から該当行を削除
   → 却下履歴に記録
   → decisions/{month}.md に却下記録を追加
   → 該当部門に差し戻し指示を送信

3. 部門エージェントが修正
   → 指摘内容に基づいて修正
   → 修正版を再度 approval-queue.md に追加
```


## decisions/{month}.md による意思決定ログ

すべての承認/却下は意思決定ログに記録される。これはAI-CEO Frameworkの「監査証跡」だ。

```markdown
# 意思決定ログ 2026年3月

## 2026-03-05

### [承認] AQ-031: A歯科 アウトリーチメール
- 部門: 営業
- 内容: HP分析に基づく改善提案メール
- 判断理由: 内容・金額ともに問題なし
- 実行結果: メール送信完了

### [却下] AQ-029: C整骨院メール
- 部門: 営業
- 内容: HP制作提案メール
- 却下理由: 月額管理費の金額が料金表と不一致。修正後に再提出
- 差し戻し先: 営業部門エージェント

### [承認] AQ-032: Zenn記事「CLAUDE.md設計パターン」
- 部門: マーケティング
- 内容: SEO記事の公開
- 判断理由: 品質基準クリア。CTAの配置も適切
- 実行結果: Zennに公開完了
```

このログが蓄積されることで、以下の価値が生まれる。

1. **判断の一貫性**: 過去の判断基準を参照できる
2. **学習データ**: どういう成果物が承認され、どういうものが却下されたかのパターンが見える
3. **監査対応**: 「いつ、誰が、何を承認したか」が明確
4. **振り返り**: 月末にレビューして、承認プロセスの改善点を発見できる


## permissions.md の詳細設計

## 自動承認ルールの設計

すべてを手動承認していてはCEOがボトルネックになる。一定の条件を満たすアクションは自動承認する仕組みを設計する。

```markdown
# 自動承認ルール

## 条件
以下の「すべて」を満たす場合、自動承認とする:

1. 権限レベルが execute であること
2. 過去10回の同種タスクでCEO却下がゼロであること
3. 金額に関わるアクションでないこと
4. 対外（社外の人間への）アクションでないこと

## 自動承認の例
- テストの追加・実行
- 内部ドキュメントの更新
- 分析レポートの生成
- リファクタリング（ロジック変更なし）

## 自動承認の対象外（必ず手動承認）
- メール送信
- SNS投稿
- デプロイ
- 請求書・見積書
- 契約書・利用規約の変更
```

自動承認の範囲は、運用しながら段階的に広げていく。最初は狭くしておき、エージェントの出力品質が信頼できるようになったら範囲を広げる。これは前章でも繰り返し述べたが、**信頼は実績の上に構築する**ものだ。


## エスカレーションフロー

エージェントがタスクを完了できない場合のフローも設計しておく。

```
タスク実行
  ↓ 失敗
1回目リトライ（エラー内容をフィードバックして再実行）
  ↓ 失敗
2回目リトライ
  ↓ 失敗
3回目リトライ
  ↓ 失敗
エスカレーション
  → approval-queue.md に「エスカレーション」として追加
  → error-log.md にエラー詳細を記録
  → /ai-ceo:morning でCEOに通知
```

```markdown
# エスカレーションの記録例

## approval-queue.md
| ID | 部門 | 内容 | ファイルパス | 優先度 | 追加日 |
|----|------|------|------------|--------|-------|
| AQ-035 | 開発 | [エスカレーション] ビルド失敗 3回 | .company/departments/dev/error-log.md | 高 | 2026-03-05 |

## error-log.md
### 2026-03-05 ビルド失敗
- タスク: PR #43 のマージ後ビルド
- エラー: TypeScriptコンパイルエラー（型不整合）
- リトライ回数: 3回
- 各試行のエラー内容:
  1. `Property 'name' does not exist on type 'User'`
  2. 型定義を修正 → 別のエラー `Cannot find module './types'`
  3. モジュールパスを修正 → 別のエラー（循環参照）
- 推定原因: 大規模な型リファクタリングが必要
- 推奨: CEOが手動で対応
```


## 実践例: 1日の承認フロー

筆者の実際の1日を再現する。

### 朝 8:30 — モーニングダイジェスト

```
/ai-ceo:morning

→ 承認待ち: 8件
  - [AQ-040] 営業: D社メール（高）
  - [AQ-041] 営業: E社メール（高）
  - [AQ-042] 営業: F社提案書（高）
  - [AQ-043] マーケ: Zenn記事（中）
  - [AQ-044] マーケ: X投稿3件（中）
  - [AQ-045] 開発: 新機能PR（中）
  - [AQ-046] 経理: G社請求書（中）
  - [AQ-047] 開発: [エスカレーション] テスト失敗（高）
```

### 8:30-8:40 — 高優先度の処理

営業メール2通と提案書1件を確認。内容に問題なし。

```
/ai-ceo:approve AQ-040
/ai-ceo:approve AQ-041
/ai-ceo:approve AQ-042
```

3件で約5分。

### 8:40-8:50 — 中優先度の処理

Zenn記事を流し読み。SEOタイトル、CTA配置、内容の正確性を確認。

```
/ai-ceo:approve AQ-043
```

X投稿3件。140字なので確認は一瞬。

```
/ai-ceo:approve AQ-044
```

開発PRの差分を確認。テスト通過、レビュー済み。

```
/ai-ceo:approve AQ-045
```

請求書の金額と宛先を確認。

```
/ai-ceo:approve AQ-046
```

4件で約10分。

### 8:50-9:00 — エスカレーション対応

テスト失敗のエスカレーションを確認。error-log.mdを読み、原因を把握。これは自分で対応する必要がある。午前中の作業リストに追加。

### 合計: 約25分

8件の承認を25分で処理。これが毎朝のルーティンだ。

ポイントは「考える時間」と「確認する時間」を分離していることだ。エージェントが成果物を用意してくれるので、CEOは「これでOKか？」の判断だけに集中できる。ゼロから作る場合と比較すると、所要時間は10分の1以下になる。

承認パイプラインは、AI-CEO Frameworkの「安全弁」であると同時に「品質管理」の仕組みだ。この仕組みがあるからこそ、エージェントに大胆にタスクを委任できる。
