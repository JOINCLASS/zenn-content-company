# KPI管理と経営ダッシュボード -- AIで経営を可視化する

## 測定できないものは改善できない

ピーター・ドラッカーの有名な言葉だ。AIエージェント経営においても、この原則は変わらない。

6部門のエージェントが動き、承認パイプラインが回り始めた。日々のタスクは自動化され、朝の承認で1日が回るようになった。しかし、「このやり方でビジネスが成長しているのか」を判断するには、数字が必要だ。

筆者の経験では、KPI管理を怠ると2つの問題が起きる。1つは「忙しいが成果が出ていない」状態に気づかないこと。もう1つは「何を優先すべきか」の判断基準がなくなることだ。

本章では、STATE.mdを使った経営状態の管理と、部門別KPIの設計、そしてAIによる自動レポートの仕組みを解説する。


## STATE.mdを使った経営状態の管理

### 会社全体のSTATE.md

AI-CEO Frameworkでは、経営状態を `.company/STATE.md` に集約する。

```markdown
# 経営状態 — 2026年3月

## 会社概要
- 社名: 合同会社ジョインクラス
- フェーズ: 成長期（プロダクト6本運用中）
- 月間売上目標: ¥500,000
- 月間コスト上限: ¥50,000（AI関連）

## KPIサマリー

| 指標 | 目標 | 実績 | 達成率 | トレンド |
|------|------|------|--------|---------|
| 月間売上 | ¥500,000 | ¥320,000 | 64% | ↑ |
| 月間PV | 10,000 | 7,500 | 75% | ↑ |
| Zenn記事公開数 | 8本/月 | 5本 | 63% | → |
| 営業アウトリーチ | 20社/月 | 12社 | 60% | ↑ |
| 受注率 | 10% | 8.3% | 83% | → |
| AI関連コスト | ¥50,000以下 | ¥38,000 | 76% | → |
| 承認処理時間 | 30分/日 | 25分/日 | 達成 | → |

## 注意事項
- 売上目標に対して36%のギャップ。HP制作案件の獲得を加速する必要あり
- 記事公開ペースが目標未達。週2本ペースに引き上げが必要
```

### 部門別STATE.md

各部門にもSTATE.mdを用意し、部門固有の状態を管理する。

```
.company/departments/dev/STATE.md       # 開発部門
.company/departments/marketing/STATE.md # マーケ部門
.company/departments/sales/STATE.md     # 営業部門
.company/departments/finance/STATE.md   # 経理部門
.company/departments/cs/STATE.md        # CS部門
.company/departments/legal/STATE.md     # 法務部門
```

各部門のSTATE.mdは、その部門のエージェントが自動更新する。タスクを完了するたびにSTATEが更新され、最新の状態が常に反映される仕組みだ。


## 部門別KPIの設計

### 開発部門のKPI

```markdown
# 開発部門 KPI

| 指標 | 測定方法 | 目標 | 意味 |
|------|---------|------|------|
| PR作成数 | GitHub PR数/週 | 5件以上 | 開発の活性度 |
| PR平均マージ時間 | PR作成→マージの平均時間 | 24時間以内 | 開発のスピード |
| バグ修正速度 | 報告→修正完了の平均時間 | 48時間以内 | 品質対応力 |
| テストカバレッジ | テストカバレッジ率 | 80%以上 | コード品質 |
| ビルド成功率 | CI/CDパイプラインの成功率 | 95%以上 | 安定性 |
```

開発KPIのポイントは「速度」と「品質」のバランスだ。PRの数だけ追うとテストが疎かになるし、品質だけ追うとリリースが遅れる。両方を同時に追跡する。

### マーケティング部門のKPI

```markdown
# マーケティング部門 KPI

| 指標 | 測定方法 | 目標 | 意味 |
|------|---------|------|------|
| 月間PV | Zenn/Google Analyticsの合算 | 10,000PV | トラフィック量 |
| 記事公開数 | Zenn記事の月間公開数 | 8本 | コンテンツ量産力 |
| 記事CTR | 検索結果でのクリック率 | 3%以上 | タイトルの訴求力 |
| SNSフォロワー増加 | X + LinkedInの純増数 | 100人/月 | ブランド認知 |
| CVR | LP訪問→登録の転換率 | 2%以上 | コンバージョン力 |
| 書籍販売部数 | Zenn Books月間販売数 | 50部 | 直接収益 |
```

マーケのKPIで最も重要なのはCVR（コンバージョンレート）だ。PVがいくら多くても、登録や購入に至らなければビジネスには貢献しない。PVは「入口」、CVRは「出口」。両方を追跡する。

### 営業部門のKPI

```markdown
# 営業部門 KPI

| 指標 | 測定方法 | 目標 | 意味 |
|------|---------|------|------|
| リスト作成数 | 新規見込み客の追加数/月 | 30社 | パイプラインの入口 |
| アウトリーチ数 | メール送信数/月 | 20通 | アプローチ量 |
| 返信率 | メール返信数/送信数 | 20%以上 | メール品質 |
| 面談数 | 月間面談回数 | 5件 | 商談進行 |
| 受注率 | 受注数/面談数 | 30%以上 | クロージング力 |
| 平均受注額 | 受注金額の平均 | ¥200,000 | 案件の価値 |
```

営業KPIは「ファネル」の形で追跡する。リスト→アウトリーチ→返信→面談→受注。各段階の転換率を見ることで、どこがボトルネックかが一目瞭然になる。

### 経理部門のKPI

```markdown
# 経理部門 KPI

| 指標 | 測定方法 | 目標 | 意味 |
|------|---------|------|------|
| AI関連コスト率 | AI費用/総売上 | 10%以下 | コスト効率 |
| 請求回収率 | 回収済み/請求済み | 100% | キャッシュフロー |
| 月次レポート作成日 | 月末から何日後か | 3日以内 | レポーティング速度 |
| 「要確認」仕訳の割合 | 要確認/全仕訳 | 5%以下 | 自動分類の精度 |
```


## /ai-ceo:morning の朝ダイジェスト

毎朝のダイジェストは、KPI管理の「ダッシュボード」の役割を果たす。

```markdown
## AI-CEO Morning Digest — 2026-03-05

### 承認待ち（5件）
- [AQ-048] 営業: G社メール | 高
- [AQ-049] マーケ: Zenn記事 | 中
- [AQ-050] マーケ: X投稿 | 中
- [AQ-051] 開発: バグ修正PR | 中
- [AQ-052] 経理: H社請求書 | 中

### KPIアラート
- [警告] 月間PV: 7,500/10,000（75%） - 記事公開ペースの向上が必要
- [警告] 営業受注率: 8.3%（目標10%） - 提案書の改善を検討
- [正常] AIコスト: ¥38,000/¥50,000（76%）
- [達成] 承認処理時間: 25分/日（目標30分以下）

### 部門状態
| 部門 | ステータス | 進行中タスク | 注意事項 |
|------|-----------|------------|---------|
| 開発 | 正常 | PR #44 レビュー中 | なし |
| マーケ | 注意 | 記事5/8本（月目標） | ペース不足 |
| 営業 | 正常 | 3社面談調整中 | G社は今週中に返信必要 |
| 経理 | 正常 | 月次レポート作成中 | なし |
| CS | 正常 | 問い合わせ2件対応中 | なし |
| 法務 | 正常 | 特になし | なし |

### 今日のおすすめアクション
1. [高] AQ-048 G社メールを承認して送信（期限: 本日）
2. [中] マーケ部門に記事公開を加速する指示を出す
3. [低] 今週の営業面談の準備を確認する
```

このダイジェストを見るだけで、会社の現在地と今日やるべきことが分かる。所要時間は2-3分だ。


## 週次・月次レビューの自動化

### 週次レビュー

毎週金曜日にエージェントが週次レビューを自動生成する。

```markdown
# 週次レビュー — 2026年3月第1週

## 今週の実績

### 数値
| 指標 | 今週 | 先週 | 変化 |
|------|------|------|------|
| 売上 | ¥85,000 | ¥60,000 | +41.7% |
| PV | 2,100 | 1,800 | +16.7% |
| 記事公開 | 2本 | 1本 | +100% |
| メール送信 | 5通 | 3通 | +66.7% |
| 承認処理 | 22件 | 18件 | +22.2% |

### ハイライト
- HP制作案件1件を受注（¥300,000）
- Zenn書籍の販売が前週比30%増
- 営業リストを10社追加

### 課題
- 記事公開ペースが目標（週2本）に届かない
- 開発のPR #43 がマージ待ち3日目

## 来週の計画
1. 記事2本の公開（火・木）
2. 営業メール5通の送信
3. PR #43 のマージとデプロイ
```

### 月次レビュー

月末にはより包括的なレビューを生成する。KPIの達成率、未達の原因分析、来月の計画を含む。経理の月次レポートと合わせて、経営の全体像を把握する。


## KPI未達の場合の自動アクション提案

KPIが目標を下回った場合、エージェントが自動的に改善アクションを提案する。

```markdown
# KPIアラート — 自動アクション提案

## アラート: 月間PV が目標の75%

### 分析
- 目標: 10,000PV
- 実績: 7,500PV（3月5日時点）
- 原因推定: 記事公開数が5/8本（目標の63%）

### 改善アクション提案

| 優先度 | アクション | 期待効果 | 工数 |
|--------|-----------|---------|------|
| 高 | 今週中に記事2本を追加公開 | +1,000PV | 2時間 |
| 中 | 既存記事のタイトル最適化（CTR向上） | +500PV | 30分 |
| 中 | SNS投稿を増やして記事への流入を増加 | +300PV | 15分 |
| 低 | 過去記事のリライト | +200PV | 1時間 |

### 推奨
1と2を今週中に実施すれば、月末に目標達成が見込まれる。
```

この提案はあくまで「参考」だ。最終的にどのアクションを取るかはCEOが判断する。しかし、データに基づいた選択肢が提示されることで、判断の質が上がる。


## KPIの罠に注意

KPI管理にはいくつかの罠がある。

### 罠1: 測定しすぎて行動が遅れる

KPIを20個も30個も設定すると、毎日の確認だけで30分以上かかる。ひとり会社では、部門ごとに2-3個の重要指標に絞るべきだ。

### 罠2: 虚栄の指標を追いかける

PVやフォロワー数は「見栄えが良い」が、必ずしもビジネスの成長を示さない。最終的に追うべきは「売上」と「利益」だ。他の指標はこの2つにつながる「先行指標」として位置づける。

### 罠3: 短期の変動に一喜一憂する

週単位の数値は変動が大きい。1週間PVが下がったからといって戦略を変える必要はない。月単位のトレンドで判断する。

```
推奨する確認頻度:
- 毎日: 承認キュー、エスカレーション
- 週1: PV、記事数、営業進捗
- 月1: 売上、利益、コスト率、KPI達成率の全体レビュー
- 四半期: 戦略の見直し、KPI目標の調整
```


## KPI管理のチェックリスト

- [ ] 会社全体のSTATE.mdにKPIサマリーを設定した
- [ ] 部門別に2-3個の重要KPIを設定した
- [ ] /ai-ceo:morningにKPIアラートを組み込んだ
- [ ] 週次レビューの自動生成を設定した
- [ ] 月次レビューの自動生成を設定した
- [ ] KPI未達時のアクション提案を設定した
- [ ] 虚栄の指標ではなく、売上に直結する指標を優先している

KPI管理は「経営の計器盤」だ。飛行機のパイロットが計器なしで飛べないように、CEOも数字なしで経営はできない。AIエージェントが計器の読み取りと異常検知を担い、CEOは操縦に集中する。この分業が、AIエージェント経営の真価だ。
