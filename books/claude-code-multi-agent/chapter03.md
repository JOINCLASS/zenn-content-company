# Claude Codeサブエージェントの仕組み

## サブエージェントの正体

前章でOrchestrator-Workerパターンを設計した。本章では、そのWorkerを実現するClaude Codeの「サブエージェント機能」を技術的に解説する。

Claude Codeには「Agent tool」と呼ばれる仕組みがある。これは、親エージェント（Orchestrator）が子エージェント（Worker）を起動し、独立したコンテキストでタスクを実行させ、結果を受け取る機能だ。

重要なのは、サブエージェントは**親とは完全に独立したコンテキスト**を持つという点だ。親のコンテキストウィンドウを消費しない。親が管理する情報と、子が処理する情報が分離される。これが、Thin Orchestrator原則を技術的に成立させる基盤になる。


## Agent toolの内部動作

Claude CodeのAgent toolは、以下の要素で構成される。

### 起動パラメータ

サブエージェントを起動する際に指定できる主要なパラメータを整理しよう。

**prompt（プロンプト）**

サブエージェントに渡す指示。これがサブエージェントの「仕事の依頼書」になる。

```
タスクの目的、参照すべきファイル、成果物の出力先、
品質基準を含む具体的な指示テキスト
```

**CLAUDE.md の継承**

サブエージェントは、プロジェクトルートのCLAUDE.mdの内容を継承する。ただし、`.claude/agents/`にエージェント固有の定義ファイルがある場合、そのファイルの内容がシステムプロンプトとして追加される。

これにより、以下の階層構造が成立する。

```
プロジェクトのCLAUDE.md（全エージェント共通のルール）
  + .claude/agents/cto.md（CTOエージェント固有のルール）
  + 起動時のprompt（今回のタスク固有の指示）
```

共通ルール（コーディング規約、ブランドガイドライン）はCLAUDE.mdに書き、エージェント固有の専門知識はagentsディレクトリに書き、タスク固有の指示はpromptで渡す。この3層構造が、設定の重複を防ぎつつ、各エージェントの専門性を保証する。

**worktree isolation（ワークツリー分離）**

サブエージェントを起動する際に`isolation: "worktree"`パラメータを指定すると、独立したgit worktreeで実行される。これにより、複数のサブエージェントが同時にファイルを変更しても、互いの変更が衝突しない。

**注意:** worktree isolationはデフォルトでは無効だ。明示的に指定しない場合、サブエージェントは親と同じワーキングディレクトリで実行される。並列でファイルを変更するリスクがある場合にのみ、worktreeオプションを使う。

詳しくは後述するが、この仕組みがマルチエージェントの並列実行を安全に実現する鍵となる。

### 実行モード

サブエージェントには2つの実行モードがある。

**foreground（フォアグラウンド）実行**

親エージェントがサブエージェントの完了を待つ。結果が返ってくるまで、親は次の処理に進まない。

**background（バックグラウンド）実行**

親エージェントがサブエージェントの完了を待たない。起動した直後に親は次の処理に進める。サブエージェントの結果は、後で確認する。


## サブエージェントのライフサイクル

サブエージェントは、以下のライフサイクルで動作する。

```
1. 起動 ─────────────────────────────────────────┐
   Orchestratorがpromptを渡してサブエージェントを起動   │
                                                   │
2. コンテキスト構築 ──────────────────────────────┤
   CLAUDE.md + agents/*.md + prompt を読み込む        │
   独立したコンテキストウィンドウが確保される           │
                                                   │
3. タスク実行 ────────────────────────────────────┤
   ファイルの読み書き、コード生成、分析など             │
   親のコンテキストとは完全に分離                      │
                                                   │
4. 結果返却 ──────────────────────────────────────┤
   実行結果のサマリを親エージェントに返す               │
   ファイルへの変更は永続化される                      │
                                                   │
5. 破棄 ──────────────────────────────────────────┘
   サブエージェントのコンテキストは破棄される
   次回は新しいコンテキストで起動される
```

ここで重要なのは**ステップ5の「破棄」**だ。サブエージェントは実行が終わるとコンテキストが破棄される。前回の実行内容を「覚えていない」。

これは制約に見えるが、実は大きなメリットだ。

- **毎回クリーンな状態で起動する**: 前回の実行でコンテキストが汚染されていても、次回には影響しない
- **ステートレスな設計を強制する**: エージェントの状態はファイルに永続化される。ファイルが「真実の唯一の情報源」になる
- **メモリリークがない**: コンテキストの肥大化を心配する必要がない

### 状態の永続化パターン

サブエージェントが「記憶を持たない」以上、必要な情報はすべてファイルに書き出す必要がある。筆者のフレームワークでは、各部門のSTATE.mdがこの役割を果たしている。

```markdown
# 開発部門 STATE

## 現在のスプリント
- タスクA: 完了
- タスクB: 進行中（PR #42 レビュー待ち）
- タスクC: 未着手

## 直近の完了タスク
- 2026-03-08: CI/CDパイプライン追加（元気ボタンアプリ）
- 2026-03-07: ShareToku パフォーマンス改善

## 注意事項
- MochiQのFlutterバージョンアップが必要（4.x → 5.x）
```

CTOエージェントは起動するたびにこのファイルを読み、現在の状態を把握する。タスクが完了したら、このファイルを更新して終了する。次回起動時には、更新されたファイルから最新の状態を読み取る。

**「ファイルがデータベースであり、Gitが変更履歴であり、Markdownがスキーマである。」**

この原則を一貫させることで、サブエージェントのステートレス性とシステム全体のステートフル性を両立できる。


## worktree isolation -- 安全な並列実行の仕組み

マルチエージェントで最も技術的に繊細な問題が、並列実行時のファイル競合だ。

CTOエージェントがコードを修正している最中に、CMOエージェントがREADMEを更新したらどうなるか。同じブランチ上で同時にファイルを変更すると、コンフリクトが発生する。

Claude Codeは、この問題を**git worktree**で解決できるオプションを提供している。

### git worktreeとは

git worktreeは、1つのリポジトリから複数の作業ディレクトリを作成する機能だ。各worktreeは独立したブランチで動作する。

```
メインリポジトリ（main ブランチ）
  │
  ├── worktree-1/ （agent/cto ブランチ）← CTOエージェントが使用
  ├── worktree-2/ （agent/cmo ブランチ）← CMOエージェントが使用
  └── worktree-3/ （agent/cfo ブランチ）← CFOエージェントが使用
```

各サブエージェントは専用のworktreeで作業するため、ファイルの競合が物理的に発生しない。作業が完了したら、変更をmainブランチにマージする。

### worktree isolationの使い方

worktree isolationを利用するには、サブエージェント起動時に`isolation: "worktree"`パラメータを指定する。**デフォルトでは、サブエージェントは親と同じワーキングディレクトリで動作する。** worktreeオプションを明示的に指定した場合のみ、独立した作業空間が作成される。

```
1. Orchestratorがサブエージェントを isolation: "worktree" で起動
2. Claude Codeが一時的なworktreeを作成
3. サブエージェントはworktree内でファイルを読み書き
4. 作業完了後、変更がコミットされる
5. 変更がなければworktreeは自動削除。変更があればブランチとパスが返される
```

### いつworktreeを使うべきか

すべてのサブエージェントにworktreeが必要なわけではない。以下の基準で判断する。

- **worktreeを使う:** 複数のエージェントが同時にファイルを変更する場合（開発エージェントの並列実行など）
- **worktreeを使わない:** 読み取り専用のタスク（朝ダイジェスト生成など）、または1つずつ順番に実行するタスク

AI-CEO Frameworkでは、状態ファイル（`.company/`配下）の読み書きは各部門エージェントの「担当ファイルだけを書き換える」ルールで競合を防いでいるため、worktreeを使わないケースが大半だ。コードの変更を伴う開発タスクで複数エージェントを並列実行する場合にのみ、worktreeを活用している。


## foreground vs background実行の使い分け

2つの実行モードは、タスクの性質によって使い分ける。

### foreground実行を選ぶケース

**結果を待って次の処理に使う場合**

```
例: 朝のダイジェスト生成

1. 各部門のSTATE.mdを読み取る（foreground）
2. 読み取った結果を統合してレポートを生成
```

ステップ2はステップ1の結果がないと実行できない。この場合、foreground実行で結果を待つ。

**単一タスクの完了を確認したい場合**

```
例: バグ修正

1. CTOエージェントにバグ修正を依頼（foreground）
2. 修正結果を確認してCEOに報告
```

### background実行を選ぶケース

**独立したタスクを並列実行する場合**

```
例: 週次レポートの同時生成

1. CTOエージェント: 開発レポート生成（background）
2. CMOエージェント: マーケレポート生成（background）
3. CFOエージェント: 経理レポート生成（background）
4. 全エージェントの完了を待って統合
```

3つのタスクは互いに依存しないため、同時に実行できる。

**即座に結果が不要な場合**

```
例: コンテンツの下書き生成

1. CMOエージェントに記事の下書きを依頼（background）
2. Orchestratorは「下書き作成中です」とCEOに報告
3. 完了したらapproval-queue.mdに追加される
```

### 判断フローチャート

```
タスクの結果が、次の処理の入力に必要か?
  ├── Yes → foreground
  └── No
        ├── 他のタスクと並列実行したいか?
        │     ├── Yes → background
        │     └── No → foreground（シンプルさ優先）
        └── 完了通知だけでよいか?
              ├── Yes → background
              └── No → foreground
```


## サブエージェントに渡すプロンプトの設計原則

サブエージェントの品質は、渡すプロンプトの品質で決まる。9ヶ月の運用で確立した5つのルールを紹介する。

### ルール1: 目的を1文で書く

冒頭で「何を達成するか」を1文で明示する。サブエージェントが「自分は何をすべきか」を即座に理解できるようにする。

```
悪い例:
「開発部門の状態を確認して、必要なら更新して、
ついでにバグがあれば修正して、レポートも出して」

良い例:
「開発部門のSTATE.mdを読み取り、現在のスプリント進捗を
サマリとして返却せよ」
```

### ルール2: 入力と出力を明示する

サブエージェントが「どのファイルを読むか」「どこに書くか」を曖昧にしない。

```markdown
## 入力
- /path/to/.company/departments/dev/STATE.md
- /path/to/.company/products/product-a/STATE.md

## 出力
- 実行結果のサマリテキスト（返却値として）
- /path/to/.company/departments/dev/STATE.md（更新がある場合）
```

### ルール3: 権限レベルを明示する

サブエージェントが「何をしてよいか」「何をしてはいけないか」を制約する。

```markdown
## 権限
- ファイルの読み取り: 許可
- STATE.mdの更新: 許可
- コードの変更: 不許可（本タスクでは状態確認のみ）
- 外部APIの呼び出し: 不許可
```

### ルール4: 品質基準を定義する

「何をもって完了とするか」を定義する。これがないと、サブエージェントは「それっぽい結果」を返して終了する。

```markdown
## 品質基準
- すべてのプロダクトの状態が含まれていること
- 各プロダクトに「フェーズ」「次のマイルストーン」が記載されていること
- 数値データ（完了タスク数、残タスク数）が含まれていること
```

### ルール5: コンテキストを最小限にする

不要な情報を渡さない。サブエージェントのコンテキストウィンドウも有限だ。

```
悪い例:
「会社の全ファイルを読んで状況を把握して」

良い例:
「以下の2ファイルだけを読み、開発部門の進捗を要約せよ:
- .company/departments/dev/STATE.md
- .company/products/sharetoku/STATE.md」
```


## 実装例: 最小限のマルチエージェント構成

ここまでの知識を統合して、実際に動作する最小限のマルチエージェント構成を示す。

### ディレクトリ構造

```
my-project/
  ├── CLAUDE.md                      ← Orchestrator定義
  ├── .claude/
  │     └── agents/
  │           ├── developer.md       ← 開発エージェント
  │           └── reviewer.md        ← レビューエージェント
  └── .project/
        ├── STATE.md                 ← プロジェクト状態
        └── tasks/
              └── current-sprint.md  ← 現在のタスク一覧
```

### CLAUDE.md（Orchestrator）

```markdown
# My Project — Orchestrator

> あなたはMy Projectの統括エージェントです。
> 開発タスクの指示と進捗管理を担当します。

## Thin Orchestratorの原則

- ファイルの中身を自分のコンテキストに読み込まず、
  ファイルパスだけを渡す
- 自分でコードを書かない。開発はdeveloperエージェントに委任する
- 自分でレビューしない。レビューはreviewerエージェントに委任する

## コマンド一覧

### /dev:implement "機能名"
1. `.project/tasks/current-sprint.md`から該当タスクの情報を確認
2. developerエージェントに実装を委任
3. 実装完了後、reviewerエージェントにレビューを委任
4. レビュー結果をユーザーに報告

### /dev:status
1. `.project/STATE.md`の内容を読み取り、現在の進捗を報告

## サブエージェントへの委任方法

タスクをサブエージェントに委任する際は、以下の情報を渡す:
1. タスクの目的（1文）
2. 参照ファイルパス
3. 成果物の出力先
4. 権限レベル（read-only / draft / execute）
5. 品質基準
```

### .claude/agents/developer.md（開発エージェント）

```markdown
# 開発エージェント

あなたはMy Projectの開発担当エージェントです。

## ペルソナ
実装力に優れたシニアエンジニア。
コードの品質、可読性、テスタビリティを重視する。

## 専門知識
- TypeScript / Next.js / React
- テスト駆動開発（TDD）
- クリーンアーキテクチャ

## 権限レベル
- **execute:** ソースコードの作成・変更、テストの作成・実行
- **draft:** package.jsonの依存関係変更、設定ファイルの変更

## ワークフロー

### 機能実装
1. タスクの要件を確認する
2. 必要なファイルを読み取り、既存コードの構造を把握する
3. 実装する（テストコードも同時に作成）
4. `.project/STATE.md`を更新する

## 品質基準
1. TypeScriptの型エラーがないこと
2. すべての新規コードにテストがあること
3. 既存のテストが壊れていないこと
4. 関数は30行以内に収めること
```

### .claude/agents/reviewer.md（レビューエージェント）

```markdown
# レビューエージェント

あなたはMy Projectのコードレビュー担当エージェントです。

## ペルソナ
品質に厳格なテックリード。
バグの予防とコードの保守性を最優先する。

## 専門知識
- コードレビューのベストプラクティス
- セキュリティ脆弱性の検出
- パフォーマンスのボトルネック分析

## 権限レベル
- **read-only:** コードの読み取り、分析、レビューコメントの生成

## ワークフロー

### コードレビュー
1. 指定されたファイルの変更内容を読み取る
2. 以下の観点でレビューする:
   - バグの可能性
   - セキュリティリスク
   - パフォーマンス問題
   - 可読性・保守性
   - テストの十分性
3. レビュー結果をサマリとして返却する

## 品質基準
1. 指摘は「問題の種類」「該当箇所」「改善案」の3点セットで記述
2. 重大度（Critical / Warning / Suggestion）を分類
3. 良い点も1つ以上指摘する（モチベーション維持）
```

### .project/STATE.md（プロジェクト状態）

```markdown
# Project STATE

## 現在のスプリント
Sprint 3（2026-03-03 〜 2026-03-14）

## タスク状態
| タスク | ステータス | 担当 |
|--------|-----------|------|
| ユーザー認証機能 | 完了 | developer |
| ダッシュボード画面 | 進行中 | developer |
| API レート制限 | 未着手 | - |

## 直近の変更
- 2026-03-08: ユーザー認証機能の実装完了、レビュー済み
```

### この構成のポイント

この最小構成には、本章で解説した要素がすべて含まれている。

1. **Orchestrator（CLAUDE.md）**: Thin原則に従い、自分ではコードを書かず、委任に徹する
2. **Worker（agents/*.md）**: それぞれが専門領域に特化し、権限レベルが明確
3. **状態ファイル（STATE.md）**: ステートレスなサブエージェントの「外部記憶装置」
4. **3層のプロンプト構造**: CLAUDE.md（共通） + agents/*.md（専門） + prompt（タスク固有）

ここから先は、エージェントを追加し、ワークフローを拡張し、承認パイプラインを組み込んでいく。それが第4章以降の内容だ。

次章では、Orchestratorの実装方法を具体的に解説する。
