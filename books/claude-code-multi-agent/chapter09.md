# エラーハンドリングとリカバリ戦略 -- 障害に強いマルチエージェントシステム

マルチエージェントシステムは、単一エージェントと比べて障害点が多い。1つのエージェントが失敗しただけなら影響は限定的だが、Orchestratorの判断ミスは全部門に波及する。STATE.mdの更新が競合すれば、複数のエージェントが古い情報を元に作業を進めてしまう。

1年以上の運用で、筆者は様々な障害パターンに遭遇してきた。本章では、マルチエージェント特有の障害パターンとその対処法を、実例とともに解説する。

## マルチエージェント特有の障害パターン

### パターン1: サブエージェントのタスク失敗

最も頻度が高い障害だ。サブエージェントが委任されたタスクを正しく完了できないケース。

**よくある原因:**

- コード生成のミス（構文エラー、ロジックエラー）
- ファイルの書き込みエラー（パスの間違い、権限不足）
- 期待するフォーマットと異なる出力
- 参照ファイルが存在しない、または古い

たとえば、CTOエージェントにLP改修を指示したとき、CSS のクラス名を間違えてスタイルが崩れるケース。エージェントは「タスク完了」と報告するが、実際にはブラウザで表示すると意図しないレイアウトになっている。

### パターン2: Orchestratorの判断ミス

CEOのコマンドを受けて、Orchestratorが間違った部門に委任するケース。

**実例:** 「月次の売上レポートを作って」というCEOの指示に対し、Orchestratorが営業エージェント（CSO）に委任した。しかし、正しくは経理エージェント（CFO）に委任すべきだった。CSOは「売上」という言葉から営業レポートを作成し、CEOが求めていた財務視点のレポートとは異なるものが出来上がった。

この種のミスは、Orchestratorの判断ロジックが曖昧な場合に起きる。CLAUDE.mdのコマンドルーティング定義が不十分だと、エッジケースで誤判断する。

### パターン3: 状態の不整合

複数のエージェントが同じファイルを更新しようとして競合するケース。

Claude Codeのサブエージェントは、バックグラウンド実行で並列に動かすことができる。ただし、worktree isolation（第3章参照）を使わない限り同じワーキングディレクトリで動作するため、同じファイルへの同時書き込みで競合が起きる可能性がある。また、フォアグラウンド実行であっても「エージェントAが読んだSTATE.mdを、次に起動したエージェントBが更新してしまい、エージェントAの成果物が古い情報に基づいている」というケースは起きる。

**実例:** 朝のダイジェスト生成中に、CTOエージェントが開発部門のSTATE.mdを読んだ。その直後、CEOが `/ai-ceo:dev:hotfix` で緊急修正を実行し、STATE.mdが更新された。しかし、ダイジェストには更新前の情報が反映されていた。

この問題の本質は、「読み取り」と「それに基づく判断」の間に時間差があることだ。

### パターン4: コンテキスト超過による品質劣化

前章で詳しく扱ったが、エラーハンドリングの文脈でも重要だ。

コンテキストの80%を超えた状態で新しいタスクを実行すると、出力の品質が目に見えて下がる。指示の一部を無視したり、参照すべきファイルを読み飛ばしたりする。

厄介なのは、この種の障害がエラーとして検出されないことだ。エージェントは「正常に完了」と報告するが、成果物の品質が低い。

### パターン5: 外部APIの障害

MCPサーバーへの接続エラー、GitHub APIのレート制限、Zenn CLIの実行失敗など、エージェント自身には責任のない外部障害。

**実例:** CSOエージェントが見込み客のWebサイトを分析するためにMCPサーバー経由でHTTPリクエストを送ったが、対象サイトがメンテナンス中で503エラーが返った。エージェントは「サイトの情報を取得できませんでした」と報告するが、提案書は不完全なまま生成された。

## リトライ戦略の設計

障害が発生したとき、最初にやるべきはリトライだ。ただし、同じプロンプトで再実行しても同じ結果になる。リトライには「フィードバック」を含めることが不可欠だ。

### フィードバック付きリトライ

AI-CEO Frameworkでは、リトライ時に前回の失敗情報をエージェントに渡す。

```
リトライ1回目:
「LP改修を再実行してください。
前回の結果: ヘッダーのCSSクラス名が間違っていた。
.header-mainではなく.site-headerを使ってください。
参照: 既存のCSSファイル src/styles/layout.css」
```

フィードバックには以下を含める。

1. **何が失敗したか:** 具体的なエラー内容
2. **なぜ失敗したか:** 原因の推定（可能なら）
3. **どう修正すべきか:** 具体的な修正方針
4. **追加の参照情報:** 前回のコンテキストに不足していた情報

単に「もう一度やって」と再実行するのは、リトライではなく「祈り」だ。

### 最大3回リトライ → エスカレーション

リトライ回数の上限を設けている。

```
1回目: タスク実行 → 失敗
2回目: フィードバック付きリトライ → 失敗
3回目: 別のアプローチでリトライ → 失敗
→ エスカレーション: CEOに報告
```

3回失敗した場合、エージェントの能力では解決できない問題である可能性が高い。それ以上リトライしてもトークンを浪費するだけだ。

2回目と3回目のリトライで重要なのは、**アプローチを変える**ことだ。

- 2回目: フィードバックを付けて同じアプローチで再試行
- 3回目: アプローチ自体を変更する（別のファイルを参照する、タスクを分割する、前提条件を変える等）

### 指数バックオフは不要

従来のソフトウェアエンジニアリングでは、リトライ時に指数バックオフ（1秒、2秒、4秒...と待機時間を増やす）が定石だ。しかし、AIエージェントのリトライでは不要だ。

理由は2つある。

1. **エージェントの失敗原因は待機時間と無関係。** APIのレート制限が原因なら待機が有効だが、エージェントの失敗の大半は「コンテキスト不足」「指示の曖昧さ」「ロジックミス」だ。待っても解決しない。
2. **人間が介在するフロー。** エスカレーション後は人間が対処する。機械的に再試行を続ける意味がない。

ただし、外部APIの障害（レート制限、一時的な接続エラー）の場合は、数秒の待機を入れることがある。これはエージェントレベルではなく、スクリプトレベルで制御する。

## エスカレーション設計

3回のリトライで解決しなかった問題は、人間にエスカレーションする。

### エスカレーションのフロー

```
サブエージェント: 3回失敗
  ↓
Orchestrator: エラー情報を収集
  ↓
error-log.md に記録
  ↓
approval-queue.md にエスカレーションとして追加
  ↓
CEO: /ai-ceo:morning で確認、対処を指示
```

### error-log.mdの記録フォーマット

各部門のディレクトリに `error-log.md` を配置し、障害情報を記録する。

```markdown
# 開発部門 エラーログ

## 2026-03-09 14:30 -- LP改修失敗

- **タスク:** Focalize LPのヒーローセクション改修
- **エージェント:** CTO
- **試行回数:** 3
- **エラー内容:**
  1回目: CSSクラス名の不一致で
    スタイルが崩れた
  2回目: クラス名修正後、
    レスポンシブ対応が崩れた
  3回目: レスポンシブ修正後、
    JavaScriptのイベントハンドラが
    動作しなくなった
- **推定原因:** 既存のCSS/JSの依存関係が
  複雑で、部分的な変更が
  副作用を引き起こしている
- **推奨対応:** 人間がブラウザで
  DevToolsを使って確認し、
  変更方針を明確にしてから再実行
- **ステータス:** エスカレーション中
```

このフォーマットのポイントは、**各試行の経過を記録している**ことだ。CEOが確認したとき、「何を試して、何がダメだったか」が一目で分かる。ゼロから問題を理解する手間が省ける。

### エスカレーションの優先度判定

エスカレーションにも優先度がある。

| 優先度 | 条件 | 対応期限 |
|--------|------|---------|
| critical | サービス停止、セキュリティインシデント | 即時 |
| high | 売上に直結するタスクの失敗、クライアント対応の遅延 | 当日中 |
| medium | 開発タスクの失敗、レポート生成の不具合 | 2-3日以内 |
| low | 内部ドキュメントの生成ミス、フォーマットの不備 | 次回のメンテナンス時 |

優先度はエージェントが自動判定する。判定基準はCLAUDE.mdに以下のように記載する。

```markdown
## エスカレーション時の優先度判定

- クライアントに影響 → high
- 本番サービスに影響 → critical
- 内部作業のみ → medium
- 品質の問題（動作はする） → low
```

## 冪等性の設計

マルチエージェントシステムでは、同じタスクが2回実行される可能性がある。ネットワークの問題でOrchestratorがサブエージェントの完了を確認できず、再度委任してしまうケース。あるいは、CEOが同じコマンドを誤って2回入力するケースだ。

同じタスクを2回実行しても安全であること、これが冪等性だ。

### ファイルベースの冪等性

AI-CEO Frameworkの成果物はすべてファイルとして出力される。ファイルの書き込みは本質的に冪等だ。同じ内容を同じファイルに2回書いても、結果は同じだ。

ただし、以下のケースでは注意が必要になる。

**承認キューへの重複追加:**

提案書作成タスクが2回実行されると、`approval-queue.md` に同じアイテムが2つ追加されてしまう。

対策として、エージェントは承認キューに追加する前に、同じファイルパスのアイテムが既に存在しないかチェックする。

```markdown
## 承認キュー追加のルール

approval-queue.md にアイテムを追加する前に、
同じファイルパスのアイテムが既に存在しないか確認する。
存在する場合は、既存のアイテムを更新する
（新規追加しない）。
```

**STATE.mdの上書き:**

STATE.mdは「現在の状態」を反映するものなので、上書きは基本的に安全だ。ただし、STATE.mdの「完了タスク」セクションに同じタスクが2回追記されないよう、エージェントは追記前にチェックを行う。

### IDベースの重複検出

より堅牢な冪等性が必要な場合は、タスクにIDを振る。

```
Orchestrator → CTOエージェント:
「タスクID: TASK-2026-03-09-001
LP改修を実行してください。
※ このタスクIDで既に成果物が存在する場合は
スキップしてください。」
```

エージェントは、成果物ファイル名にタスクIDを含めるか、完了ログでタスクIDを記録する。同じIDのタスクを受け取ったら、既に完了していることを検出してスキップする。

## ロールバック戦略

エージェントが行った変更を元に戻す必要がある場面は必ず来る。

### gitベースのロールバック

AI-CEO Frameworkの成果物はすべて `.company/` ディレクトリに格納され、gitで管理されている。ロールバックは `git checkout` や `git revert` で行える。

```bash
# 直前のコミットを取り消す
git revert HEAD

# 特定のファイルを以前の状態に戻す
git checkout HEAD~1 -- .company/departments/sales/proposals/proposal-abc.md
```

gitの利点は、**変更の粒度が柔軟**なことだ。1ファイルだけ戻すことも、特定のコミット全体を取り消すこともできる。

### git worktreeを使った安全な変更

開発タスクでリスクの高い変更を行う場合、git worktreeを使って安全な作業空間を作る方法がある。

```bash
# 別のディレクトリにworktreeを作成
git worktree add ../feature-lp-redesign feature/lp-redesign
```

worktreeは本体のリポジトリとは独立したディレクトリで作業できるため、失敗してもメインのコードベースに影響しない。変更が成功したらマージし、失敗したらworktreeごと削除すればよい。

エージェントにworktreeを使わせる場合は、CLAUDE.mdに以下のように指示する。

```markdown
## リスクの高い変更の手順

本番コードに影響する変更を行う場合:
1. feature ブランチを作成する
2. ブランチ上で変更を実装する
3. テストを実行して確認する
4. PRを作成してapproval-queue.mdに追加する
5. CEO承認後にマージする

直接mainブランチにpushしてはならない。
```

### 成果物の世代管理

ロールバックをより簡単にするために、重要な成果物はバージョンを付けて保存する運用もある。

```
proposals/
├── proposal-abc-v1.md    # 初版
├── proposal-abc-v2.md    # 価格修正版
└── proposal-abc-v3.md    # 最終版
```

gitで履歴を追えるので必須ではないが、「CEOが却下した版と修正した版を並べて比較したい」場面では便利だ。

## 実例: 開発エージェントのエラーリカバリ

実際のエラーリカバリの全体像を、具体的なシナリオで示す。

### シナリオ: PRの自動作成と修正

CEOが以下のコマンドを実行する。

```
CEO: /ai-ceo:dev:sprint
  「FocalizeにCSVエクスポート機能を追加して」
```

**1回目: 実装 → テスト失敗**

CTOエージェントがコードを生成し、PRを作成する。しかし、CI/CDパイプラインでテストが失敗する。

```
テスト結果: FAIL
失敗テスト: test_csv_export_with_japanese_chars
原因: 日本語文字のエンコーディングが
  UTF-8 BOMなしで出力されている。
  Excelで開くと文字化けする。
```

**2回目: フィードバック付きリトライ → テスト失敗**

Orchestratorが失敗情報をCTOエージェントに渡して再実行する。

```
「CSVエクスポートの日本語対応を修正してください。
前回の失敗: UTF-8 BOMなしで出力していたため、
Excelで文字化けした。
修正方針: UTF-8 BOM付きで出力するように変更する。
対象ファイル: src/exporters/csv.ts」
```

CTOエージェントがBOM付きUTF-8に修正する。しかし、今度は別のテストが失敗する。

```
テスト結果: FAIL
失敗テスト: test_csv_import_round_trip
原因: BOM付きUTF-8で出力すると、
  インポート側のパーサーがBOMを
  データの一部として認識してしまう。
```

**3回目: アプローチ変更 → テスト失敗**

3回目は、アプローチ自体を変更する。

```
「CSVエクスポートのエンコーディング戦略を
根本的に見直してください。
BOM付き/なしの問題ではなく、
エクスポート設定でエンコーディングを
選択できるようにする方針を検討してください。
参照: src/exporters/ ディレクトリ全体」
```

CTOエージェントが設定オプションを追加するが、既存のエクスポーターインターフェースとの整合性が取れず、テストが失敗する。

**エスカレーション**

3回失敗したため、エスカレーションが発動する。

1. `error-log.md` に3回の試行経過を記録
2. `approval-queue.md` にエスカレーションとして追加

```markdown
### [AQ-018] エスカレーション: CSVエクスポート実装失敗
- **部門:** 開発
- **種別:** escalation
- **ファイル:** .company/departments/dev/error-log.md
- **概要:** CSVエクスポートの日本語対応で
  3回失敗。BOM処理とインポーターの
  互換性問題。人間の設計判断が必要。
- **登録日:** 2026-03-09
- **優先度:** medium
```

CEOが朝ダイジェストで確認し、以下のような判断を行う。

```
CEO: CSVはShift_JISで出力しよう。
  日本のExcelユーザーはそれが一番確実。
  インポート側はUTF-8とShift_JIS両方に
  対応させて。
```

この判断をもとに、CTOエージェントが再度実装する。今度はCEOの明確な方針があるため、1回で成功する。

### このフローのポイント

1. **リトライのたびにフィードバックが蓄積される。** 1回目の失敗情報が2回目に、2回目の失敗情報が3回目に引き継がれる
2. **3回目はアプローチを変える。** 同じ方向で3回リトライしない
3. **エスカレーションは「失敗の報告」ではなく「判断の要求」。** CEOに「エンコーディングの方針を決めてください」と問いかけている
4. **人間の判断が入ると解決する。** エージェントに足りなかったのは「日本のExcelユーザーの慣習」という暗黙知だった

## 障害時のデバッグ手順

エスカレーションを受けたCEOが問題を調査するための手順をまとめる。

### ステップ1: error-log.mdを確認する

まず、該当部門の `error-log.md` を読む。3回の試行で何が起きたかを把握する。

### ステップ2: 成果物ファイルを確認する

エージェントが生成したファイルを直接確認する。エラーの原因が「出力の品質問題」なのか「技術的な障害」なのかを切り分ける。

### ステップ3: gitの差分を確認する

```bash
git diff HEAD~3..HEAD -- .company/departments/dev/
```

エージェントがどのような変更を行ったかを差分で確認する。意図しない変更が混入していないかをチェックする。

### ステップ4: 原因に応じた対処を選ぶ

| 原因 | 対処 |
|------|------|
| エージェントの能力不足 | CEOが方針を明確化して再実行 |
| 参照情報の不足 | 必要なファイルを追加して再実行 |
| 外部APIの障害 | 時間を置いて再実行 |
| 設計の問題 | CLAUDE.mdのワークフローを修正 |
| コンテキスト超過 | タスクを分割して再実行 |

### ステップ5: 再発防止

同じ障害が繰り返される場合は、CLAUDE.mdまたは `steering/` のルールファイルを更新する。

```markdown
## CSVエクスポートの方針（2026-03-09追加）

日本語を含むCSVは Shift_JIS で出力する。
インポート時は BOM の有無で UTF-8/Shift_JIS を
自動判定する。
```

この「障害 → 学び → ルール化」のサイクルが、システムを段階的に堅牢にしていく。

## 障害対応のまとめ

マルチエージェントシステムのエラーハンドリングで最も重要なのは、**「障害は必ず起きる」という前提で設計する**ことだ。

単一エージェントなら、失敗したら人間が気づいて対処する。しかし10のエージェントが自律的に動くシステムでは、失敗を検出し、リトライし、解決できなければ人間にエスカレーションする仕組みが不可欠だ。

設計の要点は3つだ。

1. **フィードバック付きリトライ:** 同じプロンプトで再実行しない。失敗情報を含めて品質を上げる
2. **エスカレーションの明確な基準:** 3回失敗したら人間に上げる。判断に必要な情報を構造化して伝える
3. **ファイルシステムとgitによるロールバック:** すべての変更が追跡可能で、いつでも元に戻せる安心感

この3つが揃えば、障害が起きても被害を最小限に抑え、迅速に復旧できるシステムになる。
