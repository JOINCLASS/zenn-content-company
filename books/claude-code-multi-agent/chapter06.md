# エージェント間通信の設計 -- データの受け渡しパターン

Orchestratorと専門エージェントの設計ができた。次の課題は、これらのエージェントがどうやってデータを受け渡すかだ。本章では、Claude Codeのマルチエージェントシステムにおける3つの通信パターンを解説する。

## なぜエージェント間通信が難しいか

一般的なソフトウェアでは、モジュール間の通信は関数呼び出しやAPIリクエストで行う。呼び出し元と呼び出し先が同じプロセス空間にいるか、ネットワーク越しに直接通信できる。

Claude Codeのサブエージェントは、そのどちらでもない。

**各エージェントは独立したコンテキストを持つ。** CTOエージェントが知っている情報を、CFOエージェントは知らない。Orchestratorを介して間接的に情報を渡すこともできるが、それではOrchestratorのコンテキストが膨張する。Thin Orchestrator の原則に反する。

**エージェントは独立したプロセスとして動く。** Claude Codeのサブエージェントはバックグラウンド実行で並列に動かすことができる（第3章参照）。しかし、並列実行中のエージェント同士がリアルタイムにデータを交換する仕組みはない。各エージェントは起動時に渡された情報とファイルシステムの内容だけを頼りに作業する。

この制約のもとでエージェント間通信を実現するには、**ファイルシステムを共有メモリとして使う**アプローチが有効だ。

## 3つの通信パターン

AI-CEO Framework では、3つの通信パターンを使い分けている。

### パターン1: ファイルベース通信

最もシンプルなパターン。あるエージェントがファイルに書き込み、別のエージェントがそのファイルを読む。

```
[CTO] → 書き込み → .company/products/focalize/STATE.md
                                    ↓
[CMO] ← 読み取り ← .company/products/focalize/STATE.md
```

CTOエージェントがプロダクトの状態を更新すると、CMOエージェントはそのファイルを読んで最新の機能一覧を把握し、マーケティング施策に反映する。

**使いどころ:** エージェント間で共有すべき「事実情報」がある場合。プロダクトの現在の状態、技術スタック、ブランドガイドラインなど。

**利点:**
- 実装が最もシンプル
- ファイルが残るので監査可能
- どのエージェントからも同じ情報にアクセスできる

**注意点:**
- 複数エージェントが同じファイルに書き込むと競合が起きうる
- ファイルの「鮮度」を保証する仕組みが必要

AI-CEO Framework では、各ファイルに「最終更新」のタイムスタンプを含めることで鮮度を管理している。

### パターン2: STATE.mdパターン

ファイルベース通信の発展形で、AI-CEO Framework の通信の中核を担うパターンだ。

各部門が自分のSTATE.mdを持ち、そこに現在の状態を書き込む。Orchestratorや他のエージェントは、必要なときにSTATE.mdを読むだけでその部門の状態を把握できる。

```
.company/departments/
├── dev/STATE.md        # 開発部門の状態
├── marketing/STATE.md  # マーケ部門の状態
├── sales/STATE.md      # 営業部門の状態
├── finance/STATE.md    # 経理部門の状態
├── cs/STATE.md         # CS部門の状態
└── legal/STATE.md      # 法務部門の状態
```

STATE.mdの典型的な構造は以下のようになる。

```markdown
# 開発部門 -- 状態

## 現在のステータス: 正常稼働中

## 進行中タスク
| タスク | 担当 | ステータス | 期限 |
|--------|------|-----------|------|
| LP改修 | CTO  | 実装中    | 3/15 |
| API v2 | CTO  | 設計中    | 3/20 |

## 完了タスク（今月）
- 2026-03-05: ダッシュボード機能リリース
- 2026-03-02: パフォーマンス改善

## ブロッカー
なし

## 次のマイルストーン
3/20: API v2 リリース
```

このパターンの核心は、**書き込む側と読み取る側が分離されている**ことだ。

- **書き込み責任:** 各部門のエージェントが、タスク完了時に自部門のSTATE.mdを更新する
- **読み取り責任:** Orchestratorや他部門のエージェントが、必要なときにSTATE.mdを参照する

書き込むのは常にその部門のエージェントだけなので、ファイルの競合が起きない。

```markdown
## 部門状態の更新

タスク完了時に
`.company/departments/dev/STATE.md` を更新する:
- 進行中タスクの状態変更
- 完了タスクの記録
- 次のスプリントの予定
```

**使いどころ:** 各部門の「現在の状態」をリアルタイムに共有したい場合。朝のダイジェスト生成、部門横断の調整判断など。

朝ダイジェストエージェント（`ai-ceo-morning.md`）は、このパターンの最も典型的な読み取り側だ。

```markdown
## 実行手順

1. `.company/approval-queue.md` を読み取り、
   承認待ちアイテムを収集
2. `.company/departments/*/STATE.md` を全て読み取り、
   各部門の状態を収集
3. `.company/products/*/STATE.md` を全て読み取り、
   プロダクト状態を収集
4. `.company/STATE.md` を読み取り、
   全体のコンテキストを確認
5. ダイジェストを生成
```

6つの部門と9つのプロダクトのSTATE.mdを読んでも、それぞれが20-30行の構造化されたマークダウンであれば、全体で3,000-5,000トークン程度に収まる。Thin Orchestratorの原則を守れる範囲だ。

### パターン3: 承認キュー（非同期メッセージング）

ファイルベース通信とSTATE.mdパターンは「状態の共有」だ。一方、承認キューパターンは「アクションの要求」を扱う。

```markdown
# 承認待ちキュー

## 承認待ち (2件)

### [AQ-014] 提案書送付: 株式会社ABC
- **部門:** 営業
- **種別:** proposal
- **ファイル:** .company/departments/sales/
  proposals/proposal-abc-2026-03.md
- **概要:** DX支援サービスの提案書を
  ABC社に送付する
- **登録日:** 2026-03-08
- **優先度:** high

### [AQ-015] 月次レポート公開
- **部門:** 経理
- **種別:** internal_docs
- **ファイル:** .company/departments/finance/
  monthly-2026-02.md
- **概要:** 2月の月次財務レポートを確定する
- **登録日:** 2026-03-08
- **優先度:** medium
```

このパターンの通信フローは以下のようになる。

```
[CSO] → 提案書を作成 → proposals/に保存
      → approval-queue.md にアイテム追加
                    ↓
[Orchestrator] ← approval-queue.md を読む
               → CEOに「承認待ちが2件あります」と報告
                    ↓
[CEO] → /ai-ceo:approve AQ-014
                    ↓
[Orchestrator] → approval-queue.md からアイテム削除
               → decisions/{month}.md に承認記録
               → CSOエージェントに「送付OK」を伝達
```

**使いどころ:** 対外的な影響があるアクション（メール送信、デプロイ、請求書発行）の承認フロー。人間のCEOが最終判断を挟む必要がある場面。

承認キューは、通信パターンであると同時に、**ガバナンスの仕組み**でもある。AIエージェントが自律的に動く中で、「ここだけは人間が確認する」というチェックポイントを設けている。

## パターン別の使い分けガイド

3つのパターンの使い分けをまとめる。

| パターン | データの性質 | 書き込み | 読み取り | 例 |
|---|---|---|---|---|
| ファイルベース | 共有の事実情報 | 複数可 | 複数可 | 技術スタック、ブランドガイド |
| STATE.md | 部門の現在状態 | その部門のみ | 誰でも | 進行中タスク、KPI |
| 承認キュー | アクション要求 | 誰でも追加 | Orchestratorが管理 | デプロイ申請、提案書送付 |

判断基準はシンプルだ。

- **「今の状態を伝えたい」** → STATE.md パターン
- **「これを使ってほしい」** → ファイルベース通信
- **「これを承認してほしい」** → 承認キューパターン

## ディレクトリ構造の設計

通信パターンを支えるのが、ディレクトリ構造だ。AI-CEO Framework のディレクトリ構造を見ると、通信の流れが構造に反映されていることが分かる。

```
.company/
├── STATE.md              # 会社全体の状態（全員が読む）
├── VISION.md             # ミッション・ビジョン（全員が読む）
├── ROADMAP.md            # ロードマップ（全員が読む）
├── approval-queue.md     # 承認キュー（全員が書く→CEOが読む）
│
├── steering/             # 全社ルール（全員が読む、変更はCEO承認）
│   ├── brand.md
│   ├── tech-stack.md
│   ├── policies.md
│   └── permissions.md
│
├── departments/          # 部門ごとの状態と成果物
│   ├── dev/
│   │   ├── STATE.md      # CTOが書く
│   │   └── sprint-*.md   # CTOが書く
│   ├── marketing/
│   │   ├── STATE.md      # CMOが書く
│   │   └── content-calendar-*.md
│   ├── sales/
│   │   ├── STATE.md      # CSOが書く
│   │   └── proposals/    # CSOが書く→CEOが承認
│   ├── finance/
│   │   ├── STATE.md      # CFOが書く
│   │   └── monthly-*.md  # CFOが書く
│   ├── cs/
│   │   └── STATE.md      # CS Leadが書く
│   └── legal/
│       └── STATE.md      # 法務が書く
│
├── products/             # プロダクトごとの状態
│   ├── focalize/
│   │   └── STATE.md      # CTOが書く、全員が読む
│   └── mochiq/
│       └── STATE.md
│
└── decisions/            # CEO意思決定ログ
    └── 2026-03.md        # Orchestratorが書く
```

設計の要点は3つだ。

**1. 読み取り方向は下から上**
各部門が自分の `departments/{dept}/STATE.md` を更新し、Orchestratorがそれを集約する。ボトムアップのデータ集約。

**2. 指示方向は上から下**
CEOの意思決定は `decisions/` に記録され、各部門エージェントが次のセッションで参照する。トップダウンの意思伝達。

**3. ルールは `steering/` に集約**
ブランド規約、技術スタック、権限設定など、全部門に共通するルールは `steering/` に集めている。各エージェントは自分の CLAUDE.md で `steering/` 配下のファイルを参照ファイルとして指定する。

## データ形式の標準化

エージェント間でデータをやり取りする際、形式を統一しておくことが重要だ。AI-CEO Framework では、以下の使い分けをしている。

### マークダウンテーブル（第一選択）

```markdown
| プロダクト | MRR | ユーザー数 | チャーン率 |
|-----------|-----|----------|----------|
| Focalize  | ¥0  | 15       | -        |
| MochiQ    | ¥0  | 3        | -        |
```

**利点:** 人間が読みやすい。Zenn記事やレポートにそのまま使える。Claude Codeが最もパースしやすい形式の一つ。

**使う場面:** STATE.md、レポート、提案書など、人間も読む文書。

### YAML（設定・ルール系）

```yaml
auto_approve_limit: 5000  # 円
ceo_approval_above: 5000  # 円
monthly_ai_budget: 200    # USD
```

**利点:** 構造化されたデータを簡潔に表現できる。コメントが書ける。

**使う場面:** 権限設定、閾値定義、エージェント定義のfrontmatterなど。

### JSON（プログラム連携用）

```json
{
  "site": "example.com",
  "score": 72,
  "issues": ["missing_og_image", "slow_fcp"]
}
```

**利点:** プログラムからのパースが容易。

**使う場面:** スクリプトが読み書きするデータ、API連携のデータなど。

基本方針は「人間が読む → マークダウン、設定系 → YAML、プログラム連携 → JSON」だ。迷ったらマークダウンテーブルを選ぶ。Claude Code はマークダウンテーブルのパースが得意で、行の追加・変更も自然言語の指示で正確に行える。

## 通信のタイミング設計

エージェント間の通信には、3つのタイミングパターンがある。

### 同期的通信

Orchestratorがサブエージェントを呼び出し、結果を待つ。最も基本的なパターンだ。

```
CEO: /ai-ceo:fin:monthly-report
  → Orchestrator: CFOエージェントを呼び出し
    → CFO: レポート作成、STATE.md更新
  ← Orchestrator: 完了報告をCEOに返す
```

1つのCEOコマンドに対して、1つのエージェントが応答する。依存関係がない、単一部門で完結するタスクに使う。

### 非同期的通信

承認キューを介した通信は非同期だ。

```
セッション1:
  CEO: /ai-ceo:sales:proposal "ABC社"
    → CSO: 提案書を作成、approval-queueに追加
    → Orchestrator: 「承認待ちに追加しました」

（時間経過 -- CEOが提案書を確認）

セッション2:
  CEO: /ai-ceo:approve AQ-014
    → Orchestrator: 承認処理、decisionsに記録
```

セッション1とセッション2は別のタイミングで実行される。承認キューがセッション間の橋渡し役を担う。

### イベント駆動的通信

STATE.mdの変更をトリガーにして、次のアクションが発動するパターンだ。

```
1. CTOがプロダクトの新機能をリリース
   → products/focalize/STATE.md を更新

2. 次のCMOセッションで、STATE.mdの変更を検知
   → 新機能のプロモーション記事を作成

3. CSO が新機能を営業資料に反映
```

厳密にはClaude Codeに「ファイル変更の監視」機能はないため、これは朝のダイジェスト（`/ai-ceo:morning`）を起点とした擬似的なイベント駆動だ。朝ダイジェストが各部門のSTATE.mdを読んで変更を検出し、必要なアクションをCEOに提案する。

## 実装例: 部門横断フロー

最後に、3つの通信パターンが組み合わさる実際のフローを見てみよう。「営業部門が提案書を作成し、法務部門がレビューし、CEOが承認する」というシナリオだ。

### ステップ1: 営業部門が提案書を作成

```
CEO: /ai-ceo:sales:proposal "ABC社"
```

CSOエージェントが起動し、以下を実行する。

1. `.company/products/focalize/STATE.md` を読む（ファイルベース通信）
2. `.company/steering/brand.md` を読む（ファイルベース通信）
3. 提案書を `.company/departments/sales/proposals/proposal-abc-2026-03.md` に書き出す
4. `.company/departments/sales/STATE.md` を更新（STATE.mdパターン）
5. `.company/approval-queue.md` にアイテムを追加（承認キュー）

### ステップ2: 法務部門がレビュー

```
CEO: /ai-ceo:legal:review
  ".company/departments/sales/proposals/
   proposal-abc-2026-03.md"
```

法務エージェントが起動し、以下を実行する。

1. 提案書ファイルを読む（ファイルベース通信）
2. `.company/steering/policies.md` を読む（ファイルベース通信）
3. リスク評価を実施
4. レビュー結果を `.company/departments/legal/review-abc-2026-03.md` に出力
5. `.company/departments/legal/STATE.md` を更新（STATE.mdパターン）

### ステップ3: CEOが承認

```
CEO: /ai-ceo:approve AQ-014
```

Orchestratorが以下を実行する。

1. `approval-queue.md` からアイテムを削除
2. `decisions/2026-03.md` に承認記録を追加
3. CEOに完了を報告

このフロー全体で、3つの通信パターンがすべて使われている。

- **ファイルベース通信:** プロダクト情報、ブランド規約の参照
- **STATE.mdパターン:** 各部門の状態更新
- **承認キュー:** CEOへの承認要求と承認記録

そして、Orchestratorはフロー全体を通じて、ファイルの中身を読み込んでいない。パスの受け渡しとコマンドのルーティングだけを行っている。Thin Orchestrator の原則が守られている。

## 通信設計のまとめ

エージェント間通信の設計で最も重要なのは、**ファイルシステムが唯一の共有メモリである**という制約を受け入れることだ。

この制約は一見不便に見えるが、実は大きな利点がある。

- **すべての通信が永続化される。** ファイルに残るので、あとから「どのエージェントが何を書いたか」を追跡できる。
- **通信がgitで管理できる。** `.company/` ディレクトリをgitに含めれば、全通信の履歴がバージョン管理される。
- **人間が介入しやすい。** マークダウンファイルなので、CEOがテキストエディタで直接編集して、エージェントに指示を与えることもできる。

API呼び出しやメッセージキューでは得られない、ファイルベース通信ならではの利点だ。
