# 専門エージェントの設計と実装 -- 役割を持ったAIを作る

前章でOrchestratorの設計を解説した。本章では、Orchestratorが委任する先となる「専門エージェント」の設計原則と具体的な実装方法を、6つの実例とともに解説する。

## 専門エージェントの設計原則

### 1エージェント = 1責務

マルチエージェント設計の最も重要な原則は、**1つのエージェントに1つの責務だけを持たせる**ことだ。

「マーケティングと営業を1つのエージェントにまとめたほうが効率的では？」と思うかもしれない。しかし、責務を混ぜると以下の問題が起きる。

- **CLAUDE.md が肥大化する。** 2つの役割分の専門知識、ワークフロー、成果物テンプレートを1ファイルに詰め込むと、コンテキスト消費が倍になる。
- **判断がブレる。** マーケ視点と営業視点で最適な判断が異なる場面で、エージェントが混乱する。たとえば、「価格を下げるべきか」はマーケ（認知拡大）と営業（利益確保）で答えが異なる。
- **テストが困難になる。** 責務が混ざっていると、何をテストすればいいか分からなくなる。

単一責務にすると、各エージェントのCLAUDE.mdは80-120行に収まり、コンテキストウィンドウの1%以下で動作する。

### ペルソナの設計

各エージェントには明確なペルソナを設定する。これは単なる演出ではなく、判断の一貫性に直結する設計判断だ。

```markdown
## ペルソナ

経験豊富なテックリード。実用性を重視し、
オーバーエンジニアリングを避ける。
「動くものを最速で出し、フィードバックを得て改善する」
がモットー。
```

このペルソナが設定されていると、エージェントは設計判断を迫られたとき「実用性を重視する」方向に一貫して倒す。「完璧なアーキテクチャ」ではなく「動くもの」を優先する。ペルソナがないと、同じエージェントでもセッションごとに判断がブレる。

## .claude/agents/ ディレクトリ構成

Claude Code では、サブエージェントの定義ファイルを `.claude/agents/` ディレクトリに配置する。AI-CEO Framework のディレクトリ構成は以下の通りだ。

```
.claude/agents/
├── ai-ceo-cto.md            # CTO（開発統括）
├── ai-ceo-cmo.md            # CMO（マーケ統括）
├── ai-ceo-cfo.md            # CFO（経理統括）
├── ai-ceo-cso.md            # CSO（営業統括）
├── ai-ceo-cs-lead.md        # CS部長
├── ai-ceo-legal.md          # 法務部長
├── ai-ceo-tax-accountant.md # 税理士
├── ai-ceo-content-engine.md # コンテンツ量産
├── ai-ceo-growth.md         # グロース分析
├── ai-ceo-morning.md        # 朝ダイジェスト
├── ai-ceo-hr.md             # 人事
└── ai-ceo-init.md           # 初期セットアップ
```

命名規則は `ai-ceo-{role}.md` で統一している。プレフィックスを揃えることで、エージェント一覧の把握が容易になる。

## エージェント定義ファイルの書き方

各エージェント定義ファイルは、以下の構造を持つ。

```markdown
---
name: ai-ceo-cto
description: CTO/開発部長エージェント。
  プロダクト開発全般を統括する。
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---

# CTO / 開発部長エージェント

あなたは AI-CEO Framework の CTO です。

## ペルソナ
{このエージェントの人格と判断基準}

## 担当領域
{責務の範囲}

## 権限レベル
{execute / draft の区分}

## 参照ファイル
{読むべきファイルのパス一覧}

## ワークフロー
{コマンド別の実行手順}

## 成果物テンプレート
{出力ファイルの形式}

## 品質検証
{完了条件のチェックリスト}

## 部門状態の更新
{STATE.md の更新ルール}
```

各セクションの設計意図を説明する。

### frontmatter（メタデータ）

```yaml
---
name: ai-ceo-cto
description: CTO/開発部長エージェント。
  プロダクト開発全般を統括する。
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---
```

`tools` フィールドで、このエージェントが使えるツールを制限する。たとえば、朝ダイジェストエージェントは読み取り専用なので `Read` と `Grep` だけを許可している。

```yaml
---
name: ai-ceo-morning
description: 朝ダイジェストエージェント。
  全部門の状態を収集し、CEO向けサマリを生成する。
tools:
  - Read
  - Grep
---
```

ツールを制限することで、エージェントが意図しない操作（ファイルの書き換えやコマンド実行）を行うリスクを排除できる。

### 権限レベル

各エージェントの権限は `execute` と `draft` の2段階に分ける。

```markdown
## 権限レベル

- **execute:** コスト分析、収支レポート、予算追跡
- **draft:** 請求書発行、支払い承認、予算変更
```

対外的な影響があるアクション（顧客への送信、本番デプロイ、金銭の移動）は必ず `draft` にする。エージェントが `draft` モードで生成した成果物は、承認キュー（`approval-queue.md`）に追加され、CEOの承認を待つ。

### 参照ファイル

エージェントが読むべきファイルのパスを明示する。

```markdown
## 参照ファイル

- 経理部門状態: `.company/departments/finance/STATE.md`
- コスト追跡: `.company/departments/finance/cost-tracking.md`
- 会社状態: `.company/STATE.md`
- 権限・閾値: `.company/steering/permissions.md`
```

これがエージェント間通信の起点になる。各エージェントは「自分が読むべきファイル」と「自分が書くべきファイル」を知っている。詳細は次章で解説する。

### 成果物テンプレート

出力の形式を事前に定義しておく。

```markdown
### 月次レポート
出力先: `.company/departments/finance/monthly-{YYYY-MM}.md`

# 月次財務レポート -- {YYYY年MM月}

## サマリ
| 項目 | 金額 | 前月比 |
|------|------|--------|

## プロダクト別収支
## インフラコスト
## AI運用コスト
## キャッシュフロー
## 注意事項・アクション
```

テンプレートを定義しておくと、出力の品質が安定する。エージェントがセッションごとに異なるフォーマットで出力する問題を防げる。また、他のエージェントがこの出力を読む際にも、形式が固定されているのでパースが容易になる。

## 6つの専門エージェント実装例

ここからは、AI-CEO Framework で実際に運用している6つの専門エージェントの設計を解説する。

### 1. CTO（開発統括）

CTOエージェントの特徴は、全ツールへのフルアクセスを持つことだ。`Read`, `Write`, `Edit`, `Bash`, `Grep` のすべてを使える。

```markdown
# CTO / 開発部長エージェント

あなたは AI-CEO Framework の CTO です。

## ペルソナ

経験豊富なテックリード。実用性を重視し、
オーバーエンジニアリングを避ける。
「動くものを最速で出し、
フィードバックを得て改善する」がモットー。

## ワークフロー

### /ai-ceo:dev:sprint
1. `.company/products/{name}/STATE.md`
   からバックログを確認
2. 優先度の高いタスクを選定
   （最大3タスク -- アトミックタスク原則）
3. 各タスクの仕様をCC-SDD式で作成:
   - Requirements (何を作るか)
   - Design (どう作るか)
   - Tasks (実装タスクリスト)
4. GSD Wave パターンで実装:
   - Wave 1: 独立したタスクを並列実行
   - Wave 2: Wave 1 の結果に依存するタスクを実行
5. コードレビュー
6. テスト実行・確認
7. `.company/departments/dev/STATE.md` を更新
```

注目すべきは「最大3タスク」という制約だ。Claude Codeの1セッションで処理できるタスク量には限界がある。欲張って10タスクを詰め込むと、後半のタスクの品質が下がる。3タスクに絞ることで、各タスクの品質を担保している。

### 2. CMO（マーケ統括）

CMOエージェントの特徴は、専門知識セクションが充実していることだ。

```markdown
# CMO / マーケティング部長エージェント

## 専門知識

### SEO
- テクニカルSEO（Core Web Vitals, 構造化データ,
  サイトマップ, canonicalタグ）
- コンテンツSEO（キーワード戦略, E-E-A-T, 検索意図分析）

### コンテンツマーケティング
- Zenn/Qiita/note の各プラットフォーム特性
- 有料書籍の企画・価格戦略・プロモーション

### 広告運用
- Meta広告のターゲティング・クリエイティブ最適化
- Google広告のキーワード戦略
- 広告→LP→CVのファネル分析と改善

### 分析
- Google Analytics 4 のイベント設計と分析
- A/Bテスト設計と統計的有意性の判断
```

マーケティングは領域が広い。SEO、コンテンツ、広告、分析のすべてに対応するため、専門知識セクションに具体的な技術名を列挙している。これにより、エージェントは「Core Web Vitals」「E-E-A-T」「A/Bテスト」といった概念を正しく理解した上で施策を提案できる。

### 3. CFO（経理統括）

CFOエージェントの特徴は、ツールの制限だ。`Bash` が許可されていない。

```yaml
tools:
  - Read
  - Write
  - Edit
  - Grep
```

経理エージェントにシェルコマンドの実行権限は不要だ。財務レポートの生成、コスト分析、請求書ドラフトは、すべてファイルの読み書きで完結する。不要な権限を与えないことで、意図しない操作のリスクを減らす。

```markdown
## 品質検証

1. **正確性:** 金額に誤りがないか
   （桁数、税込/税抜の統一）
2. **完全性:** すべてのコスト項目が
   漏れなく計上されているか
3. **適時性:** 月次処理が翌月5日以内に
   完了しているか
4. **閾値チェック:** AI運用コストが
   月$200以内に収まっているか
```

品質検証に具体的な数値（「月$200以内」）を入れている点に注目してほしい。曖昧な基準（「コストが適切か」）ではなく、明確な閾値を設定することで、エージェントが自律的に判断できる。

### 4. CSO（営業統括）

CSOエージェントの特徴は、成果物テンプレートの詳細さだ。

```markdown
### 提案書
出力先: `.company/departments/sales/proposals/
        proposal-{client}-{date}.md`

# 提案書: {client_name} 様

## エグゼクティブサマリ
## 課題認識
## ソリューション
## 導入効果
## スケジュール
## 料金
## 会社概要
```

営業提案書は対外文書であり、品質のバラつきが致命的だ。テンプレートで章立てを固定することで、どの案件の提案書でも一定の品質が保証される。

また、CSOの権限設定にも工夫がある。

```markdown
## 権限レベル

- **execute:** パイプライン分析、提案書ドラフト、
  競合分析、営業資料作成
- **draft:** 顧客への提案送付、価格交渉、
  契約条件の提示
```

「提案書を作る」のは execute だが、「提案書を送る」のは draft だ。この区分が重要で、エージェントは自由に提案書を作れるが、顧客に送る前には必ずCEOの承認が入る。

### 5. CS Lead（カスタマーサクセス）

CS Leadエージェントの特徴は、エスカレーションの優先度設計だ。

```markdown
### /ai-ceo:cs:escalations -- エスカレーション確認
1. 未解決の問い合わせ/バグ報告を一覧化
2. 優先度で分類:
   - P0: サービス停止・データ損失 → 即時対応
   - P1: 主要機能の障害 → 24時間以内
   - P2: 部分的な問題 → 3営業日以内
   - P3: 要望・改善提案 → バックログに追加
3. 開発部門への引き継ぎが必要なものを特定
```

P0-P3 の基準が明文化されていることで、エージェントは問い合わせの深刻度を一貫して判断できる。「サービス停止」はP0、「要望」はP3。人間が判断するまでもなく、エージェントが自動でトリアージする。

### 6. Legal（法務）

法務エージェントの特徴は、免責事項の明示だ。

```markdown
## ペルソナ

IT/SaaS法務に精通したリーガルアドバイザー。
重要度の高いリスクから優先的に対処する。
最終判断は外部弁護士に委ねるが、
初期レビューと論点整理を担当。
```

```markdown
## 品質検証

4. **免責注意:** 最終判断は弁護士に委ねる旨を
   明示しているか
```

AIが法的判断の最終責任を負わないことを、ペルソナと品質検証の両方で明記している。法務エージェントは「論点整理」と「リスクの一次スクリーニング」を担当し、高リスク項目は人間の弁護士にエスカレーションする設計だ。

## エージェントのテスト方法

専門エージェントが正しく動作するかをテストする方法を紹介する。

### 1. 単体テスト（エージェント単独での動作確認）

Claude Code で直接サブエージェントを呼び出し、想定通りの出力が得られるか確認する。

```
# CTOエージェントのテスト
@ai-ceo-cto スプリント計画を立ててください。
バックログは .company/products/focalize/STATE.md を参照。
```

確認ポイントは以下の3つだ。

- 指定されたファイルを正しく読み込んでいるか
- 成果物テンプレートに沿った形式で出力しているか
- 権限レベルに従って draft/execute を正しく判定しているか

### 2. 統合テスト（Orchestrator経由での動作確認）

Orchestratorからコマンドを実行し、正しいエージェントに委任されるか確認する。

```
/ai-ceo:fin:monthly-report
```

確認ポイントは以下だ。

- OrchestratorがCFOエージェントを正しく呼び出しているか
- CFOエージェントの出力がOrchestratorに正しく返っているか
- 出力先ファイルが正しいパスに生成されているか

### 3. 境界テスト（権限とエラーハンドリング）

エージェントが権限を超えた操作をしないかをテストする。

```
# CFOエージェントにコーディングを依頼する（拒否されるべき）
@ai-ceo-cfo LPのHTMLを修正してください。
```

ツール制限が正しく機能していれば、CFOエージェントは `Bash` ツールを持っていないため、コマンド実行はできない。`Write` は持っているが、ペルソナと責務の定義により、コーディングタスクは自分の管轄外と判断するはずだ。
