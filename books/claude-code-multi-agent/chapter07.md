# 承認パイプラインと人間-AI協調 -- 品質と安全性を担保する

マルチエージェントシステムの設計と通信パターンが整った。しかし、このまま9つのエージェントに自由に動かせたらどうなるか。

営業エージェントが、誤字だらけの提案書をクライアントに送信する。開発エージェントが、テスト不足のコードを本番にデプロイする。マーケティングエージェントが、ブランドガイドラインに反するSNS投稿を公開する。

AIエージェントは指示されたタスクを実行する能力は高い。しかし「本当にこれを実行していいのか」という判断は、現時点では人間が担うべき領域だ。本章では、AIの自律性と人間の監督をバランスさせる「承認パイプライン」の設計と実装を解説する。

## なぜ承認パイプラインが必要か

AIエージェントの自律実行には、3つのリスクがある。

**1. 対外的なダメージの不可逆性**

社内の分析レポートが多少不正確でも、あとから修正できる。しかし、クライアントに送った提案書は取り消せない。本番環境にデプロイしたバグは、ユーザーに影響する。SNSに投稿した炎上発言は、削除しても拡散される。

対外的なアクションの「やり直しコスト」は、内部アクションと比べて桁違いに高い。

**2. コンテキスト不足による判断ミス**

エージェントは、自分のコンテキストに含まれる情報だけで判断する。「先週クライアントと電話で話した内容」「業界の最新動向」「CEOが考えている戦略の方向転換」など、ファイルに書かれていない情報は考慮できない。

人間のCEOが持つ暗黙知を、承認プロセスを通じて反映する仕組みが必要だ。

**3. エージェント間の不整合**

9つのエージェントがそれぞれ独立して動くと、方向性がずれることがある。営業エージェントが「3月末に新機能リリース予定」と提案書に書いたが、開発エージェントのスプリント計画では4月中旬の予定になっている、といったケースだ。

承認プロセスは、こうした不整合を人間が検出する最後の砦になる。

## 3段階の権限レベル設計

AI-CEO Frameworkでは、すべてのアクションを3つの権限レベルに分類している。

### read-only: 自動実行OK

分析・レポート系のアクションは、承認なしで即座に実行する。

```yaml
auto_execute_readonly:
  - analytics_report      # 分析レポート生成
  - kpi_update            # KPIダッシュボード更新
  - competitor_analysis   # 競合調査レポート
  - internal_docs         # 社内ドキュメント更新
```

これらは「情報を読んで、社内向けにまとめる」だけの操作だ。外部に影響を与えないため、リスクが低い。朝のダイジェスト生成や月次KPI集計がここに該当する。

read-onlyに分類することで、CEOが承認待ちに追われる負担を減らせる。全アクションを承認必須にすると、CEOがボトルネックになり、エージェントの自律性という利点が失われる。

### draft: 人間の承認が必要

対外的な影響があるアクションは、必ずドラフトモードで生成し、承認キューに追加する。

```yaml
always_draft:
  - press_release         # プレスリリース
  - pricing_change        # 価格変更の告知
  - claim_response        # クレーム・苦情への対応
  - proposal              # クライアントへの提案書
  - contract              # 契約書・NDA
  - invoice               # 請求書
  - deploy_production     # 本番デプロイ
  - client_communication  # クライアント連絡
  - marketing_email       # マーケティングメール
```

draftモードのエージェントは、成果物をファイルとして書き出すところまでは自律的に行う。しかし「送信」「公開」「デプロイ」といった実行ステップは行わない。代わりに、承認キューにアイテムを追加し、CEOの判断を待つ。

### execute: 閾値内の自動実行

内部的なアクションのうち、影響範囲が限定的なものは、承認なしで自動実行する。

```yaml
auto_execute:
  - bugfix               # バグ修正
  - minor_feature        # マイナー機能追加
  - test                 # テスト追加・修正
  - refactor             # リファクタリング
  - docs                 # ドキュメント更新
  - dependency_patch     # 依存パッケージの更新(patch/minor)
  - internal_notification # 社内通知
```

ただし、自動実行にも閾値を設けている。

```yaml
auto_approve_limit: 5000   # 5,000円以下は自動承認
ceo_approval_above: 5000   # 5,000円超はCEO承認必要
monthly_ai_budget: 200     # USD（月間AI運用予算上限）
```

金額が閾値を超えた場合は、executeに分類されるアクションでもdraftモードにフォールバックする。

### 分類の判断基準

迷ったときの判断基準はシンプルだ。

**「取り消せるか？」で判断する。**

- 取り消せる内部アクション → execute
- 取り消せない対外アクション → draft
- 読み取りだけ → read-only

筆者の経験では、最初はdraftを多めに設定し、運用しながらexecuteに移行するのがよい。信頼が蓄積されたアクションから段階的に自動化していく。

## approval-queue.mdの設計と実装

承認パイプラインの中核は、`approval-queue.md`というMarkdownファイルだ。

### キューのフォーマット

```markdown
# 承認待ちキュー

## 承認待ち (2件)

### [AQ-014] 提案書送付: 株式会社ABC
- **部門:** 営業
- **種別:** proposal
- **ファイル:** .company/departments/sales/proposals/proposal-abc-2026-03.md
- **概要:** DX支援サービスの提案書をABC社に送付する
- **登録日:** 2026-03-08
- **優先度:** high

### [AQ-015] 月次レポート公開
- **部門:** 経理
- **種別:** internal_docs
- **ファイル:** .company/departments/finance/monthly-2026-02.md
- **概要:** 2月の月次財務レポートを確定する
- **登録日:** 2026-03-08
- **優先度:** medium
```

各アイテムには以下の情報を含める。

| フィールド | 目的 |
|-----------|------|
| ID | 一意の識別子。`AQ-{連番}` 形式 |
| 部門 | どのエージェントが生成したか |
| 種別 | アクションの分類（deploy, email, proposal等） |
| ファイル | 成果物の実体ファイルへのパス |
| 概要 | 1-2文の説明。CEOが一覧で見て判断できる粒度 |
| 登録日 | キューに追加された日付 |
| 優先度 | high / medium / low |

重要なのは、**承認キューには成果物そのものを含めない**ことだ。成果物はファイルとして別に保存し、キューにはパスだけを記録する。これにより、キューファイルが肥大化せず、Orchestratorのコンテキストを節約できる。

### 承認ワークフロー

承認操作は2つのCEOコマンドで行う。

**承認の場合:**

```
CEO: /ai-ceo:approve AQ-014
```

Orchestratorが以下を実行する。

1. `approval-queue.md` から `[AQ-014]` のブロックを削除
2. `decisions/2026-03.md` に承認記録を追加
3. CEOに完了を報告

**却下の場合:**

```
CEO: /ai-ceo:reject AQ-014 "価格体系を再検討。月額プランを追加して"
```

1. `approval-queue.md` から `[AQ-014]` のブロックを削除
2. `decisions/2026-03.md` に却下理由を含めて記録
3. 該当部門に差し戻し指示を伝達

却下時に理由を必ず含める設計にしているのは、エージェントが次のイテレーションで改善できるようにするためだ。「却下」だけでは、何を直せばいいか分からない。

### decisions/{month}.md への記録

すべての承認・却下は月次の意思決定ログに記録する。

```markdown
# CEO意思決定ログ -- 2026年3月

## 2026-03-08

### 承認: [AQ-014] 提案書送付 -- 株式会社ABC
- **判断:** 承認
- **備考:** 内容確認済み。価格は適正。

### 却下: [AQ-015] 月次レポート公開
- **判断:** 却下
- **理由:** 2月の広告費の内訳が不明確。
  Google Adsの費目を分解して再提出のこと。
```

この記録は2つの目的を果たす。

1. **監査証跡:** あとから「なぜこの判断をしたか」を追跡できる
2. **エージェントの学習材料:** 次回のセッションでエージェントがdecisionsログを参照し、CEOの判断傾向を把握できる

## permissions.mdでの権限閾値の設計

権限の設計は `permissions.md` に集約する。実際のAI-CEO Frameworkで使用している構造を示す。

### 費用閾値

```yaml
auto_approve_limit: 5000  # 円
ceo_approval_above: 5000  # 円
monthly_ai_budget: 200    # USD
```

5,000円以下の支出は自動承認。これは「コーヒー代を毎回CEOに聞くのは非効率」という感覚だ。ただし月間予算の上限は設けている。

### デプロイ権限

```yaml
deploy:
  local: execute       # ローカル開発は自由
  staging: execute     # ステージングは自動デプロイ可
  production: draft    # 本番は必ずCEO承認
```

ステージング環境までは自動実行を許可し、本番デプロイだけCEO承認を必須にしている。ステージングで問題が発生しても、ユーザーに影響しないからだ。

### メール送信権限

```yaml
email:
  internal_notification: execute   # 社内通知は自動送信可
  client_communication: draft      # クライアント連絡は承認必須
  marketing_email: draft           # マーケメールは承認必須
  system_notification: execute     # システム通知は自動送信可
```

メール送信は「相手が誰か」で権限を切り分けている。社内通知やシステム通知は自動化し、クライアント向けのメールだけ承認を通す。

### 緊急時のバイパスルール

通常の承認フローを迂回する必要がある場面は存在する。

```yaml
emergency_bypass:
  conditions:
    - security_incident    # セキュリティインシデント
    - service_down         # サービス停止
  actions_allowed:
    - hotfix_deploy        # 緊急パッチのデプロイ
    - service_restart      # サービス再起動
  post_action:
    - notify_ceo           # 事後報告必須
    - log_to_decisions     # 意思決定ログに記録
```

セキュリティインシデントやサービス停止時は、承認を待っていては被害が拡大する。事前に定義した条件に合致する場合に限り、エージェントが自律的に対処し、事後報告する仕組みだ。

ただし筆者の運用では、この緊急バイパスを実際に発動したことはまだない。存在するだけで安心感がある、保険のような設計だ。

## 人間がボトルネックにならない設計

承認パイプラインの最大の課題は、「CEOが承認しないとエージェントが止まる」問題だ。これを解決する4つの工夫を紹介する。

### 1. バッチ承認

朝のダイジェスト（`/ai-ceo:morning`）で承認待ちを一覧表示し、まとめて処理する運用にしている。

```
## 承認待ち (3件)
- [AQ-014] 営業: 提案書送付 -- ABC社 | high
- [AQ-015] 経理: 月次レポート公開 | medium
- [AQ-016] マーケ: SNS投稿 -- 新機能告知 | low
```

1件ずつ個別に判断するのではなく、朝と夕方の1日2回、まとめて承認する。これで承認のオーバーヘッドを最小化できる。

### 2. 承認待ちの優先度設計

すべての承認待ちが同じ緊急度ではない。優先度を3段階に分けることで、CEOは重要なものから処理できる。

- **high:** 期限が迫っている、または売上に直結する（クライアント提案書、緊急デプロイ等）
- **medium:** 今週中に処理すればよい（月次レポート、定期SNS投稿等）
- **low:** いつ処理しても問題ない（社内ドキュメント更新等）

### 3. テンプレート承認

一度承認したパターンを「承認済みテンプレート」として登録し、同じパターンのアクションは自動承認にする。

```yaml
auto_after_approval:
  - scheduled_sns     # 承認済みテンプレートに基づくSNS投稿
  - faq_response      # 承認済みテンプレートに基づくFAQ回答
  - tech_article      # レビュー済みの技術記事公開
```

たとえば、SNS投稿のフォーマットとトーンを一度承認しておけば、同じフォーマットの投稿は以降自動公開できる。承認の手間を増やさずにエージェントの自律性を段階的に広げていく仕組みだ。

### 4. 承認のSLA

これは未実装だが、設計として検討している仕組みを紹介する。

```yaml
approval_sla:
  high: 4h      # 4時間以内に承認/却下
  medium: 24h   # 24時間以内
  low: 72h      # 72時間以内
  timeout_action: notify  # SLA超過時は再通知
```

一定時間以内に承認・却下がなければ再通知する仕組みだ。「デフォルト承認」（n時間以内に却下がなければ自動承認）も検討したが、リスクが高いため現時点では採用していない。承認を忘れていただけなのに、意図しないメールが送信される可能性があるからだ。

## 実装例: 営業メール送信の承認フロー

承認パイプラインの全体像を、具体的なフローで見てみよう。「HP リニューアル提案のメールを見込み客に送信する」というシナリオだ。

### ステップ1: 営業エージェントが提案書とメールを生成

```
CEO: /ai-ceo:sales:proposal "大城歯科"
```

CSOエージェントが起動し、以下を実行する。

1. 見込み客リスト（`.company/departments/sales/hp-renewal/prospect-list.md`）から対象情報を取得
2. 提案書テンプレート（`templates/proposal-template.md`）を参照
3. 対象のWebサイトを分析
4. 提案書を `proposals/ohshiro-dental-proposal.md` に出力
5. アウトリーチメールを `outreach/05-ohshiro-dental-email.md` に出力
6. `approval-queue.md` にアイテムを追加

```markdown
### [AQ-017] HP リニューアル提案: 大城歯科
- **部門:** 営業
- **種別:** proposal
- **ファイル:** .company/departments/sales/hp-renewal/proposals/ohshiro-dental-proposal.md
- **概要:** 大城歯科医院へのHPリニューアル提案書とアウトリーチメールのドラフト
- **登録日:** 2026-03-09
- **優先度:** high
```

この時点で、メールは送信されない。ファイルとして保存されるだけだ。

### ステップ2: CEOが朝ダイジェストで確認

翌朝の `/ai-ceo:morning` で、承認待ちに `[AQ-017]` が表示される。

CEOは提案書ファイルとメール文面を確認する。ここで人間にしかできない判断を行う。

- 提案内容はこのクライアントに合っているか
- 価格は適正か
- メールのトーンは適切か
- 送信タイミングは今がベストか

### ステップ3: 承認または却下

```
CEO: /ai-ceo:approve AQ-017
```

承認されると、以下が実行される。

1. `approval-queue.md` から削除
2. `decisions/2026-03.md` に記録
3. メールが「送信可能」ステータスになる

もし問題があれば却下する。

```
CEO: /ai-ceo:reject AQ-017 "初期費用を¥150,000に下げて。
競合の見積もりが安いと聞いている"
```

却下理由がdecisionsログに記録され、CSOエージェントが次のセッションで修正版を作成する。

### このフローの要点

このフローで重要なのは、**エージェントの自律性と人間の監督が適切に分離されている**ことだ。

- **エージェントが自律的に行う部分:** Webサイト分析、提案書作成、メール文面生成、キューへの追加
- **人間が判断する部分:** 内容の妥当性確認、価格の適正性、送信タイミング

エージェントは「作る」ことに集中し、人間は「判断する」ことに集中する。この分業が、品質と効率を両立させる。

## 承認パイプラインの運用から得た教訓

9ヶ月の運用で得た3つの教訓を共有する。

**教訓1: 最初はdraftを多めにして、段階的にexecuteに移行する**

最初はほぼすべてのアクションをdraftにしていた。2-3ヶ月運用して「このパターンは毎回承認している」と気づいたら、executeに変更する。逆方向（execute → draft）に戻すのも簡単だ。permissions.mdを書き換えるだけでよい。

**教訓2: 却下理由は具体的に書く**

「やり直し」ではなく「価格を15万円に下げて、競合の見積もりが安いため」と書く。エージェントは次回、この却下理由を参照して改善版を作る。曖昧な却下理由は、何度も差し戻しのループを生む。

**教訓3: 承認キューはgitで管理する**

`approval-queue.md` をgitに含めることで、「いつ、何が承認され、何が却下されたか」の完全な履歴が残る。過去の承認パターンを分析して、権限設計を最適化するデータにもなる。

---

承認パイプラインは、マルチエージェントシステムの「ブレーキ」だ。ブレーキのない車は速く走れるが、安全には走れない。しかし、ブレーキの踏みすぎもまた問題だ。権限レベルの3段階設計と、バッチ承認や段階的自動化の仕組みで、「安全に速く走る」バランスを実現する。次章では、マルチエージェントシステムのもう一つの重要課題、コンテキストウィンドウの管理を掘り下げる。
