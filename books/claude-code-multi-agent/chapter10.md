# 実践 -- 10部門AIエージェント経営システムの全貌

ここまでの章で、マルチエージェントの設計原則、実装パターン、エラーハンドリング、権限制御を個別に解説してきた。本章では、それらすべてが統合された実システムの全貌を公開する。

筆者が合同会社ジョインクラスで実際に運用している「AI-CEO Framework」だ。CEO1名とAIエージェント10部門で会社を回している。理論ではなく、日々の経営判断に使われている実装をそのまま見せる。

## 全体アーキテクチャ

本書で解説してきたOrchestrator-Workerパターン（第2章）、専門エージェント（第5章）、通信パターン（第6章）、承認パイプライン（第7章）が、実際のシステムとしてどう統合されているかを見てみよう。

```
CEO（人間）
  │
  │ 自然言語で指示（/ai-ceo:* コマンド）
  │
  ▼
C-Suite Orchestrator（CLAUDE.md）
  │
  │ タスク委任（Agent tool）
  │
  ├─── CTO部門 ──── プロダクト開発、コードレビュー、CI/CD
  ├─── CMO部門 ──── SEO記事、SNS投稿、LP改善、広告運用
  ├─── CFO部門 ──── 経理処理、月次決算、予算管理
  ├─── CSO部門 ──── 提案書作成、顧客管理、営業メール
  ├─── CS部門 ──── ユーザー対応、エスカレーション管理
  ├─── 法務部門 ── 契約書レビュー、OSSライセンス監査
  ├─── 税務部門 ── 仕訳チェック、確定申告準備、節税対策
  ├─── コンテンツ部門 ── Zenn書籍執筆、技術記事量産
  ├─── 広告部門 ── 広告運用、広告文作成、入札最適化
  └─── HR部門 ──── エージェント管理、オンボーディング設計
```

### なぜ「10部門」なのか

最初から10部門だったわけではない。筆者の場合、以下の順で段階的に追加していった。

1. **開発部門（CTO）** -- 最初に作った。コードレビューとスプリント管理を委任
2. **マーケ部門（CMO）** -- SEO記事の量産ニーズから追加
3. **経理部門（CFO）** -- freeeとの連携、月次決算の自動化
4. **営業部門（CSO）** -- HPリニューアル営業で提案書を21社分作る必要があった
5. **法務部門** -- 契約書レビューとOSSライセンス監査
6. **税務部門** -- 確定申告準備で仕訳チェックが大量に発生
7. **CS部門** -- プロダクトのユーザー対応（現在は待機中）
8. **コンテンツ部門** -- Zenn書籍の量産フェーズで分離
9. **広告部門** -- Google Ads運用やAmazon Ads運用を分離
10. **HR部門** -- エージェント自体の管理・メンテナンス

部門を追加するタイミングの判断基準は明確だ。「その作業に週3時間以上使っているなら、エージェントにすべき」。逆に言えば、週1時間程度の作業なら既存のエージェントに含めたほうがよい。部門を増やすとOrchestratorの判断コストも増えるからだ。

この構成が優れている理由は3つある。

**1. Gitで全履歴が残る。** すべてがMarkdownファイルなので、`git log`で経営判断の履歴を追跡できる。「なぜこの営業方針に変えたのか」を半年後に振り返れる。

**2. エージェントが直接読み書きできる。** データベースやAPIを介さないため、エージェントはReadとWriteツールだけで全社の情報にアクセスできる。セットアップコストがゼロだ。

**3. 人間も読める。** CEOがVSCodeで`.company/STATE.md`を開けば、エージェントを介さずとも経営状態を確認できる。AIに依存しない可読性が担保されている。


## 各部門エージェントの実装詳細

エージェント定義ファイルの実際の構造を見てみよう。以下はCTOエージェントの冒頭部分だ。

```yaml
---
name: ai-ceo-cto
description: CTO/開発部長エージェント。プロダクト開発全般を統括する。
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
---
```

YAMLフロントマターで、エージェント名と利用可能なツールを定義する。CTO（開発）にはBashが許可されているが、CSO（営業）にはBashを渡していない。営業エージェントがシェルコマンドを実行する必要はないからだ。ツールの制限はセキュリティの最小権限原則に直結する。

本文には以下の要素を記述する。

```markdown
# CTO / 開発部長エージェント

## ペルソナ
経験豊富なテックリード。実用性を重視し、
オーバーエンジニアリングを避ける。

## 担当領域
- プロダクト開発全般（設計、実装、テスト、デプロイ）
- 技術的意思決定
- スプリント管理

## 権限レベル
- execute: コーディング、テスト、ステージングデプロイ
- draft: 本番デプロイ、アーキテクチャ変更

## 参照ファイル
- 技術スタック: .company/steering/tech-stack.md
- プロダクト状態: .company/products/{name}/STATE.md
- 開発部門状態: .company/departments/dev/STATE.md

## ワークフロー
### /ai-ceo:dev:sprint
1. バックログ確認
2. 優先タスク選定（最大3タスク）
3. CC-SDD式の仕様作成
4. GSD Waveパターンで実装
```

ここで重要なのは「参照ファイル」セクションだ。エージェントに必要なファイルパスを明示することで、自分の管轄外のファイルを読みに行かないように制約をかけている。CTOエージェントが営業の提案書を勝手に編集するリスクを構造的に排除する。

### エージェント定義の設計パターン

9部門のエージェント定義で共通するパターンがある。

| セクション | 内容 | 目的 |
|-----------|------|------|
| ペルソナ | そのエージェントの「人格」 | 出力のトーンと判断基準の統一 |
| 専門知識 | 業務に必要な具体的知識 | コンテキストの専門化 |
| 担当領域 | 何をやるか（スコープ定義） | 責務の明確化 |
| 権限レベル | execute / draft の区分 | セキュリティの最小権限 |
| 参照ファイル | 読むべきファイルのパス | 情報アクセスの制限 |
| ワークフロー | コマンド別の手順書 | 再現性のある実行 |
| 品質基準 | 完了条件の定義 | 成果物の品質担保 |

このパターンを守ることで、新しい部門を追加するときも迷わない。テンプレートに沿って埋めるだけでエージェントが立ち上がる。


## 1日の運用フロー

実際の1日がどう回っているかを時系列で追う。

### 朝（5分）: 経営状態の把握

CEOがターミナルを開き、`/ai-ceo:morning`と入力する。

Orchestratorは朝ダイジェスト専用エージェント（`ai-ceo-morning.md`）を起動する。このエージェントは全部門のSTATE.mdと承認キューを読み取り、以下のようなレポートを返す。

```
AI-CEO Morning Digest -- 2026-03-09 (日)

承認待ち (0件)
  承認待ちはありません

部門状態
| 部門   | 状態    | 進行中           | 注意事項                 |
|--------|---------|-----------------|--------------------------|
| 営業   | 停滞    | HP営業21社       | 20社未送信。最優先で再開  |
| コンサル| 準備中  | LP開発中         | LP公開が先決             |
| マーケ  | 稼働   | X投稿、広告準備   | Google Ads設定待ち       |
| 開発   | 保守中心 | AIコンサルLP     | LP仕様書完成、実装待ち    |
| 経理   | 稼働   | 3月分証憑収集     | 税理士確認待ち           |
| 法務   | 待機中  | 契約書レビュー    | 外部レビュー未依頼       |

今日のおすすめアクション
1. HP営業のメール送信を再開（20社未送信は機会損失）
2. AIコンサルLPの実装着手
3. 3月分の経費証憑をfreeeに登録
```

このダイジェストを見て、CEOは1日の優先順位を決める。所要時間は1-2分だ。

### 午前（10-30分）: タスクの委任

CEOは判断に基づいてコマンドを発行する。

```
/ai-ceo:sales:proposal "ohshiro-dental"
```

Orchestratorはこのコマンドを受け取り、CSOエージェントに委任する。CSOエージェントは顧客のWebサイトを分析し、提案書をドラフトとして生成し、承認キューに追加する。CEOは他の作業に移れる。

複数のタスクを同時に出すことも日常的だ。

```
/ai-ceo:dev:sprint        -- CTOエージェントが起動
/ai-ceo:mkt:content-plan  -- CMOエージェントが起動
/ai-ceo:fin:monthly-report -- CFOエージェントが起動
```

これら3つのエージェントは並列で動く。お互いに依存関係がないため、同時に進行する。

### 午後（5分）: 承認と確認

作業が終わると、各エージェントは成果物をdraftとして出力し、承認キューに追加する。CEOは承認キューを確認し、1件ずつ承認または却下する。

```
/ai-ceo:approve AQ-015    -- 提案書を承認
/ai-ceo:reject AQ-016 "価格を15万円に修正して"  -- 見積を差し戻し
```

承認されたアイテムはdecisions/に記録される。却下されたアイテムは、理由とともに該当部門に差し戻される。このパイプラインがあることで、AIが勝手にメールを送ったり、デプロイしたりするリスクをゼロにしている。

### 夕方（2分）: STATE.mdの更新

1日の終わりに、各部門のSTATE.mdが更新される。これは手動のこともあれば、エージェントが自動で行うこともある。翌朝のmorningダイジェストに反映される。

**CEOの1日の作業時間: 合計20-40分。**

残りの時間は、エージェントが処理できない創造的な仕事、つまりプロダクトの方向性の検討、パートナーとの対話、新しい事業アイデアの構想に使える。


## 部門間連携の実例

マルチエージェントの真価は、部門間のワークフローで発揮される。実際に運用している3つの連携パターンを紹介する。

### パターン1: 営業 → 開発 → 営業（HP分析から提案書まで）

HPリニューアル営業では、以下のフローが自動化されている。

```
1. CSOエージェント: ターゲット企業のWebサイトURLを受け取る
2. CTOエージェント: Lighthouseでパフォーマンス分析を実行
3. CSOエージェント: 分析結果を元に提案書を生成
4. コンテンツエージェント: 営業メールの文面をドラフト
5. 承認キューに追加 → CEO承認 → 送信
```

このフローで21社分の提案書とメールを作成した。人間が1社ずつやっていたら、1社あたり2-3時間、21社で40時間以上かかる。エージェントに委任したことで、CEOの実作業は承認確認の30分程度に圧縮された。

### パターン2: マーケ → コンテンツ → 営業（記事からリードへ）

```
1. CMOエージェント: 月間コンテンツカレンダーを生成
2. コンテンツエージェント: SEO記事を執筆（CTA付き）
3. CMOエージェント: SNS投稿を一括生成
4. CSOエージェント: 記事経由のリードをパイプラインに登録
```

記事にはAI業務自動化コンサルティングへのCTAが自然に埋め込まれる。コンテンツエージェントの定義ファイルに「すべてのコンテンツにプロダクトへの導線を含めること」と明記しているからだ。

### パターン3: 経理 → 税務（月次処理から確定申告へ）

```
1. CFOエージェント: freeeから取引データを取得
2. 税務エージェント: 仕訳の正確性をチェック
3. 税務エージェント: 節税対策の提案を生成
4. CFOエージェント: コスト追跡ファイルを更新
```

税務エージェントは、勘定科目の誤りや経費計上漏れを検知すると、CFOエージェントにフィードバックする。この連携はファイルを介して行われる。税務エージェントが`.company/departments/finance/review-notes.md`にフィードバックを書き、CFOエージェントがそれを読み取る。リアルタイム通信ではなく非同期のファイルベース通信だが、日次のバッチ処理には十分だ。


## Thin Orchestratorと承認パイプラインの実運用結果

Thin Orchestrator原則（第2章で設計、第4章で実装）と承認パイプライン（第7章）が、実運用でどう機能しているかを具体的に示す。

### Orchestratorのコンテキスト消費の実測値

1年以上の運用で、Orchestratorのコンテキスト消費は常に10-15%以内に収まっている。CLAUDE.mdが約150行（3,000トークン）、1回の委任指示が200-300トークン、1セッションで10回前後のサブエージェント呼び出し。200Kトークンのコンテキストウィンドウに対して十分な余裕がある。

コンテキストが15%を超えたケースは、CEOとの会話が長くなった場合のみだ。その場合はセッションを分割して対処している。

### 承認パイプラインの事故ゼロの実績

1年以上の運用で、エージェントが勝手にメールを送ったことはない。勝手にデプロイしたこともない。承認キューの実際の内容は以下のようになる。

```markdown
### [AQ-013] Google Ads検索広告キーワード・広告テキストドラフト
- 部門: マーケ
- 種別: other
- ファイル: .company/departments/marketing/ads/google-ads-draft.md
- 概要: AI業務自動化コンサル向けの検索広告テキスト
- 登録日: 2026-03-04
- 優先度: high
```

シンプルな仕組みこそが、信頼性の源泉だ。承認パイプラインの詳細な設計は第7章を参照してほしい。

## 実運用で学んだ教訓

1年以上の運用で、うまくいったことと失敗したことの両方がある。

### うまくいったこと

**1. ファイルベースの状態管理は極めて堅牢。** データベースを使わず、Markdownファイルで全社の状態を管理する設計は、当初は「雑すぎる」と思った。しかし実運用では、ファイルが壊れることがなく、Gitで全履歴が残り、人間もAIも同じファイルを読み書きできる。障害ポイントがゼロに近い。

**2. 承認パイプラインは安心感を生む。** AIに任せる不安の9割は「勝手に何かされるのでは」という恐怖だ。承認パイプラインがあることで、CEOは安心してタスクを委任できる。この心理的安全性がなければ、エージェント活用は進まない。

**3. エージェント定義のテンプレート化で追加が容易。** 新しい部門（税務、法務）を追加するとき、既存のエージェント定義をコピーして中身を書き換えるだけで済んだ。30分で新部門が稼働する。

### 失敗したこと

**1. 最初からフル構成にしようとした時期がある。** 初期に9部門を一気に作ろうとして、各エージェントの定義が薄くなった。結局、2-3部門から始めて段階的に追加するほうが各エージェントの品質が高くなる。

**2. STATE.mdの更新を忘れる。** エージェントがタスクを完了しても、STATE.mdの更新を明示的に指示しないと古い情報が残る。現在はワークフローの最終ステップに「STATE.md更新」を必ず含めるようにしている。

**3. 営業活動の「実行」はAIだけでは完結しない。** 提案書とメールをAIが生成しても、実際にメールを送信し、電話をかけるのは人間だ。21社分の提案書を作っても、送信したのは1社だけという状況が生まれた。AIはあくまで「準備」を加速するツールであり、「行動」は人間がしなければならない。

この教訓は本書を読んでいるすべての人に当てはまる。AIエージェントは魔法ではない。ただし、正しく設計すれば、1人の人間の能力を10倍に拡張する確かな仕組みになる。
