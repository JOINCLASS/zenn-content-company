# コンテキストウィンドウ管理の実践テクニック

マルチエージェントシステムを運用していると、ある問題に必ず直面する。コンテキストウィンドウの枯渇だ。

単一エージェントの場合、コンテキスト管理は「長い会話を避ける」程度の意識で済む。しかし、10の部門を統括するOrchestratorがすべての部門情報を読み込もうとすると、それだけでコンテキストの大半を消費してしまう。肝心のタスク実行に使える余裕が残らない。

本章では、AI-CEO Frameworkの実運用から得た、コンテキストウィンドウを効率的に管理する5つのテクニックを解説する。

## マルチエージェントにおけるコンテキストの課題

まず、マルチエージェント特有のコンテキスト問題を整理する。

### 各エージェントのコンテキストは独立している

Claude Codeのサブエージェントは、呼び出し元のOrchestratorとは別のコンテキストで動く。CTOエージェントが読んだコードの内容を、CMOエージェントは知らない。逆もまた然りだ。

これは設計上の制約だが、実はメリットでもある。各エージェントが自分の専門領域だけにコンテキストを集中できるからだ。問題は、Orchestratorがこの独立性を無視して、すべての情報を自分のコンテキストに集約しようとしたときに起きる。

### Orchestratorのコンテキスト膨張

素朴な設計をすると、Orchestratorは以下のすべてを読み込もうとする。

- 9部門のSTATE.md（各20-30行）
- 6プロダクトのSTATE.md（各20-30行）
- 承認キュー
- 権限設定
- ブランドガイドライン
- 技術スタック
- 四半期ロードマップ
- 今月の意思決定ログ

これだけで5,000-10,000トークン。さらにCEOとの会話、サブエージェントの呼び出し結果、エラーハンドリングが加わると、コンテキストの30-50%をOrchestrator自身が消費してしまう。

### 長時間セッションでのコンテキスト圧縮

Claude Codeは、コンテキストが上限に近づくと自動的に古い情報を圧縮する。この圧縮は多くの場合うまく機能するが、圧縮後に失われる情報もある。

特に問題になるのが、セッション序盤でCEOが伝えた「暗黙の前提」だ。「今月は営業を優先したい」「ABC社の提案は慎重に」といった指示が、セッション後半で圧縮され、エージェントが当初の意図と異なる判断をすることがある。

## コンテキスト効率化の5つのテクニック

### テクニック1: ファイルパス参照戦略

第2章で解説したThin Orchestrator原則の中核となるテクニックだ。Orchestratorがファイルの中身を読まず、パスだけを渡す。

第4章で実装例を示したが、改めて悪い例と良い例を比較する。

**悪い例（コンテキストを浪費）:**
ロードマップとSTATE.mdの全文（80行以上）をOrchestrator自身が読み込み、それをサブエージェントへの指示に含める。Orchestratorのコンテキストが無駄に消費される。

**良い例（パスだけ渡す）:**

```
Orchestrator → CTOエージェントに委任:
「次のスプリント計画を作成してください。
参照ファイル:
- .company/ROADMAP.md
- .company/departments/dev/STATE.md
- .company/products/focalize/STATE.md
出力先: .company/departments/dev/sprint-2026-03-w2.md」
```

Orchestratorはパスを3行渡すだけ。CTOエージェントが自分のコンテキストでファイルを読み込み、作業する。Orchestratorのコンテキスト消費は最小限だ。

このテクニックの効果は絶大で、筆者の実運用ではOrchestrator自身が実作業（コーディング、記事執筆など）を行わないルールと組み合わせることで、コンテキスト使用率を常に10-15%以内に維持できている。

### テクニック2: サマリ層の設計

STATE.mdは、各部門の状態を「人間とAIの両方が効率よく読める形」で要約するための仕組みだ。

設計のポイントは、**STATE.mdに含めるのは「現在の状態」だけ**ということだ。過去の履歴や詳細な作業ログは含めない。

**STATE.mdに含めるもの:**

```markdown
## 現在のステータス: 正常稼働中

## 進行中タスク
| タスク | ステータス | 期限 |
|--------|-----------|------|
| LP改修 | 実装中    | 3/15 |

## ブロッカー
なし

## 次のマイルストーン
3/20: API v2 リリース
```

**STATE.mdに含めないもの:**

- 過去のスプリント履歴（別ファイルに保存）
- 詳細な技術仕様（コードベースのドキュメントに記載）
- 議事録や意思決定の経緯（decisions/に記録）

この設計により、1つのSTATE.mdは20-30行（200-400トークン）に収まる。10部門+6プロダクトの全STATE.mdを読んでも、3,000-6,000トークンで済む。

サマリ層のもう一つの利点は、**更新頻度のコントロール**だ。STATE.mdは各部門のエージェントがタスク完了時に更新する。つまり、最新の情報だけが常にSTATE.mdに反映されている状態を維持できる。

### テクニック3: サブエージェントへの情報フィルタリング

サブエージェントに委任する際、そのタスクに必要な情報だけを渡す。

**悪い例:**

```
「提案書を作成してください。
参照: 会社の全ファイル一覧
  .company/VISION.md
  .company/STATE.md
  .company/ROADMAP.md
  .company/steering/brand.md
  .company/steering/tech-stack.md
  .company/steering/policies.md
  .company/steering/permissions.md
  .company/departments/dev/STATE.md
  .company/departments/marketing/STATE.md
  .company/departments/sales/STATE.md
  ...（20ファイル）」
```

CSOエージェントが提案書を作るのに、permissions.mdや開発部門のSTATE.mdは不要だ。

**良い例:**

```
「大城歯科へのHP リニューアル提案書を作成してください。
参照:
  - .company/departments/sales/hp-renewal/prospect-list.md
  - .company/departments/sales/hp-renewal/templates/proposal-template.md
  - .company/steering/brand.md
出力先: .company/departments/sales/hp-renewal/proposals/ohshiro-dental-proposal.md」
```

3ファイルに絞られている。CSOエージェントのコンテキストは提案書作成に最大限活用される。

フィルタリングの判断基準は、「このファイルがなかったら、タスクの品質が落ちるか？」だ。答えがNoなら、渡さない。

### テクニック4: 結果の圧縮

サブエージェントの実行結果をOrchestratorが受け取る際も、情報量を最小限にする。

サブエージェントがどれだけ複雑な作業をしても、Orchestratorが受け取るのは以下の3点だけでよい。

1. **成功/失敗のステータス**
2. **成果物のファイルパス**
3. **CEOへの報告に必要な要約（1-3文）**

エージェント定義のCLAUDE.mdに、出力フォーマットを明記しておくと確実だ。

```markdown
## 完了時の報告フォーマット

タスク完了時は以下のフォーマットで報告すること:

- ステータス: 成功 / 失敗
- 成果物: {ファイルパス}
- サマリ: {1-3文の要約}
- 承認要否: はい(approval-queue.mdに追加済み) / いいえ
```

CTOエージェントが2時間かけて実装したPRの詳細を、Orchestratorが全部読む必要はない。「成功。PR #42を作成。Focalizeのダッシュボード機能を追加。テスト全件パス。」これだけで十分だ。

### テクニック5: セッション分割

長いタスクを1つのセッションで完了させようとしない。

**アンチパターン: 1セッションで全部やろうとする**

```
CEO: 今月のコンテンツカレンダーを作成して、
  記事を5本書いて、SNS投稿も生成して、
  LPのコピーも改善して
```

このような指示を1セッションで処理すると、後半のタスクほどコンテキストが圧迫され、品質が劣化する。

**推奨: タスク単位でセッションを分ける**

```
セッション1: /ai-ceo:mkt:content-plan
  → コンテンツカレンダーを生成、ファイルに保存

セッション2: /ai-ceo:content:article "テーマA"
  → 記事1本を執筆、ファイルに保存

セッション3: /ai-ceo:content:article "テーマB"
  → 記事1本を執筆、ファイルに保存
```

各セッションがクリーンなコンテキストで開始されるため、品質が安定する。セッション間の情報共有はファイルシステム（STATE.md、成果物ファイル）が担う。

## コンテキスト使用率の計測と監視

コンテキスト使用率を意識的に監視することは重要だ。

Claude Codeはコンテキストの使用状況を表示する機能がある。会話が長くなると「context window usage」の表示が出る。これを定期的に確認し、以下の目安で運用する。

| 使用率 | 状態 | 対処 |
|--------|------|------|
| 0-30% | 健全 | そのまま続行 |
| 30-60% | 注意 | 新しい大きなタスクは新セッションで |
| 60-80% | 警告 | 現在のタスクを完了させて終了 |
| 80%以上 | 危険 | 品質劣化のリスクあり。即座にセッション分割 |

Orchestratorの場合、10-15%に維持するのが理想だ。これは「CEOとの会話+サブエージェントの呼び出し数回」に相当する。15%を超えたら、ファイルパス参照戦略と結果の圧縮を見直す。

## 実例: 9部門統括時のコンテキスト配分設計

AI-CEO Frameworkでの実際のコンテキスト配分を示す。

### Orchestrator: 10-15%

Orchestratorが持つ情報は以下に限定する。

- CLAUDE.md（エージェント定義）: 約150行
- CEOとの会話: 5-10ターン分
- サブエージェントへの委任指示と結果の要約

Orchestratorは**一切のファイル内容を読まない**。部門のSTATE.mdすら読まない。朝のダイジェストが必要なら、`ai-ceo-morning.md` エージェントに委任し、そのエージェントがSTATE.mdを読んでダイジェストを生成する。Orchestratorは生成されたダイジェストの「要約」だけを受け取る。

### 専門エージェント: 60-80%

各専門エージェントは、自分の専門領域のファイルをフルに読み込む。

たとえばCSOエージェント（営業）の場合:

- 自身のCLAUDE.md定義: 5%
- 見込み客リスト: 10%
- 提案書テンプレート: 5%
- ブランドガイドライン: 3%
- 対象企業の分析データ: 15%
- 提案書の執筆作業: 30-40%

合計60-80%をタスク実行に使える。Orchestratorとは比較にならない潤沢なコンテキストで、高品質な成果物を生成できる。

### 配分設計の要点

```
Orchestrator  [====............] 10-15%  ルーティングと判断のみ
専門エージェント [============....] 60-80%  専門領域にフル活用
未使用        [....====........] 20-30%  エラーハンドリング用バッファ
```

20-30%を未使用バッファとして残しておくのがコツだ。エラーが発生したときのリトライや、予想外の追加情報が必要になったときの余裕として機能する。

## 永続メモリの活用

Claude Codeには、セッションをまたいで情報を保持する仕組みがある。CLAUDE.mdファイルだ。これを「永続メモリ」として活用する。

### CLAUDE.mdに何を保存すべきか

CLAUDE.mdは「セッション開始時に毎回読み込まれる情報」だ。したがって、以下の性質を持つ情報を保存する。

**保存すべき情報:**

- エージェントの役割とペルソナ
- ワークフローの手順
- 参照すべきファイルのパス一覧
- 品質基準と制約条件
- 過去の重大な学びや教訓

**保存すべきでない情報:**

- 頻繁に変わるデータ（今月の売上、進行中タスクの詳細）
- 大量のデータ（顧客リスト、コードベースの説明）
- 一時的な情報（今日のタスクリスト）

頻繁に変わるデータはSTATE.mdに、大量のデータは専用ファイルに、一時的な情報はセッション内で処理する。CLAUDE.mdは「変わりにくいルールと手順」だけを保持する。

### セッションをまたぐ知識の管理

10部門のエージェントが日々タスクを実行する中で、「学び」が蓄積される。たとえば:

- 「ABC社の提案書は毎回価格が高いと却下される。初回提案は控えめに」
- 「Zennの記事タイトルに数字を入れるとクリック率が上がる」
- 「freeeのAPI呼び出しは1秒間に3回が上限」

これらの学びは、CLAUDE.mdの末尾に追記するか、`steering/` ディレクトリに専用ファイルを作成する。エージェントが同じ失敗を繰り返さないための仕組みだ。

## アンチパターン: コンテキストを浪費する5つの間違い

最後に、実運用で遭遇した「やってはいけない」パターンを紹介する。

### 1. Orchestratorがファイルを読む

最も致命的なアンチパターン。Orchestratorが「状況を把握するために」と全部門のSTATE.mdを読み込むと、それだけで10-20%を消費する。把握は専門エージェントに委任し、要約だけ受け取る。

### 2. 会話の中でファイル内容を繰り返す

CEOが「このファイルの内容を見せて」と頼み、Orchestratorが全文を表示する。その時点でファイル全文がコンテキストに乗る。代わりに「このファイルを直接開いて確認してください」とパスを伝えるほうが効率的だ。

### 3. サブエージェントの結果をそのまま保持する

CTOエージェントが200行のPR説明文を返し、Orchestratorがそれをコンテキストに保持したままCMOエージェントを呼ぶ。200行分がまるごと無駄になる。結果は「3行の要約+ファイルパス」に圧縮してから次のステップに進む。

### 4. 長大なCLAUDE.mdを書く

CLAUDE.mdが500行を超えると、エージェント定義だけでコンテキストの10%以上を消費する。80-150行に収める。詳細な手順は外部ファイルに分離し、CLAUDE.mdにはパスだけ記載する。

### 5. 1セッションに複数の大タスクを詰め込む

「記事を5本書いて、SNSも作って、LPも直して」は、1セッションの限界を超える。タスクごとにセッションを分け、ファイルシステムで成果物を引き継ぐ。

---

コンテキストウィンドウは、マルチエージェントシステムにおける最も貴重なリソースだ。「コンテキストを使わない方法」を常に考える。Orchestratorはパスだけ渡す。サブエージェントは要約だけ返す。詳細はファイルシステムに任せる。この徹底が、10部門を同時に運用できるかどうかの分水嶺になる。次章では、もう一つの重要課題であるエラーハンドリングとリカバリ戦略を解説する。
