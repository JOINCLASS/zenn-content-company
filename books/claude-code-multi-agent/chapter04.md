# Orchestratorパターンの実装 -- 指揮者エージェントを作る

マルチエージェントシステムの中核には、タスクを分解し、適切なエージェントに委任し、結果を集約する「指揮者」が必要だ。本章では、Claude Codeで Orchestrator パターンを実装する具体的な方法を、AI-CEO Framework の実コードとともに解説する。

## Orchestratorの4つの責務

Orchestratorエージェントが担う責務は、大きく4つに分かれる。

**1. タスク分解（Decomposition）**
CEOからの指示を、各部門エージェントが実行可能な単位に分解する。たとえば「新プロダクトを立ち上げたい」という指示は、開発部門への技術調査、マーケ部門への市場分析、法務部門への規約確認、経理部門へのコスト試算に分解される。

**2. 委任（Delegation）**
分解したタスクを適切なサブエージェントに振り分ける。ここで重要なのは、依存関係の把握だ。市場分析が終わらないと価格設計ができない、といった順序を Orchestrator が管理する。

**3. 結果集約（Aggregation）**
各エージェントの成果物を集め、矛盾がないか確認し、統合されたレポートとしてCEOに返す。

**4. 品質判定（Quality Gate）**
成果物が基準を満たしているか判断する。不十分であればエージェントに差し戻す。3回失敗したらエスカレーションする。

## CLAUDE.mdでのOrchestrator定義

AI-CEO Framework では、プロジェクトルートの `CLAUDE.md` に Orchestrator の全定義を書いている。以下は実際に運用している設定の構造だ。

```markdown
# AI-CEO Framework -- C-Suite Orchestrator

> あなたはAI-CEO Frameworkの「C-Suite Orchestrator」です。
> CEOの経営判断を支援し、9部門のAIエージェントを統括します。

## あなたの役割

あなたはCEOと直接対話する唯一のインターフェースです。
以下を担当します:

1. **CEOコマンドの受付と実行**
   — `/ai-ceo:*` コマンドを解釈し、適切な部門エージェントに委任
2. **部門間調整**
   — 依存関係の解決、Wave実行計画の生成
3. **承認管理**
   — draft成果物の承認キュー管理、CEOへの報告
4. **プロダクト横断管理**
   — 複数プロダクト間のリソース配分・優先度判断
```

ここで注目すべきは、Orchestrator自身が「コーディングをする」「記事を書く」といった実作業を担当していない点だ。あくまで「指揮者」として、受付・分解・委任・判定に徹する。

## Thin Orchestrator戦略の実装

第2章で解説したThin Orchestrator原則を、CLAUDE.mdの実装に落とし込む方法を見ていこう。原則の要点は「Orchestratorはコンテキスト使用率を10-15%に維持し、ファイルの中身を読み込まず、パスだけを渡す」ことだ。

ここでは、この原則を実現する3つの具体的なテクニックに焦点を当てる。

### 技術1: ファイルパスだけを渡す

Orchestratorは、サブエージェントにタスクを委任するとき、必要なファイルの中身を読み込まない。パスだけを渡す。

たとえば、月次レポートの作成を CFO エージェントに委任する場合、Orchestrator は次のような指示を出す。

```
目的: 2026年2月の月次財務レポートを作成する
参照ファイル:
  - .company/departments/finance/cost-tracking.md
  - .company/STATE.md
出力先: .company/departments/finance/monthly-2026-02.md
権限: execute
品質基準: 全コスト項目が計上されており、前月比が算出されていること
```

Orchestrator自身は `cost-tracking.md` の中身を知らない。CFOエージェントが読んで処理する。これにより、Orchestratorのコンテキストは数百トークンの「指示書」だけで済む。

### 技術2: 結果のサマリだけを受け取る

サブエージェントの実行が完了したあとも、成果物全体を読み込まない。成果物ファイルのパスと、1-2行の完了サマリだけを受け取る。

詳細を確認したければ、別のサブエージェント（レビュアー）に成果物のレビューを委任する。Orchestrator はレビュー結果のサマリだけを見て判断する。

### 技術3: 判断基準をCLAUDE.mdに明文化する

Orchestratorがその場で「この成果物は合格か？」と考えると、判断のためにコンテキストを消費する。代わりに、判断基準を事前にCLAUDE.mdに書いておく。

```markdown
## 権限制御ルール

全てのアクションは
`.company/steering/permissions.md` の閾値に従う。

- **read-only:** 分析・レポート系は承認不要で自動実行
- **draft:** 対外アクション（メール送信、請求書発行、
  SNS投稿、デプロイ等）は必ずdraftモードで生成し、
  approval-queue.md に追加
- **execute:** 閾値内の内部アクション
  （バグ修正、テスト実行等）は自動実行
```

Orchestratorは「このタスクは draft か execute か」を CLAUDE.md のルールに照らして即座に判断できる。都度考える必要がない。

## Orchestratorが持つべき情報と持つべきでない情報

設計の要点を表にまとめる。

| 持つべき情報 | 持つべきでない情報 |
|---|---|
| コマンド体系（どのコマンドがどの部門か） | 各部門の詳細な作業手順 |
| サブエージェントのファイルパス一覧 | サブエージェント定義の全文 |
| 権限判定ルール（draft/execute/read-only） | 個別ファイルの中身 |
| エラー時のエスカレーション手順 | 過去の全成果物 |
| ファイル参照先のディレクトリ構造 | コードの実装詳細 |

原則として、**メタ情報（どこに何があるか）は持ち、実データ（何が書いてあるか）は持たない**。

## 実装例: CEOエージェントのCLAUDE.md全体構造

AI-CEO Framework の Orchestrator 用 CLAUDE.md は、以下の構造で設計されている。

```
CLAUDE.md
├── あなたの役割（4行）
├── Thin Orchestratorの原則（4行）
├── 会社情報の参照先（9行 -- パスの一覧のみ）
├── CEOコマンド一覧（20行 -- コマンドと1行説明）
├── コマンド実行ルール
│   ├── /ai-ceo:init の実行フロー
│   ├── /ai-ceo:morning の実行フロー
│   └── 承認操作のルール
├── 権限制御ルール（10行）
├── エラー時の振る舞い（5行）
└── サブエージェントへの委任方法（5行）
```

全体で約160行、トークン数にして約3,000トークン。200Kトークンのコンテキストウィンドウの1.5%に収まっている。

この160行の中に、10部門のエージェント定義は含まれていない。それぞれ `.claude/agents/` 配下の独立したファイルに分離されている。

```
.claude/agents/
├── ai-ceo-cto.md        # CTO / 開発部長
├── ai-ceo-cmo.md        # CMO / マーケ部長
├── ai-ceo-cfo.md        # CFO / 経理部長
├── ai-ceo-cso.md        # CSO / 営業部長
├── ai-ceo-cs-lead.md    # CS部長
├── ai-ceo-legal.md      # 法務部長
├── ai-ceo-morning.md    # 朝ダイジェスト
├── ai-ceo-content-engine.md  # コンテンツ量産
└── ai-ceo-growth.md     # グロース分析
```

Orchestratorがサブエージェントを呼ぶとき、Claude Codeの `@agent` 機能を使う。このとき、エージェント定義ファイルはサブエージェント側のコンテキストに読み込まれ、Orchestrator側のコンテキストには影響しない。

## コマンドルーティングの設計

Orchestratorの最も頻繁な仕事は「コマンドルーティング」だ。CEOが入力したコマンドを、正しいサブエージェントに振り分ける。

```markdown
## CEOコマンド一覧

### 部門コマンド
- `/ai-ceo:dev:sprint`    — スプリント計画・実行
- `/ai-ceo:dev:hotfix`    — 緊急バグ修正
- `/ai-ceo:mkt:campaign`  — マーケキャンペーン
- `/ai-ceo:mkt:content-plan` — コンテンツカレンダー
- `/ai-ceo:sales:proposal` — 提案書自動生成
- `/ai-ceo:fin:monthly-report` — 月次決算
- `/ai-ceo:cs:escalations` — エスカレーション確認
- `/ai-ceo:legal:review`   — 契約書レビュー
```

コマンドの名前空間がそのまま部門へのルーティングテーブルになっている。`dev:` で始まるコマンドは CTO エージェントに、`mkt:` は CMO エージェントに振り分ける。Orchestrator は複雑なルーティングロジックを持つ必要がなく、プレフィックスを見るだけで判断できる。

この設計が Thin Orchestrator を実現する鍵だ。ルーティングに思考リソースを使わないことで、Orchestrator はもっと重要な判断、つまり「部門横断の調整」や「承認の判定」に集中できる。

## コンテキストウィンドウ効率の計測

Thin Orchestrator が実際に機能しているかを確認するには、コンテキストの使用効率を計測する必要がある。

実用的な計測方法は以下の通りだ。

**1. CLAUDE.md のトークン数を計測する**

```bash
# CLAUDE.md のおおよそのトークン数を確認
# 日本語は1文字 ≒ 1-2トークン、英語は1単語 ≒ 1トークン
wc -c CLAUDE.md
# 約8,000バイト → 約3,000-4,000トークン
```

**2. サブエージェント呼び出し時の指示サイズを確認する**

1回の委任指示が500トークン以下に収まっているかを確認する。目的1行、参照パス3-5行、出力先1行、品質基準2-3行であれば、200-300トークンに収まる。

**3. 1セッションで何回のサブエージェント呼び出しが可能かを逆算する**

200Kトークンのコンテキストウィンドウで、Orchestratorが使うのが10-15%（20K-30Kトークン）。1回の委任+結果受信で約500トークンとすると、1セッションで40-60回のサブエージェント呼び出しが可能だ。

実際の運用では、朝のダイジェスト生成（1回）、承認処理（2-3回）、部門タスク実行（3-5回）で、1日あたり10回前後の呼び出しで収まっている。コンテキストには十分な余裕がある。

## Orchestratorのアンチパターン

最後に、陥りがちなアンチパターンを3つ挙げる。

**1. Fat Orchestrator（太った指揮者）**
全部門のエージェント定義をCLAUDE.mdに書いてしまうパターン。10部門のエージェント定義を全部書くと、それだけで15,000-20,000トークンになる。コンテキストの10%を超え、セッション後半でパフォーマンスが劣化する。

**2. Micro-manager（過干渉）**
サブエージェントの成果物を毎回全文読み込んで細かくチェックするパターン。Orchestratorのコンテキストが実データで埋まり、判断能力が下がる。品質基準はサブエージェント側の定義に書き、Orchestratorはパス/フェイルの結果だけを受け取るべきだ。

**3. 委任なしの直接実行**
「簡単なタスクだから自分でやろう」とOrchestratorが実作業をしてしまうパターン。1回は問題なくても、積み重なるとコンテキストが膨張する。どんなに小さなタスクでも、対応するサブエージェントがあるならそちらに委任する。
