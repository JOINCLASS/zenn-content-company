# パフォーマンスとコスト最適化

マルチエージェントシステムは強力だが、無計画に運用すればコストが膨れ上がる。本章では、AI-CEO Frameworkの実運用で確立したコスト構造の理解と最適化テクニック、そしてROI計算の方法を解説する。


## マルチエージェントのコスト構造

### Claude Code Maxプラン（$200/月）の実力

2026年3月時点で、筆者はClaude Code Maxプラン（月額$200）を利用している。このプランはAPIトークン課金ではなく定額制のため、エージェントの起動回数やトークン消費量を気にせず使える。

ただし「無制限」ではない。5時間あたりの使用量上限があり、これを超えると一時的に制限がかかる。筆者の実運用では、1日に10-15回のエージェント起動であれば制限にかかることはほとんどない。1日に30回以上起動すると、午後に制限がかかることがある。

### トークン消費の内訳

定額制でもトークン消費の内訳を把握することは重要だ。どこにコストが集中しているかを知れば、最適化の余地がわかる。

```
トークン消費の分布（筆者の実測値）

Orchestrator（ルーティング判断）:     5-10%
CTO（コード生成・レビュー）:         25-30%
CMO/コンテンツ（記事・SNS生成）:     25-30%
CSO（提案書・メール生成）:           15-20%
CFO/税務（経理処理）:               10-15%
その他（CS, 法務, HR）:              5-10%
```

最もトークンを消費するのは「長文を生成するタスク」だ。コードの生成、記事の執筆、提案書の作成がこれに該当する。一方、状態の読み取りやKPIの集計は比較的軽い。

### コスト効率の高いタスク vs 低いタスク

| コスト効率が高い | コスト効率が低い |
|-----------------|-----------------|
| テンプレートベースの生成（提案書、メール） | 自由形式の長文生成（書籍の章） |
| 定型分析（KPI集計、ステータス確認） | 探索的な調査（競合分析、市場調査） |
| 既存コードの修正・リファクタリング | ゼロからの大規模コード生成 |
| 構造化データの処理（CSV→仕訳） | 非構造化データの解釈（契約書レビュー） |

テンプレートがあるタスクは、エージェントが「何を出力すべきか」を明確に理解できるため、試行錯誤が少なく、トークン消費が少ない。逆に、ゼロから創造するタスクはトークン消費が大きい。


## パフォーマンス最適化の実践テクニック

### 1. 並列実行の最大化

独立したタスクは必ず同時に起動する。依存関係がないタスクを直列に実行するのは、時間の無駄だ。

```
悪い例（直列実行）:
  /ai-ceo:dev:sprint → 完了を待つ → /ai-ceo:mkt:content-plan → 完了を待つ
  合計: 60分

良い例（並列実行）:
  /ai-ceo:dev:sprint     ──┐
  /ai-ceo:mkt:content-plan ──┤ 同時起動
  /ai-ceo:fin:monthly-report──┘
  合計: 30分（最も遅いタスクの所要時間）
```

並列実行の判断基準は単純だ。「タスクAの出力がタスクBの入力に必要か？」。Noなら並列にできる。Yesなら直列にするしかない。

筆者の実運用では、朝のmorningダイジェスト生成時に6-8部門のSTATE読み取りを並列で行っている。これにより、9部門の状態収集が1部門分の時間で完了する。

### 2. バッチ処理パターン

似たタスクをまとめて1回のセッションで処理する。エージェントの起動にはオーバーヘッドがある。コンテキストの読み込み、エージェント定義の解釈、参照ファイルの読み取り。これらを毎回繰り返すのは非効率だ。

```
悪い例（1件ずつ起動）:
  /ai-ceo:sales:proposal "A社"  → 起動コスト + 実行
  /ai-ceo:sales:proposal "B社"  → 起動コスト + 実行
  /ai-ceo:sales:proposal "C社"  → 起動コスト + 実行
  起動オーバーヘッド: 3回分

良い例（バッチ処理）:
  "A社、B社、C社の提案書を一括生成して"
  → CSOエージェントが1回の起動で3社分を処理
  起動オーバーヘッド: 1回分
```

筆者がHPリニューアル営業で21社分の提案書を作成したときも、このバッチ処理パターンを使った。5社ずつ4バッチ+1社の合計5回の起動で21社分を処理した。21回起動する場合と比較して、起動オーバーヘッドを76%削減した。

### 3. STATE.mdを中間キャッシュとして活用

STATE.mdは単なる状態記録ではなく、エージェント間の「キャッシュ」として機能する。

例えば、CTOエージェントがコードレビューを実行した結果を`departments/dev/STATE.md`に記録しておけば、翌日のmorningダイジェストでは「コードレビュー完了」という情報をSTATE.mdから読み取るだけで済む。コードベースを再度スキャンする必要がない。

```markdown
# 開発部門 STATE -- 2026-03-09

## 直近の完了タスク
- [x] Focalize: セキュリティパッチ適用（2026-03-08）
- [x] AIコンサルLP: 仕様書作成完了（2026-03-07）

## 進行中
- [ ] AIコンサルLP: 実装（着手待ち）

## ブロッカー
- LP実装のデザインカンプがない → CEOに確認
```

このSTATE.mdがあることで、他のエージェントは開発部門の状態を「ファイル1つ読むだけ」で把握できる。コードベース全体をスキャンして状態を推定する必要がない。

### 4. 不要なエージェント起動の削減

CLAUDE.md（Orchestrator）に判断基準のif-thenルールを書くことで、不要なエージェント起動を防ぐ。

```markdown
## エージェント起動の判断基準

- 承認キューが空の場合 → morningダイジェストで
  「承認待ちなし」と表示するだけ。承認エージェントは起動しない。
- CS部門の問い合わせがゼロの場合 → CSエージェントは起動しない。
- 月末でない場合 → CFOの月次決算エージェントは起動しない。
```

筆者の場合、CS部門はユーザー数が少ないため、ほとんど起動されない。それでもエージェント定義は用意してある。ユーザーが増えたときに即座に稼働できるようにするためだ。エージェント定義のコスト（ファイル1つ）は無視できるほど小さい。

### 5. 軽量エージェントと重量エージェントの使い分け

すべてのエージェントが同じ「重さ」である必要はない。

```
軽量エージェント（ツール最小限）:
  ai-ceo-morning.md → tools: [Read, Grep]
  → STATE.mdを読むだけ。Write不要。Bash不要。

重量エージェント（フルツールセット）:
  ai-ceo-cto.md → tools: [Read, Write, Edit, Bash, Grep]
  → コード生成、テスト実行、ファイル編集すべてが必要。
```

朝ダイジェストのような読み取り専用タスクには、ReadとGrepだけを持つ軽量エージェントで十分だ。ツールが少ないほうが起動が速く、エージェントの判断も単純になる。


## ROI計算

### 月$200で節約できる工数の試算

AI-CEO Frameworkの投資対効果を計算してみよう。

```
月間コスト:
  Claude Code Max: $200/月（約30,000円）

月間の工数削減効果（筆者の実測値）:
  提案書作成: 3時間/件 × 5件/月 = 15時間 → 1.25時間に削減（13.75時間削減）
  記事執筆:   2時間/本 × 4本/月 = 8時間  → 0.7時間に削減（7.3時間削減）
  経理処理:   4時間/月           = 4時間  → 0.5時間に削減（3.5時間削減）
  営業メール: 0.5時間/通 × 10通 = 5時間  → 0.8時間に削減（4.2時間削減）
  経営管理:   2時間/日 × 20日   = 40時間 → 10時間に削減（30時間削減）

  合計削減: 約59時間/月
```

エンジニアの時給を5,000円として計算すると、月間の工数削減効果は約295,000円。月額30,000円の投資に対して、約10倍のリターンだ。

### 損益分岐点

| 組織規模 | 月間コスト | 必要な工数削減 | 損益分岐 |
|---------|-----------|--------------|---------|
| 1人社長 | 30,000円 | 6時間/月 | 2-3タスクの自動化で達成 |
| 5人チーム | 30,000円 | 6時間/月 | 1人が月6時間節約すれば達成 |
| 20人組織 | 30,000円 | 6時間/月 | 全員が月18分節約すれば達成 |

損益分岐点のハードルは驚くほど低い。1人社長の場合、提案書2件分の自動化だけで元が取れる。5人チームなら、1人あたり月6時間の節約。20人組織なら、1人あたり月18分の節約で損益分岐する。

### 隠れた価値: 速度と機会損失

ROI計算に表れない価値がある。それは「速度」だ。

提案書を3時間で書く代わりに15分で仕上がるなら、その日のうちに提案を送れる。翌週まで待たせることがない。営業の世界では、レスポンスの速さが受注率に直結する。

HPリニューアル営業で21社分の提案書を1週間で作れたのは、エージェントのおかげだ。人間が1社ずつ作っていたら2ヶ月かかる。その2ヶ月の間に競合に先を越されるリスクを考えれば、月30,000円は保険料としても安い。


## コスト監視の仕組み

定額制とはいえ、使用量の監視は重要だ。5時間あたりの上限に近づいていることを把握し、重要なタスクを優先的に実行するためだ。

筆者は`.company/scripts/cost-tracker.json`で月間の起動回数を追跡している。

```json
{
  "month": "2026-03",
  "total_usd": 200,
  "calls": {
    "cto": 45,
    "cmo": 38,
    "cso": 32,
    "cfo": 18,
    "content": 42,
    "tax": 12,
    "legal": 5,
    "cs": 2,
    "hr": 3,
    "morning": 28
  }
}
```

このデータから、どの部門がリソースを多く消費しているかが一目でわかる。コンテンツ部門とCTO部門が多いのは、書籍執筆とプロダクト開発が活発な時期だからだ。

予算アラートの設計はシンプルだ。permissions.mdに月間予算上限を定義し、80%に到達したらCEOに通知する。

```yaml
monthly_ai_budget: 200  # USD
alert_threshold: 0.8     # 80%で通知
```


## スケーリング戦略

組織の成長に合わせて、エージェント構成はどう変わるべきか。筆者の経験と推奨構成を示す。

### フェーズ1: 1人（個人開発者・フリーランス）

```
CEO（人間）── Orchestrator
  ├─ CTO（開発）
  └─ CMO（マーケ）
```

まず2部門から始める。開発とマーケは、1人で事業をやる場合に最も時間を食う領域だ。コードを書く時間と、書いたものを売る時間。この2つをエージェントに委任するだけで、1日3-4時間を確保できる。

### フェーズ2: 1-3人（スモールチーム）

```
CEO（人間）── Orchestrator
  ├─ CTO（開発）
  ├─ CMO（マーケ）
  ├─ CFO（経理）
  └─ CSO（営業）
```

4部門構成。経理と営業を追加する。チームが3人になると、経費処理の量が増え、営業活動も本格化する。この段階で承認パイプラインを導入し、対外アクションのガバナンスを確立する。

### フェーズ3: 3-10人（成長期）

```
CEO（人間）── Orchestrator
  ├─ CTO（開発）
  ├─ CMO（マーケ）
  ├─ CFO（経理）
  ├─ CSO（営業）
  ├─ CS（カスタマーサポート）
  ├─ 法務
  └─ HR
```

7部門構成。ユーザーが増えるとCS部門が必要になり、契約書が増えると法務が必要になる。HRは人間の従業員管理だけでなく、エージェント自体の管理（定義の更新、パフォーマンス評価）も担う。

### フェーズ4: 10人以上（安定期）

```
CEO（人間）── Orchestrator
  ├─ CTO（開発）── 複数のサブエージェント（architect, coder, reviewer）
  ├─ CMO（マーケ）── 複数のサブエージェント（SEO, SNS, ads）
  ├─ CFO（経理）
  ├─ CSO（営業）
  ├─ CS
  ├─ 法務
  ├─ 税務
  ├─ コンテンツ
  └─ HR
```

9部門フル構成。さらに、CTO部門やCMO部門の中にサブエージェントを配置する階層構造に移行する。CTOエージェントがarchitect、coder、reviewerの3つのサブエージェントを使い分ける。Orchestratorパターンの再帰的適用だ。

各フェーズで重要なのは、「部門を増やすタイミング」を間違えないことだ。週3時間以上その作業に使っているなら追加。そうでなければ既存部門に含める。この基準を守れば、過剰な複雑さを避けられる。
