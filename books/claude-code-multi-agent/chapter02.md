# マルチエージェントのアーキテクチャ設計

## アーキテクチャ選定は最初の分岐点

マルチエージェントシステムを構築する際、最初に決めなければならないことがある。「エージェント同士をどう繋ぐか」だ。

この判断を誤ると、後から修正するコストが非常に大きい。データベースの正規化をやり直すような痛みと言えば、エンジニアには伝わるだろう。

本章では、マルチエージェントの3つの基本パターンを比較し、Claude Codeに最適なアーキテクチャを導き出す。そして、筆者が1年以上の実運用で磨き上げた「Thin Orchestrator原則」と「エージェント責務分割の設計原則」を解説する。


## 3つの基本パターン

マルチエージェントのアーキテクチャは、大きく3つのパターンに分類できる。

### パターン1: Orchestrator-Worker

```
         ┌─────────────────┐
         │  Orchestrator    │
         │  (指揮者)        │
         └────┬───┬───┬────┘
              │   │   │
         ┌────┘   │   └────┐
         ▼        ▼        ▼
      ┌─────┐ ┌─────┐ ┌─────┐
      │ CTO │ │ CMO │ │ CFO │
      └─────┘ └─────┘ └─────┘
```

1つの「指揮者」エージェントがタスクを受け取り、適切な「作業者」エージェントに委任する。作業者はOrchestratorにのみ結果を返し、作業者同士は直接やり取りしない。

**メリット:**

- 制御フローが明確。誰が何を指示し、誰が何を返すかが追跡しやすい
- エージェントの追加・削除が容易。Orchestratorの委任ロジックを変更するだけ
- 権限制御がしやすい。Orchestratorが全アクションのゲートキーパーになる

**デメリット:**

- Orchestratorがボトルネックになりうる。すべてのタスクがOrchestratorを経由する
- Orchestratorのコンテキストが肥大化しやすい

### パターン2: Pipeline（パイプライン）

```
  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
  │ 企画 │ →  │ 執筆 │ →  │ 校正 │ →  │ 公開 │
  └──────┘    └──────┘    └──────┘    └──────┘
```

タスクが直線的に流れる。前工程の出力が、次工程の入力になる。製造業のアセンブリラインと同じ発想だ。

**メリット:**

- 処理の順序が明確。デバッグしやすい
- 各ステージの責務がきれいに分離される

**デメリット:**

- 並列処理ができない。全工程が直列実行
- 途中のステージで失敗すると、以降のすべてが止まる
- 工程数が多いと、全体のレイテンシが大きくなる

### パターン3: Peer-to-Peer

```
      ┌─────┐
      │ CTO │ ←─────────────┐
      └──┬──┘               │
         │                  │
         ▼                  │
      ┌─────┐           ┌─────┐
      │ CMO │ ─────────→ │ CFO │
      └─────┘           └─────┘
```

すべてのエージェントが対等で、必要に応じて直接やり取りする。中央の管理者は存在しない。

**メリット:**

- 柔軟性が高い。エージェント同士が自律的に協調できる
- 単一障害点がない

**デメリット:**

- 制御フローの追跡が困難。誰が誰に何を依頼したかを把握しにくい
- デバッグが極めて難しい。問題の原因特定に時間がかかる
- エージェント数が増えると、通信経路が指数的に増加する（n*(n-1)/2）
- 権限制御が複雑。各エージェントが自律的に判断するため、暴走リスクがある


## Claude Codeに最適なパターン

結論から言うと、Claude Codeで構築するマルチエージェントシステムには**Orchestrator-Workerパターン**が最適だ。理由は3つある。

### 理由1: Claude Codeの設計思想との親和性

Claude Codeには「Agent tool」というサブエージェント起動の仕組みが組み込まれている。親エージェント（Orchestrator）が子エージェント（Worker）を起動し、タスクを委任し、結果を受け取る。この構造は、まさにOrchestrator-Workerパターンそのものだ。

Claude Codeの設計思想に逆らわず、自然に乗るパターンを選ぶ。これがアーキテクチャ選定の最も重要な原則だ。

### 理由2: ファイルベース通信との相性

Claude Codeのサブエージェントは、ファイルの読み書きを通じて仕事をする。データベースやメッセージキューは使わない。Markdownファイルが「組織の共有メモリ」になる。

Orchestrator-Workerパターンでは、以下のようなファイルベース通信が自然に成立する。

```
.company/
  ├── STATE.md              ← Orchestratorが読む（全体状態）
  ├── approval-queue.md     ← Workerが書き込み、CEOが読む
  ├── departments/
  │     ├── dev/STATE.md    ← CTOエージェントが読み書き
  │     ├── marketing/STATE.md ← CMOエージェントが読み書き
  │     └── finance/STATE.md   ← CFOエージェントが読み書き
  └── products/
        ├── product-a/STATE.md ← 各エージェントが参照
        └── product-b/STATE.md
```

各Workerエージェントは「自分の部門のファイル」を読み書きし、他部門のファイルは読み取り専用で参照する。このルールだけで、エージェント間の通信と状態同期が成立する。

### 理由3: 承認パイプラインの実装が容易

経営におけるマルチエージェントでは、「AIが勝手に対外アクションを実行しない」ことが絶対条件だ。Orchestrator-Workerパターンなら、Orchestratorが全アクションのゲートキーパーとなり、承認パイプラインを自然に組み込める。

```
Worker → draftファイル生成 → approval-queue.mdに追加
CEO → 承認/却下
Orchestrator → 承認されたアクションを実行
```

Peer-to-Peerでこれを実現しようとすると、すべてのエージェントに承認チェックのロジックを組み込む必要があり、漏れのリスクが高い。


## Thin Orchestrator原則 -- 指揮者は自分で演奏しない

Orchestrator-Workerパターンを採用した上で、次に重要なのがOrchestratorの設計だ。

筆者が1年以上の運用で得た最も重要な教訓を、1つの原則にまとめた。

**「Thin Orchestrator原則: Orchestratorは指揮者であり、自分で演奏してはならない。」**

### 原則の内容

Orchestratorがやるべきことは3つだけだ。

1. **CEOのコマンドを解釈する** -- 「何を」「どの部門に」「どの優先度で」やらせるかを判断する
2. **適切なWorkerに委任する** -- タスクの目的、参照ファイルパス、成果物の出力先、権限レベル、品質基準を渡す
3. **結果を統合して報告する** -- 複数のWorkerからの結果をまとめ、CEOに報告する

Orchestratorがやってはいけないことは以下だ。

- **ファイルの中身を読み込む**: ファイルパスだけを管理し、中身はWorkerが読む
- **コードを書く**: コーディングはCTOエージェントの仕事
- **文章を書く**: ライティングはCMOエージェントの仕事
- **計算をする**: 経理処理はCFOエージェントの仕事

### なぜThinでなければならないか

理由はコンテキストウィンドウの制約に尽きる。

Orchestratorがファイルの中身を読み込み始めると、コンテキストがすぐに埋まる。6つのプロダクトのSTATE.md、9部門のエージェント定義、承認待ちアイテムの詳細。これらをすべて読み込むと、それだけで数万トークンを消費する。

Orchestratorのコンテキストが埋まると、新しいコマンドを受け付けられなくなる。つまり、CEOがAIに指示を出せなくなる。これは経営として致命的だ。

**Thin Orchestratorは、コンテキスト使用率を10-15%に維持する。** 残りの85-90%は、CEOとの対話とコマンドの解釈に使える。ファイルの中身が必要な作業は、すべてWorkerに委任する。

### 実装例: Orchestratorの委任テンプレート

筆者のCLAUDE.mdでは、Workerへの委任時に以下の5項目を必ず渡すルールにしている。

```markdown
## サブエージェントへの委任方法

タスクをサブエージェントに委任する際は、以下の情報を渡す:
1. **タスクの目的** — 何を達成するか（1文で）
2. **参照ファイルパス** — 必要な入力ファイルのパス一覧
3. **成果物の出力先** — 出力ファイルのパスとフォーマット
4. **権限レベル** — read-only / draft / execute
5. **品質基準** — 完了条件と検証方法
```

これにより、Orchestratorは「何をWorkerに渡すか」だけに集中し、「Workerが何をどう処理するか」の詳細には関与しない。


## エージェントの責務分割 -- Single Responsibility for Agents

ソフトウェア設計のSingle Responsibility Principle（SRP: 単一責任原則）は、エージェント設計にもそのまま適用できる。

**「1つのエージェントは、1つの責務だけを持つ。」**

### 責務分割の4つの基準

エージェントの責務をどこで分割するかは、以下の4つの基準で判断する。

**基準1: ドメイン知識の境界**

必要な専門知識が異なるなら、エージェントを分ける。コーディング規約の知識と、SEOライティングの知識は、まったく別のドメインだ。1つのエージェントに両方を持たせると、どちらも中途半端になる。

```
良い分割:
  CTO エージェント → コーディング規約、テスト方針、CI/CD
  CMO エージェント → SEO、コンテンツ戦略、SNS運用

悪い分割:
  「開発もマーケもやるエージェント」→ どちらも品質が落ちる
```

**基準2: 権限の境界**

必要な権限レベルが異なるなら、エージェントを分ける。コードの読み取りだけで完結するレビューと、ファイルの変更を伴うバグ修正は、権限レベルが異なる。

**基準3: 実行頻度の境界**

実行頻度が大きく異なるタスクは、エージェントを分ける。毎日実行するタスク（状態レポート）と、月1回のタスク（月次決算）を同じエージェントに持たせると、定義ファイルが肥大化する。

**基準4: 障害の影響範囲**

障害が起きた時の影響を局所化したいなら、エージェントを分ける。デプロイのエージェントと経理のエージェントが同一だと、デプロイ障害で経理も止まる。

### 分割しすぎの罠

ただし、エージェントを細かく分割しすぎるのも問題だ。

エージェントの数が増えると、Orchestratorの委任ロジックが複雑になる。「このタスクはどのエージェントに渡すべきか」の判断に迷う場面が増える。エージェント間の依存関係も増え、調整コストが上がる。

筆者の経験則では、**5-10個が適切な範囲**だ。筆者は10部門で運用しているが、これは9つの事業機能が明確に分かれているからだ。3つのプロダクトを1人で開発している場合なら、3-5個のエージェントで十分だろう。

分割の判断に迷ったら、以下の問いを立てる。

**「このエージェントのCLAUDE.mdは、1画面（50行以内）に収まるか?」**

収まるなら、適切な粒度だ。収まらないなら、分割を検討する。逆に、10行しかないなら、他のエージェントと統合してもよい。


## 設計図の描き方: CLAUDE.mdでのエージェント定義構造

アーキテクチャが決まったら、具体的な設計図をCLAUDE.mdとエージェント定義ファイルで描く。

### プロジェクト全体の構造

```
project-root/
  ├── CLAUDE.md                    ← Orchestratorの定義
  ├── .claude/
  │     └── agents/                ← Workerエージェントの定義
  │           ├── cto.md           ← CTO部門エージェント
  │           ├── cmo.md           ← CMO部門エージェント
  │           ├── cfo.md           ← CFO部門エージェント
  │           └── content.md       ← コンテンツ部門エージェント
  └── .company/                    ← 共有状態ファイル
        ├── STATE.md               ← 全体状態
        ├── approval-queue.md      ← 承認キュー
        ├── steering/              ← 経営方針
        ├── departments/           ← 部門別状態
        └── products/              ← プロダクト別状態
```

### CLAUDE.md（Orchestrator）の基本構造

```markdown
# プロジェクト名 — Orchestrator

> あなたは{プロジェクト名}のOrchestratorです。
> {役割の1文要約}

## あなたの役割
1. コマンドの受付と解釈
2. 適切なWorkerへの委任
3. 結果の統合と報告

## Thin Orchestratorの原則
- ファイルの中身を読み込まない
- 自分で実作業をしない
- コンテキスト使用率を10-15%に維持する

## コマンド一覧
- `/command:action` — 説明

## 委任ルール
タスクを委任する際は以下を渡す:
1. タスクの目的
2. 参照ファイルパス
3. 成果物の出力先
4. 権限レベル
5. 品質基準

## エラー時の振る舞い
- 3回リトライして失敗 → エスカレーション
```

### Worker エージェント定義の基本構造

```markdown
# {部門名}エージェント

> あなたは{部門名}の専門エージェントです。

## ペルソナ
{この部門の専門性を1-2文で定義}

## 専門知識
- {知識領域1}
- {知識領域2}

## 権限レベル
- **execute:** {自動実行できるアクション}
- **draft:** {承認が必要なアクション}

## ワークフロー
### {コマンド名}
1. {手順1}
2. {手順2}

## 品質基準
1. {基準1}
2. {基準2}
```

この構造に従えば、新しいエージェントの追加は「`.claude/agents/`にファイルを1つ追加し、CLAUDE.mdのコマンド一覧に1行追加する」だけで完了する。


## アーキテクチャ決定のチェックリスト

最後に、マルチエージェントアーキテクチャを設計する際のチェックリストをまとめておく。プロジェクトの初期設計時に、各項目を確認してほしい。

### 全体設計

- [ ] アーキテクチャパターンを選定したか（Orchestrator-Worker推奨）
- [ ] Orchestratorの責務を「指揮のみ」に制限したか
- [ ] エージェントの数は5-10個の範囲に収まっているか
- [ ] 各エージェントの責務は1つの専門領域に絞られているか

### エージェント定義

- [ ] 各エージェントのCLAUDE.mdは50行以内に収まっているか
- [ ] 各エージェントの権限レベル（read-only / draft / execute）を定義したか
- [ ] 各エージェントの品質基準を定義したか
- [ ] 各エージェントのエラー時の振る舞いを定義したか

### 通信設計

- [ ] ファイルベースの状態管理ディレクトリ構造を設計したか
- [ ] 各エージェントが読み書きするファイルの範囲を明確にしたか
- [ ] 承認パイプラインのフロー（draft → 承認 → 実行）を設計したか
- [ ] エージェント間の依存関係を洗い出したか

### 運用設計

- [ ] Orchestratorのコンテキスト使用率の監視方法を決めたか
- [ ] エラー時のリトライ回数とエスカレーション先を決めたか
- [ ] エージェントの追加・変更・削除の手順を文書化したか
- [ ] コスト（トークン消費）の見積もりをしたか

これらを満たした上で実装に入れば、後から「設計を根本から見直す」事態は避けられるはずだ。
