# なぜマルチエージェントなのか -- 単一エージェントの限界を超える

:::message
**本書の対象バージョン:** 本書はClaude Code 2026年3月時点の機能に基づいている。Claude Codeは急速に進化しているため、一部のパラメータや挙動が変更されている可能性がある。最新情報は[Anthropic公式ドキュメント](https://docs.anthropic.com/en/docs/claude-code)を参照してほしい。
:::

## Claude Codeは「1人の優秀な部下」である

Claude Codeを日常的に使っている人なら、その有能さを実感しているはずだ。コードの生成、バグの修正、テストの作成、ドキュメントの執筆。1つのタスクを渡せば、驚くほどの精度で仕上げてくれる。

しかし、プロジェクトの規模が大きくなると、ある壁にぶつかる。

「1人の部下にすべてを任せても、同時にこなせる仕事には限界がある。」

筆者は合同会社ジョインクラスのCEOとして、6つの自社プロダクトと複数の事業ラインを運営している。従業員は自分1人。しかし社内には10の部門が存在し、それぞれにAIエージェントが配置されている。この体制を1年以上にわたって実運用してきた。

本書は、その実運用から得た設計・実装・運用のノウハウを体系化したものだ。Claude Codeのサブエージェント機能を使い、複数のAIエージェントが協調して動くシステムをどう作るか。抽象論ではなく、実装に踏み込んで解説する。


## 単一エージェントでできること・できないこと

まず、単一のClaude Codeセッションで何ができて、何ができないかを明確にしておこう。

### 単一エージェントが得意なこと

- **1つのタスクに集中した作業**: 「このファイルをリファクタリングして」「このバグを修正して」
- **コンテキストが1ファイルに収まる作業**: 関連コードがすべて見渡せる範囲での実装
- **対話的な作業**: 人間と往復しながら仕様を詰め、段階的に実装する
- **単一ドメインの知識で完結する作業**: フロントエンドだけ、バックエンドだけ、といった範囲

### 単一エージェントの限界

問題は、実際の業務が「1つのタスク」で完結することが少ないという点だ。

**限界1: コンテキストウィンドウの壁**

Claude Codeのコンテキストウィンドウは有限だ。会話が長くなれば、古い情報は押し出される。大規模なプロジェクトで「全体像を把握したまま」作業を続けることは、物理的に不可能になる。

筆者のケースで言えば、6つのプロダクトの状態管理ファイル、10部門のエージェント定義、経営方針ドキュメント、四半期ロードマップ。これらをすべて1つのセッションに読み込もうとすると、それだけでコンテキストの大部分を消費してしまい、肝心の作業に使える容量が残らない。

**限界2: 専門性の希薄化**

1つのエージェントにすべてを任せると、「何でもそこそこできるが、何も深くできない」状態になる。

SEOに最適化された記事を書かせたい場合、SEOの専門知識（検索意図分析、E-E-A-T対策、構造化マークアップ）をプロンプトに含める必要がある。同じエージェントにコードレビューもさせたい場合、コーディング規約、テスト方針、パフォーマンス基準もプロンプトに含める。経理処理もさせたい場合、勘定科目体系、税務ルール、freeeの操作知識も必要になる。

結果、プロンプトが肥大化し、各分野の指示が薄まり、どの仕事も中途半端な品質になる。

**限界3: 順次実行のボトルネック**

単一エージェントは、タスクを1つずつ順番にしか処理できない。コードレビューが終わるまで記事執筆は始められない。記事執筆が終わるまで経理処理は始められない。人間のCEOが6つの部門を「1人で回す」のと同じ問題が、AIにも起きる。

**限界4: 障害の連鎖**

1つのセッションですべてを処理している場合、途中でエラーが起きると全体が止まる。コードレビュー中にファイルの読み取りエラーが発生したら、その後に予定していた記事執筆も経理処理も巻き添えで中断される。


## マルチエージェントで解決できる3つの課題

これらの限界を突破するのが、マルチエージェントアーキテクチャだ。解決できる課題は大きく3つある。

### 1. 並列処理 -- 同時に複数の仕事を進める

マルチエージェントでは、独立したタスクを複数のエージェントが同時に実行できる。

```
単一エージェント:
  コードレビュー(30分) → 記事執筆(30分) → 経理処理(20分) = 80分

マルチエージェント:
  CTOエージェント: コードレビュー(30分) ──┐
  CMOエージェント: 記事執筆(30分) ────────┤ = 30分
  CFOエージェント: 経理処理(20分) ────────┘
```

筆者の実運用では、朝の`/ai-ceo:morning`コマンド1つで、全10部門の状態を同時に収集し、統合レポートを生成する。各部門エージェントが並列で動くため、10部門の情報収集が1部門分の時間で完了する。

### 2. 専門分化 -- 各エージェントがプロフェッショナルになる

エージェントごとに専門知識を分離すると、各エージェントのCLAUDE.md（またはエージェント定義ファイル）には、その分野に特化した知識だけを記述できる。

- CTOエージェント: コーディング規約、テスト方針、CI/CDパイプラインの知識
- CMOエージェント: SEOライティング、コンバージョン最適化、SNS戦略の知識
- CFOエージェント: 勘定科目体系、税務ルール、会計ソフト連携の知識

コンテキストウィンドウを1つの専門領域に集中できるため、各タスクの品質が上がる。「広く浅く」から「狭く深く」へのシフトだ。

### 3. リスク分離 -- 障害を局所化する

エージェントが独立したプロセスとして動くため、1つのエージェントで障害が起きても、他のエージェントには影響しない。

CTOエージェントがコードレビュー中にエラーを起こしても、CMOエージェントの記事執筆は中断されない。障害が起きたエージェントだけをリトライすればよい。筆者のフレームワークでは、3回リトライして失敗した場合のみCEOにエスカレーションする設計にしている（詳しくは第9章で解説する）。


## 実例: 1人社長が9部門を同時に動かすユースケース

マルチエージェントの威力を具体的に示そう。筆者が実際に運用している9部門の構成を紹介する。

```
CEO（人間）── CLAUDE.md Orchestrator
  │
  ├─ CTO部門 ──── コードレビュー、CI/CD、テスト、スプリント管理
  ├─ CMO部門 ──── SEO記事、SNS投稿、LP改善、コンテンツカレンダー
  ├─ CFO部門 ──── 経理処理、月次決算、予算管理、コスト追跡
  ├─ CSO部門 ──── 提案書作成、顧客リスト管理、営業メール下書き
  ├─ CS部門 ──── ユーザー問い合わせ対応、エスカレーション管理
  ├─ 法務部門 ── 契約書レビュー、OSSライセンス監査、コンプライアンス
  ├─ 税務部門 ── 仕訳チェック、確定申告準備、節税対策
  ├─ コンテンツ部門 ── Zenn書籍執筆、技術記事量産
  ├─ 広告部門 ── Google Ads運用、広告文作成、入札最適化
  └─ HR部門 ── エージェント管理、オンボーディング設計
```

これらのエージェントが、ファイルベースの状態管理と承認パイプラインで結合されている。

典型的な1日の流れはこうだ。

1. CEOが`/ai-ceo:morning`を実行
2. Orchestratorが10部門のエージェントを並列起動し、各部門の状態を収集
3. 統合レポートが生成され、承認待ちアイテムの一覧が表示される
4. CEOは承認待ちを確認し、承認または却下する
5. 承認されたアイテム（記事公開、メール送信、デプロイなど）が実行される
6. CEOは新たな指示（「来週のSNS投稿を準備して」「ShareTokuのLP改善して」）を出す
7. 該当部門のエージェントが作業を開始し、完了したらdraftとして承認キューに追加

このサイクルが1日に何度も回る。CEOの仕事は「指示と判断」に集中し、「作業」はエージェントが担う。

単一エージェントでこれを実現しようとすると、10部門の知識を1つのプロンプトに詰め込み、タスクを1つずつ順番に処理し、全体の状態を1つのコンテキストで管理しなければならない。物理的に不可能だ。


## この書籍で学べること

本書は全12章で構成されている。以下のマップに沿って、マルチエージェントシステムの設計から実装、運用までを順に解説する。

### Part 1: 設計編（第1章〜第3章）

- **第1章**（本章）: なぜマルチエージェントなのか -- 単一エージェントの限界と、マルチエージェントで解決できる課題
- **第2章**: マルチエージェントのアーキテクチャ設計 -- Orchestrator-Workerパターン、Thin Orchestrator原則、責務分割の設計
- **第3章**: Claude Codeサブエージェントの仕組み -- Agent toolの内部動作、ライフサイクル、プロンプト設計原則

### Part 2: 実装編（第4章〜第9章）

- **第4章**: Orchestratorパターンの実装 -- コマンドルーティング、Thin Orchestrator戦略、コンテキスト効率の計測
- **第5章**: 専門エージェントの設計と実装 -- 6つの実例でエージェント定義ファイルの書き方を解説
- **第6章**: エージェント間通信の設計 -- ファイルベース通信、STATE.mdパターン、承認キュー
- **第7章**: 承認パイプラインと人間-AI協調 -- 3段階の権限レベル設計、permissions.md、承認ワークフロー
- **第8章**: コンテキストウィンドウ管理の実践テクニック -- ファイルパス参照戦略、サマリ層、セッション分割
- **第9章**: エラーハンドリングとリカバリ戦略 -- リトライ設計、エスカレーション、冪等性、ロールバック

### Part 3: 運用編（第10章〜第12章）

- **第10章**: 実践 -- 10部門AIエージェント経営システムの全貌 -- 1日の運用フロー、部門間連携の実例、運用データの公開
- **第11章**: パフォーマンスとコスト最適化 -- ROI計算、並列実行の最大化、スケーリング戦略

各章には、実際に動作するCLAUDE.mdの設定例、ディレクトリ構造、エージェント定義ファイルのコードを豊富に含めている。本書を読みながら手元で再現し、あなた自身のマルチエージェントシステムを構築できるように設計した。

それでは、次章でアーキテクチャの設計から始めよう。
