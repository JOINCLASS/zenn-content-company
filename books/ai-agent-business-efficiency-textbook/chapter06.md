# PoCから本番運用への移行

## この章で学ぶこと

- PoCと本番運用のギャップを構造的に理解する5観点
- 「PoCで止まる」を防ぐための運用設計の進め方
- インフラ構成（クラウド/オンプレ/ハイブリッド）の選定基準
- 監視・アラート・SLO（Service Level Objective）の設計
- 段階リリース戦略（カナリア・グリーン/ブルー・段階展開）
- セキュリティレビュー・コンプライアンスチェックの実務

## はじめに -- 「PoCで止まる」の正体

業界全体で、AI PoCが本番運用に到達する割合は3割程度と言われている。残りの7割は何が原因で止まっているのか。

筆者が実際に「PoCで止まった案件」を観察した結果、原因は以下に集約される。

1. **PoC環境と本番環境のギャップが大きすぎた**（接続するシステムが違う、データ量が違う等）
2. **セキュリティレビューでブロックされた**（情報セキュリティ・法務の懸念）
3. **運用体制が決まらない**（誰が見るのか・誰が改修するのか）
4. **コスト試算が現実とずれた**（PoCより本番のほうがはるかに高コスト）
5. **現場が使わない**（「便利だが今のやり方で十分」と言われる）

これらはすべて、**Phase 4（PoC開発）の段階で予防できる**問題だ。本章では、PoCを「**本番に乗せる前提**」で進めるための具体的な手法を解説する。

:::message
**筆者の実践メモ**
筆者が支援する案件では、PoC開始時に「**本番移行のためのチェックリスト**」を最初に作る。PoCが終わるまでに、このチェックリストの全項目を埋めることを目標とする。これを徹底すると、PoCから本番への移行率が大幅に上がる。
:::

## PoCと本番運用のギャップ（5観点）

PoCと本番運用には、以下の5つのギャップがある。

### ギャップ1: データ量・データ種類

- **PoC**: 50-100件の選別されたサンプルデータ
- **本番**: 月数千〜数万件、品質バラバラ、想定外パターン多数

### ギャップ2: ユーザー数・利用パターン

- **PoC**: 開発者・業務担当数名が手動でテスト
- **本番**: 全社員が利用、同時アクセス、業務繁忙期のスパイク

### ギャップ3: システム連携

- **PoC**: テスト用サンドボックスAPI、開発DBコピー
- **本番**: 本番CRM・ERP、リアルタイムデータ、SLO制約

### ギャップ4: セキュリティ要求

- **PoC**: 開発者権限で全て触れる
- **本番**: 個人情報マスキング、監査ログ、アクセス制御、SOC 2等の準拠

### ギャップ5: 運用責任

- **PoC**: 開発者がエラー対応、平日昼間のみ
- **本番**: 24/7運用、SLA、エスカレーション体制

これら5つのギャップを「**PoC設計時から**」意識して埋めていくのが本章のテーマだ。

## 本番移行のためのチェックリスト

PoC開始時に作る、本番移行のためのマスターチェックリストを示す。

### 機能要件チェックリスト

```
□ 業務フローの全ステップが実装されている
□ エラー時のフォールバック経路が動作する
□ 確信度低下時の人エスカレーションが動作する
□ 既存システムとの連携が本番APIで動作する
□ バッチ処理・リアルタイム処理の両方に対応している（必要なら）
```

### 非機能要件チェックリスト

```
□ レスポンス時間が目標値以内（例: 95%が3秒以内）
□ 同時接続数の目標を満たす
□ 月間処理量の上限が見積もられている
□ ピーク時の負荷試験が完了している
□ コスト試算（月額）が確定している
```

### セキュリティチェックリスト

```
□ 認証方式が決まっている（OAuth/SAML/APIキー等）
□ 認可（権限管理）が実装されている
□ 通信が暗号化されている（HTTPS）
□ データ保存時の暗号化が実装されている
□ 個人情報マスキングが動作している
□ 監査ログが90日以上保存される
□ APIキーがコードに埋め込まれていない
□ 情シス・法務のセキュリティレビューを受けた
□ 脆弱性スキャンが実施された
□ インシデント対応手順書が用意されている
```

### 運用チェックリスト

```
□ 監視ツールがセットアップされている
□ アラート条件・通知先が設定されている
□ 一次対応者・エスカレーション先が決まっている
□ 障害対応手順書（Runbook）がある
□ バックアップ・リストア手順がある
□ 定期メンテナンス計画がある
□ 利用者向けマニュアルがある
□ 問い合わせ窓口が決まっている
```

### コンプライアンスチェックリスト

```
□ 個人情報保護法に準拠している
□ 業界固有規制への準拠を確認した（金商法/医療法等）
□ AI出力の根拠説明が後追い可能
□ 利用規約・プライバシーポリシーが整備されている
□ データ取扱合意（DPA）が締結されている（外部AI使用時）
```

これらをPoC期間中に**並行して埋めていく**。PoCが終わった時点で、これらが揃っていれば本番移行は最短で進む。

## インフラ構成の選定

AIエージェントを本番運用する際のインフラ構成は、大きく3パターンある。

### パターン1: フルクラウド

**構成**: AWS / Azure / GCP上に全コンポーネントをデプロイ。AI APIはAnthropic等の外部サービス。

**メリット**:
- スケーラビリティが高い
- 構築コスト・期間が短い
- 運用保守が比較的軽量

**デメリット**:
- 業務データを外部に送信することへの懸念
- 月額固定費がかかる

**適用業種**: SaaS企業、ITサービス、コンプライアンス要件が緩い業界

### パターン2: ハイブリッド

**構成**: 業務システムはオンプレ、AIゲートウェイ層をクラウドに配置。データは暗号化してやり取り。

**メリット**:
- 既存システムを変えなくて良い
- 段階的にクラウド移行できる

**デメリット**:
- ネットワーク設計が複雑
- 障害ポイントが増える

**適用業種**: 製造業、中堅企業

### パターン3: フルオンプレ（プライベートLLM）

**構成**: AI推論サーバーも自社内（GPU搭載サーバー）に設置。

**メリット**:
- データを外部に出さない
- 規制要件をクリアしやすい

**デメリット**:
- 初期投資が大きい（GPU・サーバー）
- 運用専門知識が必要
- 最新モデルへの追従が難しい

**適用業種**: 金融機関、医療、防衛、政府機関

### 選定フローチャート

```
[データを社外に出して良いか？]
   ↓ Yes
   [フルクラウド] ← SaaS企業、IT企業
   ↓ No
   [既存システムをクラウド化したいか？]
       ↓ Yes
       [ハイブリッド + データ匿名化]
       ↓ No
       [規制要件が厳しいか？]
           ↓ Yes
           [フルオンプレ]
           ↓ No
           [ハイブリッド]
```

選定時の判断材料:
- データ機密度（個人情報・財務情報・営業秘密）
- 業界規制
- 初期投資の予算
- 運用体制（インフラ専門家の有無）
- スケーラビリティ要求

## 監視・アラート・SLO設計

本番運用には**監視**が必須だ。「**動いているか分からない**」状態では、問題発見が遅れる。

### 監視すべき4つの観点

#### 観点1: 可用性（Availability）

「**システムが動いているか**」を計測する。

- エンドポイントの応答状況（HTTP 200の比率）
- 依存システム（CRM・DB・AI API）の応答状況

**SLO例**: 月間稼働率99.5%以上

#### 観点2: パフォーマンス（Performance）

「**速く動いているか**」を計測する。

- 平均レスポンス時間
- パーセンタイル（95パーセンタイル、99パーセンタイル）
- スループット（秒間処理件数）

**SLO例**: 95パーセンタイルで3秒以内

#### 観点3: 品質（Quality）

「**正しく動いているか**」を計測する。

- AI精度（人レビュー時の修正率）
- 確信度の分布
- フォールバック発生率

**SLO例**: AI出力の人修正率20%以下

#### 観点4: コスト（Cost）

「**予算内で動いているか**」を計測する。

- API呼び出し回数・トークン消費
- インフラコスト
- 1処理あたりのコスト

**SLO例**: 月額コスト上限50,000円

### アラート設計

すべてを通知するとアラート疲れが起きる。**Severityレベル**で分けて設計する。

| レベル | 通知方法 | 例 |
|--------|---------|-----|
| Critical | 即時電話/PagerDuty | サービス完全停止、データ破損 |
| High | 5分以内Slack/Teams | エラー率5%超、レイテンシ大幅悪化 |
| Medium | 30分以内Slack | エラー率1%超、コスト見込み超過 |
| Low | 翌営業日メール | 利用パターン変化、月次レポート |

### Runbook（障害対応手順書）の作成

各アラートに対応した**Runbook**を作る。これがあると、夜間・休日でも初動対応ができる。

```markdown
# Runbook: API応答エラー率が5%を超えた場合

## 一次切り分け
1. ダッシュボードでエラー種別を確認
   - 4xx系: 入力データの問題か、認証問題
   - 5xx系: サーバー側問題
   - タイムアウト: 依存サービスの問題

## 対応手順
### 4xx系の場合
1. 直近10件のエラーログを確認
2. 認証エラーなら、APIキーの有効期限を確認
3. 入力データエラーなら、現場に確認

### 5xx系の場合
1. インフラ監視ダッシュボードを確認
2. 影響範囲を特定
3. ロールバック判断（直近30分以内のデプロイがあれば検討）

### タイムアウトの場合
1. AI API側のステータスページを確認
2. 自社ネットワーク状況を確認
3. フォールバック動作（人エスカレーション）が動いているか確認

## エスカレーション
- 30分以内に解決しない場合: [担当者A]
- 1時間以内に解決しない場合: [担当者B]
- ビジネス影響が出た場合: [部長]
```

## 段階リリース戦略

PoCから本番への移行は**一気にやらない**。段階的にリスクを下げながら進める。

### 戦略1: シャドウモード

**概要**: 本番データを使ってAIに処理させるが、結果は使わず人の判断を採用。AIの出力と人の判断を比較してログだけ取る。

**期間**: 2-4週間

**メリット**:
- 本番データでの精度測定が可能
- 業務に影響を与えない
- 本番システムとの統合動作確認

**実装例**:

```python
def shadow_mode_handler(email_body, human_classification):
    """シャドウモード: AIの結果はログするだけ"""
    ai_result = classify_email(email_body)
    
    # 比較ログを残す
    log_comparison({
        "email_id": email_body["id"],
        "ai_category": ai_result["category"],
        "ai_confidence": ai_result["confidence"],
        "human_category": human_classification,
        "match": ai_result["category"] == human_classification
    })
    
    # 業務には人の判断を返す
    return human_classification
```

### 戦略2: カナリアリリース

**概要**: 全体の5%だけAI処理にして、残り95%は従来通り。問題なければ徐々に割合を増やす。

**期間**: 5% → 10% → 25% → 50% → 100% を1-2週間ずつ

**メリット**:
- 問題発生時の影響範囲が小さい
- 利用者が段階的に慣れる

### 戦略3: ブルー/グリーンデプロイ

**概要**: 本番環境を2系統用意し、片方で運用、もう片方で次バージョンを準備。問題なければ切り替え。

**用途**: AIモデル・プロンプトの更新時

**メリット**:
- 切り戻しが瞬時にできる
- ダウンタイムが短い

### 戦略の組み合わせ

実プロジェクトでは、これらを組み合わせる。

```
Week 1-2: シャドウモード（精度確認）
Week 3:   カナリア5%（影響少ない範囲）
Week 4:   カナリア25%（拡大）
Week 5:   カナリア50%
Week 6:   100%（本番完全移行）
```

このスケジュールで進めると、6週間かけて段階的にリスクを下げながら移行できる。

## セキュリティレビューの進め方

セキュリティレビューでブロックされて止まる案件が多いため、**早めに**情シス・法務を巻き込む。

### セキュリティレビューの主な観点

#### 観点1: データ取扱

- 個人情報を外部AI APIに送信するか
- 送信する場合、マスキング・匿名化はどう行うか
- データの保存場所・期間
- データ削除のフロー

#### 観点2: アクセス制御

- 誰がAIを利用できるか
- どの業務システムにアクセスできるか
- 権限変更のフロー

#### 観点3: 監査

- AIの操作ログが取られているか
- ログの保存期間（最低90日推奨）
- 不正利用の検知メカニズム

#### 観点4: 障害対応

- AI APIが停止した場合の業務影響
- 復旧手順
- 連絡体制

### セキュリティレビュー資料テンプレート

```markdown
# セキュリティレビュー資料 - [プロジェクト名]

## 1. システム概要
- 用途:
- 利用者数:
- 取り扱うデータ:

## 2. データフロー図
[業務システム → AIエージェント → AI API → 結果出力]

## 3. データ取扱
- 個人情報の有無: 
- 個人情報の処理方法:
- 送信先（AI API ベンダー）:
- データ保存期間:

## 4. AIベンダーの認証取得状況
- SOC 2 Type II:
- ISO 27001:
- 各国規制対応:

## 5. アクセス制御
- 認証方式:
- 権限管理:
- 監査ログ:

## 6. 障害対応
- フォールバック動作:
- 復旧目標時間（RTO）:
- データ復旧目標点（RPO）:

|-------|-----|---------|------|

- ポリシーA: 適合
- ポリシーB: 適合（条件: ...）
```

### レビュー指摘への対応

典型的な指摘と対応パターン:

| 指摘 | 対応 |
|------|------|
| 個人情報を外部に送るのは不安 | マスキング処理、または匿名化処理を入れる |
| AI出力の責任所在が不明確 | 「AI出力は参考情報、最終判断は人」と明文化 |
| ログ保存先が不明 | 自社管理のログサーバーまたはクラウドDB |
| ベンダー側の認証取得状況が不明 | ベンダーの認証取得書類を取得・提示 |
| 障害時の業務影響が大きい | フォールバック経路の動作テスト結果を提示 |

## コンプライアンスチェック

業界別の規制対応も、本番移行前に必ず確認する。

### 金融業界

- **金商法**: 顧客情報の取扱、商品説明の責任
- **マネーロンダリング対策**: KYC、不審取引検知
- **AIモデルガバナンス**: 金融庁の指針（モデルリスク管理）

### 医療業界

- **個人情報保護法（医療版）**: 要配慮個人情報の取扱
- **医療機器規制**: 診断補助は医療機器扱いになる場合あり
- **3省2ガイドライン**: 医療情報システムのセキュリティ要件

### 製造業

- **品質管理規格**: ISO 9001 / IATF 16949
- **トレーサビリティ**: 製品リコール時の遡及調査
- **PL法**: AI判定ミスによる製品事故時の責任

### IT・SaaS

- **個人情報保護法**: 一般要件
- **GDPR等の国際規制**: グローバル展開時
- **業界自主規制**: SaaS連合のガイドライン等

## 本番移行完了の判断基準（チェックリスト）

```
□ 上記の機能要件・非機能要件チェックリストが満たされている
□ セキュリティレビューが完了している
□ コンプライアンスチェックが完了している
□ 監視・アラート・Runbookが整備されている
□ シャドウモード・カナリアでの段階リリースが完了している
□ 業務担当者の利用マニュアルが完成している
□ 一次対応・エスカレーション体制が決まっている
□ 障害復旧テスト（実際に止めて復旧確認）が完了している
□ コスト試算が確定し、予算承認が取れている
□ 経営層・現場・情シスの全員が承認している
```

10項目すべてのチェックが入ってから、100%本番運用にする。

## ケーススタディ: 金融機関の段階移行

筆者が支援した、ある地域金融機関の顧客対応AIプロジェクトの段階移行を紹介する。

### プロジェクト概要

- 業務: 顧客からの電話・メール対応
- 月間処理件数: 約8,000件
- 利用部署: コールセンター30名
- 制約: 個人情報を外部に出さない（オンプレLLM選択）

### 移行スケジュール

| 期間 | フェーズ | 内容 |
|------|--------|------|
| 1ヶ月目 | PoC | 50件のテストデータで精度検証 |
| 2ヶ月目 | シャドウモード | 過去1ヶ月のメールで精度測定（人の判断と比較） |
| 3ヶ月目 | カナリア5% | 影響の少ない問い合わせ種別から開始 |
| 4ヶ月目 | カナリア25% | 種別を拡大 |
| 5ヶ月目 | カナリア50% | コールセンターの半分が利用 |
| 6ヶ月目 | 100% | 全件利用 |

### 各フェーズで得られた知見

- **シャドウモード**: AIと人の判断が一致したのは78%。20-30%の差異があるカテゴリを特定し、プロンプトを修正。
- **カナリア5%**: 初期は1日2-3件の誤回答が発生。フォールバック経路が機能した。
- **カナリア25%**: コスト見積もりを修正（PoC時の試算より15%増）。
- **カナリア50%**: 現場の声を反映し、UIを改修（ボタン配置の変更等）。
- **100%**: 月間処理時間が30%削減。コールセンター残業がほぼゼロに。

このような**段階的な移行**により、リスクを下げながら本番運用に到達できた。

## まとめ

- PoCと本番運用には5つのギャップ（データ・利用パターン・連携・セキュリティ・運用）がある
- PoC開始時に「**本番移行チェックリスト**」を作り、PoC期間中に並行で埋めていく
- インフラは業界・データ機密度・予算で3パターンから選ぶ
- 監視は可用性・パフォーマンス・品質・コストの4観点で設計
- 本番移行は**シャドウモード → カナリア → 100%**の段階移行が原則
- セキュリティレビューは早めに情シス・法務を巻き込む

## 次章への導線

第7章では、本番移行の関門である**UAT（User Acceptance Test: 利用者受入テスト）**の設計と実施を扱う。

UATは「**現場が本当に使えるか**」を確認する最後の関門。テストケース設計、現場の巻き込み方、結果の評価、追加改修の判断など、UATの実務を具体的に解説する。

技術的に動くことと、現場が使えることは別問題だ。UATを軽視すると、せっかく作ったAIが「使われない遺産」になる。
