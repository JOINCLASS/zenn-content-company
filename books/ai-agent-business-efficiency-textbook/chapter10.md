# プロジェクト管理・品質管理フレームワーク

## この章で学ぶこと

- AI業務効率化プロジェクト特有のプロジェクト管理（PM）の難しさと対処法
- スケジュール・スコープ・コスト・リスク・品質・コミュニケーションの6軸統合管理
- プロジェクト推進者が手元に持っておくべき5つのドキュメント
- 「**仮説検証型プロジェクト**」のスケジューリング技法
- 品質管理（QA）の3層構造（技術品質・業務品質・運用品質）
- 本書全体の総まとめと、プロジェクト着手のための行動チェックリスト

## はじめに -- AIプロジェクト管理の特殊性

最終章では、本書全体を貫いてきた**プロジェクト管理・品質管理**を統合的に整理する。

第1-9章では、業務理解からPoC、UAT、内製化までの各フェーズを深掘りしてきた。各フェーズは独立しているのではなく、**プロジェクト管理・品質管理**という横串によって結びついている。この横串の質が、プロジェクト全体の成否を決める。

AI業務効率化プロジェクトのPMは、従来のシステム開発PMと以下の点で本質的に異なる。

| 観点 | 従来PM | AIプロジェクトPM |
|------|------|----------------|
| 要件 | 上流で固定 | **継続的に変化**（PoC結果で見直し） |
| スケジュール | ガントチャート的 | **マイルストーンベース** |
| 成果物 | 明確に定義可能 | 確率的な性質を含む |
| 品質基準 | パスかフェイル | 多次元評価 |
| リスク | 主に技術リスク | **業務適合性・現場受入性が大** |
| ステークホルダー | システム部門中心 | **業務・経営・現場・情シス**の全方位 |

これらの違いを理解しないままウォーターフォール的にPMすると、ほぼ確実に失敗する。

:::message
**筆者の実践メモ**
筆者が観察した範囲で、AIプロジェクトの最大の失敗要因は「**PMが従来のPM経験で運用しようとした**」ことだ。「要件定義書を作って承認を取って、その後変更させない」というスタイルは、AIプロジェクトでは破綻する。本章のフレームワークは、この失敗パターンを回避するためのものだ。
:::

## プロジェクト管理の6軸統合

PMBOK等の伝統的PM論では、プロジェクトを多軸で管理する。AIプロジェクトでは、それぞれの軸に特有のポイントがある。

### 軸1: スケジュール管理

#### 特徴
- フェーズ間の依存が**緩やか**（PoCの結果次第で前後する）
- 不確実性が高いため、**バッファ**を必須にする
- マイルストーンベースで管理する

#### 推奨スケジューリング手法

**「不確実性錐」**を意識する。

```
[Phase 1: 業務理解] -- 不確実性: 大
   見積精度: ±100%（1週間〜1ヶ月）
   
[Phase 2-3: As-Is/To-Be設計] -- 不確実性: 中
   見積精度: ±50%（2-4週間）
   
[Phase 4: PoC開発] -- 不確実性: 中
   見積精度: ±30%（3-5週間）
   
[Phase 5: UAT・本番移行] -- 不確実性: 小
   見積精度: ±20%（3-4週間）
   
[Phase 6: 内製化] -- 不確実性: 小
   見積精度: ±20%（3-4週間）
```

上流ほど不確実性が大きい。最初から全フェーズの正確なスケジュールを引くのではなく、**直近フェーズだけ詳細に、後フェーズは粗く**引く。

#### マイルストーンベースのスケジュール例

```markdown
# プロジェクトスケジュール

## マイルストーン

| MS | 内容 | 目安日 | 必達条件 |
|----|------|-------|--------|
| MS1 | 業務理解完了 | 2026-07-15 | ヒアリング議事録、業務リスト確定 |
| MS2 | As-Is/To-Be合意 | 2026-08-15 | フロー図、AI適用判断、ROI試算承認 |
| MS3 | PoC GO判断 | 2026-08-20 | 経営層承認、予算確保 |
| MS4 | PoC完了 | 2026-09-30 | 動作プロトタイプ、精度80%達成 |
| MS5 | UAT合格 | 2026-10-31 | 本番GO判断 |
| MS6 | 本番リリース | 2026-11-30 | 段階リリース完了 |
| MS7 | 内製化卒業 | 2027-01-31 | 卒業判定チェックリスト合格 |

## 進捗ステータス
- MS1: 完了 (2026-07-12)
- MS2: 進行中（80%）
- MS3-7: 未着手
```

### 軸2: スコープ管理

#### 特徴
- スコープ追加要求が頻発（「**ついでにあれもできない？**」）
- AIで何ができるか分からず、スコープが拡大しがち
- 「**やらない**」判断が極めて重要

#### スコープ管理の3原則

**原則1: スコープ外を明文化**

「やる」リストと同じ重さで「**やらない**」リストを作る。

```markdown
# プロジェクトスコープ

## スコープ内（やること）
- 顧客問い合わせメールの分類
- 返信下書きの自動生成
- CRM自動記録

## スコープ外（このプロジェクトではやらない）
- 電話応対のAI化（次フェーズ検討）
- チャットボット（別プロジェクト）
- 営業活動支援（要望は受けるが今期はやらない）
- 海外言語対応（来期検討）
```

**原則2: 変更管理プロセスを定義**

新規要求が出たら、必ず以下のプロセスを通す。

```
[新規要求発生]
   ↓
[影響評価] - スケジュール・コスト・リスク
   ↓
[判断] - GO / 後回し / 別プロジェクト化 / 却下
   ↓
[文書化] - 決定の根拠を記録
```

**原則3: 段階拡張を前提に設計する**

最初から全部やろうとせず、**v1で動かし、v2で拡張**することを最初から組み込む。

### 軸3: コスト管理

#### 特徴
- AI APIコストが**変動費**（利用量で変わる）
- 隠れコスト（運用工数、教育費）が大きい
- ROI実測まで真の評価ができない

#### コスト構造の見える化

```markdown
# プロジェクトコスト構造

## 初期コスト（一括）
- コンサル費: 500万円
- システム構築: 200万円
- 教育費: 50万円
- 計: 750万円

## 運用コスト（月額）
- AI API利用料: 30,000円（変動）
- インフラ: 20,000円
- 保守ベンダー: 50,000円
- 内製化担当者人件費: 80,000円（業務時間20%換算）
- 計: 180,000円/月、年間216万円

## 改善コスト（年間）
- プロンプト改善: 50万円
- 新機能追加: 100万円
- 計: 150万円

## 総保有コスト（年間）
- 1年目: 750 + 216 + 150 = 1,116万円
- 2年目以降: 366万円/年
```

このように**5年スパン**で見ると、真のコスト構造が見える。

#### コスト変動の管理

AI API利用料は使うほど増える。**月次で見ながら制御**する。

```python
# コスト監視の例
class CostMonitor:
    def __init__(self, monthly_budget_jpy: int):
        self.budget = monthly_budget_jpy
        self.warning_threshold = 0.8  # 80%でアラート
        self.stop_threshold = 1.0     # 100%で停止
    
    def check(self, current_spend: int) -> str:
        ratio = current_spend / self.budget
        if ratio >= self.stop_threshold:
            return "STOP"  # 利用を停止
        elif ratio >= self.warning_threshold:
            return "WARN"  # アラート発出
        return "OK"
```

### 軸5: 品質管理（QA）

AIプロジェクトの品質は**3層構造**で管理する。

#### 層1: 技術品質

- コード品質（テスト網羅率、コードレビュー）
- セキュリティ品質（脆弱性スキャン、ペネトレーションテスト）
- パフォーマンス品質（負荷試験、レイテンシ）

#### 層2: 業務品質

- AI出力の精度
- 業務ルールへの適合性
- エッジケースへの対応
- 業務フローへの統合度

#### 層3: 運用品質

- 可用性（稼働率）
- 保守性（変更のしやすさ）
- 監視性（問題の発見しやすさ）
- 教育性（新メンバーが習得しやすいか）

各フェーズの完了時にQAチェックを実施する。

## 仮説検証型プロジェクトのスケジューリング

AI業務効率化プロジェクトは「**仮説検証型**」である。これに合わせたスケジューリング技法を解説する。

### 技法1: タイムボックスを切る

不確実性が高い作業は、**「いつまでに何をやる」を期限で区切る**。

例: 「**PoC開発は4週間。完璧でなくても4週間で評価する**」

期限を切らないと、開発が延々と続いて意思決定が遅れる。

### 技法2: ステージゲート

フェーズ間に**判断ポイント**を設ける。

```
[Phase 1] → [Gate 1: GO/NO-GO判断] → [Phase 2] → [Gate 2] → ...
```

各ゲートで、続けるか・撤退するか・スコープ変更するかを判断する。

### 技法3: 学習スプリント

1-2週間ごとに「**何を学んだか**」を振り返り、計画を調整する。

```markdown
# スプリント振り返り

## 学び
- AI精度がドメイン別に大きく異なる（特に「クレーム」が低い）
- 既存システムのAPI制限が想定以上に厳しい
- 現場の習熟スピードが速い

## 計画への反映
- クレームカテゴリにより多くの工数を割く
- 既存システム連携を簡略化する
- OJT期間を4週間→3週間に短縮可能
```

### 技法4: バッファの確保

不確実性に備えて、各フェーズに**20-30%のバッファ**を組み込む。

```
[計画] PoC開発: 4週間
[実際の計画] PoC開発: 4週間 + バッファ1週間 = 5週間で計画
```

バッファを最初から見せておくと、現場が「**バッファ使わないと損**」と感じて使い切ってしまう。なので、バッファは**PM個人で管理**する。

## 品質管理（QA）の3層構造

### 層1: 技術品質のチェック

#### コード品質

```
□ コードレビューが全コミットで実施されている
□ テスト網羅率（重要関数）が80%以上
□ 静的解析（ESLint、Pylint等）でエラーゼロ
□ 主要依存ライブラリのバージョンが固定されている
□ 機密情報がコードにハードコードされていない
```

#### セキュリティ品質

```
□ 認証・認可が実装されている
□ 通信暗号化（HTTPS）
□ 個人情報マスキングが実装されている
□ 監査ログが保存される
□ 脆弱性スキャンを実施した
□ ペネトレーションテストを実施した（重要システム）
```

#### パフォーマンス品質

```
□ 想定ピーク時の負荷試験を実施した
□ レスポンス時間SLOを満たす
□ 同時接続数の上限を確認した
□ メモリリーク・タイムアウトの問題がない
```

### 層2: 業務品質のチェック

#### AI出力品質

```
□ テストケース全体での精度が目標値以上
□ カテゴリ別精度に大きな偏りがない
□ 確信度の分布が妥当
□ ハルシネーションが検出されない
□ エッジケースへの対応が確認できている
```

#### 業務適合品質

```
□ 業務ルールへの適合性が業務オーナー承認済み
□ 業務フローへの統合がスムーズ
□ 既存業務システムとの連携が安定
□ 例外ケースへの対応が確認できている
```

### 層3: 運用品質のチェック

#### 可用性

```
□ 月間稼働率が99.5%以上
□ 主要障害シナリオへの対応が確認できている
```

## 本書のまとめ

本書では、AIエージェント業務効率化プロジェクトを推進するための一連のプロセスを、10章にわたって解説してきた。

### 各章の要点

- **第1章**: プロジェクトの全体像、失敗を避ける5原則
- **第2章**: 信頼構築型ヒアリング、質問テンプレート30問
- **第3章**: As-Is可視化、4点セット（業務フロー/課題マップ/データフロー/ステークホルダー）
- **第4章**: AI適用判断フレームワーク、4類型（全自動/半自動/アシスト/並列）、ROI3要素
- **第5章**: Claude Agent SDK + MCP、3層構造（業務知識/ツール/実行制御）、本番移行5要素
- **第6章**: PoCから本番への5ギャップ、段階リリース戦略、セキュリティレビュー
- **第7章**: UATは合意形成、業務シナリオベース、3軸評価（精度/速度/満足度）
- **第8章**: KPI 3層（システム/業務/ビジネス）、月次レビュー、PDCAサイクル
- **第9章**: 内製化はPhase 1から、3類型混成チーム、ドキュメント10種類、OJT 4週間
- **第10章**: PM 6軸統合管理、QA 3層、仮説検証型スケジューリング

### 本書を貫く5つの原則

1. **「業務側を変える覚悟」**を最初に持つ
2. **「半自動」**から始める
3. **「現場を巻き込む」**を最優先する
4. **「PoCで止まらない設計」**を最初から組み込む
5. **「内製化」**をプロジェクト初期から並行する

これらの原則を貫けば、「**作って終わり**」ではなく「**継続的に価値を生む**」AI業務効率化プロジェクトが実現できる。
