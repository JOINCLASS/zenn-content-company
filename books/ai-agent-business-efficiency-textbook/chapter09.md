# 内製化支援

## この章で学ぶこと

- 内製化を「**プロジェクト終了時のイベント**」ではなく「**Phase 1から並行する活動**」として捉える本質理解
- 内製化が失敗する5つの典型パターンと、その回避策
- 内製化担当者の育成パス（業務理解者・プロジェクトリーダー・技術者の3類型）
- 知識移転のためのドキュメント体系（10種類の必須ドキュメント）
- OJTの設計（4週間プログラム）
- 「卒業」を判定するためのチェックリスト
- 内製化後の継続改善体制

## はじめに -- なぜ内製化は失敗するのか

AIプロジェクトの最大の失敗パターンは「**コンサルが抜けたら誰も触れなくなった**」だ。

「プロジェクト中はうまく動いていた」「コンサルが優秀で頼りになった」と評価される一方、半年〜1年後には:

- AI出力の精度が落ちても直せない
- 業務の変化にAIが追従できない
- API障害が起きても原因が分からない
- 結局、別のコンサルに頼んで改修する

こうなると、年間ROIどころか保守コストの肥大化でマイナスに転じる。AIプロジェクトが「**続かない投資**」と認識されてしまう。

なぜ内製化はこれほど難しいのか。理由は4つある。

1. **「内製化」をプロジェクト最終フェーズだけの作業と捉えている**
2. **内製化担当者が早期に確定されない**
3. **ドキュメントが「コンサルが作ったもの」のままで、現場の言葉に翻訳されていない**
4. **OJT期間が短すぎる（通常1-2週間しか取らない）**

本章では、これらの落とし穴を回避し、**確実に内製化を成功させる手法**を解説する。

:::message
**筆者の実践メモ**
筆者が支援した案件で内製化が最も成功したのは、**Phase 1（ヒアリング）の段階から内製化担当者を巻き込んでいた案件**だ。プロジェクトの全工程に内製化担当者が参加することで、プロジェクト終了時には自然と「中身を理解した人」が育つ。後付けの内製化研修では、絶対にこの状態は作れない。
:::

## 内製化の3つのレベル

「内製化」と一括りに言っても、レベルが3段階ある。プロジェクトのゴールをどこに置くかで、必要な準備が大きく変わる。

### レベル1: 運用内製化

**範囲**: 日々の運用（監視、利用者問い合わせ対応、軽微な設定変更）を自社で行える状態。

**必要なスキル**:
- 監視ダッシュボードの読み方
- 障害時の一次切り分け
- ベンダーへのエスカレーション判断

**育成期間**: 1-2ヶ月

### レベル2: 改善内製化

**範囲**: AI精度の改善、プロンプト調整、新シナリオへの拡張を自社で行える状態。

**必要なスキル**:
- プロンプトエンジニアリング
- 業務ルールの定義
- テスト・評価の実施
- 簡単なコード修正

**育成期間**: 3-6ヶ月

### レベル3: 新規開発内製化

**範囲**: 新しいAIエージェント開発を、ベンダー支援なしで実施できる状態。

**必要なスキル**:
- システム設計
- Claude Agent SDK / MCPの実装
- インフラ構築・運用
- セキュリティレビュー

**育成期間**: 1年以上

多くの中小企業では**レベル2まで**を目標にするのが現実的。レベル3を狙うには、IT部門の本格的な体制構築が必要になる。

## 内製化が失敗する5つの典型パターン

### パターン1: 内製化担当者が決まらない

**症状**: プロジェクト終盤になっても「**この後誰が運用するのか**」が決まっていない。

**原因**: 業務部門が「IT部門が見るだろう」、IT部門が「業務部門が見るだろう」と互いに押し付け合っている。

**対策**: **Phase 1の時点で内製化担当者を明示的に指名**する。経営層の了承を取り、辞令で確定する。

### パターン2: 担当者の業務時間が確保されない

**症状**: 内製化担当者は決まったが、本業が忙しくてプロジェクトに時間を割けない。

**原因**: 「兼任で何とかなる」という誤算。実際にはAI運用には最低でも業務時間の20-30%が必要。

**対策**: **業務時間の何%をAI運用に振り向けるか**を最初に決め、上司を巻き込んで業務調整する。

### パターン3: ドキュメントが「コンサル目線」のまま

**症状**: コンサルが作った技術ドキュメントは網羅的だが、現場が読んでも意味が分からない。

**原因**: ドキュメント作成時に**読み手を意識していない**。

**対策**: ドキュメントを**読み手別に分けて作る**（後述）。

### パターン4: OJTが短すぎる

**症状**: 「**3日間の引き継ぎ**」で内製化完了とみなす。

**原因**: 予算・スケジュールの都合で、内製化期間を圧縮する。

**対策**: **最低4週間**のOJT期間を確保する。

### パターン5: 「**何かあったら聞いてくださいね**」で終わる

**症状**: コンサルが抜けた後、「**質問はメールで**」と言って実質サポート無し。

**原因**: コンサル契約の終了で関係が切れる。

**対策**: **卒業後3-6ヶ月の**フォローアップ契約を結ぶ。月数時間で良いので、相談できる体制を残す。

## 内製化担当者の育成パス

### 内製化担当者の3類型

内製化担当者は、以下の3つのタイプから選定する。それぞれ得意領域が違う。

#### 類型A: 業務理解者

**プロフィール**: 対象業務を5年以上経験。業務の隅々まで知っている。IT知識は基礎レベル。

**強み**:
- 業務ルールの判断ができる
- 現場との関係性が良い
- AIへの指示（プロンプト）を業務目線で書ける

**弱み**:
- 技術的な詳細理解は時間がかかる
- システムトラブル時に対応しにくい

**推奨業務**: プロンプト調整、業務ルール更新、現場サポート

#### 類型B: プロジェクトリーダー

**プロフィール**: プロジェクトマネジメント経験者。業務とITの両方をある程度知っている。

**強み**:
- 関係者調整ができる
- ベンダー管理ができる
- 全体最適の視点を持っている

**弱み**:
- 個別の深い専門性は他類型に劣る
- 全部自分でやろうとすると破綻する

**推奨業務**: 内製化チームのリーダー、ベンダー折衝、KPI管理

#### 類型C: 技術者

**プロフィール**: IT部門所属。プログラミングまたはシステム運用の経験あり。

**強み**:
- システム設計・実装ができる
- 障害対応ができる
- 新規拡張を担当できる

**弱み**:
- 業務理解が浅いことが多い
- 現場との距離が遠い

**推奨業務**: コード修正、インフラ運用、新規開発

### 推奨チーム構成

最も成功率が高いのは、**3類型の混成チーム**だ。

```
[内製化チーム（推奨3-4名）]
- リーダー: 類型B（プロジェクトリーダー）× 1名
- 業務理解者: 類型A × 1-2名
- 技術者: 類型C × 1名
```

このチームをPhase 1の段階で組成し、プロジェクトの全工程に参加してもらう。

### 育成スケジュール

| 期間 | フェーズ | 育成内容 |
|------|--------|--------|
| Phase 1-3（業務理解〜To-Be設計） | プロジェクト参加 | 業務ヒアリング・設計に同席。理解を深める |
| Phase 4（PoC開発） | 技術理解 | コードレビュー、プロンプト調整に関与 |
| Phase 5（UAT） | 運用準備 | 監視ダッシュボード操作、Runbook理解 |
| Phase 6（内製化） | 自走 | OJT・ハンズオン |
| 卒業後3ヶ月 | フォロー | 月数時間のメンタリング |

このスケジュールで進めると、卒業時には内製化チームが**完全に自走できる**状態になる。

## 知識移転のためのドキュメント体系

内製化のためのドキュメントは、**読み手別**に分けて整備する。

### 必須ドキュメント10種類

#### 1. 業務ルール集（読み手: 業務理解者）

業務上の判断ルールを自然言語で記述。プロンプトの元になる。

```markdown
# 業務ルール: 顧客問い合わせ分類

## カテゴリ分類ルール
- 新規見積依頼: 「料金」「見積もり」「価格」を含む & 既存顧客でない
- 既存契約確認: 既存顧客 & 「契約」「請求」「サポート」を含む
- クレーム: 「困っている」「不満」「対応してほしい」等の表現

## 優先度判定ルール
- 「至急」「緊急」を含む: 最優先
- 大口顧客（年間100万円以上）: 高
- それ以外: 通常

## エスカレーション基準
- 確信度0.7未満: 必ず人間レビュー
- 法的問題を含む: 法務部門に連携
```

#### 2. プロンプト集（読み手: プロンプト調整担当）

AIエージェントに渡すプロンプトの一覧と変更履歴。

```markdown
# プロンプト集

## 1. 分類プロンプト（v1.3 / 2026-06-11）
[現在のプロンプト本文]

## 変更履歴
- v1.3 (2026-06-11): クレーム判定基準を追加
- v1.2 (2026-05-15): 優先度判定を改善
- v1.1 (2026-04-20): カテゴリの定義を明確化
- v1.0 (2026-04-01): 初版
```

#### 3. 運用手順書（読み手: 運用担当）

日々の運用業務の手順。

```markdown
# 運用手順書

## 1. 朝の確認業務（5分）
1. 監視ダッシュボードを開く
2. 前日エラー件数を確認
3. 5件以上ならエスカレーション

## 2. 週次レビュー（30分・毎週月曜）
1. 週次レポートを開く
2. KPI推移を確認
3. 異常値があれば原因調査
4. 改善項目をTODOに追加

## 3. 月次レポート作成（2時間・月初）
[詳細手順]
```

#### 4. Runbook（障害対応手順書）（読み手: 一次対応者）

障害時の対応手順。第6章で扱った内容を本書の中で整理。

#### 5. システム構成図（読み手: 技術者）

システム全体のアーキテクチャ図。

#### 6. データフロー図（読み手: 技術者・業務理解者）

データの流れを可視化した図。

#### 7. APIリファレンス（読み手: 技術者）

MCPサーバー・連携先APIの仕様。

#### 8. テストケース集（読み手: 改善担当）

UATで使ったテストケース + 継続テスト用のケース。

#### 9. KPIダッシュボード操作手順（読み手: 全員）

ダッシュボードの見方・操作方法。

#### 10. FAQ（読み手: 全員）

よくある質問と答え。

```markdown
# FAQ

## Q: AIの精度が下がった気がする
A: ダッシュボードの「精度トレンド」を確認。1ヶ月で5%以上低下なら原因調査が必要。

## Q: 利用者から「使い方が分からない」と言われた
A: 利用者マニュアル（リンク）を案内。それでも解決しない場合は、画面録画で操作確認。

## Q: ベンダーAPIが応答しない
A: ベンダーステータスページを確認 → ベンダーに問い合わせ → フォールバック動作確認。
```

### ドキュメント整備のコツ

#### コツ1: 「現場の言葉」で書く

「**Tokenizer**」「**Inference**」のような技術用語は、業務担当者には伝わらない。「**文章を細切れにする処理**」「**AIに答えを聞く処理**」と書き換える。

#### コツ2: 操作スクリーンショットを多用する

文字だけのドキュメントは読まれない。各操作にスクリーンショットを添付する。

#### コツ3: バージョン管理する

GitHubやNotionの履歴機能で、変更が追えるようにする。

#### コツ4: 「これだけ読めば最低限OK」を1ドキュメントに集約する

10種類すべて読むのは現実的でない。「**まずこれを読んでください**」という1ドキュメントを作る。

### 「最低限ドキュメント」のテンプレート

```markdown
# AIエージェント運用 - 最低限の知識

## 1. このシステムは何をするか（5行）
[業務内容を簡潔に]

## 2. 毎日やること（3つ）
1. ダッシュボードを朝確認
2. アラートが来たら対応
3. 利用者からの問い合わせに答える

## 3. 困った時に見るところ
- 操作の疑問 → 利用者マニュアル
- 障害の疑問 → Runbook
- 精度の疑問 → プロンプト集

## 4. エスカレーション先
- 業務の判断 → [業務オーナー]
- 技術トラブル → [技術リード]
- ベンダー対応 → [契約管理者]

## 5. 詳しく知りたい人へ
- 業務ルールを変えたい → 業務ルール集
- AI動作を改善したい → プロンプト集
- 新機能を追加したい → システム構成図
```

A4 1-2枚に収まる。新任の担当者は、まずこれを読む。

## OJT 4週間プログラム

内製化のOJTは、**最低4週間**で設計する。

### Week 1: 業務理解・基礎操作

**目標**: システムの全体像を理解し、基本操作ができる。

**プログラム**:
- Day 1: プロジェクト全体説明（2時間）
- Day 2: 業務ルール集・プロンプト集の読み込み（4時間）
- Day 3: ダッシュボード操作（2時間）
- Day 4: テスト環境での操作（4時間）
- Day 5: 質疑応答・振り返り（2時間）

**到達基準**: 利用者マニュアルレベルの操作ができる。

### Week 2: 運用業務

**目標**: 日次・週次の運用業務を独力でできる。

**プログラム**:
- Day 1-2: 朝の確認業務をコンサルと並行実施
- Day 3-4: 障害対応のシミュレーション（Runbookに従って対応）
- Day 5: 週次レビュー会の運営

**到達基準**: 日常運用を独力で実施できる。

### Week 3: 改善業務

**目標**: 簡単な改善ができる。

**プログラム**:
- Day 1-2: プロンプトの修正実習（テスト環境）
- Day 3: テストケースの追加
- Day 4: KPIの集計・分析
- Day 5: 改善提案書の作成

**到達基準**: 軽微な改善を独力で実施できる。

### Week 4: 自走確認

**目標**: コンサルが見守る中、独力で運用する。

**プログラム**:
- Day 1-5: 全ての運用業務を独力で実施
- 毎日30分の振り返り
- 質問はSlack/Teamsで（即答ではなく、自分で考える時間を取る）

**到達基準**: コンサルなしで1週間運用できる。

### OJT後のフォロー

OJT完了後、3-6ヶ月のフォローアップ契約を結ぶ。

| 期間 | フォロー内容 |
|------|----------|
| 1-3ヶ月目 | 月2回の定例ミーティング（各1時間）+ Slack/Teams質問対応 |
| 4-6ヶ月目 | 月1回の定例 + メール対応 |
| それ以降 | 必要に応じてスポット相談 |

「**完全に切る**」のではなく、**段階的に距離を取る**のが成功のコツ。

## 「卒業」の判定基準

内製化が完了したと判定する基準を、明確に決めておく。

### 卒業判定チェックリスト

```
## 業務面
□ 業務ルール集が最新に保たれている
□ プロンプト変更の意思決定を内製で行えている
□ 月次KPIレビューを内製で実施している
□ 現場からの問い合わせに内製で対応できている

## システム面
□ 監視ダッシュボードを毎日確認している
□ 直近1ヶ月以内に1回以上の障害対応を内製で実施した
□ ベンダー連携の窓口担当が決まっている
□ コスト管理を内製で行っている

## ドキュメント面
□ 全10種類のドキュメントが整備されている
□ ドキュメントの更新が内製で行われている
□ FAQが最新に保たれている

## 組織面
□ 内製化チームが3-4名で組成されている
□ 各メンバーの業務時間が確保されている
□ 経営層から内製化チームへの権限委譲が完了している
□ 改善サイクル（PDCA）が回っている

## 自律性
□ 直近1ヶ月、コンサル介入なしで運用できている
□ 軽微な改善を独力で実施した実績がある
□ 新規シナリオへの拡張計画を内製で立てている
```

20項目中16項目以上にチェックが入れば、卒業と判定できる。

## 内製化後の継続改善体制

卒業後も、改善活動を継続するための体制を作る。

### 月次改善サイクル

```
[月初]
- 前月KPIレビュー
- 改善項目の優先順位付け

[月中]
- 改善実装（プロンプト調整、テストケース追加等）
- 効果検証

[月末]
- 効果測定結果のまとめ
- 経営層への報告
```

### 四半期改善サイクル

```
[Q初]
- 前期総括
- 今期の重点改善テーマ

[Q中]
- テーマに沿った大型改善（新カテゴリ追加、新業務への展開等）

[Q末]
- 効果測定
- 経営層への報告
- 来期計画策定
```

### 年次改善サイクル

```
[年初]
- 前年度総括
- 今年度の戦略（拡大 / 深化 / 撤退）

[年央]
- 中間レビュー

[年末]
- 全社的なROI評価
- 来年度予算策定
```

### コミュニティ・社外ネットワーク

内製化チームは「**社内孤立**」しがち。社外ネットワークが重要。

- 業界カンファレンス参加（年2-3回）
- 同業他社との情報交換
- Anthropic等ベンダー主催のイベント参加
- 関連書籍・記事の継続学習

筆者の支援案件では、これらを推奨し、必要な予算を確保するよう経営層に提案している。

## ケーススタディ: SIerの内製化支援

筆者が支援した、従業員300名のSIerの事例を紹介する。

### プロジェクト概要

- 業務: 社内ヘルプデスク（プログラム開発の質問対応）
- 月間問い合わせ数: 約500件
- 既存体制: ヘルプデスク担当2名（兼任）

### 内製化計画

- Phase 1から内製化担当者3名を指名
  - 業務理解者: ヘルプデスク担当 × 1名
  - リーダー: 情シス課長 × 1名
  - 技術者: ジュニアエンジニア × 1名
- 各メンバーの業務時間30%をプロジェクトに振り向け

### プロジェクト中の内製化活動

| Phase | 内製化チームの活動 |
|------|---------------|
| 1. 業務理解 | コンサルのヒアリングに全員同席 |
| 2. As-Is設計 | フロー図作成を一緒に実施 |
| 3. To-Be設計 | AI適用判断に参加、ROI計算を学ぶ |
| 4. PoC開発 | プロンプト設計に業務理解者が参加 |
| 5. UAT | 全員でテスト実施 |
| 6. 内製化 | OJT 4週間 |

### 卒業時の状態

- ドキュメント10種類を内製化チームが理解
- 月次KPIレビューを独力で実施
- ベンダーAPI障害を1回独力対応した実績
- プロンプト調整を3回独力実施した実績

### 卒業後の状況（6ヶ月）

- 月次KPIが安定。ROI 250%を維持
- 新カテゴリ（インフラ問い合わせ）への展開を独力で実施
- 他部署からの引き合いが3件発生
- 内製化チームが社内で評価され、メンバーが昇進

このように、**内製化担当者の早期指名と全工程への巻き込み**が、卒業後の成功を決める。

## まとめ

- 内製化は「**Phase 1から並行する活動**」。プロジェクト終了時の作業ではない
- 内製化担当者は3類型（業務理解者・リーダー・技術者）の混成チームが理想
- 内製化レベルは3段階（運用 / 改善 / 新規開発）。多くの中小企業はレベル2を目標に
- ドキュメントは10種類を**読み手別**に整備する
- OJTは**最低4週間**。卒業後3-6ヶ月のフォローアップで段階的に距離を取る
- 卒業判定は20項目のチェックリストで明文化
