# 現場ヒアリングと業務理解

## この章で学ぶこと

- ヒアリングを「情報収集」ではなく「**信頼構築と仮説検証**」の場として設計する考え方
- 1回90分のヒアリングを最大限に活かす質問テンプレート（30問構成）
- 現場の警戒心を解き、暗黙知を引き出す話法
- ヒアリング議事録を「次のフェーズに使える資産」に変えるドキュメント化手法
- ヒアリングで陥りがちな5つの罠と回避方法

## はじめに -- ヒアリングは「業務理解」より「関係構築」が先

AIプロジェクトのヒアリングで最も多い失敗は、初回から「業務をどう自動化するか」「どの工程に時間がかかっているか」と直球で聞いてしまうことだ。

これは現場の警戒心を最大限に煽る質問である。なぜなら、現場担当者にとってAI導入は「**自分の仕事が奪われるかもしれない**」「**今までのやり方を否定されるかもしれない**」というネガティブな出来事だからだ。

筆者が支援した案件で、最も成功率が高いヒアリングのパターンは、初回の半分以上を「業務の話**以外**」に充てているケースである。

> 「いつからこの業務を担当されていますか？」
> 「この業務の中で、ご自身が工夫されているところはどこですか？」
> 「これまでに改善活動はされてきましたか？」

このような質問で「あなたの仕事を理解したい」「あなたの工夫を尊重したい」という姿勢を示すと、現場は驚くほど協力的になる。**業務情報の質と量は、関係構築の質に比例する**。

:::message
**筆者の実践メモ**
筆者は初回ヒアリングの最初の30分を意図的に「雑談・経歴・業務歴」に充てている。一見遠回りに見えるが、結果としてこの30分が、その後の業務情報の引き出しやすさを2-3倍に高める。
:::

## ヒアリングの目的を再定義する

### よくある誤った目的設定

| 誤った目的 | 起きる失敗 |
|----------|----------|
| 業務の全工程を網羅的に把握する | ヒアリング時間が足りず、表面情報のみで終わる |
| 課題を一度に全部聞き出す | 「全部困っている」という答えしか出ない |
| 自動化候補を初回で特定する | 現場の信頼を失い、その後の情報が出てこない |

### 正しい目的設定（5段階）

ヒアリングの目的は、以下の5段階を順番に達成することだ。

1. **信頼を獲得する** — 「この人なら本音を話せる」
2. **業務の全体像を粗く掴む** — 細部は後回し
3. **痛みのある工程を特定する** — どこに人の負担が集中しているか
4. **暗黙知を引き出す** — マニュアルに書かれていない判断
5. **次のヒアリングへの宿題を作る** — 一度で完結させない

この5段階を意識すると、ヒアリングの設計が大きく変わる。

## ヒアリング前の準備

### 準備1: 業務領域の事前学習

ヒアリングの前に、対象業務の業界用語・標準的な業務フロー・関連法規を最低限学習する。

例えば、金融業の「顧客応対業務」をヒアリングするなら、以下を事前に把握しておく。

- 業界用語: 「DM」「TC」「コール」「アウトバウンド/インバウンド」等
- 標準業務フロー: 着電 → 本人確認 → 用件確認 → 対応 → 記録
- 関連法規: 金商法・個人情報保護法・会話録音規制

これを知らずにヒアリングすると、現場から「**素人に説明する時間がない**」と判断され、深い情報が出てこない。

### 準備2: 関係者マップの作成

ヒアリング対象者だけでなく、その周辺の関係者を事前に整理する。

```
[業務オーナー: 部長] ← 意思決定権者
    ↓
[現場リーダー: 課長] ← ヒアリング主対象
    ↓
[現場担当: 一般職3名] ← 実作業者
    ↑
[システム管理: 情シス1名] ← 既存システムの担当
    ↑
[隣接業務: 営業1名] ← 業務の前後工程担当
```

このマップを書き出すと、「**誰にいつ何を聞くべきか**」が見えてくる。

### 準備3: 仮説の事前整理

「白紙でヒアリングに臨む」のは間違いだ。事前情報から仮説を立て、その仮説を**検証する形でヒアリングを進める**ほうが、深い情報が引き出せる。

仮説の例:

- 仮説1: メール対応に最も時間がかかっているのではないか
- 仮説2: 担当者ごとに対応品質のばらつきがあるのではないか
- 仮説3: マニュアルに書かれていない判断基準があるのではないか

ヒアリングではこれらを「**仮説をぶつけて反応を見る**」形で確認する。

## ヒアリング質問テンプレート（30問構成）

筆者が実プロジェクトで使っている標準的な質問テンプレートを示す。1回90分のヒアリングで、すべてを聞こうとせず、その日の流れに合わせて選択する。

### ブロック1: アイスブレイク・経歴（5問・10分）

```
Q1. お名前と現在のお役職を教えていただけますか？
Q2. この業務にはいつから関わられていますか？
Q3. 前職や前部署でのご経験を教えていただけますか？
Q4. この業務で、ご自身が一番得意とされているのはどこですか？
Q5. 最近の業務で、印象に残っている出来事はありますか？
```

このブロックは「業務情報の収集」より「**話しやすい雰囲気作り**」が目的。答えは深掘りせず、相手のペースに合わせる。

### ブロック2: 業務の全体像（5問・15分）

```
Q6. 1日の業務の流れを、朝出社してから退社まで、ざっくり教えてください。
Q7. 1週間・1ヶ月のサイクルで、定期的に発生する業務はありますか？
Q8. 業務で関わる関係者（社内・社外）はどなたですか？
Q9. 業務で使っているシステム・ツールを教えてください。
Q10. この業務の最終的なアウトプット（成果物）は何ですか？
```

このブロックで業務の「**地図**」を描く。詳細は後で深掘りする。

### ブロック3: 痛みの特定（6問・20分）

```
Q11. 1日の業務で、最も時間を取られているのはどの作業ですか？
Q12. 「これさえなければ早く帰れるのに」と思う作業はありますか？
Q13. 月末や繁忙期に、特に負担が増す業務はありますか？
Q14. 過去にミスや手戻りが発生した業務を教えてください。
Q15. 新人さんが入ったときに、最も教えるのに時間がかかるのはどの業務ですか？
Q16. 業務上で「これは属人化している」と感じる部分はありますか？
```

ここが本丸。**「効率化したい業務」ではなく「痛みのある業務」**を聞く。痛みのある業務こそが、AI導入で価値を出せるポイントだ。

### ブロック4: 暗黙知の引き出し（6問・20分）

```
Q17. マニュアルに書かれていないけれど、現場で気をつけていることはありますか？
Q18. 判断に迷ったとき、何を基準に決めていますか？
Q19. 例外的なケースが発生したとき、どう対応しますか？
Q20. ベテランと若手で、対応の差が出るのはどんな場面ですか？
Q21. 「これは現場の常識だけど、外の人には分かりにくい」と思うことは？
Q22. 過去に「これはAIには無理だろうな」と思った業務はありますか？
```

このブロックが**最も難しく、最も価値が高い**。マニュアルや業務フロー図には絶対に出てこない情報がここにある。

### ブロック5: 改善への思い（4問・15分）

```
Q23. これまでに業務改善の取り組みはありましたか？うまくいきましたか？
Q24. もし業務を一つだけ変えられるとしたら、何を変えたいですか？
Q25. AIや自動化に対して、どんなイメージをお持ちですか？
Q26. AI導入で、不安に感じることはありますか？
```

Q26は特に重要。**不安を口にしてもらう**ことで、その後の信頼関係が安定する。

### ブロック6: 次のステップ（4問・10分）

```
Q27. 今日お話しいただいた内容で、追加で説明したいことはありますか？
Q28. 次回お時間をいただけるとしたら、どの業務を深掘りしましょうか？
Q29. 業務の現場を見学させていただくことは可能ですか？
Q30. このプロジェクトに関連する資料（マニュアル等）を共有いただけますか？
```

**「次回」を必ず作る**。一度のヒアリングで完結させないのが原則。

## 現場の警戒心を解く5つの話法

質問テンプレートを使っても、現場が心を開かないと深い情報は出ない。以下の5つの話法を意識する。

### 話法1: 「教えてください」を多用する

「**ご教示いただけますか**」「**教えてください**」は、相手の専門性を尊重する魔法のフレーズだ。「分析しに来ました」「課題を聞きに来ました」よりも、はるかに心を開いてもらえる。

### 話法2: AIの限界を先に開示する

「AIは万能ではない」「現場の判断は引き続き必要」と最初に明言する。これにより「**自分の仕事は奪われない**」と理解してもらえる。

```
「実は、AIは判断業務が苦手なんです。なので、皆さんの判断業務をAIに置き換えるというより、皆さんが判断する時間を増やすために、データ整理や定型作業をAIに任せる、というイメージで考えています。」
```

### 話法3: 「失敗事例」を自分から開示する

「過去に他社で導入したけど、現場が使ってくれずに失敗した事例があります」と先に話す。これにより**「失敗したくないなら現場の声が大事」**という共通認識が生まれる。

### 話法4: 「分かりません」を尊重する

ヒアリング中に「分かりません」「答えにくいです」と言われたら、それを尊重する。無理に深掘りしない。**「分からないことが分かった」**こと自体が貴重な情報だ。

### 話法5: 議事録を見せる

ヒアリング中にメモを取り、最後にそのメモを画面共有する。「**私が理解した内容に誤解はないですか？**」と確認する。これにより、認識ズレを即座に修正でき、「**ちゃんと聞いてくれている**」という信頼が生まれる。

## ヒアリング議事録のドキュメント化

ヒアリング後、その情報を**次のフェーズに使える資産**として残す。

### 議事録テンプレート

```markdown
# ヒアリング議事録 - YYYY-MM-DD - [対象者名]

## 基本情報
- 日時:
- 場所/形式: (対面/オンライン)
- 参加者: (こちら側/相手側)
- 担当業務: 

## 業務概要
- 業務の目的:
- 主なアウトプット:
- 関係者:
- 使用システム:

## 業務フロー（粗）
1. 朝の流れ
2. 午前中の主業務
3. 午後の主業務
4. 締めの業務

## 痛みのある業務（重要度順）
| 業務 | 月間時間 | 痛みの種類 | 備考 |
|------|---------|----------|------|
| A    | 40h     | 単純作業  | ...  |
| B    | 20h     | 判断負担  | ...  |
| C    | 15h     | 手戻り    | ...  |

## 暗黙知メモ
- マニュアル外の判断基準:
- 例外ケースの扱い:
- ベテランと若手の差:

## キーワード抽出
- 業界用語: 
- システム名: 
- 関係者の役職: 

## AI適用候補（仮説）
| 候補業務 | 期待効果 | 難易度 | 優先度 |
|---------|---------|-------|--------|
| ...     | ...     | ...   | ...    |

## 不安・反対意見メモ
- 現場の懸念事項:
- 過去の失敗体験:

## 次のアクション
- 追加ヒアリング: (誰に・何を)
- 共有依頼資料:
- 次回日程:

## オフレコ情報
(他者には共有しない情報をここに明示)
```

### 議事録作成のコツ

- **その日のうちに作る**（記憶が新鮮なうち）
- **発言の引用を残す**（後で解釈の根拠になる）
- **解釈と事実を分ける**（事実: 「Aさんが言った」 / 解釈: 「これは...という意味だろう」）
- **疑問点を明示する**（次のヒアリングで確認する事項）

## ヒアリングで陥りがちな5つの罠

### 罠1: 「課題は何ですか？」と直接聞く

業務担当者は、自分の業務の課題を**抽象的にしか言語化できない**。「忙しい」「人手が足りない」「ミスが出る」といった答えしか返ってこない。

**対処法**: 課題を直接聞かず、「具体的な作業手順」を聞いて、その中で**話者が無意識に発する不満や手間の表現**を拾う。「これは手作業で...」「いつもここで時間が取られて...」といった一言が金鉱だ。

### 罠2: 経営層のヒアリングだけで進める

「現場に時間を取らせるのは申し訳ない」という配慮から、経営層や管理職だけにヒアリングし、現場をスキップするケース。

**対処法**: 現場ヒアリングは**必ず実施する**。経営層の認識と現場の実態は驚くほど乖離している。

### 罠3: 一度に多人数を集めて聞く

ヒアリング対象者を会議室に5-6人集めて、まとめて質問する。これでは個人の本音が出ない。

**対処法**: **1対1または1対2を原則**にする。集団ヒアリングは「概要把握」までに留め、深掘りは個別に行う。

### 罠4: メモを取らずに録音だけする

「後で聞き直せばいい」と思って録音だけに頼ると、後で聞き直す時間が膨大になる。

**対処法**: **メモを主、録音を補助**にする。重要な発言は手書きで書き留め、録音は確認用にする。

### 罠5: ヒアリング後すぐにAI実装に走る

ヒアリングが終わると、すぐに「じゃあプロトタイプ作ります」と実装に走るケース。これは早すぎる。

**対処法**: ヒアリング後は必ず**As-Is業務フロー図**を作り、ヒアリング対象者に**レビューしてもらう**。認識ズレを潰してから次へ進む。

## ヒアリング完了の判断基準（チェックリスト）

```
□ 業務の全工程が把握できている（粗くてもよい）
□ 痛みのある業務トップ5が特定できている
□ 各痛みのある業務の月間時間が把握できている
□ マニュアル外の判断基準が3つ以上引き出せている
□ 関係者マップが完成している
□ 主要な業界用語・システム名が理解できている
□ 現場担当者の不安・懸念事項を把握できている
□ ヒアリング議事録が共有・レビュー済みである
□ 次フェーズ（As-Is設計）に進む合意が取れている
```

このチェックリストの8項目以上にチェックが入れば、次のフェーズに進んでよい。

## 業種別のヒアリングのコツ

### 金融業

- **コンプライアンスの話を先に通す**（法務・コンプラ部門への確認）
- **データの社外持ち出し可否を最初に確認**
- 「**録音録画の同意取得**」を毎回確認

### 製造業

- **現場見学を最優先**（言葉より目で見たほうが速い）
- **熟練工の暗黙知**が中心。「**勘とコツ**」の聞き取り
- 5S・安全教育の文化を尊重

### IT・SaaS企業

- **既存ツール**との連携前提で聞く（Slack・Notion・Salesforce等）
- **エンジニア視点**と**業務担当視点**を分けて聞く
- スクリプト・マクロ等の**既存自動化資産**を確認

### 小売・サービス業

- **シフト勤務**を考慮した時間設計
- **店長と本部**で関心が異なる（店長=現場負担、本部=売上向上）
- 顧客対応データの**個人情報配慮**

## まとめ

- ヒアリングの目的は「情報収集」より「**信頼構築**」が先
- 質問テンプレート30問を準備し、その日の流れに合わせて選択する
- **痛みのある業務**と**暗黙知**を引き出すことが最重要
- 議事録は「事実と解釈を分ける」「次のアクションを明示する」形で残す
- 5つの罠を回避し、現場ヒアリングを必ず1対1で実施する

## 次章への導線

第3章では、ヒアリングで得た情報を**As-Is業務フロー図**として可視化する手法を解説する。BPMN風のシンプル記法、ツール選定、可視化の粒度、現場とのレビュー手順など、ヒアリング結果を「**プロジェクト資産**」に変換する具体的な技術を扱う。

ヒアリング情報が頭の中・議事録の中に留まっている限り、プロジェクトは進まない。可視化こそが、次のフェーズへの橋渡しになる。
