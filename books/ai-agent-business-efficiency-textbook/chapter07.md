# UAT設計と実施

## この章で学ぶこと

- UAT（User Acceptance Test）が「**動作確認**」ではなく「**業務受入の合意形成**」である本質
- AI業務効率化プロジェクト特有のUATの難しさ（確率的な出力をどう評価するか）
- 業務シナリオベースのテストケース設計手法
- UAT実施計画書のテンプレートと進め方
- UAT結果を評価する3つの軸（精度・速度・現場満足度）
- 追加改修・本番GO判断のための意思決定フレームワーク

## はじめに -- なぜAIプロジェクトのUATは難しいのか

従来のシステム開発のUATは比較的シンプルだ。「**ボタンを押したら、想定した結果が出るか**」を確認すれば良い。入力と出力が決定論的に対応しているため、テストケースを作りやすい。

ところがAIエージェントのUATは様相が違う。**入力に対して出力が一意に決まらない**からだ。

例えば「**顧客からの問い合わせメールをAIで分類する**」業務をUATする場合:

- 同じメールでも、AIは異なる分類結果を返すことがある
- 「正解」を一意に決められないケースが多い（人間でも判断が分かれる）
- 確信度・出力の長さ・ニュアンスなど、評価軸が複数ある

このため、AIプロジェクトのUATは「**合格 / 不合格**」の二択ではなく、「**全体的にどの程度業務に使えるか**」という多面的評価になる。

:::message
**筆者の実践メモ**
筆者が支援した案件で、UATを「**動作テスト**」と捉えた案件は、ほぼ全て本番運用後に問題が露呈した。「**業務シナリオでの実践試用**」として設計した案件は、本番移行がスムーズだった。UATは「**業務との接続テスト**」と捉えるべきだ。
:::

## UATの目的を再定義する

### よくある誤解

| 誤解 | 起きる問題 |
|------|---------|
| UAT=動作確認 | 業務に乗せた時の問題が見えない |
| UAT=精度測定 | 数字は良くても現場が使ってくれない |
| UAT=エンジニアのテスト工程 | 現場の声が反映されない |
| UAT=本番前の最後のチェック | 改修期間が短くなり、妥協で本番リリース |

### 正しい目的設定（4段階）

1. **動作確認**: 想定通りに動くか（最低限のレベル）
2. **業務適合**: 実際の業務フローに乗せられるか
3. **現場合意**: 現場が「これなら使える」と納得するか
4. **本番GO判断材料**: 経営層・情シス含めた合意形成

UATはこの4段階を全て満たすために設計する。

## AI業務効率化プロジェクト特有のUATの観点

通常のシステムUATに加えて、AIプロジェクトでは以下の追加観点が必要だ。

### 観点1: 確率的出力への対応

同じ入力でも異なる出力が出る可能性がある。これをどう評価するか。

**対応方法**:
- 同じテストケースを3回ずつ実行し、結果のばらつきを記録
- 重要なケースは10-20回実行し、確率分布を確認
- 「ばらつきの許容範囲」を事前に決めておく

### 観点2: エッジケース・想定外入力への耐性

実業務では想定外の入力が必ず来る。これにAIがどう反応するか。

**対応方法**:
- 通常パターンに加えて、エッジケースを意図的にテストする
  - 空文字列、極端に長い文字、特殊文字、絵文字
  - 業務外の問い合わせ（雑談、宣伝メール等）
  - 複数の業務が混在した問い合わせ
- AIが「分からない」と返せるか、誤った自信を持たないかを評価

### 観点3: ハルシネーション（虚構生成）の検知

AIが事実と異なる情報を自信ありげに出力するケースがある。

**対応方法**:
- 業務知識ベースに無い情報を生成していないか確認
- 出典・根拠を明示できるか確認
- 確信度の妥当性（高い確信度で間違える等）を評価

### 観点4: 操作性・UI/UX

技術的に動いても、現場が操作しにくければ使われない。

**対応方法**:
- 現場担当者が「初見」で何分で操作できるか測定
- 必要な情報がパッと見て分かる画面構成か
- エラー時のメッセージが理解できるか

### 観点5: 業務フロー全体への影響

AIが処理した結果が、後工程に正しく繋がるか。

**対応方法**:
- AI処理 → 人レビュー → 業務システム反映 までの一連の流れを通しでテスト
- 既存業務との接続点でデータ齟齬が起きないか
- 異常時に業務全体が止まらないか

## テストケース設計

### 業務シナリオベースの設計

AIプロジェクトのテストケースは「**機能単位**」ではなく「**業務シナリオ単位**」で設計する。

**機能単位のテストケース（避ける）**:
```
TC1: 分類ボタンを押すと分類結果が表示される
TC2: 確信度80%以上の場合、青色で表示される
```

これでは「業務として使えるか」が分からない。

**業務シナリオ単位のテストケース（推奨）**:
```
シナリオ: 新規見積依頼メールの対応
1. 顧客から「サービス料金を教えてください」というメールを受信した
2. オペレーターがAI分類結果を見る → 「新規見積」と表示される
3. オペレーターが返信下書きを生成 → 料金プラン説明文が生成される
4. オペレーターが内容を確認し、顧客固有情報を追記する
5. 送信ボタンを押す → メール送信 + CRM記録
6. CRMで顧客レコードを確認 → 対応履歴が追加されている
```

このように、**実際の業務の流れに沿ったシナリオ**でテストする。

### テストケース管理表のテンプレート

```markdown
# UATテストケース管理表

## シナリオ1: 新規見積依頼メールの対応
| 項目 | 内容 |
|------|------|
| シナリオID | TC-001 |
| 業務 | 顧客問い合わせ対応 |
| 想定実行頻度 | 1日あたり20件程度 |
| 重要度 | 高 |

### ステップ
| # | ステップ | 期待結果 | 実際結果 | 判定 |
|---|---------|---------|---------|------|
| 1 | テストメール送信 | 受信される | | |
| 2 | AI分類実行 | 「新規見積」と分類 / 確信度0.8以上 | | |
| 3 | 返信下書き生成 | 料金プラン情報が含まれる | | |
| 4 | 下書き確認 | 編集可能 | | |
| 5 | 送信 | 送信成功 + CRM記録 | | |

### 評価
- 動作: ◎ / ○ / △ / ×
- 業務適合: ◎ / ○ / △ / ×
- 操作性: ◎ / ○ / △ / ×
- 速度: 計測値（秒）
- コメント:
```

### テストケースの優先度設計

すべてのケースを同じ深さでテストできない。優先度で設計する。

| 優先度 | 対象 | テスト回数 | テスト深度 |
|--------|-----|-----------|----------|
| Critical | コア業務（毎日発生） | 各ケース10回以上 | エッジケース含む |
| High | 重要業務（週に数回） | 各ケース5回 | 通常+主要例外 |
| Medium | 通常業務（月に数回） | 各ケース3回 | 通常パターンのみ |
| Low | レアケース（年に数回） | 各ケース1回 | 動作確認のみ |

### テストデータの作り方

UATで使うテストデータは、以下の4種類を準備する。

#### データ1: 過去実データ（匿名化）

過去6ヶ月の実業務データから100-200件サンプリングし、個人情報を匿名化したもの。**最も重要**。

#### データ2: 想定典型ケース

業務担当者が「**典型的にはこういう感じ**」というケースを作る。20-30件程度。

#### データ3: エッジケース・想定外データ

意図的に作る困らせデータ。20-30件程度。
- 業務外の問い合わせ
- 複数業務が混在
- 情報不足
- 矛盾する情報

#### データ4: 過去のクレーム・障害事例

過去に問題が起きたケースを再現したデータ。10-20件程度。

合計150-280件のテストデータを準備すれば、UAT期間中に十分な検証が可能だ。

## UAT実施の進め方

### Step 1: キックオフ（30分）

UAT開始時に、関係者全員を集めてキックオフする。

- UATの目的・期間・体制の共有
- テスター向けマニュアルの説明
- 質問・懸念事項のヒアリング
- 「**遠慮なく不具合を報告してください**」の宣言

### Step 2: トレーニング（1-2時間）

現場テスターに、AIエージェントの使い方を教える。

- 画面の見方
- 操作手順
- 不具合報告方法
- 評価シートの記入方法

**トレーニング後のフィードバック**: 「ここが分かりにくい」という声を最初に聞く。これだけでUI改善のヒントになる。

### Step 3: テストデータの準備

テスト期間中に使うデータを、テスター個別に配布する。

- 通常業務シナリオ用データ
- エッジケース用データ
- 障害再現用データ

### Step 4: 並走テスト（推奨）

可能であれば、**通常業務をしながらAIも併用する**形でテストする。シャドウモード（第6章参照）と組み合わせると効果的。

「**人とAIの両方が処理し、結果を比較する**」状態を作ると、現実に近い評価ができる。

### Step 5: 日次進捗確認

毎日15-30分の進捗確認を行う。

- 進捗状況（何件テスト済か）
- 発生した不具合
- 改修依頼
- 翌日の予定

### Step 6: 中間レビュー（2週間目）

UAT期間の中間で、結果をまとめてレビューする。

- ここまでの精度
- 主な不具合
- 改修方針
- スケジュール再確認

中間時点で大きな問題があれば、後半のスケジュールを調整する。

### Step 7: 結果評価・本番GO判断

UAT終了後、結果を評価し、本番GO/NO-GOを判断する。

## 結果評価の3つの軸

### 軸1: 精度（Accuracy）

「**AIの出力がどれくらい正しいか**」を定量評価する。

**測定方法**:
```
精度 = AIの判定が人の判定と一致した件数 / 全件数

例: 200件中160件一致 → 精度80%
```

**カテゴリ別の精度を出す**:

| カテゴリ | 件数 | 一致 | 精度 |
|---------|-----|-----|-----|
| 新規見積依頼 | 50 | 48 | 96% |
| 既存契約確認 | 60 | 51 | 85% |
| クレーム | 30 | 22 | 73% |
| その他 | 60 | 45 | 75% |
| **全体** | **200** | **166** | **83%** |

カテゴリ別に見ると、「**クレームの精度が低い**」「**その他の分類が曖昧**」など、改善ポイントが見える。

### 軸2: 速度（Speed）

「**AIの処理が業務に支障ない速さか**」を測る。

**測定方法**:
- 平均レスポンス時間
- 95パーセンタイル
- 99パーセンタイル
- 最大値

```
処理時間統計（メール分類）:
- 平均: 1.8秒
- 中央値: 1.5秒
- 95%: 2.9秒
- 99%: 4.2秒
- 最大: 8.1秒
```

「**95パーセンタイル3秒以内**」のようなSLOと比較して判定。

### 軸3: 現場満足度（Satisfaction）

「**現場が使いたいと思うか**」を定性+定量で測る。

**測定方法**:

#### 定量: NPS（Net Promoter Score）的な質問

「**このAIを同僚に勧めたいと思いますか？**」を10段階で評価。

| スコア | 解釈 |
|-------|-----|
| 9-10 | Promoter（推進者） |
| 7-8 | Passive（受動的） |
| 0-6 | Detractor（批判者） |

NPS = Promoter比率 - Detractor比率

例: 10人中、Promoter 6人、Passive 3人、Detractor 1人
NPS = 60% - 10% = 50

#### 定性: 自由記述

- 良かった点
- 改善したい点
- 業務にどう変化したか
- 同僚にどう説明するか

定量だけでなく、定性的な意見が改善に直結する。

### 総合評価表

3軸を組み合わせた評価表で本番GO判断する。

| 軸 | 目標 | 実績 | 判定 |
|----|------|-----|------|
| 精度（全体） | 80%以上 | 83% | ◎ |
| 精度（Criticalカテゴリ） | 90%以上 | 88% | △ |
| 速度（95%ile） | 3秒以内 | 2.9秒 | ○ |
| NPS | +20以上 | +50 | ◎ |
| 現場の懸念 | 主要懸念なし | 「クレーム対応の精度」 | △ |

「△」がある項目について、追加改修するか、本番リリース後に改善するか判断する。

## 追加改修の判断フレームワーク

UATで見つかった問題への対応は、以下のフレームワークで判断する。

### Critical Issue（致命的問題）

- 業務が止まる
- データ破損
- セキュリティリスク

**対応**: 必ず本番リリース前に修正

### Major Issue（重要問題）

- 業務効率が想定より低い
- 現場の主要懸念事項
- カテゴリ別精度が目標値の80%未満

**対応**: 本番リリース前に修正、または本番後に2週間以内に改修

### Minor Issue（軽微問題）

- UI/UXの細かい問題
- まれにしか発生しないエッジケース
- 「あれば嬉しい」程度の改善要望

**対応**: 本番後にバックログに積む

### Enhancement（追加要望）

- 当初スコープ外の機能要望
- 「他の業務にも展開したい」

**対応**: 別プロジェクトとして扱う

### 判断時のチェックリスト

```
□ 問題の影響範囲を定量化したか（影響業務件数、影響時間等）
□ 修正コストを見積もったか（工数、リリース遅延期間）
□ 修正しない場合のリスクを評価したか
□ ステークホルダー（業務オーナー、情シス、経営層）に共有したか
□ 本番後の改修計画があるか（Minor / Enhancementの場合）
```

## ケーススタディ: 製造業の品質検査UAT

筆者が支援した製造業の品質検査AIプロジェクトのUATを紹介する。

### プロジェクト概要

- 業務: 製品の外観検査
- AI機能: 画像認識による良品/不良品の一次判定
- UAT期間: 4週間
- テスター: 検査担当者5名

### UATで実施したこと

1. 過去6ヶ月の検査画像 500枚をテストデータとして準備
2. 通常品 300枚、不良品 150枚、判断難しい品 50枚
3. AIと検査員の判定を比較
4. 検査員に「**操作性**」「**業務適合性**」「**信頼性**」を10段階評価

### 結果

| 軸 | 目標 | 実績 |
|----|------|-----|
| 良品判定精度 | 95% | 97% |
| 不良品判定精度 | 99%（見逃しゼロ）| 96%（一部見逃し） |
| 判断難しい品の人エスカレ率 | 80%以上 | 92% |
| 平均処理時間 | 2秒以内 | 1.3秒 |
| 検査員のNPS | +20 | +40 |

### 判明した問題

- **不良品見逃しが目標未達** → Critical Issue
  - 特定のキズパターンで誤判定が多かった
  - プロンプト・学習データ追加で対応
  - 2週間の追加改修

- **画面表示が小さい**（検査員からの指摘） → Minor
  - 本番後の改修に回す

### 本番リリース判断

- Critical Issueを修正後、再UATで不良品判定精度99%達成
- 本番リリース実施
- リリース後3ヶ月でROI予測通り達成

UATを丁寧に行うことで、本番後の大きな問題を未然に防げた事例だ。

## UAT完了の判断基準（チェックリスト）

```
□ 全Criticalテストケースが◎または○判定
□ Major Issueが全て対応済み、または対応計画が明確
□ 業務担当者の80%以上が「業務で使える」と評価
□ NPSが+20以上
□ 精度が目標値以上、または改善計画あり
□ 速度SLOを満たしている
□ 想定外パターンへの対応（フォールバック）が確認できている
□ セキュリティ・コンプライアンス問題が無い
□ 業務オーナーが本番GOを承認
□ 経営層・情シス・現場の合意が取れている
```

10項目中9項目以上で本番GO。

## まとめ

- AIプロジェクトのUATは「**動作確認**」ではなく「**業務受入の合意形成**」
- 確率的出力・エッジケース・ハルシネーションへの対応が特有の難しさ
- テストケースは**業務シナリオベース**で設計する
- テストデータは過去実データ・典型ケース・エッジケース・障害事例の4種類を準備
- 結果は**精度・速度・現場満足度**の3軸で評価
- 問題はCritical/Major/Minor/Enhancementで分類して対応

第8章では、本番運用開始後の**効果測定・KPI設計**を扱う。

UATで本番GOしただけでは終わらない。実際に業務にどんな変化があったか、ROIは試算通り出ているか、を継続的に計測する仕組みが必要だ。KPIの設計、ダッシュボード構築、月次レビューの進め方、改善サイクルの回し方を解説する。

「**作って終わり**」のプロジェクトは1年後に成果が見えない。計測する仕組みこそが、AIプロジェクトを持続可能にする。
