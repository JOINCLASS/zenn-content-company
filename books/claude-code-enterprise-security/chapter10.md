## テンプレート1: Claude Code企業利用ポリシー

```markdown
# Claude Code 企業利用ポリシー

文書番号: SEC-AI-001
バージョン: 1.0
最終更新日: YYYY-MM-DD
承認者: [CTO名/CISO名]
次回レビュー予定: YYYY-MM-DD

---

## 1. 目的

本ポリシーは、当社におけるClaude Code（Anthropic社提供のAIコーディングツール）
の安全な利用方法を定め、情報セキュリティリスクを適切に管理することを目的とする。

## 2. 適用範囲

- 当社の全従業員（正社員、契約社員、派遣社員を含む）
- 当社の業務委託先で、当社のソースコードにアクセスする者
- 当社のインターンシップ参加者

## 3. 利用条件

### 3.1 承認されたプラン
- 当社で利用が承認されたプランは [Business/Enterprise/API] である
- 個人のFree/Proアカウントでの業務利用は禁止する

### 3.2 利用開始手続き
1. 所属部門の上長の承認を得る
2. IT部門に利用申請を提出する
3. IT部門がアカウントをプロビジョニングする
4. 利用開始前に本ポリシーの研修を受講する

### 3.3 利用可能なプロジェクト
- セキュリティ分類が「一般」「社外秘」のプロジェクトで利用可能
- セキュリティ分類が「極秘」のプロジェクトでは利用禁止
- 顧客のソースコードに対する利用は、顧客との契約でAIツール利用が
  許可されている場合に限る

## 4. 禁止事項

以下の行為を禁止する。

1. 個人情報（氏名、メールアドレス、電話番号、住所等）を含むファイルを
   Claude Codeに読み取らせること
2. 認証情報（パスワード、APIキー、トークン等）をClaude Codeの
   プロンプトに入力すること
3. 本番環境のデータベースに対するクエリをClaude Codeに実行させること
4. IT部門が配布したPermissions設定を変更すること
5. IT部門が承認していないMCPサーバーを接続すること
6. Claude Codeの出力をレビューせずに本番環境にデプロイすること
7. 企業のPermissions/Hooks設定を迂回する試みを行うこと

## 5. 技術的制御

### 5.1 Permissions
- IT部門が全社共通のPermissions設定を管理・配布する
- ロール別（ジュニア/シニア/リード）のPermissionsを適用する
- Permissionsの変更はセキュリティチームの承認を必要とする

### 5.2 CLAUDE.md
- 各プロジェクトにセキュリティポリシーを含むCLAUDE.mdを配置する
- CLAUDE.mdの変更はPRレビューを必須とする

### 5.3 Hooks
- 全プロジェクトに監査ログHookを適用する
- 機密情報のフィルタリングHookを適用する

### 5.4 ネットワーク
- Claude Codeの通信は社内プロキシを経由する
- プロキシでコンテンツフィルタリングとログ記録を行う

## 6. 監査とログ

- 全てのClaude Code操作は監査ログに記録される
- ログはセキュリティチームが月次でレビューする
- 異常な利用パターンは自動検知しアラートする

## 7. インシデント対応

- Claude Code経由でのデータ漏洩が疑われる場合は、
  直ちにIT部門（[連絡先]）に報告する
- IT部門はインシデント対応手順に従い対処する
- 必要に応じてAnthropicサポートにエスカレーションする

## 8. 違反時の対応

- 本ポリシーへの違反は、当社の就業規則に基づき処分の対象となる
- 悪意のある違反は、懲戒処分の対象となる場合がある

## 9. ポリシーのレビュー

- 本ポリシーは [四半期/半期/年次] でレビューする
- Claude Codeの大幅なアップデートがあった場合は臨時レビューを行う
- レビュー結果はセキュリティ委員会に報告する
```


## テンプレート2: 社内利用規約（同意書）

```markdown
# Claude Code 利用同意書

私は、Claude Code（Anthropic社提供のAIコーディングツール）の利用にあたり、
以下の事項を理解し遵守することに同意します。

## 理解事項

1. Claude Codeに入力したプロンプトおよびClaude Codeが読み取った
   ファイルの内容は、Anthropic社のAPIサーバーに送信されます。

2. 当社は [Business/Enterprise/API] プランを利用しており、
   送信されたデータはAIモデルのトレーニングには使用されません。

3. 送信されたデータは安全性評価の目的で一定期間保持された後、
   削除されます。

## 遵守事項

1. 「Claude Code企業利用ポリシー（SEC-AI-001）」を遵守します
2. 個人情報や認証情報をClaude Codeに入力しません
3. IT部門が配布したPermissions設定を変更しません
4. Claude Codeの出力は必ずレビューしてから使用します
5. セキュリティインシデントが発生した場合は直ちに報告します

## 署名

氏名: ___________________
部署: ___________________
日付: ___________________
署名: ___________________
```

## 1. 設定管理

### Permissions
- [ ] Enterprise Policy / Organization settingsが最新か
- [ ] denyリストに必須項目（.env, force push, rm -rf, sudo）が含まれるか
- [ ] ロール別の設定が適切に適用されているか
- [ ] 設定変更のCI検知が機能しているか

### CLAUDE.md
- [ ] 全プロジェクトにセキュリティポリシーを含むCLAUDE.mdがあるか
- [ ] CLAUDE.mdの内容が最新のポリシーと整合しているか
- [ ] 変更履歴が適切に管理されているか

### Hooks
- [ ] 監査ログHookが全プロジェクトで有効か
- [ ] 機密情報フィルタリングHookが正常に動作するか
- [ ] Hookスクリプトのテストが定期的に実施されているか

## 2. ログ分析

### 監査ログ
- [ ] ログが継続的に記録されているか
- [ ] ログの保持期間が規定に準拠しているか
- [ ] 異常な利用パターンがないか
  - 大量のファイル読み取り
  - 深夜・休日の利用
  - ブロックされた操作の頻発

### ブロックされた操作
- [ ] ブロック理由の分析が行われているか
- [ ] 意図的な回避の試みがないか
- [ ] ブロックルールの過不足がないか

## 3. アクセス管理

- [ ] 退職者のアカウントが無効化されているか
- [ ] ロール変更に伴うPermissionsの更新が行われているか
- [ ] 不要なアカウントが存在しないか

## 4. ネットワーク

- [ ] プロキシサーバーが正常に動作しているか
- [ ] コンテンツフィルタリングが機能しているか
- [ ] ファイアウォールルールが最新か
- [ ] 不正な外部通信がないか

## 5. シークレット管理

- [ ] git-secrets / detect-secretsが全リポジトリで有効か
- [ ] pre-commitフックが機能しているか
- [ ] シークレットのローテーションが実施されているか
- [ ] .envファイルがgitにコミットされていないか

## 6. コンプライアンス

- [ ] DPA/利用規約が最新か
- [ ] プライバシーポリシーの記載が最新か
- [ ] 利用者の同意書が取得されているか
- [ ] 教育・研修が実施されているか

## 7. インシデント対応

- [ ] インシデント対応手順が最新か
- [ ] 連絡先一覧が最新か
- [ ] インシデント訓練が定期的に実施されているか

## テンプレートの活用ガイド

### カスタマイズのポイント

1. **会社の規模に合わせる**: スタートアップなら簡略版で十分。大企業なら詳細版を使う
2. **業種に合わせる**: 金融・医療はコンプライアンスセクションを強化する
3. **既存のポリシーと整合させる**: 情報セキュリティポリシーの体系に合わせて文書番号を付与する
4. **段階的に導入する**: 最初は基本ポリシーのみ導入し、運用しながら詳細を追加する

### テンプレート使用時の注意

- テンプレートはあくまで出発点。自社の状況に合わせてカスタマイズすること
- 法的な効力が求められる文書（DPA等）は、必ず法務部門のレビューを経ること
- 技術的な設定は、実際の環境でテストしてから展開すること


## 明日から実行すべき3ステップ

本書を読み終えた方に、明日から実行してほしい3つのアクションを提示する。

### Step 1: Permissions の deny リストを設定する（10分）

第1章のクイックスタートで紹介した最小限の `.claude/settings.json` をプロジェクトに追加する。`rm -rf`、`git push --force`、`DROP` の3つをブロックするだけで、最も致命的な事故を防げる。

### Step 2: 監査ログHookを1つ追加する（20分）

第6章のパターン2（監査ログ）を設定する。全てのTool実行がログファイルに記録され、「誰が、いつ、何を実行したか」が追跡可能になる。セキュリティインシデント発生時の原因特定に不可欠。

### Step 3: CLAUDE.md にセキュリティポリシーを3行書く（5分）

第5章のテンプレートから、最低限の3行を CLAUDE.md に追加する:
- シークレットをコードにハードコードしない
- 本番データベースへの直接アクセスを禁止
- 外部APIの認証情報は環境変数から読み込む

この3ステップで、合計35分。明日の朝一番に実行すれば、午前中にはClaude Codeの基本的なセキュリティ基盤が整う。


## おわりに -- 安全なAI活用の第一歩

本書を通じて、Claude Codeの企業導入に必要なセキュリティの全体像を解説した。

振り返ると、以下の7つのレイヤーで多層防御を構築してきた。

**Layer 7**: コンプライアンス証跡の管理（第9章）

**Layer 6**: ネットワークレベルの制御（第8章）

**Layer 5**: シークレットの保護（第7章）

**Layer 4**: Hooksによる動的制御（第6章）

**Layer 3**: CLAUDE.mdによる行動規範（第5章）

**Layer 2**: Permissionsによるアクセス制御（第4章）

**Layer 1**: プラン選定とデータフローの理解（第2-3章）

これら全てを一度に導入する必要はない。まずはLayer 1（プラン選定）とLayer 2（Permissions）から始め、段階的にレイヤーを追加していくのが現実的だ。

AIツールの活用は、もはや「導入するかどうか」ではなく「いかに安全に導入するか」の段階に入っている。本書が、あなたの企業における安全なClaude Code導入の一助となれば幸いだ。
