# MCPとは何か -- AI開発の新しいパラダイム

## この章で学ぶこと

- MCPが登場した背景と解決する課題
- MCPのアーキテクチャ（ホスト・クライアント・サーバーの関係）
- MCPが提供する3つの機能（Tools・Resources・Prompts）の概要
- なぜ今MCPを学ぶべきなのか

## AIツールの「連携問題」

Claude CodeやCursorを日常的に使っているエンジニアなら、こんな場面に心当たりがあるはずです。

「Slackのメッセージを取得して、その内容をもとにGitHubのIssueを作成して、結果をNotionにまとめたい。」

これを実現しようとすると、従来は3つのAPIクライアントを個別に実装し、認証を管理し、エラーハンドリングを書き、それらを繋ぎ合わせるグルーコードを用意する必要がありました。AIモデルが賢くなっても、外部システムとの「接続部分」は人間がゴリゴリ書くしかなかった。

MCPは、この問題を根本から解決します。

## MCP（Model Context Protocol）とは

MCPは、Anthropicが2024年末にオープンソースとして公開したプロトコルです。一言でいえば、**AIモデルと外部ツール・データソースを接続するための標準規格**です。

従来のAIツール連携が「個別のAPIを直接叩く」アプローチだったのに対し、MCPは「標準化されたプロトコルを介してサーバーと通信する」アプローチをとります。

```
従来のアプローチ:
  AIモデル → Slack API（個別実装）
  AIモデル → GitHub API（個別実装）
  AIモデル → Notion API（個別実装）

MCPのアプローチ:
  AIモデル → MCPクライアント → MCPサーバー(Slack)
                              → MCPサーバー(GitHub)
                              → MCPサーバー(Notion)
```

USBが周辺機器の接続を標準化したように、MCPはAIと外部ツールの接続を標準化します。一度MCPサーバーを作れば、Claude Code、Cursor、その他MCP対応のあらゆるクライアントから利用できます。

## MCPのアーキテクチャ

MCPは3つの構成要素で成り立っています。

### ホスト（Host）

ユーザーが直接操作するアプリケーションです。Claude Desktop、Claude Code、Cursorなどがこれにあたります。ホストはMCPクライアントを内蔵しており、ユーザーの指示に応じてMCPサーバーと通信します。

### クライアント（Client）

ホストの内部で動作し、MCPサーバーとの1対1の接続を管理するコンポーネントです。プロトコルのネゴシエーション、メッセージのルーティング、機能の管理を担当します。通常、開発者がクライアントを直接実装することはありません。

### サーバー（Server）

外部ツールやデータソースへのアクセスを提供するプログラムです。**本書で作るのは、主にこのサーバー部分です。** MCPサーバーは軽量なプロセスとして動作し、Tools・Resources・Promptsという3つの機能をクライアントに公開します。

```
┌─────────────────────────────────────────┐
│ ホスト (Claude Code / Cursor)           │
│  ┌───────────┐  ┌───────────┐          │
│  │ MCPクライアント │  │ MCPクライアント │       │
│  └─────┬─────┘  └─────┬─────┘          │
└────────┼───────────────┼────────────────┘
         │               │
    ┌────┴────┐     ┌────┴────┐
    │MCPサーバーA│     │MCPサーバーB│
    │(ファイル操作)│     │(DB連携)  │
    └─────────┘     └─────────┘
```

## MCPの3つの機能

MCPサーバーが提供できる機能は3つに分類されます。詳細は第3章で解説しますが、ここでは概要を押さえましょう。

### Tools（ツール）

AIモデルが**実行**できるアクション。「ファイルを作成する」「データベースにクエリを投げる」「Slackにメッセージを送る」など、副作用を伴う操作です。Function Callingに近い概念ですが、MCPでは標準化されたインターフェースで定義します。

### Resources（リソース）

AIモデルが**読み取れる**データソース。「設定ファイルの内容」「データベースのスキーマ」「APIのドキュメント」など、コンテキストとして参照する情報です。RESTでいうGETリクエストに相当します。

### Prompts（プロンプト）

再利用可能な**プロンプトテンプレート**。「コードレビューの定型プロンプト」「SQLクエリ生成のテンプレート」など、特定のワークフローに最適化された指示セットです。

## なぜ今MCPを学ぶべきか

3つの理由があります。

**1. エコシステムが急速に拡大している**

2025年に入ってから、MCPサーバーのエコシステムは爆発的に成長しました。GitHub、Slack、Notion、PostgreSQL、ファイルシステムなど、主要なツールの公式・準公式MCPサーバーが続々と公開されています。しかし、自社固有のシステムや独自のワークフローに対応するには、自分でMCPサーバーを作る必要があります。

**2. Claude Codeとの相性が抜群に良い**

Claude CodeはMCPのネイティブサポートを備えており、MCPサーバーを追加するだけで新しい能力を獲得できます。たとえば、社内DBへのアクセス、独自の分析ツール、外部サービスとの連携。MCPサーバーを書ける人とそうでない人で、Claude Codeの活用度に大きな差が出ます。

**3. 「作れる側」の市場価値が高い**

MCPサーバーの**利用者**は増えていますが、**開発者**はまだ少ない。自分の業務に合わせたMCPサーバーをゼロから設計・実装できるスキルは、今後数年で大きな差別化要因になります。

## 本書のアプローチ

本書では、実際に動くMCPサーバーを手を動かして作りながら学びます。

- 第2章: 最小のMCPサーバーを5分で作る（Hello World）
- 第3-4章: MCPの3つの機能を深掘りし、ファイルシステム操作のサーバーを作る
- 第5-6章: データベースや外部APIと連携するサーバーを作る
- 第7章: 認証とセキュリティの実装
- 第8-9章: Claude Codeでの実践的な活用パターン
- 第10-11章: テスト、デバッグ、本番運用

各章は独立して読めるように書いていますが、第2章から順に進めることを推奨します。コード例はすべてTypeScriptで記述し、`@modelcontextprotocol/sdk`を使います。

## まとめ

- MCPは、AIモデルと外部ツール・データソースを標準化されたプロトコルで接続する仕組み
- ホスト・クライアント・サーバーの3層アーキテクチャで構成される
- サーバーはTools・Resources・Promptsの3つの機能を提供する
- MCPサーバーを自作できるスキルは、AIネイティブ開発時代の重要な武器になる

次の章では、実際に手を動かして最初のMCPサーバーを作ります。5分で「Hello, MCP!」を体験しましょう。
